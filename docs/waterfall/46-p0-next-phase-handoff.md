# 46 P0 下一阶段需求交接说明

## 1. 定位

本文件定义 ChainTrace P0 视觉原型阶段冻结后，如何把已经确认的业务需求和原型口径交接给后续阶段。

P0 下一阶段需求交接说明不是技术设计文档，不是开发任务清单，不是接口定义，不是数据库设计，也不是智能合约实现说明。

它的作用是：

> 在视觉原型阶段收口后，用业务语言说明哪些冻结内容可以交接、交给谁、交接后可以做什么、不能做什么，以及如果后续阶段发现需求缺口应该如何回流 waterfall 文档。

本文件不写组件、样式、布局尺寸、图表库、页面路由、接口路径、请求响应格式、数据库、Solidity、ABI、钱包签名、hash 或 proof 细节。

## 2. 交接原则

| 原则 | 说明 |
|---|---|
| 冻结先于交接 | 只有通过 45 号收口检查的内容才允许交接。 |
| 业务口径优先 | 后续阶段以已冻结业务需求为输入，不反向重写核心定位。 |
| 接收方分工明确 | 产品、技术接口、数据、合约、Agent、原型实现准备各接收不同业务输入。 |
| 后续阶段不越界 | 接收方不能把后置项、技术细节或旧版口径塞回 P0。 |
| 缺口回流 waterfall | 后续阶段发现业务缺口，必须回到需求文档补充，不得直接改实现。 |
| 保持角色边界 | 所有后续设计必须继承单钱包单角色和角色权限边界。 |
| 保持路径差异 | 银行辅助、纯 DeFi、低人工和人工升级路径不能被合并。 |
| 保持 Agent 边界 | Agent 仍是角色身边业务助理，不是事实制造者或高风险审批人。 |
| 保持 DeFi 定位 | 智能合约仍是业务规则和资金执行约束层。 |

## 3. 交接前置条件

| 前置条件 | 通过标准 |
|---|---|
| 产品定位冻结 | ChainTrace 是完整供应链金融系统，不是供应链金融前站。 |
| 控制塔表达冻结 | 源头到终端、节点状态、卡点和资金结果已经讲清。 |
| 角色视角冻结 | 每个角色能看、不能看、能做、不能做已经讲清。 |
| 路径差异冻结 | 银行辅助、纯 DeFi、低人工和人工升级已经区分。 |
| Agent 模型冻结 | 三种 Agent 模式和人工升级边界已经讲清。 |
| DeFi 输出冻结 | 允许、等待、阻断、资金池不足、暂停、失败、完成已经讲清。 |
| 异常回流冻结 | 主要异常已有原因、责任方、解除条件和下一步。 |
| 变更处理完成 | D 类阻断关闭，B 类需求缺口已补或明确不得交接。 |
| 后置清单完成 | P1/P2 和技术设计后置项已记录。 |
| 交付包完成 | 44 号交付包材料齐全。 |

## 4. 交接对象总览

| 接收方 | 接收内容 | 输出方向 | 不允许做 |
|---|---|---|---|
| 产品细化 | 冻结的业务口径、角色、路径、异常和交付包。 | 后续 PRD、用例、业务说明。 | 重写核心定位或强行新增 P0 范围。 |
| 技术接口设计 | 业务动作、业务事件、状态、错误解释和交接边界。 | 后续接口设计输入。 | 直接定义 endpoint、请求响应或鉴权实现。 |
| 数据设计 | 业务信息域、状态、记录、审计和可见范围。 | 后续数据模型输入。 | 直接设计表结构、字段或存储方案。 |
| 合约技术设计 | DeFi 业务规则、执行条件、阻断、暂停和资金结果。 | 后续合约技术设计输入。 | 直接写 Solidity、ABI、部署或钱包签名。 |
| Agent 规则设计 | Agent 模式、摘要、缺口、待办、建议、自动推进和升级边界。 | 后续 Agent 策略设计输入。 | 让 Agent 伪造事实、越权或裁决重大争议。 |
| 原型实现准备 | 控制塔表达、角色视角、路径差异、异常和交付包。 | 后续实现准备输入。 | 直接继承旧页面或绕过业务口径。 |
| 测试验收准备 | 走读脚本、评审清单、收口门禁、异常和越权场景。 | 后续测试和验收输入。 | 只测页面点击，不测业务闭环。 |

## 5. 产品细化交接

| 交接内容 | 产品细化应使用 | 不得改变 |
|---|---|---|
| 产品定位 | 完整供应链金融系统，Agent Native + Crypto Native / DeFi。 | 不得改回供应链金融前站。 |
| 角色模型 | 单钱包注册后永久绑定单一业务身份。 | 不得提前引入多组织多成员复杂权限。 |
| P0 核心闭环 | 出口商意图、Agent 整理、事实确认、资金方/规则、DeFi 执行、资金结果。 | 不得强行新增 P1/P2 深度流程。 |
| 路径模型 | 银行辅助、纯 DeFi、低人工、人工升级。 | 不得让银行成为所有场景必经。 |
| 异常模型 | 材料不足、事实冲突、dispute、规则阻断、资金池不足、暂停、失败。 | 不得把异常简化成单一失败状态。 |
| 后置清单 | P1/P2 和技术设计后置项。 | 不得把后置项重新塞回 P0。 |

产品细化阶段可以输出：

| 可输出 | 边界 |
|---|---|
| 更细的业务用例 | 必须继承已冻结角色和路径。 |
| 更清晰的业务文案 | 不改变业务规则和权限。 |
| 更完整的业务场景说明 | 不扩大 P0 范围。 |
| 更明确的验收口径 | 不进入技术实现。 |

## 6. 技术接口设计交接

技术接口设计接收的是业务动作和业务结果，不是直接接收页面或代码任务。

| 交接内容 | 技术接口设计应理解 | 当前不得定义 |
|---|---|---|
| 业务动作 | 创建融资/结算意图、补材料、确认事实、授权、请求复核、暂停、恢复、执行检查。 | endpoint 名称。 |
| 业务事件 | 材料已提交、事实已确认、授权已给出、规则阻断、资金池不足、执行完成。 | 事件传输协议。 |
| 业务状态 | Waiting、In progress、Completed、Warning、Dispute、Blocked、Failed、Skipped。 | 状态字段结构。 |
| 错误解释 | 用户应理解的业务原因、责任方和下一步。 | 错误码格式。 |
| 权限边界 | 每个角色能调用什么业务动作。 | 鉴权中间件实现。 |
| Agent 边界 | Agent 可生成摘要、待办、提醒、升级。 | Agent 调用接口细节。 |

技术接口设计阶段如果发现以下问题，必须回流 waterfall：

| 回流问题 | 回流原因 |
|---|---|
| 某业务动作没有责任角色 | 需求不完整。 |
| 某状态无法解释业务含义 | 状态定义不完整。 |
| 某异常没有解除条件 | 异常回流不完整。 |
| 某角色权限冲突 | 权限矩阵不完整。 |
| 某路径需要新增业务动作 | 路径矩阵需要补充。 |

## 7. 数据设计交接

数据设计接收的是业务信息域、记录责任和可见范围，不是直接接收表结构。

| 交接内容 | 数据设计应理解 | 当前不得定义 |
|---|---|---|
| 案件信息 | 融资/结算案件、路径、金额、状态、责任角色。 | 表名、字段名、索引。 |
| 贸易材料 | 发票、合同、装箱单、贸易背景和材料缺口。 | 文件存储结构。 |
| 事实记录 | 质检、物流、通关、保险、买方等事实。 | 数据库关系模型。 |
| Agent 记录 | 摘要、缺口、待办、建议、提醒、升级原因。 | Agent memory 结构。 |
| DeFi 业务结果 | 允许、等待、阻断、暂停、失败、完成。 | 链上/链下存储方案。 |
| 审计记录 | 授权、事实确认、规则阻断、暂停、恢复、执行结果。 | 审计表结构。 |
| 可见范围 | 角色相关可见、无关案件不可见。 | 权限表设计。 |

数据设计阶段必须保持：

| 要求 | 说明 |
|---|---|
| 事实来源独立 | 谁确认事实，谁负责事实。 |
| 关键记录可追踪 | 关键动作、授权、阻断、恢复和执行结果不可随意删除。 |
| Agent 记录不等于事实 | Agent 摘要不能替代原始事实。 |
| 合约结果不等于贸易判断 | DeFi 输出只说明执行条件和资金动作。 |

## 8. 合约技术设计交接

合约技术设计接收的是智能合约业务角色和 DeFi 业务规则，不是直接进入 Solidity。

| 交接内容 | 合约技术设计应理解 | 当前不得定义 |
|---|---|---|
| 合约业务定位 | 业务规则和资金执行约束层。 | Solidity 代码。 |
| 执行前条件 | 授权、事实、规则、资金池、暂停、保证条件。 | 函数签名。 |
| 业务输出 | 允许执行、不允许执行、等待条件、规则阻断、资金池不足、平台暂停、执行失败、执行完成。 | ABI。 |
| 暂停逻辑 | 平台、规则、资金池、案件或路径暂停。 | 合约权限实现。 |
| 资金池状态 | 可用、不足、暂停、币种不支持。 | 资金池合约结构。 |
| 关键记录 | 状态变化、执行结果、阻断和恢复。 | 事件编码。 |

合约技术设计不得改变：

| 不得改变 | 原因 |
|---|---|
| 合约不判断贸易真实性 | 贸易真实性来自业务材料和事实方。 |
| 合约不替资金方审批 | 银行辅助路径必须保留资金方授权。 |
| 合约不替质检方出结论 | 质检事实来自质检方。 |
| 合约不替 Agent 解释材料 | Agent 负责摘要和建议。 |
| 合约不绕过平台暂停 | 暂停是资金执行约束。 |

## 9. Agent 规则设计交接

Agent 规则设计接收的是 Agent Native operating model 和各角色 Agent 边界。

| 交接内容 | Agent 规则设计应理解 | 当前不得定义 |
|---|---|---|
| 三种模式 | 全人工确认、半自动托管、全自动 Agent。 | 模型选择、提示词或工具调用实现。 |
| 业务助理定位 | Agent 属于角色身边，不是独立业务用户。 | 独立 Agent 账号体系。 |
| 可做事项 | 整理材料、生成摘要、检查缺口、生成待办、提醒、请求补材料、低风险推进、异常升级。 | Agent 运行框架。 |
| 不可做事项 | 不伪造事实、不越权、不高风险放款决策、不绕过规则、不裁决重大争议。 | 自动裁决逻辑。 |
| 升级条件 | 争议、高风险、越权、置信不足、执行失败、规则冲突。 | 具体模型阈值。 |
| 角色继承 | Agent 权限继承所属角色。 | 权限实现方式。 |

Agent 规则设计阶段如果希望扩大 Agent 权限，必须回流 waterfall 重新评审。

## 10. 原型实现准备交接

原型实现准备接收的是业务表达要求，不是直接复制旧代码或旧页面。

| 交接内容 | 原型实现准备应理解 | 当前不得做 |
|---|---|---|
| 控制塔表达 | 大数据控制塔风格，源头到终端，中间节点可观察。 | 直接实现页面。 |
| 角色视角 | 不同角色看到不同范围。 | 复用旧版页面口径。 |
| 路径差异 | 银行辅助、纯 DeFi、低人工、人工升级要被区分。 | 合并成单一路径模板。 |
| Agent 表达 | 摘要、缺口、待办、建议、升级。 | 让 Agent 成为审批人。 |
| DeFi 表达 | 允许、等待、阻断、资金池不足、暂停、失败、完成。 | 展示合约技术细节。 |
| 异常表达 | 原因、责任方、解除条件、下一步。 | 只显示失败或处理中。 |
| 后置范围 | P1/P2 和技术内容暂不展开。 | 抢进 P0。 |

原型实现准备阶段不得继承：

| 不得继承 | 原因 |
|---|---|
| 旧版 app 页面口径 | 当前分支是全新瀑布重建设计分支。 |
| 旧版 components 组件结构 | 当前不做组件设计。 |
| 旧版 lib 业务逻辑 | 当前先按新需求重建。 |
| 旧版 contracts 细节 | 当前不进入合约实现。 |
| 旧版 scripts/demo/dashboard/proof/wallet/hash 页面 | 当前不进入旧证明产品路径。 |

## 11. 测试验收准备交接

测试验收准备接收的是业务验收口径，不是代码测试任务。

| 交接内容 | 测试验收准备应覆盖 | 当前不得定义 |
|---|---|---|
| 正常路径 | 出口商创建意图、Agent 整理、事实确认、资金方/规则、DeFi 执行、资金结果。 | 测试框架或测试代码。 |
| 银行辅助路径 | 资金方审查、补材料、授权、合约检查、资金结果。 | API 测试实现。 |
| 纯 DeFi 路径 | 无银行、规则和资金池检查、条件满足后执行。 | 链上测试脚本。 |
| 低人工路径 | Agent 自动推进、合约执行、异常升级。 | 自动化测试工具。 |
| 异常路径 | dispute、规则阻断、资金池不足、平台暂停、执行失败。 | 技术断言格式。 |
| 权限路径 | 无关案件不可见、越权动作阻断。 | 鉴权实现测试。 |

## 12. 后置项交接

| 后置项 | 交接去向 | P0 处理方式 |
|---|---|---|
| 物流完整流程 | P1 产品细化 | P0 保留节点和异常入口。 |
| 保险完整核保/理赔 | P1 产品细化 | P0 保留承保、拒保、加费、理赔状态。 |
| 买方完整付款系统 | P1 产品细化 | P0 保留确认、争议和付款责任。 |
| 海关真实接入 | P2 接入规划 | P0 保留通关事实节点或占位。 |
| 多组织多成员权限 | P1/P2 权限设计 | P0 保持单钱包单角色。 |
| 复杂法律自动执行 | P2 规则和法律场景设计 | P0 只保留低人工边界和人工升级。 |
| 技术接口细节 | 技术接口设计阶段 | P0 只提供业务动作和业务结果。 |
| 数据库结构 | 数据设计阶段 | P0 只提供业务信息域和记录责任。 |
| 合约实现 | 合约技术设计阶段 | P0 只提供合约业务输出和约束。 |
| 钱包签名 | 钱包实现阶段 | P0 只保留角色身份和执行授权口径。 |
| hash / proof 产品页 | 后续证明产品阶段 | P0 不作为主流程。 |

## 13. 需求缺口回流机制

后续阶段如果发现需求缺口，应按以下方式回流。

| 缺口类型 | 回流文档 | 不得直接做 |
|---|---|---|
| 产品定位冲突 | P0 PRD 总纲或阶段边界文档。 | 直接改原型或实现。 |
| 新角色 | 角色需求文档、权限矩阵、MVP 范围。 | 直接新增角色入口。 |
| 新路径 | 场景路径矩阵、业务规则清单。 | 直接改流程。 |
| 新权限 | 权限矩阵。 | 直接放开可见范围。 |
| 新状态 | 业务事件清单、控制塔要求、验收脚本。 | 直接新增状态展示。 |
| 新异常 | 业务规则清单、异常回流说明。 | 直接新增失败提示。 |
| 新合约输出 | 智能合约业务角色、DeFi 规则清单。 | 直接改合约设计。 |
| Agent 权限扩大 | Agent Operating Model、权限矩阵。 | 直接让 Agent 自动执行。 |
| P1/P2 提前 | MVP 范围和后置清单。 | 直接塞入 P0。 |

## 15. 交接通过标准

| 通过项 | 通过标准 |
|---|---|
| 交接范围清楚 | 接收方知道拿到什么、不拿什么。 |
| 冻结内容清楚 | 接收方知道哪些业务口径不能随意改。 |
| 后置内容清楚 | P1/P2 和技术细节没有混入 P0。 |
| 接收方责任清楚 | 每个后续阶段知道自己的输出方向。 |
| 回流机制清楚 | 发现需求缺口时知道回到哪个 waterfall 文档。 |
| 边界清楚 | 不进入代码、接口、数据库、Solidity、钱包签名、hash/proof 实现。 |

## 16. 交接阻断条件

出现以下情况，不得向后续阶段交接。

| 阻断条件 | 原因 |
|---|---|
| 45 号收口检查未通过 | 视觉原型阶段尚未冻结。 |
| D 类阻断变更未关闭 | 核心业务风险未解除。 |
| B 类需求缺口未补文档 | waterfall 需求不完整。 |
| 产品定位仍有争议 | 后续阶段会反复返工。 |
| 角色权限仍不清楚 | 后续接口、数据和原型都无法确定边界。 |
| 路径差异仍不清楚 | 银行辅助、纯 DeFi 和低人工会被混淆。 |
| Agent 权限仍不清楚 | 容易产生越权自动化。 |
| DeFi 输出仍不清楚 | 合约技术设计无法接收稳定业务输入。 |
| 后置清单缺失 | P1/P2 和技术内容会抢进 P0。 |

## 17. MVP 范围

### 17.1 P0 必须交接

| 范围 | 交接去向 |
|---|---|
| 产品定位和核心闭环 | 产品细化、原型实现准备、测试验收准备。 |
| 控制塔表达 | 产品细化、原型实现准备、测试验收准备。 |
| 角色视角和权限边界 | 产品细化、技术接口设计、数据设计、原型实现准备。 |

| DeFi 业务输出 | 合约技术设计、技术接口设计、测试验收准备。 |
| 异常回流 | 产品细化、技术接口设计、测试验收准备。 |

## 18. 本步结论

P0 下一阶段需求交接的本质不是“开始开发”。

它的本质是：

> 把已经冻结的 ChainTrace P0 业务原型口径，按接收方拆分成交接输入；后续产品、技术接口、数据、合约、Agent、原型实现准备和测试验收都必须继承这些业务边界，发现缺口时回流 waterfall 文档，而不是直接绕过需求进入实现。

下一步应进入：

> P0 技术设计准备说明，用于定义在不写代码的前提下，后续技术设计阶段应接收哪些业务输入、保持哪些业务边界，以及哪些内容仍然不得提前实现。
