# 11 Agent Native Operating Model

## 1. 定位

ChainTrace 的 Agent 不是一个单独用户，也不是一个独立角色。

Agent 的核心定位是：

> 每个业务角色身边的业务助理，负责整理、检查、提醒、生成待办、请求补充和推进低风险流程。

Agent 不替代出口商、资金方、质检方、物流方、保险方、买方、平台方的业务身份。

每个角色仍然由钱包注册后永久绑定为单一业务身份。Agent 只在该角色授权范围内工作。

## 2. 进入条件

任何角色使用 Agent 前，必须已经完成：

| 条件 | 要求 |
|---|---|
| 钱包已连接 | 用户证明自己控制该钱包。 |
| 钱包已注册 | 钱包已经完成 ChainTrace 注册。 |
| 角色已锁定 | 钱包已经永久绑定为一个业务角色。 |
| 账户已激活 | 可以进入对应角色工作区。 |
| Agent 模式已选择 | 全人工确认、半自动托管、全自动 Agent。 |
| 授权边界已确认 | 明确 Agent 可以自动做什么，必须让人确认什么。 |

Agent 不处理登录前展示页。
Agent 不处理未注册用户的营销转化页面。

## 3. 全系统最想完成的业务结果

ChainTrace 的 Agent Native 目标不是让用户看到一个聊天机器人。

它要完成的业务结果是：

| 目标 | 说明 |
|---|---|
| 让正常案件自动推进 | 大部分材料完整、风险正常、规则清楚的案件由 Agent + 规则 + DeFi 智能合约推进。 |
| 让人工只处理关键判断 | 人主要处理授权、异常、争议、高风险判断。 |
| 减少重复整理材料 | 合同、发票、物流、质检、保险、买方确认等由 Agent 先整理。 |
| 减少角色之间来回沟通 | 缺什么、找谁补、什么时候补，由 Agent 转成待办。 |
| 让卡点可见 | 案件卡在材料、质检、物流、买方、保险、资金池、规则还是平台状态，要清楚。 |
| 保持业务事实边界 | Agent 可以整理事实，但不能伪造事实。 |

全系统目标：

```text
70% 左右正常案件由 Agent + 规则 + DeFi 智能合约自动推进；
不超过 30% 需要人工处理；
人工主要处理异常、授权、争议、高风险判断。
```

## 4. 三种 Agent 模式

每个角色都可以选择三种 Agent 模式：

| 模式 | 含义 | 适用场景 |
|---|---|---|
| 全人工确认 | Agent 只提示和整理，每一步都需要用户确认。 | 新用户、高风险角色、高金额案件、监管敏感案件。 |
| 半自动托管 | Agent 自动处理常规整理和提醒，关键节点需要用户确认。 | 默认推荐模式，适合大多数业务用户。 |
| 全自动 Agent | Agent 在授权范围内自动推进低风险流程，只在异常和关键授权时提醒用户。 | 标准化、低风险、材料完整、规则清楚的案件。 |

### 4.1 全人工确认

| Agent 可以做 | 人必须做 |
|---|---|
| 整理材料摘要 | 确认每个关键动作。 |
| 检查缺口 | 决定是否提交、批准、承保、确认付款等。 |
| 生成待办 | 处理异常和争议。 |
| 提醒用户 | 做最终业务判断。 |

适合：

| 场景 | 说明 |
|---|---|
| 首次使用 | 用户还没有形成信任。 |
| 高金额案件 | 任何自动化都需要谨慎。 |
| 高风险国家 / 品类 | 需要更多人工判断。 |
| 重大争议 | Agent 只能整理，不能裁决。 |

### 4.2 半自动托管

| Agent 可以做 | 人必须做 |
|---|---|
| 自动整理材料 | 确认提交融资、银行参与场景的允许放款、承保、付款责任等关键动作。 |
| 自动生成摘要 | 处理异常和争议。 |
| 自动检查缺口 | 处理高风险判断。 |
| 自动请求补材料 | 确认会影响权利义务的动作。 |
| 自动提醒状态变化 | 处理平台暂停、规则升级等系统级动作。 |

半自动托管是 MVP 默认推荐模式。

它的目标是：

```text
让 Agent 做重复工作，让人做最终判断。
```

### 4.3 全自动 Agent

| Agent 可以做 | 人必须做 |
|---|---|
| 自动推进低风险流程 | 高风险放款授权，或需要银行参与场景的银行授权。 |
| 自动分发补材料任务 | 重大争议裁决。 |
| 自动提醒相关角色 | 平台系统级暂停 / 恢复。 |
| 自动生成业务摘要 | 承保拒绝、加费、重大理赔。 |
| 自动识别异常并升级人工 | 付款责任确认等法律或业务责任动作。 |

全自动 Agent 必须受到授权边界限制。

它不代表系统可以自动完成所有事情。越接近高金额资金、重大责任、争议、监管和平台控制，越需要人工确认。

但在规则清楚、授权边界明确、资金池条件满足、事实材料结构化的场景中，Agent 可以与 DeFi 智能合约一起完成低人工流程，例如自动小额融资、保证金释放、付款到期提醒、按条件结算、法律条款明确的资金锁定/释放。

## 5. Agent 可以做什么

Agent 可以做的事情必须围绕业务助理能力。

| 能力 | 说明 |
|---|---|
| 整理材料 | 将合同、订单、发票、物流、质检、保险、买方确认等整理成业务摘要。 |
| 生成摘要 | 给不同角色生成对应视角摘要。 |
| 检查缺口 | 判断还缺什么材料、缺谁的确认、缺什么说明。 |
| 生成待办 | 把缺口转成出口商、质检方、物流方、买方、保险方、资金方、平台方的任务。 |
| 提醒用户 | 提醒确认、补材料、到期、异常、争议和风险。 |
| 请求相关角色补材料 | 在授权范围内向相关角色发出补充请求。 |
| 自动推进低风险流程 | 对材料完整、规则清楚、风险正常的常规节点自动推进。 |
| 配合智能合约执行低人工场景 | 在授权范围内，把规则清楚的融资、结算、保证金或法律/运营流程推进到 DeFi 智能合约执行。 |
| 识别异常并交给人 | 金额不一致、质检 dispute、物流损坏、监管异常、买方争议等交给人处理。 |
| 生成业务解释 | 用人话说明为什么卡住、缺什么、下一步找谁。 |
| 记录操作原因 | 说明 Agent 为什么生成待办、提醒或建议。 |

## 6. Agent 不可以做什么

Agent 的能力边界必须清楚。

| 禁止事项 | 原因 |
|---|---|
| 伪造贸易事实 | 贸易事实必须来自出口商和相关业务材料。 |
| 伪造质检事实 | 质检结论必须来自质检方。 |
| 伪造物流事实 | 运输状态必须来自物流方或可确认材料。 |
| 伪造海关 / 报关事实 | 通关事实必须来自对应事实方。 |
| 伪造保险结论 | 承保结论必须来自保险公司或授权规则。 |
| 替资金方做高风险放款决策 | 需要银行/资金方参与的场景中，放款权属于资金方。 |
| 替平台绕过规则 | 平台和合约规则是执行边界。 |
| 自动裁决重大争议 | 重大争议必须人工处理。 |
| 自动裁决重大法律责任 | Agent 可以整理法律事实和条款，但不能自动裁决重大法律争议。 |
| 删除或篡改关键记录 | 审计记录必须保留。 |
| 自动确认付款责任 | 买方付款责任必须由买方确认。 |
| 自动执行系统级暂停 / 恢复 | 平台级高影响动作必须人工确认。 |

## 7. 角色视角下的 Agent 责任

| 角色 | Agent 主要帮助 | Agent 不替代 |
|---|---|---|
| 出口商 / 资金需求方 | 整理贸易材料、检查融资缺口、请求补材料。 | 不伪造贸易事实，不替出口商承担材料真实性责任。 |
| 资金方 / 银行 / 保理机构 | 在需要银行参与的场景中，汇总风险摘要、完整性检查、生成审批建议。 | 不替资金方做高风险放款决策。 |
| DeFi 智能合约场景 | 整理事实、检查规则适配、生成执行前缺口和阻断解释。 | 不绕过合约规则，不伪造执行条件。 |
| 质检方 | 生成质检清单、检查照片和报告完整性。 | 不替质检方出具真实质检结论。 |
| 平台方 / 合约运营 | 汇总异常、监控 Agent 任务、提示规则影响。 | 不绕过规则，不自动执行重大平台控制。 |
| 物流方 | 整理提单、运单、签收、异常说明。 | 不伪造揽收、出港、签收等物流事实。 |
| 海关 / 报关 / 通关数据方 | 检查通关材料完整性和字段一致性。 | 不伪造报关、清关或监管状态。 |
| 保险公司 / 承保方 | 整理核保、风险和理赔材料。 | 不绕过保险规则承保，不裁决重大理赔。 |
| 买方 / 进口商 / 付款方 | 提醒订单、收货、争议、付款责任确认。 | 不替买方确认付款责任。 |

## 8. 自动推进和人工升级规则

### 8.1 可以自动推进的常规场景

| 场景 | 条件 |
|---|---|
| 材料整理 | 文件已上传，字段清楚，无明显冲突。 |
| 缺口提醒 | 缺口明确，责任角色明确。 |
| 补材料请求 | 缺少物流、质检、保险、买方确认等常规材料。 |
| 状态提醒 | 案件状态变化、截止时间临近、付款到期。 |
| 低风险流程推进 | 金额正常、材料完整、无争议、规则允许。 |

### 8.2 必须升级人工的场景

| 场景 | 原因 |
|---|---|
| 高金额融资 | 影响资金安全。 |
| 银行参与场景的资金方放款最终授权 | 决策权属于资金方。 |
| 纯 DeFi 规则例外 | 无银行场景下，规则例外不能由 Agent 自行绕过。 |
| 重大法律责任争议 | Agent 和智能合约不能自动裁决重大法律争议。 |
| 质检 Fail / Dispute | 影响贸易真实性和风险判断。 |
| 物流损坏 / 丢失 / 改港 | 可能影响交付、保险、付款责任。 |
| 监管异常 | 可能存在合规风险。 |
| 买方付款责任争议 | 涉及真实付款义务。 |
| 保险拒保 / 加费 / 重大理赔 | 涉及风险缓释和赔付责任。 |
| 平台暂停 / 恢复 / 规则升级 | 系统级影响大。 |
| 疑似欺诈 | 必须人工调查。 |

## 9. MVP 范围

### P0 必须做

| 功能 | 说明 |
|---|---|
| 每个角色选择 Agent 模式 | 全人工确认、半自动托管、全自动 Agent。 |
| Agent 材料整理 | 生成角色视角摘要。 |
| Agent 缺口检查 | 告诉用户缺什么、找谁补。 |
| Agent 待办生成 | 将缺口和异常转成任务。 |
| Agent 状态提醒 | 提醒案件卡点、补材料、确认、异常。 |
| Agent 低风险推进 | 对常规、低风险、规则清楚流程自动推进。 |
| Agent + 合约低人工推进 | 对纯 DeFi、保证金、结算、法律条款明确等场景推进到智能合约执行。 |
| Agent 人工升级 | 异常、争议、高风险转给人。 |
| Agent 操作原因说明 | 说明为什么提醒、请求或建议。 |

### P1 后续做

| 功能 | 说明 |
|---|---|
| 更细的角色级授权模板 | 不同角色可配置更细的自动化范围。 |
| Agent 自动化效果监控 | 统计自动推进比例、失败原因、人工升级比例。 |
| Agent 建议质量反馈 | 用户可以反馈建议是否有用。 |
| 多 Agent 协同策略 | 不同角色 Agent 之间协作补材料。 |

### 暂不做

| 功能 | 原因 |
|---|---|
| 登录前展示页 | 已明确延后。 |
| 聊天机器人产品设计 | 当前写业务操作模型，不做 UI。 |
| Agent 技术架构 | 当前仍处于业务需求阶段。 |
| 数据库设计 | 当前仍处于功能需求阶段。 |
| 智能合约实现 | 还没有进入技术实现阶段。 |

## 10. 本步结论

ChainTrace 的 Agent Native 不是“加一个 AI 聊天窗口”。

它的本质是：

> 每个业务角色都有自己的 Agent 助理；Agent 做整理、检查、提醒、待办、低风险推进和异常升级，并可与 DeFi 智能合约协同完成纯 DeFi 或低人工流程。人主要处理需要银行参与的授权、异常、争议、高风险和重大法律判断。
