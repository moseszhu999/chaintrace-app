# 31 P0 非功能业务要求

## 1. 定位

本文件定义 ChainTrace P0 阶段的非功能业务要求。

非功能业务要求不是技术架构，不是性能指标设计，不是数据库审计表，也不是智能合约实现。

它的作用是：

> 明确 P0 系统在可解释性、审计、权限隔离、人工确认、异常升级、Agent 自动化边界和 DeFi 资金执行可信度方面必须满足的业务质量要求。

本文件不写页面、组件、API、数据库、Solidity、钱包签名、hash 或 proof 细节。

## 2. 非功能业务原则

| 原则 | 说明 |
|---|---|
| 可解释优先 | 用户必须知道当前状态、原因、责任方和下一步。 |
| 审计优先 | 关键动作、事实、授权、阻断和执行结果必须可追踪。 |
| 权限隔离优先 | 单钱包单角色，角色只能看到和操作自己的业务范围。 |
| 人工边界清楚 | 授权、异常、争议、高风险、监管和重大法律责任交给人。 |
| Agent 有边界 | Agent 可以整理、检查、提醒、推进低风险流程，但不能伪造事实或越权裁决。 |
| DeFi 合约有边界 | 智能合约负责资金执行约束，不负责判断贸易真实性或解释全部业务材料。 |
| 历史不覆盖 | 补充、修正、争议和关闭都形成新记录，不覆盖关键历史。 |

## 3. 可解释性要求

### 3.1 状态可解释

| 要求 | 说明 |
|---|---|
| 每个案件状态必须有业务解释 | 不能只显示 Draft、Blocked、Failed 等状态名。 |
| 每个卡点必须有原因 | 缺材料、事实争议、规则阻断、资金池不足、平台暂停等原因必须清楚。 |
| 每个下一步必须有责任角色 | 用户必须知道下一步由谁处理。 |
| 每个阻断必须有恢复条件 | 用户必须知道是补材料、等待、人工复核、平台恢复还是规则调整。 |

状态解释至少回答：

```text
现在是什么状态？
为什么是这个状态？
谁需要处理？
下一步是什么？
如果不能继续，解除条件是什么？
```

### 3.2 决策可解释

| 决策 | 必须解释 |
|---|---|
| 资金方允许放款 | 通过依据、金额、条件、执行前仍需检查项。 |
| 资金方拒绝融资 | 拒绝原因、是否可补材料、是否可重新提交。 |
| Agent 建议补材料 | 缺什么、为什么缺、谁补、补完回到哪里。 |
| DeFi 合约允许执行 | 哪些规则、资金池、授权和事实条件满足。 |
| DeFi 合约阻断执行 | 阻断类型、责任方、解除条件。 |
| 平台暂停 | 暂停对象、原因、影响范围、恢复条件。 |

## 4. 审计要求

P0 审计要求是业务审计，不是数据库或链上实现设计。

| 审计对象 | 必须记录的业务内容 |
|---|---|
| 案件生命周期 | 创建、提交、补材料、审查、执行、阻断、关闭。 |
| 贸易材料 | 谁提交、何时提交、补充了什么、是否形成缺口。 |
| 事实结论 | 质检、物流、通关、保险、买方事实由谁确认。 |
| Agent 动作 | 摘要、缺口、待办、提醒、自动推进、异常升级。 |
| 资金方决策 | 允许、拒绝、补材料、人工复核及原因。 |
| DeFi 执行 | 规则检查、资金池检查、授权检查、执行、阻断、失败。 |
| 平台动作 | 规则暂停、资金池暂停、恢复、执行失败处理。 |
| 权限异常 | 越权查看、越权操作、Agent 越权风险。 |

审计记录必须满足：

| 要求 | 说明 |
|---|---|
| 来源清楚 | 哪个角色、Agent、规则、平台或合约触发。 |
| 时间顺序清楚 | 后续补充不能覆盖历史。 |
| 影响范围清楚 | 影响哪个案件、任务、事实、规则、资金池或执行状态。 |
| 原因清楚 | 业务原因不能只写技术失败。 |
| 可回看 | 相关角色能回看自己权限范围内的关键历史。 |

## 5. 权限隔离要求

| 要求 | 说明 |
|---|---|
| 单钱包单角色 | 注册后永久绑定一个业务身份。 |
| 角色工作区隔离 | 每个角色只进入自己的业务工作区。 |
| 案件相关可见 | 只能看与自己有关的案件、任务、事实和结果。 |
| 无关案件不可见 | 不能因为同属平台或同属业务网络而查看无关案件。 |
| 事实来源隔离 | 质检、物流、通关、保险、买方事实由各自角色确认。 |
| 决策权限隔离 | 银行辅助路径中资金方决策；纯 DeFi 路径中合约按规则执行。 |
| 平台权限受限 | 平台管规则和秩序，不伪造业务事实，不替资金方审批。 |
| Agent 权限继承 | Agent 只拥有所属角色授权范围内的权限。 |

必须阻断的权限风险：

| 风险 | 阻断原因 |
|---|---|
| 出口商修改质检结论 | 破坏事实独立性。 |
| 资金方修改事实方结论 | 破坏审查可信度。 |
| 平台替资金方审批 | 角色越界。 |
| Agent 查看所属角色不可见案件 | 权限继承错误。 |
| 纯 DeFi 场景被强制银行审批 | 场景路径错误。 |
| 银行辅助场景绕过资金方授权 | 执行边界错误。 |

## 6. 人工确认要求

正常案件尽量由 Agent + 规则自动推进，但 P0 必须明确哪些动作不能完全自动化。

### 6.1 必须人工确认

| 动作 | 人工确认方 |
|---|---|
| 银行辅助路径中的放款授权 | 资金方 / 银行 / 保理机构 |
| 高金额融资 | 资金方 / 平台方 |
| 重大规则变更 | 平台方 |
| 平台级暂停和恢复 | 平台方 |
| 质检 dispute 处理 | 质检方 / 资金方 / 平台方 |
| 买方重大争议 | 买方 / 出口商 / 资金方 / 仲裁机制 |
| 重大法律责任争议 | 人工角色 / 约定仲裁 |
| Agent 越权风险处理 | 所属角色 / 平台方 |

### 6.2 可以半自动推进

| 动作 | 条件 |
|---|---|
| 材料摘要生成 | 原始材料已提交。 |
| 缺口检查 | 规则和最低材料要求清楚。 |
| 待办生成 | 责任角色和完成标准清楚。 |
| 低风险提醒 | 状态变化、超时或缺口明确。 |
| 低风险流程推进 | 授权范围内、事实完整、规则明确。 |

### 6.3 可以全自动推进

| 场景 | 条件 |
|---|---|
| 小额规则化纯 DeFi 融资 | 金额在授权范围内，资金池充足，事实满足，规则未暂停。 |
| 保证金按条件释放 | 条件明确，事实状态满足，无争议。 |
| 到期提醒和结算 | 到期条件明确，付款责任无争议。 |
| 材料完整的标准低风险流程 | Agent 置信足够，规则允许自动推进。 |

## 7. 异常升级要求

P0 必须保证异常不会被 Agent、规则或合约静默吞掉。

| 异常 | 升级对象 | 升级要求 |
|---|---|---|
| 材料缺口 | 出口商 / 事实方 | 生成待办，说明缺什么。 |
| 事实冲突 | 相关事实方 / 资金方 / 平台方 | 标出冲突字段和来源。 |
| 质检 fail | 质检方 / 资金方 | 给出原因，进入人工处理。 |
| 质检 dispute | 质检方 / 资金方 / 平台方 | 进入争议处理。 |
| 买方争议 | 买方 / 出口商 / 资金方 | 暂停自动执行。 |
| 资金池不足 | 平台方 | 返回资金池不足原因。 |
| 规则阻断 | 平台方 / 责任角色 | 说明规则类型和解除条件。 |
| 平台暂停 | 平台方 | 说明影响范围和恢复条件。 |
| 执行失败 | 平台方 | 记录失败原因和处理动作。 |
| Agent 置信不足 | 所属角色 | 降级到人工确认。 |
| 重大法律责任争议 | 人工角色 / 约定仲裁 | 不允许自动裁决。 |

## 8. Agent 自动化质量要求

| 要求 | 说明 |
|---|---|
| 摘要可追溯 | Agent 摘要必须来自已有材料或事实。 |
| 缺口可操作 | 缺口必须说明责任角色和补充标准。 |
| 建议可解释 | Agent 建议必须说明依据。 |
| 自动推进可控 | 全自动只在授权、低风险、事实完整、规则明确时发生。 |
| 异常必须升级 | Agent 不得为了推进流程而忽略异常。 |
| 模式可切换 | 用户或平台可将 Agent 从全自动降级为半自动或人工确认。 |
| 不伪造事实 | Agent 不得编造贸易、质检、物流、保险、买方、通关或法律事实。 |

Agent 质量目标：

```text
70% 左右正常案件由 Agent + 规则自动推进；
不超过 30% 需要人工处理；
人工主要处理异常、授权、争议、高风险和重大法律判断。
```

## 9. DeFi 资金执行可信要求

本节只写业务可信要求，不写合约实现。

| 要求 | 说明 |
|---|---|
| 执行前检查 | 放款、结算、锁定、释放前必须检查规则、资金池、授权和暂停状态。 |
| 银行辅助路径不绕权 | 需要银行参与时，合约不能绕过资金方授权。 |
| 纯 DeFi 路径不等银行 | 无银行场景中，条件满足即可按规则执行。 |
| 阻断必须有原因 | 规则阻断、资金池不足、授权不足、平台暂停必须可解释。 |
| 失败必须可处理 | 执行失败后平台方必须知道失败原因和下一步。 |
| 关键状态可追踪 | 允许、不允许、等待、阻断、失败、完成必须形成业务事件。 |
| 不判断贸易真实性 | 贸易真实性来自事实方和业务材料，不由合约自行判断。 |

## 10. 业务连续性要求

| 场景 | 要求 |
|---|---|
| Agent 暂时不可用 | 案件可以降级为人工确认，不应丢失状态。 |
| 资金池暂停 | 案件进入等待或阻断，恢复后可重新检查。 |
| 规则暂停 | 相关路径暂停，用户看到暂停原因。 |
| 事实方未响应 | 生成提醒或超时待办，不静默停滞。 |
| 资金方未响应 | 银行辅助路径显示等待资金方，不影响纯 DeFi 路径。 |
| 执行失败 | 保留失败记录，平台方处理，用户看到下一步。 |

## 11. 用户信任要求

| 要求 | 说明 |
|---|---|
| 不制造黑盒 | 用户不能只看到“系统处理中”而不知道责任方。 |
| 不隐藏卡点 | 缺材料、规则阻断、资金池不足、平台暂停必须明确。 |
| 不混淆角色 | 银行、平台、Agent、合约、事实方职责必须清楚。 |
| 不伪装自动化能力 | Agent 做不到的判断必须升级人。 |
| 不把技术细节当业务解释 | 用户看到的是业务原因，不是技术错误码。 |
| 不承诺全自动覆盖所有案件 | 高风险、争议、监管、法律责任必须保留人工边界。 |

## 12. MVP 范围

### 12.1 P0 必须定义

| 范围 | 说明 |
|---|---|
| 可解释性 | 状态、决策、卡点、下一步清楚。 |
| 审计要求 | 关键动作、事实、授权、阻断、执行可追踪。 |
| 权限隔离 | 单钱包单角色、案件相关可见、事实来源独立。 |
| 人工确认 | 高风险、授权、争议、监管、重大法律责任交给人。 |
| 异常升级 | 异常不能被 Agent、规则或合约静默吞掉。 |
| Agent 质量 | 摘要、缺口、建议、自动推进和越权边界清楚。 |
| DeFi 执行可信 | 执行前检查、阻断有原因、失败可处理。 |

### 12.2 暂不做

| 范围 | 原因 |
|---|---|
| 性能指标 | 当前不做技术容量设计。 |
| 数据库审计表 | 当前不做数据库设计。 |
| 链上审计实现 | 当前不写合约或 proof。 |
| 权限中间件 | 当前不写代码。 |
| UI 可用性细节 | 当前不做页面和组件。 |

## 13. 本步结论

P0 非功能业务要求的核心不是“系统性能清单”。

它的核心是：

> 让 ChainTrace 在多角色、多路径、Agent 自动化和 DeFi 资金执行并存的情况下，仍然保持可解释、可审计、权限隔离、人工边界清楚、异常可升级、执行结果可信。

下一步应进入：

> P0 PRD 冻结检查清单，确认进入产品原型或技术业务契约前还缺什么。
