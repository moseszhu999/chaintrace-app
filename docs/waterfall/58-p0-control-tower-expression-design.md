# 58 P0 控制塔业务表达设计

## 1. 定位

本文件定义 ChainTrace P0 技术设计阶段的控制塔业务表达设计。

它不是页面设计，不是 UI 组件说明，不是路由规划，不是前端状态管理，不是样式规范，不是图表库选型，也不是实现代码。

它的作用是：

> 用业务语言定义 P0 中不同角色如何理解“源头到终端”的融资链路、当前节点状态、卡点原因、责任方、下一步、资金结果和 DeFi/银行双路径差异，为后续原型和界面实现提供业务表达边界。

本文件只定义业务表达，不进入 UI 实现。

## 2. 控制塔业务定位

ChainTrace 的控制塔不是普通后台首页，也不是展示型 dashboard。

控制塔的核心定位是：

> 让不同业务角色在一个案件或一组案件中，快速知道当前贸易融资链路走到哪里、卡在哪里、谁负责、下一步是什么、资金结果如何。

控制塔要回答的核心问题：

```text
这笔贸易融资现在处于什么状态？
从源头到终端哪些事实已经确认？
哪些事实还缺？
资金方是否已决策？
合约/规则是否允许执行？
钱是否到账？
如果没有到账，卡在哪里？
```

## 3. 表达原则

| 原则 | 说明 |
|---|---|
| 业务结果优先 | 优先表达能不能借、能借多少、是否到账、为什么卡住。 |
| 源头到终端 | 从出口商融资意图到资金结果的完整链路必须可理解。 |
| 角色视角分离 | 不同角色看到不同范围，不是同一个后台换菜单。 |
| 节点状态清楚 | 每个节点是完成、等待、阻断、争议、失败还是后置。 |
| 卡点必须可处理 | 卡点必须说明原因、责任方、下一步。 |
| Agent 解释业务语言 | Agent 把技术状态翻译成业务语言。 |
| 合约输出可理解 | 合约输出必须表达成允许、等待、阻断、资金池不足、暂停、失败、完成。 |
| 路径差异可见 | 银行辅助、纯 DeFi、低人工、人工升级不能被混成一个流程。 |
| 不进入 UI 实现 | 不写页面、组件、样式、颜色、路由、前端状态。 |

## 4. P0 控制塔主链路

P0 控制塔表达的主链路如下：

```text
出口商创建融资意图
→ 贸易材料准备
→ Agent 材料检查
→ 事实节点确认
→ 资金方审查或 DeFi 规则路径
→ 合约/规则执行检查
→ 资金执行状态
→ 出口商资金结果
```

每个节点都必须能表达：

| 表达项 | 说明 |
|---|---|
| 当前状态 | 完成、等待、补材料、争议、阻断、失败、已完成。 |
| 责任方 | 当前由谁处理。 |
| 缺口 | 缺什么材料、事实或确认。 |
| 影响 | 对融资、放款、到账有什么影响。 |
| 下一步 | 谁要做什么。 |
| Agent 解释 | Agent 用业务语言解释当前状态。 |

## 5. 控制塔节点状态语言

| 状态 | 业务含义 | 用户应理解 |
|---|---|---|
| Not started | 还未开始。 | 该节点尚未进入。 |
| In preparation | 准备中。 | 当前正在准备材料或信息。 |
| Ready | 已准备好。 | 可以进入下一步。 |
| Submitted | 已提交。 | 当前节点已提交给下一责任方。 |
| Waiting | 等待中。 | 还需要其他角色或条件。 |
| Need materials | 需补材料。 | 缺少材料或事实。 |
| Need confirmation | 需确认。 | 需要用户或责任方确认。 |
| Under review | 审查中。 | 资金方、平台或责任方正在处理。 |
| Pass | 已通过。 | 事实或条件满足要求。 |
| Fail | 未通过。 | 当前事实或条件不满足要求。 |
| Dispute | 有争议。 | 需要补充说明、复检或人工处理。 |
| Rule blocked | 规则阻断。 | 当前规则不允许继续。 |
| Pool insufficient | 资金池不足。 | 可执行条件受资金池影响。 |
| Platform paused | 平台暂停。 | 平台暂停了相关能力。 |
| Execution failed | 执行失败。 | 需要平台或责任方处理。 |
| Completed | 已完成。 | 该节点已闭环。 |

## 6. 卡点表达模型

控制塔中任何卡点都不能只显示“失败”或“异常”。

每个卡点必须表达：

| 卡点表达项 | 说明 |
|---|---|
| 卡点名称 | 例如：缺物流文件、质检 dispute、资金池不足。 |
| 卡点原因 | 为什么卡住。 |
| 当前责任方 | 谁需要处理。 |
| 影响范围 | 影响提交、审核、放款、到账还是贷后。 |
| 解除条件 | 满足什么才能继续。 |
| 下一步动作 | 谁做什么。 |
| 是否可自动推进 | Agent 是否可以继续处理。 |
| 是否需人工升级 | 是否需要人接管。 |

示例：

```text
卡点：质检 Dispute
原因：现场数量与装箱单不一致。
责任方：质检方、出口商。
影响：资金方无法完成融资决策。
解除条件：质检方补充说明，或出口商上传数量差异解释。
下一步：Agent 已生成补充说明待办。
自动推进：不允许。
人工升级：需要资金方人工复核。
```

## 7. 出口商控制塔表达

### 7.1 出口商最关心的问题

| 问题 | 控制塔必须表达 |
|---|---|
| 我这笔贸易能不能融资？ | 可融资 / 不可融资 / 需补材料 / 等待 / 人工复核。 |
| 能融资多少？ | 申请金额、建议金额、资金方调整金额。 |
| 钱什么时候到账？ | 放款中、资金池等待、执行失败、已到账。 |
| 现在卡在哪里？ | 材料、质检、物流、资金方、规则、资金池、平台暂停等。 |
| 谁需要处理？ | 出口商、质检方、资金方、平台、买方、物流方等。 |
| 我下一步做什么？ | 补材料、解释异常、等待、确认、联系相关方。 |

### 7.2 出口商视角节点

| 节点 | 出口商应看到 |
|---|---|
| 融资意图 | 草稿、待补信息、已创建。 |
| 贸易材料 | 完整、缺合同、缺发票、金额不一致、需说明。 |
| Agent 检查 | 建议提交、建议补材料、建议人工确认。 |
| 质检状态 | 待质检、Pass、Fail、Dispute、需复检。 |
| 物流状态 | 待确认、运输中、已签收、异常、后置。 |
| 资金方状态 | 待审、补材料、审核中、已批准、已拒绝、人工复核。 |
| 合约/规则状态 | 允许、等待、规则阻断、资金池不足、平台暂停。 |
| 资金结果 | 放款中、已到账、执行失败、等待资金池。 |

### 7.3 出口商不应看到

| 不应看到 | 原因 |
|---|---|
| 其他出口商案件 | 数据隔离。 |
| 资金方内部风险偏好 | 商业机密。 |
| 平台内部规则维护细节 | 只需看到业务影响。 |
| 无关资金池内部明细 | 只需看到影响自己案件的状态。 |
| 质检方内部任务管理细节 | 只需看到本案质检结果和缺口。 |

## 8. 资金方控制塔表达

### 8.1 资金方最关心的问题

| 问题 | 控制塔必须表达 |
|---|---|
| 哪些案件值得看？ | 待审、风险较低、材料完整、金额匹配。 |
| 证据是否完整？ | 合同、订单、发票、质检、物流、买方、保险。 |
| 风险是否过线？ | 高风险、低风险、需人工、需补材料。 |
| 我是否可以批准？ | 可批准、需补材料、需人工复核、建议拒绝。 |
| 批准后能不能执行？ | 合约检查、资金池、规则、平台暂停。 |
| 放款后是否正常？ | 回款、逾期、争议、保险。 |

### 8.2 资金方视角节点

| 节点 | 资金方应看到 |
|---|---|
| 案件摘要 | 出口商、买方、金额、商品、融资类型。 |
| 证据完整性 | 完整、不完整、缺口、冲突。 |
| 事实状态 | 质检、物流、买方、保险等状态。 |
| Agent 风险建议 | 建议允许、补材料、人工复核、拒绝。 |
| 决策动作 | 允许、拒绝、补材料、降额、人工复核。 |
| 执行条件 | 已满足、等待、规则阻断、资金池不足、平台暂停。 |
| 贷后状态 | 回款中、逾期、争议、保险处理。 |

### 8.3 资金方不应看到或不应做

| 边界 | 原因 |
|---|---|
| 不应修改出口商原始贸易材料 | 防止证据污染。 |
| 不应冒充质检方确认事实 | 事实来源隔离。 |
| 不应查看其他资金方专属案件 | 商业隔离。 |
| 不应绕过资金池和规则执行 | 资金安全。 |

## 9. 质检方控制塔表达

### 9.1 质检方最关心的问题

| 问题 | 控制塔必须表达 |
|---|---|
| 哪些任务需要我处理？ | 待接收、待上传、待补充、争议中。 |
| 要检查什么？ | 商品、数量、包装、标签、照片、现场说明。 |
| 我的材料是否完整？ | 报告、照片、结论、异常说明是否足够。 |
| 我的结论是什么状态？ | Pass、Fail、Dispute、需补充、已接受。 |
| 是否有复检或争议？ | 出口商异议、资金方复核、平台异常。 |

### 9.2 质检方视角节点

| 节点 | 质检方应看到 |
|---|---|
| 质检任务 | 待接收、已接收、拒绝、需改派。 |
| 质检清单 | 必查项目、建议照片、异常说明要求。 |
| 质检材料 | 已上传、缺照片、缺报告、缺结论。 |
| 质检结论 | Pass、Fail、Dispute。 |
| 补充请求 | 资金方/平台/Agent 要求补充什么。 |
| 争议状态 | 谁提出、原因、下一步。 |

### 9.3 质检方不应看到或不应做

| 边界 | 原因 |
|---|---|
| 不应看到资金池状态 | 与质检任务无关。 |
| 不应决定放款 | 资金方职责。 |
| 不应修改出口商贸易材料 | 防止证据污染。 |
| 不应查看无关案件 | 数据隔离。 |

## 10. 平台控制塔表达

### 10.1 平台最关心的问题

| 问题 | 控制塔必须表达 |
|---|---|
| 系统是否安全？ | 资金池、合约、规则、Agent、异常总览。 |
| 哪些案件异常？ | dispute、资金池不足、规则阻断、执行失败、疑似欺诈。 |
| 哪些规则影响案件？ | 规则暂停、规则过期、规则升级影响范围。 |
| 资金池是否正常？ | 可用、不足、暂停、冻结、执行堆积、失败。 |
| Agent 是否失控或堆积？ | 自动化失败、任务堆积、需降级人工。 |
| 是否需要暂停/恢复？ | 对象、原因、影响范围、恢复条件。 |

### 10.2 平台视角节点

| 节点 | 平台应看到 |
|---|---|
| 运行总览 | 资金池、规则、合约、异常、Agent 状态。 |
| 资金池状态 | 可用、不足、暂停、冻结、堆积、失败。 |
| 规则版本状态 | Active、Paused、Expired、Pending upgrade。 |
| 合约执行状态 | 正常、暂停、执行失败、完成。 |
| 异常队列 | 缺材料、dispute、规则阻断、资金池不足、执行失败。 |
| Agent 状态 | 正常、失败、堆积、需人工接管。 |
| 高影响操作 | 暂停、恢复、升级、分派异常。 |

### 10.3 平台不应做的表达误导

| 不应表达成 | 原因 |
|---|---|
| 平台逐单审批所有案件 | 破坏平台定位。 |
| 平台可以修改全部业务事实 | 破坏事实来源边界。 |
| 平台可以替资金方批准放款 | 破坏资金方责任。 |
| 平台异常队列等于正常案件列表 | 正常案件应由 Agent + 规则推进。 |

## 11. Agent 控制塔表达

Agent 在控制塔中的表达不是聊天窗口优先，而是业务辅助优先。

| 表达类型 | 业务作用 |
|---|---|
| 摘要 | 用一句或几句话说明案件是什么。 |
| 缺口 | 告诉用户还缺什么。 |
| 待办 | 告诉谁要做什么。 |
| 建议 | 建议提交、补材料、等待、人工复核、拒绝或升级。 |
| 风险提示 | 提醒金额不一致、质检 dispute、资金池不足等。 |
| 状态解释 | 把规则阻断、资金池不足、平台暂停等解释成人话。 |
| 升级原因 | 告诉用户为什么不能自动继续。 |

Agent 表达边界：

| 边界 | 说明 |
|---|---|
| Agent 摘要不是事实 | 原始事实仍来自材料和事实方。 |
| Agent 建议不是审批 | 资金方或规则路径仍负责决策。 |
| Agent 缺口不是拒绝 | 缺口只是下一步需要补齐。 |
| Agent 升级不是失败 | 升级表示需要人处理。 |
| Agent 不能掩盖合约阻断 | 必须如实解释阻断原因。 |

## 12. 合约 / DeFi 控制塔表达

合约输出必须转成业务语言。

| 合约输出 | 控制塔表达 |
|---|---|
| Allowed | 当前满足执行条件，可以进入资金执行。 |
| Waiting | 条件未满足，还在等待某个确认或状态恢复。 |
| Need materials | 缺少必要材料或事实。 |
| Need review | 需要人工复核，不能自动继续。 |
| Rule blocked | 当前规则不允许继续，需查看规则原因。 |
| Pool insufficient | 资金池不足或不可用，批准不等于能立即放款。 |
| Platform paused | 平台暂停了相关案件、资金池、规则或执行能力。 |
| Dispute | 存在争议，需人工处理或补充事实。 |
| Execution failed | 执行失败，需要平台或责任方处理。 |
| Completed | 执行完成，可以查看资金结果。 |

控制塔不能把合约输出表达成：

| 不应表达 | 原因 |
|---|---|
| 合约认定贸易真实 | 合约不判断贸易真实性。 |
| 合约自动批准融资 | 合约不是资金方。 |
| 合约裁决争议 | dispute 不由合约裁决。 |
| 合约失败等于案件欺诈 | 执行失败可能是资金池、规则、暂停或技术执行原因。 |

## 13. 路径差异表达

### 13.1 银行辅助路径表达

| 节点 | 表达重点 |
|---|---|
| 资金方审查 | 资金方正在看证据完整性、风险和执行条件。 |
| 资金方决策 | 允许、拒绝、补材料、人工复核、降额。 |
| 合约执行检查 | 资金方批准后仍需检查资金池、规则和暂停状态。 |
| 资金结果 | 已到账、等待资金池、执行失败、平台暂停。 |

### 13.2 纯 DeFi 路径表达

| 节点 | 表达重点 |
|---|---|
| 规则路径 | 当前不经过银行人工审批，而是按规则和资金池条件推进。 |
| 事实要求 | 必要事实节点必须满足规则。 |
| 合约检查 | 合约检查规则、资金池、暂停状态。 |
| 阻断原因 | 规则不足、事实不足、资金池不足、平台暂停。 |

### 13.3 低人工路径表达

| 节点 | 表达重点 |
|---|---|
| Agent 推进 | Agent 在授权范围内处理常规事项。 |
| 自动条件 | 低风险、事实完整、规则明确。 |
| 人工边界 | 高风险、dispute、置信不足、执行失败时升级。 |
| 用户感知 | 用户看到 Agent 正在处理什么，何时需要自己确认。 |

### 13.4 人工升级路径表达

| 节点 | 表达重点 |
|---|---|
| 升级原因 | 为什么不能自动继续。 |
| 责任方 | 谁要处理。 |
| 解除条件 | 满足什么可以继续。 |
| 结果 | 解除、继续阻断、拒绝、复检、补材料、平台处理。 |

## 14. 业务对象表达

| 业务对象 | 控制塔表达 |
|---|---|
| 融资案件 | 当前状态、融资金额、路径、卡点、资金结果。 |
| 贸易材料 | 完整性、缺口、冲突、补充说明。 |
| 质检事实 | Pass、Fail、Dispute、待补充、复检。 |
| 物流事实 | 待确认、运输中、已签收、异常、后置。 |
| 保险事实 | 已承保、拒保、加费、需补材料、理赔中。 |
| 买方事实 | 已确认、未确认、有争议、付款责任待确认。 |
| 资金方决策 | 允许、拒绝、补材料、降额、人工复核。 |
| 合约输出 | 允许、等待、阻断、资金池不足、平台暂停、失败、完成。 |
| Agent 待办 | 谁做什么、什么时候做、为什么做。 |
| 异常 | 原因、责任方、影响范围、解除条件、下一步。 |

## 15. 状态到业务语言转换

| 技术/系统状态 | 控制塔业务语言 |
|---|---|
| Missing evidence | 还缺少关键材料或事实，当前不能继续。 |
| Validation conflict | 材料之间存在不一致，需要解释或修正。 |
| Review pending | 正在等待资金方或责任方审查。 |
| Inspection dispute | 质检存在争议，需要补充说明或复检。 |
| Rule blocked | 当前规则不允许继续。 |
| Pool insufficient | 资金池不足，暂时不能执行放款。 |
| Paused | 平台暂停了相关能力。 |
| Execution failed | 执行失败，需要平台或责任方处理。 |
| Completed | 当前节点已经完成。 |

## 19. 通过标准

| 检查项 | 通过标准 |
|---|---|
| 主链路清楚 | 从融资意图到资金结果的链路可理解。 |
| 角色视角清楚 | 出口商、资金方、质检方、平台方看到不同业务范围。 |
| 状态语言清楚 | 节点状态能被业务用户理解。 |
| 卡点表达完整 | 卡点有原因、责任方、影响、解除条件、下一步。 |
| 资金结果清楚 | 出口商知道能不能借、能借多少、钱是否到账。 |
| 路径差异保留 | 银行辅助、纯 DeFi、低人工、人工升级都可表达。 |
| Agent 边界清楚 | Agent 摘要和建议不等于事实或审批。 |
| 合约边界清楚 | 合约输出不被表达成贸易真实性判断。 |
| UI 实现后置 | 未出现页面路由、组件、样式或前端代码。 |

## 20. 一票否决项

| 一票否决项 | 原因 |
|---|---|
| 控制塔被写成普通后台列表 | 破坏控制塔定位。 |
| 出口商看不到资金结果和卡点 | 破坏核心用户价值。 |
| 资金方看不到证据完整性和执行条件 | 破坏融资决策。 |
| 平台被表达成逐单审批员 | 破坏平台运营定位。 |
| Agent 建议被表达成最终审批 | 破坏 Agent 边界。 |
| 合约输出被表达成贸易真实性判断 | 破坏合约边界。 |
| 路径差异被合并成单一审批流程 | 破坏 DeFi/银行双路径。 |
| 文档直接进入页面路由、UI 组件、样式或前端代码 | 阶段越界。 |
| P1/P2 控制塔能力被塞入 P0 必做 | 破坏 MVP 范围。 |

## 21. 本步结论

P0 控制塔业务表达设计的本质不是“画页面”。

它的本质是：

> 在业务层定义不同角色如何理解一笔融资案件从源头到终端的状态、事实、卡点、责任方、下一步和资金结果，并确保 Agent、合约、平台、资金方和事实方的边界在控制塔表达中不被混淆。
