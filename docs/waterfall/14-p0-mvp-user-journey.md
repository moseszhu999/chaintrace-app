# 14 P0 MVP 用户旅程和融资闭环流程

## 1. 定位

本文件定义 ChainTrace P0 MVP 的核心用户旅程。

P0 MVP 不追求覆盖全部供应链金融角色，而是先跑通最小融资/资金执行闭环：

> 出口商或业务方提出融资/结算需求，Agent 整理材料和缺口，事实方提供独立事实；需要银行的场景由资金方做放款判断，不需要银行的场景由 DeFi 智能合约按规则检查和执行；平台和规则维护系统边界，用户最终看到到账、结算完成或卡点。

本文件只写业务流程，不写页面、组件、数据库、智能合约实现、钱包签名、hash、proof 或底层技术细节。

## 2. 进入条件

P0 MVP 流程开始前，必须满足：

| 条件 | 要求 |
|---|---|
| 出口商已注册 | 钱包已经永久绑定为出口商 / 资金需求方。 |
| 资金方已注册 | 需要银行辅助的场景中，钱包已经永久绑定为资金方 / 银行 / 保理机构。纯 DeFi 场景不要求资金方参与。 |
| 质检方已注册 | 钱包已经永久绑定为质检方 / 第三方检验机构。 |
| 平台方已注册 | 钱包已经永久绑定为平台方 / 合约运营。 |
| Agent 模式已选择 | 每个角色选择全人工确认、半自动托管或全自动 Agent。 |
| 基础规则可用 | 平台已经定义当前 MVP 使用的基础业务规则和 DeFi 执行边界。 |

不处理登录前展示页。
不处理未注册用户的营销转化页面。

## 3. P0 角色和业务目标

| 角色 | P0 目标 | 不负责 |
|---|---|---|
| 出口商 / 资金需求方 | 创建融资意图、提交贸易材料、查看融资结果和卡点。 | 不审批放款，不修改质检结论，不修改平台规则。 |
| 质检方 / 第三方检验机构 | 提供 pass / fail / dispute 的独立货物事实。 | 不决定放款，不修改出口商贸易材料。 |
| 资金方 / 银行 / 保理机构 | 在银行辅助场景中查看摘要、完整性、风险建议，并决定允许放款 / 拒绝 / 补材料。 | 不伪造贸易事实，不绕过规则执行。 |
| 平台方 / 合约运营 | 维护资金池、规则、暂停、异常和执行约束。 | 不替资金方审批，不伪造业务事实。 |
| DeFi 智能合约 | 检查规则、资金池、抵押/保证、授权边界和执行条件，并执行或阻断资金动作。 | 不判断贸易真实性，不替代需要银行参与的审批，不裁决重大争议。 |
| Agent | 整理、检查、提醒、生成待办、推进低风险流程、升级异常。 | 不伪造事实，不裁决重大争议，不替资金方做高风险放款决策。 |

## 4. P0 业务旅程

### 4.1 银行辅助总流程

| 步骤 | 业务动作 | 主要角色 | 业务输出 |
|---|---|---|---|
| 1 | 创建融资意图 | 出口商 | 案件草稿。 |
| 2 | 上传或补充贸易材料 | 出口商 | 合同、订单、发票、装箱单、基础说明。 |
| 3 | Agent 整理材料 | 出口商 Agent | 贸易摘要、缺口清单、建议动作。 |
| 4 | Agent 判断是否需要质检 | Agent / 出口商 | 质检任务或不需要质检说明。 |
| 5 | 质检方接收任务 | 质检方 | 接受 / 拒绝 / 需要补充。 |
| 6 | 质检方提交结论 | 质检方 | pass / fail / dispute。 |
| 7 | Agent 生成融资案件摘要 | Agent | 给资金方的审查摘要。 |
| 8 | 资金方审查案件 | 资金方 | 允许放款 / 拒绝 / 补材料 / 人工复核。 |
| 9 | 执行条件检查 | 平台 / 规则 / DeFi 智能合约 | 允许执行 / 等待条件 / 阻断。 |
| 10 | 出口商查看结果 | 出口商 | 到账、等待、拒绝、补材料或卡点。 |

### 4.2 纯 DeFi 总流程

| 步骤 | 业务动作 | 主要角色 | 业务输出 |
|---|---|---|---|
| 1 | 创建融资/结算意图 | 出口商 / 业务方 | 案件草稿。 |
| 2 | Agent 整理材料和事实 | Agent | 摘要、缺口、规则适配建议。 |
| 3 | 必要事实确认 | 质检方 / 物流方 / 买方 / 其他事实方 | 事实状态。 |
| 4 | DeFi 执行条件检查 | DeFi 智能合约 | 允许执行 / 等待条件 / 规则阻断。 |
| 5 | 自动执行或阻断 | DeFi 智能合约 | 到账、结算完成、锁定、释放、阻断或失败。 |
| 6 | 异常升级 | Agent / 平台方 / 人工角色 | 争议、高风险或规则例外处理。 |

### 4.3 出口商旅程

| 阶段 | 出口商想知道 | 系统必须回答 |
|---|---|---|
| 创建前 | 我能不能基于这笔贸易融资？ | 需要哪些材料、适合哪类融资。 |
| 准备中 | 我还缺什么？ | 缺合同、订单、发票、质检、说明或其他材料。 |
| 提交前 | 现在提交会不会被拒？ | Agent 建议提交、补材料或暂不提交。 |
| 审查中 | 案件卡在哪里？ | 银行审查、DeFi 规则检查、质检、补材料、规则或资金池。 |
| 决策后 | 能不能借、能借多少？ | 允许放款、自动执行、拒绝、补材料、建议金额、批准金额或规则可执行金额。 |
| 执行中 | 钱什么时候到账？ | 等待执行、资金池不足、平台暂停、执行完成。 |

### 4.4 质检方旅程

| 阶段 | 质检方想知道 | 系统必须回答 |
|---|---|---|
| 接收任务 | 哪些案件需要我质检？ | 任务列表、货物摘要、截止时间。 |
| 执行质检 | 我要检查什么？ | 通用质检清单和关键风险点。 |
| 提交材料 | 我的材料够不够？ | 缺报告、缺照片、缺结论、结论冲突等。 |
| 提交结论 | 结论如何进入融资流程？ | pass / fail / dispute 及原因，可进入银行审查或 DeFi 规则检查。 |
| 补充处理 | 为什么还要补？ | 资金方或 Agent 指出的缺口。 |

### 4.5 资金方旅程

资金方旅程只适用于需要银行/机构参与的场景。

| 阶段 | 资金方想知道 | 系统必须回答 |
|---|---|---|
| 接收案件 | 哪些案件值得看？ | Agent 分组：可审、缺材料、高风险、建议拒绝。 |
| 初审 | 这个案件是什么？ | 出口商、买方、货物、金额、材料、质检状态摘要。 |
| 风险判断 | 风险是否过线？ | 正常、警示、高风险、阻断及原因。 |
| 决策 | 我该怎么处理？ | 允许放款、拒绝、补材料、人工复核。 |
| 执行前 | 我批准后能不能放？ | 资金池、规则、平台、执行条件是否允许。 |
| 执行后 | 结果是什么？ | 执行完成、资金池不足、规则阻断、平台暂停、执行失败。 |

### 4.6 平台方旅程

| 阶段 | 平台方想知道 | 系统必须回答 |
|---|---|---|
| 运行监控 | 系统是否安全？ | 资金池、规则、暂停、执行失败和异常数量。 |
| 异常处理 | 哪些案件需要平台介入？ | 规则阻断、资金池不足、执行失败、Agent 堆积。 |
| 规则维护 | 当前按哪套规则执行？ | 规则版本、状态、适用范围、暂停状态。 |
| 执行控制 | 是否需要暂停或恢复？ | 暂停对象、影响范围、原因和下一步。 |

## 5. 正常案件自动推进路径

正常案件应尽量由 Agent + 规则自动推进。

| 条件 | 自动推进方式 |
|---|---|
| 出口商材料齐全 | Agent 生成摘要并建议提交。 |
| 质检结论为 pass | Agent 将质检事实纳入资金方摘要。 |
| 风险为正常或低警示 | Agent 建议资金方快速审查。 |
| 银行辅助场景中资金方允许放款 | 进入执行条件检查。 |
| 纯 DeFi 场景中规则和资金池条件满足 | 进入智能合约自动执行。 |
| 资金池和规则正常 | 返回允许执行或执行完成。 |
| 无争议和高风险异常 | 人只处理授权和关键确认。 |

目标：

```text
70% 左右正常案件由 Agent + 规则自动推进；
人工主要处理授权、异常、争议和高风险判断。
```

## 6. 人工介入路径

以下场景必须进入人工处理：

| 场景 | 人工处理方 | 原因 |
|---|---|---|
| 贸易金额明显不一致 | 出口商 / 资金方 / Agent | 需要解释或判断风险。 |
| 质检 fail | 质检方 / 资金方 | 可能阻断融资。 |
| 质检 dispute | 质检方 / 资金方 / 平台方 | 需要争议处理。 |
| 资金方要求补材料 | 出口商 / 相关事实方 | 银行辅助场景需要补充证据。 |
| 高金额融资 | 资金方 / 平台方 | 资金风险高。 |
| 规则阻断 | 平台方 / 资金方 / DeFi 智能合约 | 需要规则或业务判断。 |
| 资金池不足 | 平台方 / 资金方 / DeFi 智能合约 | 需要资金池处理。 |
| 法律或责任争议 | 人工角色 / 仲裁机制 | Agent 和智能合约不能自动裁决重大争议。 |
| 执行失败 | 平台方 | 系统级运营处理。 |

## 7. P0 MVP 范围

### P0 必须做

| 范围 | 说明 |
|---|---|
| 出口商创建融资意图 | 形成案件草稿。 |
| 出口商提交贸易材料 | 支持融资判断所需核心材料。 |
| Agent 整理和检查缺口 | 输出摘要、缺口、建议动作。 |
| 质检方接收和提交结论 | pass / fail / dispute。 |
| 银行辅助审查和决策 | 需要银行参与时，允许放款 / 拒绝 / 补材料 / 人工复核。 |
| 纯 DeFi 规则检查和执行 | 不需要银行时，智能合约检查规则、资金池、抵押/保证和事实状态。 |
| 平台/规则/DeFi 合约业务检查 | 是否允许执行。 |
| 出口商查看结果 | 到账、等待、拒绝、补材料、卡点。 |

## 8. 本步结论

P0 MVP 用户旅程的本质不是“做一个后台流程图”。

它的本质是：

> 让出口商、事实方、银行/资金方、平台方、Agent 和 DeFi 智能合约围绕一笔融资或结算案件形成最小可用闭环；需要银行的场景进入银行辅助路径，不需要银行的场景进入纯 DeFi 或低人工路径。Agent 负责整理、检查、提醒和推进，人负责必要授权、异常、争议和高风险判断。
