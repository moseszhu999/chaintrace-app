# 46 P0 下一阶段需求交接说明

## 1. 定位

本文件定义 ChainTrace P0 视觉原型在正式业务走读通过后，如何把已评审、可冻结的业务原型口径交接给后续产品、视觉 / 交互、技术接口、数据、DeFi 合约、Agent 和实现准备阶段。

本文件是 business handoff，不是实现计划。

本文件明确属于 not implementation：

```text
不写代码
不写 API endpoint
不写数据库表或 database schema
不写 Solidity
不写 ABI
不写钱包签名流程或 wallet signature 细节
不写 hash/proof 实现
不写前端路由
不写组件规格
```

当前 V9 状态只能表述为：

```text
V9 is ready for business walkthrough review, not formally frozen.
```

因此，本文件说明的是“如果正式业务走读通过，应如何交接”，不是宣布 V9 已经冻结，也不是自动进入开发。

## 2. 交接前提

只有同时满足以下条件，P0 视觉原型阶段才允许进入下一阶段需求交接。

| 前提 | 说明 |
|---|---|
| 正式业务走读完成 | V9 或后续视觉原型完成正式业务走读，而不是内部桌面预演。 |
| 42 号评审清单无阻塞项 | 产品定位、control tower、role boundary、path difference、Agent boundary、DeFi business output、exception recovery 和 deferred scope 均无阻塞。 |
| 43 号 D 类阻断关闭 | 所有破坏产品定位、路径、权限、Agent、DeFi 或异常逻辑的问题必须关闭。 |
| B 类需求缺口已补文档 | 任何影响核心业务口径的需求缺口必须先补 waterfall 文档。 |
| 44 号交付包完整 | 业务定位、控制塔表达、角色视角、路径差异、Agent、DeFi、异常、评审、变更和后置清单齐备。 |
| 45 号收口检查允许冻结 | 收口结论允许冻结业务表达。 |
| 阶段性质清楚 | 当前状态不是自动进入开发，而是允许进入下一阶段设计。 |

如果任一前提未满足，交接应暂停，并回到 42、43、44 或 45 号文档补齐。

## 3. 可交接的冻结业务口径

正式走读通过后，可交接的是业务口径，不是实现细节。

| 冻结业务口径 | 可交接含义 |
|---|---|
| 产品定位 | ChainTrace 是完整供应链金融系统，不是供应链金融前站、销售页或单点 proof/hash 工具。 |
| 控制塔主表达 | control tower 以源头到终端为主线，表达节点状态、当前卡点、责任方、解除条件和资金结果。 |
| 单钱包单角色 | 一个钱包只绑定一个业务角色；Agent 模式不是角色切换。 |
| 角色视角和权限边界 | 每个角色只能看到与自己相关的案件、事实、待办、状态和业务输出。 |
| 路径差异 | 银行辅助、纯 DeFi、低人工和人工升级路径必须保持 path difference，不能合并成单一路径。 |
| Agent Native 边界 | Agent 负责摘要、缺口、待办、建议、提醒、自动推进和升级，但不审批、不制造事实、不裁决重大争议。 |
| DeFi 执行业务输出 | DeFi business output 以允许、等待、阻断、资金池不足、平台暂停、执行失败和执行完成表达。 |
| 异常回流 | exception recovery 必须说明异常原因、责任方、解除条件和下一步。 |
| 后置范围 | deferred scope 包括 P1/P2 深度流程和技术实现细节，不抢进 P0 视觉原型冻结口径。 |

这些口径一旦冻结，后续阶段只能在边界内细化，不得用实现便利性改写业务规则。

## 4. 交接对象

下一阶段接收方按业务输入物接收，不按开发任务接收。

| 接收方 | 接收输入物 | 不接收内容 |
|---|---|---|
| 产品需求细化 | 已冻结业务口径、角色边界、路径差异、异常回流和后置清单。 | 未评审角色、未定义路径、临时实现方案。 |
| 视觉 / 交互原型 | 控制塔表达、角色视角、走读场景、异常状态和权限阻断表达。 | 前端组件、路由、设计系统或代码实现。 |
| 技术接口设计 | 业务动作、业务输入输出含义、权限失败解释、状态变化和事件影响。 | API endpoint、HTTP method、JSON schema、鉴权中间件或代码。 |
| 数据需求 / 数据设计 | 案件、角色、组织、节点、事实、待办、决策、规则、资金池状态、异常和审计记录等业务信息域。 | database schema、字段、索引、数据库选型。 |
| DeFi 合约技术设计 | 合约需要承载的业务输出和阻断语义。 | Solidity、ABI、部署、Gas、钱包签名细节。 |
| Agent 任务设计 | 摘要、缺口、待办、建议、提醒、自动推进和异常升级业务边界。 | 独立审批、事实制造、重大争议裁决、绕过规则。 |
| 后续实现阶段 | 已冻结业务口径和设计阶段产物。 | 直接从视觉原型跳到旧代码或纯技术证明。 |

## 5. 交接给产品需求细化

产品需求细化可以继续展开以下内容：

| 可细化内容 | 说明 |
|---|---|
| 工作区内容 | 每个角色进入工作区后应看到的业务信息、卡点和下一步。 |
| 角色动作 | 每个角色能做、不能做、何时能做、做完产生什么业务结果。 |
| 业务状态 | 案件、节点、事实、审查、规则、资金和异常的业务状态。 |
| 异常回流 | 异常原因、责任角色、解除条件、复核路径和升级路径。 |
| 业务事件 | 哪些业务动作会改变状态、形成记录或触发待办。 |
| 评审记录 | 正式走读反馈、通过项、调整项、后置项和阻断关闭证据。 |
| 后置范围 | P1/P2 和技术内容继续保持 deferred scope。 |

产品侧不得直接新增未经过 waterfall 的角色、路径、权限或规则。若发现缺口，必须按 43 号文档归类，必要时先补 waterfall 需求文档。

## 6. 交接给视觉 / 交互原型

视觉 / 交互阶段可以基于冻结业务口径继续表达：

| 可表达内容 | 说明 |
|---|---|
| 控制塔画面组织 | 源头到终端、节点状态、卡点、责任方、资金结果。 |
| 角色视角组织 | Exporter、Funder、Fact party、Platform / Operator、Agent 的可见范围和任务差异。 |
| 走读场景组织 | 银行辅助、纯 DeFi、低人工、人工升级和异常回流。 |
| 异常状态表达 | 材料不足、事实冲突、规则阻断、资金池不足、平台暂停、执行失败。 |
| 权限阻断表达 | 角色看不到什么、不能做什么、为什么被阻断。 |

视觉 / 交互阶段不得提前写前端组件、页面路由、设计系统、组件库或代码实现。它接收的是业务表达，不是开发规格。

## 7. 交接给技术接口设计

技术接口设计只能在业务契约基础上继续定义。

| 可设计方向 | 业务含义 |
|---|---|
| 业务动作承载 | 系统如何承载创建案件、提交材料、确认事实、审查、阻断、恢复和完成等业务动作。 |
| 输入输出业务含义 | 每个动作需要哪些业务信息、输出什么业务结果、对用户意味着什么。 |
| 权限和失败解释 | 角色不允许、状态不满足、规则阻断、资金池不足或平台暂停时如何解释。 |
| 状态变化 | 动作如何影响案件、节点、待办、异常和资金结果。 |
| 事件影响 | 哪些动作产生业务事件、审计线索或后续待办。 |

本文件不写 API endpoint、HTTP method、JSON schema、鉴权中间件或代码。接口设计阶段也必须保留 role boundary、path difference 和 Agent boundary。

## 8. 交接给数据设计

数据设计接收的是业务数据需求，不是表结构。

可交接的信息域包括：

| 信息域 | 业务说明 |
|---|---|
| 案件 | 融资 / 结算案件的业务主体。 |
| 角色 | 出口商、资金方、事实方、平台方和 Agent 所属角色。 |
| 组织 | 业务主体、机构和责任归属。 |
| 节点 | 控制塔中的贸易、材料、事实、审查、规则、资金和结果节点。 |
| 事实 | 质检、物流、海关、保险、买方确认等外部业务事实。 |
| 待办 | Agent 或系统生成的角色责任动作。 |
| 决策 | 资金方、平台或人工升级场景下的业务决策记录。 |
| 规则 | 影响路径、阻断、升级和执行条件的业务规则。 |
| 资金池状态 | 可用、等待、资金池不足、暂停或异常等业务状态。 |
| 异常 | 业务卡点、原因、责任方、恢复条件和下一步。 |
| 审计记录 | 关键动作、来源、时间和业务影响。 |

本文件不得写字段、表结构、索引、database schema 或数据库选型。

## 9. 交接给 DeFi 合约技术设计

DeFi 合约技术设计接收的是业务输出，不是合约实现。

| 合约业务输出 | 交接含义 |
|---|---|
| 允许执行 | 当前业务条件满足，可以进入资金动作。 |
| 不允许执行 | 当前条件不满足，必须说明业务原因。 |
| 等待条件 | 等待事实、授权、规则、资金池、保证或平台状态。 |
| 规则阻断 | 规则不允许继续，必须说明阻断范围和解除条件。 |
| 资金池不足 | 资金池状态不足以支持当前执行。 |
| 平台暂停 | 平台暂停影响当前路径或资金动作。 |
| 执行失败 | 资金动作未完成，需要责任方处理。 |
| 执行完成 | 放款、结算、锁定、释放或到账等业务结果完成。 |

不得在本文件中写 Solidity、ABI、部署、Gas、wallet signature、钱包签名或 hash/proof 技术细节。

## 10. 交接给 Agent 任务设计

Agent 任务设计接收以下业务能力：

| Agent 输出 | 交接说明 |
|---|---|
| 摘要 | 对案件、材料、事实、风险、状态和下一步进行可追溯摘要。 |
| 缺口 | 说明缺什么、为什么影响推进、谁负责补齐。 |
| 待办 | 生成角色责任明确、完成标准清楚的业务待办。 |
| 建议 | 给出下一步建议和原因。 |
| 提醒 | 对卡点、超时、缺材料、越权风险或执行失败进行提醒。 |
| 自动推进 | 在授权范围、事实完整、规则明确、低风险场景下推进。 |
| 异常升级 | 对争议、高风险、越权、置信不足和执行失败进行人工升级。 |

Agent boundary 必须保留：

```text
Agent 不是独立审批人
Agent 不制造事实
Agent 不裁决重大争议
Agent 不绕过规则或合约约束
Agent 不改变角色权限
```

## 11. 不允许交接的内容

以下内容不得从视觉原型阶段直接进入实现。

| 不允许交接内容 | 原因 |
|---|---|
| 未冻结的业务口径 | 还没有正式评审通过。 |
| 未评审的角色 | 会破坏权限和工作区边界。 |
| 未定义的路径 | 会破坏路径矩阵和验收口径。 |
| 未关闭的 D 类阻断 | 视觉原型阶段尚未收口。 |
| 未补文档的 B 类需求缺口 | waterfall 需求链未闭合。 |
| 登录前营销首页 | 当前阶段是注册后业务控制塔，不是营销页。 |
| 旧版 app / components / lib / contracts / scripts 代码 | 不得继承旧口径或旧实现路径。 |
| hash/proof product page | no hash/proof product page 进入当前 P0 视觉原型交接。 |
| API / DB / Solidity / wallet 细节 | 当前只交接业务口径，不交接实现细节。 |
| 静态销售 demo 口径 | 销售演示材料不得替代业务原型冻结口径。 |

## 12. 交接风险

| 风险 | 影响 | 控制方式 |
|---|---|---|
| 把视觉原型误当开发任务 | 提前进入实现，丢失业务边界。 | 明确 not implementation。 |
| 把销售 demo 当产品原型 | 回到销售叙事，弱化业务闭环。 | 以 V9 正式走读记录为准。 |
| 把所有路径做成银行审批 | 破坏纯 DeFi 和低人工路径。 | 保留 path difference。 |
| 让 Agent 变成审批人 | 破坏 Agent Native 边界。 | 保留 Agent boundary。 |
| 让合约变成页面或数据库 | 破坏 DeFi 执行业务定位。 | 只交接 DeFi business output。 |
| 让平台方伪造事实 | 破坏可信来源。 | 平台只处理规则、资金池、暂停和异常运营。 |
| 让事实方决定融资 | 破坏资金方和执行边界。 | 事实方只确认事实。 |
| 越权可见 | 破坏角色隔离。 | 强化 role boundary。 |
| 技术实现抢进 P0 | 阶段跑偏。 | 明确 no API endpoint、no database schema、no Solidity、no wallet signature。 |
| proof/hash 抢占主线 | 回到旧证明产品。 | 明确 no hash/proof product page。 |

| 交接项 | 是否已冻结 | 接收方 | 后续动作 | 禁止提前进入的范围 |
|---|---|---|---|---|
| 产品定位 | 待正式走读确认 | 产品需求细化 | 记录正式评审结论。 | 登录前营销页、销售 demo 替代产品原型。 |
| 控制塔主表达 | 待正式走读确认 | 视觉 / 交互原型 | 围绕源头到终端、节点、卡点、责任方、资金结果继续表达。 | 前端组件、路由、实现代码。 |
| 角色边界 | 待正式走读确认 | 产品需求细化、数据设计、接口设计 | 细化角色可见范围和不可做事项。 | 多组织多成员权限提前进入 P0。 |
| 路径差异 | 待正式走读确认 | 产品需求细化、合约技术设计、Agent 任务设计 | 保持银行辅助、纯 DeFi、低人工、人工升级差异。 | 单一路径审批系统。 |
| Agent 边界 | 待正式走读确认 | Agent 任务设计 | 细化摘要、缺口、待办、建议、升级。 | Agent 审批、Agent 事实制造、Agent 裁决。 |
| DeFi 业务输出 | 待正式走读确认 | DeFi 合约技术设计 | 将允许、等待、阻断、失败、完成作为业务输出。 | Solidity、ABI、部署、Gas、钱包签名流程。 |
| 异常回流 | 待正式走读确认 | 产品需求细化、视觉 / 交互原型 | 细化原因、责任方、解除条件和下一步。 | 只有失败状态、没有恢复路径。 |
| 后置范围 | 待正式走读确认 | 全部接收方 | 维护 P1/P2 和技术设计后置清单。 | API endpoint、database schema、hash/proof product page。 |
| 走读记录 | not formally frozen | 正式评审记录 | 执行 formal walkthrough result record。 | 用内部预演替代正式评审。 |
| 冻结决策 | not formally frozen | 收口 / 冻结记录 | 形成 visual prototype freeze decision record。 | 未关闭 B/D 项就进入实现。 |

该交接必须防止实现阶段重新偏回：


当前 V9 仍然是：

```text
V9 is ready for business walkthrough review, not formally frozen.
```

```text

P0 visual prototype freeze decision record
```
