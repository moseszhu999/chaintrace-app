# 56 P0 智能合约业务设计

## 1. 定位

本文件定义 ChainTrace P0 技术设计阶段中智能合约 / 规则执行层的业务设计。

它不是 Solidity 设计，不是 ABI 说明，不是函数签名，不是部署网络选择，不是链上存储模型，不是钱包签名流程，也不是合约测试计划。

它的作用是：

> 用业务语言定义智能合约在 P0 中负责什么、不负责什么、何时允许执行、何时必须阻断、如何表达资金池/规则/暂停状态，以及如何把执行结果反馈给出口商、资金方、平台和 Agent。

本文件只定义合约的业务角色和业务输出，不进入实现。

## 2. 合约业务定位

智能合约不是页面，不是数据库，不是 Agent，也不是融资审批人。

智能合约在 ChainTrace P0 中的业务定位是：

> 业务规则和资金执行约束层。

它负责在融资案件进入执行阶段时，根据已确认的业务事实、资金方决策或 DeFi 规则路径、资金池状态、规则版本和平台暂停状态，输出是否允许继续执行。

## 3. 合约负责什么

| 负责事项 | 业务含义 |
|---|---|
| 检查执行条件 | 判断当前案件是否满足进入资金执行的业务条件。 |
| 检查资金池状态 | 判断资金池是否可用、余额是否足够、币种是否支持、是否暂停。 |
| 检查规则版本 | 判断当前案件适用的规则是否有效、暂停、过期或不适用。 |
| 检查平台暂停状态 | 判断案件、规则、资金池或合约执行能力是否被平台暂停。 |
| 输出业务执行结果 | 允许、等待、阻断、资金池不足、平台暂停、执行失败、执行完成。 |
| 记录关键状态变化 | 关键执行结果必须可追踪。 |
| 阻断不满足条件的执行 | 防止绕过资金池、规则、授权和暂停状态。 |

## 4. 合约不负责什么

| 不负责事项 | 原因 |
|---|---|
| 判断贸易真实性 | 贸易真实性来自出口商、质检方、物流方、买方、保险方、海关等事实来源。 |
| 替资金方审批 | 银行辅助路径下，资金决策属于资金方。 |
| 替质检方给出 pass/fail/dispute | 质检结论属于质检方。 |
| 替物流方确认运输状态 | 物流事实属于物流方或相关事实来源。 |
| 替买方确认付款责任 | 买方事实属于买方。 |
| 替保险方承保或理赔 | 保险判断属于保险方。 |
| 自动裁决重大争议 | dispute 必须由相关角色或人工复核处理。 |
| 解释业务材料 | 材料摘要和解释由 Agent 和业务角色处理。 |
| 绕过平台暂停 | 平台暂停是系统级边界。 |

## 5. 合约参与的 P0 业务路径

### 5.1 银行辅助路径

```text
出口商创建融资意图
→ Agent 整理材料和事实
→ 质检方提供质检事实
→ 资金方审查真实性、完整性和风险
→ 资金方允许放款
→ 合约检查执行条件、资金池、规则和暂停状态
→ 输出允许/等待/阻断/失败/完成
```

合约在该路径中不是审批人，而是资金方批准后的执行约束层。

### 5.2 纯 DeFi 路径

```text
出口商创建融资意图
→ Agent 整理材料和事实
→ 事实节点达到规则要求
→ DeFi 规则路径判断是否可进入执行
→ 合约检查资金池、规则版本、暂停状态和执行条件
→ 输出允许/等待/阻断/失败/完成
```

合约在该路径中仍然不判断贸易真实性。它只检查规则条件是否满足，以及资金执行是否允许。

### 5.3 低人工路径

```text
授权范围明确
→ 事实完整
→ 风险较低
→ Agent 自动推进常规步骤
→ 合约执行条件检查
→ 正常执行或异常升级
```

合约在该路径中负责防止 Agent 自动化越过规则、资金池或暂停状态。

### 5.4 人工升级路径

```text
出现 dispute / 高风险 / 事实不足 / 资金池不足 / 规则冲突 / 执行失败
→ 合约或 Agent 输出阻断/失败/需人工
→ 对应责任角色处理
→ 条件解除后再重新检查
```

合约在该路径中负责明确阻断原因和下一步责任，不负责自行解决争议。

## 6. 执行前业务条件

案件进入合约执行检查前，必须有清楚的业务前提。

| 条件 | 说明 | 不满足时结果 |
|---|---|---|
| 角色授权明确 | 触发方必须有权触发执行检查。 | Permission denied / Role mismatch。 |
| 案件状态允许 | 案件必须进入可执行检查阶段。 | Waiting / Rule blocked。 |
| 关键事实存在 | 必要事实节点已达到当前路径最低要求。 | Need materials / Need review。 |
| 资金方决策或 DeFi 规则路径明确 | 银行辅助路径需资金方批准；纯 DeFi 路径需规则路径满足。 | Waiting / Need review / Rule blocked。 |
| 质检状态满足要求 | P0 至少要知道 pass/fail/dispute/pending。 | Waiting / Dispute / Rule blocked。 |
| 资金池可用 | 资金池未暂停，余额和币种满足要求。 | Pool insufficient / Platform paused。 |
| 规则版本有效 | 当前案件适用规则有效。 | Rule blocked。 |
| 平台未暂停相关对象 | 案件、资金池、规则、合约能力未被暂停。 | Platform paused。 |
| 未存在未关闭关键异常 | 重大异常未解除时不得执行。 | Need review / Dispute / Rule blocked。 |

## 7. 合约业务输出

合约输出必须是业务角色能理解的结果，不能只输出技术成功或失败。

| 业务输出 | 含义 | 典型下一步 |
|---|---|---|
| Allowed | 当前条件允许进入执行。 | 继续执行资金动作或进入执行中状态。 |
| Not allowed | 当前条件不允许执行。 | 查看原因，由责任方处理。 |
| Waiting | 条件还未满足，但不是失败。 | 等待事实、确认、资金方决策或平台恢复。 |
| Need materials | 缺少必要材料或事实。 | 出口商或事实方补充。 |
| Need review | 需要人工复核。 | 资金方、平台或相关角色处理。 |
| Rule blocked | 规则不允许继续。 | 查看规则原因，平台或责任方处理。 |
| Pool insufficient | 资金池不足或币种不支持。 | 等待资金池恢复、切换路径或人工处理。 |
| Platform paused | 平台暂停了案件、规则、资金池或执行能力。 | 等待平台恢复或查看暂停原因。 |
| Dispute | 存在争议状态。 | 进入人工争议处理或补充事实。 |
| Execution failed | 执行动作失败。 | 平台/执行责任方处理。 |
| Completed | 执行完成。 | 更新资金结果和案件状态。 |

## 8. 合约输出对角色的业务解释

### 8.1 对出口商

| 合约输出 | 出口商看到的业务解释 |
|---|---|
| Allowed | 当前案件已满足执行条件，正在进入放款流程。 |
| Waiting | 还在等待某个条件满足，例如资金方确认、质检完成或资金池恢复。 |
| Need materials | 当前还缺少材料或事实，补齐后才能继续。 |
| Rule blocked | 当前规则不允许这笔案件继续，查看阻断原因。 |
| Pool insufficient | 案件可能已通过，但资金池暂时不足或不可用。 |
| Platform paused | 平台已暂停相关执行能力，需要等待恢复。 |
| Execution failed | 放款执行失败，平台或资金方需要处理。 |
| Completed | 资金执行已完成，可以查看到账结果。 |

### 8.2 对资金方

| 合约输出 | 资金方看到的业务解释 |
|---|---|
| Allowed | 你已批准的案件满足执行条件。 |
| Waiting | 你的决策已记录，但仍等待事实、规则或资金池条件。 |
| Need review | 当前执行存在不确定，需要你或平台人工复核。 |
| Rule blocked | 当前规则不允许执行，需要查看规则影响。 |
| Pool insufficient | 资金池条件不足，已批准不等于可以执行。 |
| Platform paused | 平台暂停影响本案执行。 |
| Execution failed | 执行失败，需要平台或相关责任方处理。 |
| Completed | 放款执行完成，进入贷后跟踪。 |

### 8.3 对平台方

| 合约输出 | 平台看到的业务解释 |
|---|---|
| Rule blocked | 某规则正在阻断案件，需要查看规则影响范围。 |
| Pool insufficient | 资金池不足或异常，需要查看资金池状态。 |
| Platform paused | 平台暂停生效，需明确影响范围和恢复条件。 |
| Execution failed | 执行层失败，需要排查或分派处理。 |
| Completed | 执行完成，保留审计记录。 |

### 8.4 对 Agent

| 合约输出 | Agent 应做的业务动作 |
|---|---|
| Allowed | 继续推进或提醒用户进入执行中。 |
| Waiting | 告诉用户等待什么条件。 |
| Need materials | 生成补材料待办。 |
| Need review | 升级给责任角色。 |
| Rule blocked | 解释规则阻断原因并分派给平台或责任方。 |
| Pool insufficient | 提醒资金方/平台，出口商看到资金池卡点。 |
| Platform paused | 不得继续自动推进，等待平台恢复。 |
| Execution failed | 升级平台或执行责任方。 |
| Completed | 通知相关角色查看资金结果。 |

## 9. 资金池业务边界

| 资金池状态 | 业务含义 | 影响 |
|---|---|---|
| Available | 资金池可用。 | 可继续执行符合条件的案件。 |
| Low balance | 余额不足。 | 部分案件等待或需要资金方/平台处理。 |
| Currency unsupported | 当前币种不支持。 | 案件不能以该币种执行。 |
| Paused | 资金池被平台暂停。 | 暂停相关放款执行。 |
| Frozen | 资金池异常冻结。 | 需要平台人工处理。 |
| Execution backlog | 执行堆积。 | 可能延迟到账，需要平台和资金方关注。 |
| Failed execution | 执行失败。 | 需要平台或执行责任方处理。 |

资金池边界：

- 资金池状态影响能否执行，不等于判断案件真实性。
- 出口商不能直接修改资金池状态。
- 资金方批准不能绕过资金池不足。
- 平台可以暂停或恢复资金池，但必须有业务原因和审计记录。

## 10. 规则版本业务边界

| 规则状态 | 业务含义 | 影响 |
|---|---|---|
| Active | 当前规则有效。 | 可用于执行条件检查。 |
| Paused | 规则被暂停。 | 相关案件被阻断或等待。 |
| Expired | 规则已过期。 | 不能继续按该规则执行。 |
| Pending upgrade | 规则待升级。 | 可能需要等待或人工复核。 |
| Not applicable | 规则不适用该案件。 | 案件需换路径或人工处理。 |

规则边界：

- 规则版本必须有业务适用范围。
- 规则阻断必须能解释原因。
- 规则升级或暂停属于平台运营边界。
- 技术设计阶段不写规则代码或函数。

## 11. 暂停业务边界

| 暂停对象 | 谁可以暂停 | 业务含义 | 恢复条件 |
|---|---|---|---|
| 案件 | 平台方 | 单个案件存在异常或疑似风险。 | 异常解释清楚或风险解除。 |
| 资金池 | 平台方 | 资金池不足、异常或执行失败。 | 资金池恢复正常。 |
| 规则 | 平台方 | 规则存在风险、需维护或升级。 | 规则复核通过。 |
| 合约执行能力 | 平台方 | 系统级执行风险或维护。 | 系统风险解除。 |
| 角色账户 | 平台方 | 账户异常、疑似欺诈或合规风险。 | 平台复核后恢复。 |
| Agent 自动化 | 平台方 / 角色授权 | Agent 误判、失败或需降级人工。 | 自动化风险解除。 |

暂停边界：

- 暂停不是删除事实。
- 暂停必须有原因、影响范围和恢复条件。
- Agent 不能绕过暂停继续推进。
- 合约必须尊重暂停状态。

## 12. 与权限设计的关系

| 权限边界 | 合约业务要求 |
|---|---|
| 出口商不能放款 | 合约不能接受出口商直接绕过决策触发放款。 |
| 资金方有决策权 | 银行辅助路径需资金方决策作为条件之一。 |
| 平台有暂停/恢复权 | 合约执行检查必须尊重平台暂停状态。 |
| 质检方只提供事实 | 合约可以使用质检状态作为条件，但不生成质检结论。 |
| Agent 继承角色权限 | Agent 触发执行检查不得越过所属角色授权。 |
| 无关案件不可见 | 合约业务输出只能被相关角色看到。 |

## 13. 与数据信息域的关系

| 信息域 | 合约如何使用 | 合约不做什么 |
|---|---|---|
| 融资案件 | 使用案件状态判断是否进入执行检查。 | 不创建贸易事实。 |
| 贸易材料 | 使用材料完整性结果作为条件。 | 不解析和判断材料真假。 |
| 质检事实 | 使用 pass/fail/dispute/pending 状态。 | 不出质检结论。 |
| 资金方决策 | 使用允许/拒绝/补材料等结果。 | 不替资金方审批。 |
| 平台运营 | 使用暂停、恢复、规则、资金池状态。 | 不替平台做全部运营判断。 |
| Agent 输出 | 可使用 Agent 提示的缺口或升级状态。 | 不把 Agent 摘要当事实。 |
| 审计记录 | 产生执行结果记录。 | 不删除历史记录。 |

## 14. P0 必须覆盖的合约业务场景

| 场景 | 必须输出 |
|---|---|
| 资金方已批准且条件满足 | Allowed / Completed。 |
| 资金方已批准但资金池不足 | Pool insufficient。 |
| 资金方已批准但规则暂停 | Rule blocked / Platform paused。 |
| 质检 dispute 未解除 | Dispute / Need review。 |
| 关键材料缺失 | Need materials / Waiting。 |
| 平台暂停资金池 | Platform paused。 |
| 平台暂停案件 | Platform paused / Rule blocked。 |
| 纯 DeFi 路径条件满足 | Allowed / Completed。 |
| 纯 DeFi 路径条件不足 | Waiting / Rule blocked。 |
| 执行失败 | Execution failed。 |
| 执行完成 | Completed。 |

## 16. 不得进入的合约实现内容

| 禁止内容 | 后置阶段 |

| 函数名和函数签名 | 合约规格阶段。 |
| modifier / event / storage | 合约实现阶段。 |
| 链上数据结构 | 架构和实现阶段。 |

| 钱包签名细节 | 钱包实现阶段。 |
| gas、交易、nonce、revert 细节 | 合约实现阶段。 |
| 测试脚本 | 测试实现阶段。 |

## 17. 通过标准

| 检查项 | 通过标准 |
|---|---|
| 合约定位清楚 | 是业务规则和资金执行约束层。 |
| 不负责事项清楚 | 不判断贸易真实性、不替资金方审批、不裁决争议。 |
| 执行条件清楚 | 授权、事实、规则、资金池、暂停状态明确。 |
| 业务输出清楚 | 输出能被出口商、资金方、平台和 Agent 理解。 |
| 路径差异保留 | 银行辅助、纯 DeFi、低人工、人工升级都保留。 |
| 暂停边界清楚 | 案件、资金池、规则、合约执行、角色账户、Agent 自动化边界清楚。 |
| 失败解释清楚 | 执行失败有责任方和下一步。 |
| 实现细节后置 | 未出现 Solidity、ABI、函数签名、部署或链上存储。 |

## 18. 一票否决项

| 一票否决项 | 原因 |
|---|---|
| 合约被设计成贸易真实性判断者 | 破坏事实来源边界。 |
| 合约替资金方做银行辅助路径的最终审批 | 破坏资金方责任。 |
| 合约替质检方输出 pass/fail/dispute | 破坏质检事实边界。 |
| 合约绕过平台暂停继续执行 | 破坏平台运营边界。 |
| 合约接受 Agent 越权触发高风险执行 | 破坏 Agent 权限边界。 |
| 合约输出只有技术成功/失败 | 业务角色无法理解。 |
| 合约设计直接出现 Solidity、ABI、函数签名或部署细节 | 阶段越界。 |
| P1/P2 合约能力被塞入 P0 必做 | 破坏 MVP 范围。 |

## 19. 本步结论

P0 智能合约业务设计的本质不是“写合约”。

它的本质是：

> 在业务层定义 ChainTrace 的合约 / 规则执行层如何约束资金执行，如何检查授权、事实、规则、资金池和暂停状态，如何输出可解释的业务结果，并明确合约不判断贸易真实性、不替资金方审批、不替事实方确认、不裁决重大争议。
