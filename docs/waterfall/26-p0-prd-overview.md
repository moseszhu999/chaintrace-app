# 26 P0 PRD 总纲

## 1. 定位

本文件是 ChainTrace P0 阶段的产品需求说明总纲。

它不是 UI 设计稿，不是技术方案，不是数据库设计，也不是智能合约实现说明。

它的作用是：

> 把前序瀑布需求文档收敛成 P0 产品范围、角色、业务路径、状态、权限、验收和排除范围，为后续 PRD 冻结和产品原型阶段提供统一口径。

ChainTrace 当前定位是：

> Agent Native + Crypto Native 的供应链金融系统。

它不是供应链金融前站，也不是单纯银行审批系统。

它同时支持：

| 路径 | 说明 |
|---|---|
| 银行辅助路径 | 需要银行、保理机构或资金方审查和授权的融资场景。 |
| 纯 DeFi 路径 | 不需要银行审批，由规则、资金池、抵押/保证和事实状态驱动的融资或结算场景。 |
| Agent + 智能合约低人工路径 | 规则清楚、授权边界明确、事实结构化的结算、保证金、付款责任或法律/运营场景。 |

## 2. P0 产品目标

P0 的目标不是覆盖所有供应链金融业务。

P0 的目标是先跑通一个可解释、可分工、可自动推进的最小业务闭环：

```text
业务方创建融资/结算意图
→ Agent 整理材料、事实、缺口和风险摘要
→ 事实方提供必要事实
→ 根据场景进入银行辅助、纯 DeFi 或低人工路径
→ 银行/资金方授权，或 DeFi 智能合约检查执行条件
→ 执行、等待、阻断、失败或到账
→ 用户看到结果、原因、卡点和下一步
```

P0 必须回答四个业务问题：

| 问题 | P0 回答 |
|---|---|
| 这笔业务能不能继续？ | 可以、等待条件、补材料、阻断、拒绝或人工复核。 |
| 为什么能或不能？ | 材料、事实、规则、资金池、授权、风险或平台状态原因清楚。 |
| 当前卡在哪里？ | 出口商、事实方、资金方、Agent、平台、规则、资金池或合约执行。 |
| 下一步谁处理？ | 每个待办必须有责任角色、动作、完成标准和回流状态。 |

## 3. P0 核心角色

| 角色 / 能力 | P0 定位 | 是否所有场景必需 |
|---|---|---:|
| 出口商 / 资金需求方 | 创建融资意图、提交贸易材料、查看结果和卡点。 | 是 |
| Agent | 每个角色身边的业务助理，整理、检查、提醒、推进和升级异常。 | 是 |
| DeFi 智能合约资金执行层 | 检查资金池、规则、抵押/保证、授权边界和执行条件，执行或阻断资金动作。 | 是 |
| 平台方 / 合约运营 | 维护规则、资金池、暂停、异常、执行秩序和系统边界。 | 是 |
| 质检方 / 第三方检验机构 | 提供独立货物事实。 | P0 核心事实方 |
| 资金方 / 银行 / 保理机构 | 在需要银行参与的场景中审查和授权。 | 否，按场景出现 |
| 物流方 | 提供运输事实。 | P1 |
| 保险公司 | 提供风险缓释和理赔事实。 | P1 |
| 买方 / 进口商 | 确认订单、收货、争议和付款责任。 | P1 |
| 海关 / 报关方 | 提供通关和监管事实。 | P2 |

## 4. P0 场景范围

### 4.1 P0 必须支持

| 场景 | 路径 | 说明 |
|---|---|---|
| 出口商基于贸易材料创建融资意图 | 通用入口 | 形成业务案件。 |
| 贸易材料整理和缺口检查 | Agent Native | Agent 输出摘要、缺口、待办和建议。 |
| 质检事实确认 | 事实方协作 | 质检方提交 pass / fail / dispute。 |
| 银行辅助融资 | 银行辅助路径 | 资金方做允许放款、拒绝、补材料、人工复核。 |
| 纯 DeFi 自动执行 | 纯 DeFi 路径 | 规则和资金池满足时由合约执行。 |
| 资金池不足或规则阻断 | 合约约束 | 输出业务原因和下一步。 |
| 异常和补材料回流 | 人工/Agent 协作 | 缺口能回到对应角色处理。 |
| 执行结果解释 | 通用结果 | 到账、等待、失败、阻断、关闭等状态可解释。 |

### 4.2 P0 不强求完整支持

| 场景 | 原因 |
|---|---|
| 完整物流链路管理 | 物流方在 P1 作为事实增强。 |
| 完整保险承保和理赔系统 | 保险在 P1 作为风险缓释增强。 |
| 买方完整付款系统 | 买方在 P1 先确认付款责任和争议。 |
| 真实海关系统集成 | 海关/报关价值高但接入复杂，P2 后置。 |
| 所有法律条款自动执行 | P0 只定义低人工业务边界，不写法律执行代码。 |

## 5. P0 业务状态

P0 状态必须用业务语言解释，不用技术错误码替代。

| 状态 | 业务含义 |
|---|---|
| Draft | 案件草稿，材料仍在准备。 |
| Submitted | 案件已提交，等待 Agent 或相关角色处理。 |
| Material checking | Agent 正在检查材料完整性和缺口。 |
| Need materials | 需要补材料或补说明。 |
| Fact confirming | 事实方正在确认质检、物流、买方或其他事实。 |
| Under review | 银行辅助场景中，资金方正在审查。 |
| DeFi checking | 纯 DeFi 或低人工场景中，合约正在检查执行条件。 |
| Manual review | 高风险、争议、异常或授权节点需要人工处理。 |
| Approved | 已允许进入执行检查。 |
| Rejected | 已拒绝继续。 |
| Execution pending | 等待执行条件或资金动作。 |
| Blocked | 被规则、资金池、平台、事实冲突或风险阻断。 |
| Funds arrived | 资金到账或结算完成。 |
| Closed | 案件关闭。 |

## 6. P0 功能范围总表

| 功能域 | P0 必须有 | P0 不做 |
|---|---|---|
| 注册和角色 | 钱包永久绑定单一业务身份。 | 多角色切换、复杂组织权限。 |
| 案件创建 | 创建融资/结算意图和基础材料。 | 复杂合同编辑器。 |
| Agent 协作 | 摘要、缺口、待办、提醒、异常升级。 | 伪造事实、裁决重大争议。 |
| 事实确认 | 质检事实进入 P0 闭环。 | 全量物流、保险、海关系统。 |
| 银行辅助 | 需要银行时支持审查和决策。 | 把所有场景都强制交给银行。 |
| 纯 DeFi 执行 | 无银行场景下按规则和资金池执行或阻断。 | Solidity、ABI、部署细节。 |
| 平台运营 | 规则、资金池、暂停、异常、执行失败。 | 替资金方审批或伪造事实。 |
| 状态解释 | 用户看到当前状态、原因、下一步。 | 只显示技术错误码。 |

## 7. Agent 模式要求

每个角色都有三种 Agent 模式：

| 模式 | P0 行为 |
|---|---|
| 全人工确认 | Agent 只整理、提醒、建议，所有关键动作都需人确认。 |
| 半自动托管 | Agent 自动整理和推进低风险动作，关键授权和异常需人确认。 |
| 全自动 Agent | 在授权范围、低风险、规则明确、事实完整时自动推进。 |

P0 默认推荐：

> 半自动托管。

全自动 Agent 只适用于低风险、授权清楚、事实完整、规则清楚的场景。

## 8. 权限原则

P0 权限遵循以下原则：

| 原则 | 说明 |
|---|---|
| 单一业务身份 | 一个钱包注册后永久绑定一个角色。 |
| 案件相关可见 | 角色只能看与自己相关的案件、任务和事实。 |
| 事实来源不可篡改 | 出口商不能改质检事实，质检方不能改贸易材料。 |
| 决策权按场景归属 | 银行辅助场景中资金方决策；纯 DeFi 场景中合约按规则执行。 |
| 平台管规则不管事实 | 平台维护规则和执行秩序，不伪造业务事实。 |
| Agent 不独立拥有权力 | Agent 权限来自所属角色和授权模式。 |

## 9. 验收口径

P0 PRD 通过的业务标准：

| 标准 | 通过要求 |
|---|---|
| 闭环可讲通 | 从创建意图到到账、阻断或关闭可完整解释。 |
| 多路径清楚 | 银行辅助、纯 DeFi、低人工路径边界清楚。 |
| 银行不是必经角色 | 无银行场景不会强制等待资金方审批。 |
| DeFi 合约是资金执行核心 | 合约可执行或阻断资金动作，而不是只做展示。 |
| Agent 边界清楚 | 能整理和推进，不能伪造事实或裁决重大争议。 |
| 人工边界清楚 | 异常、争议、高风险、监管、授权由人处理。 |
| 权限边界清楚 | 角色不能越权查看、修改或决策。 |
| 排除范围清楚 | 不写 UI、数据库、API、Solidity、hash/proof 细节。 |

## 10. 本步结论

P0 PRD 总纲把 ChainTrace 从“需求文档集合”收敛为一个可进入产品需求冻结的业务模型：

> ChainTrace P0 是一个以融资/结算案件为中心、由 Agent 整理和推进、由事实方提供业务事实、按场景选择银行辅助或 DeFi 智能合约执行、由平台维护规则和异常边界的供应链金融系统。

下一步应进入：

> P0 场景执行路径矩阵、权限矩阵、业务动作清单和业务事件清单。
