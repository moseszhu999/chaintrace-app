# 25 场景执行路径模型

## 1. 定位

本文件修正并明确 ChainTrace 的多场景执行模型。

ChainTrace 不是所有场景都必须经过银行审批的系统。

它的核心定位是：

> 一个 Agent Native + Crypto Native 的供应链金融系统，根据不同业务场景选择银行辅助、纯 DeFi、或 Agent + 智能合约低人工执行路径。

本文件只写业务需求，不写页面、组件、数据库、Solidity、ABI、部署、钱包签名、hash 或 proof 细节。

## 2. 核心原则

| 原则 | 说明 |
|---|---|
| 场景决定路径 | 不同融资、结算、担保、法律和运营场景可以选择不同执行路径。 |
| 银行不是永远必须 | 银行/资金方是部分场景的参与方，不是所有场景的必经角色。 |
| DeFi 智能合约是资金执行核心 | 资金池、规则、抵押/保证、释放、结算和阻断由合约执行或约束。 |
| Agent 降低人工参与 | Agent 整理材料、检查缺口、生成待办、提醒、推进低风险流程。 |
| 人处理关键例外 | 人主要处理异常、争议、高风险授权、监管判断和重大法律责任。 |
| 事实来源仍要清楚 | Agent 和合约不能伪造贸易、质检、物流、保险、买方或法律事实。 |

## 3. 三类执行路径

### 3.1 银行辅助路径

| 项目 | 说明 |
|---|---|
| 适用场景 | 银行、保理机构或资金方需要审查和授权的融资。 |
| 核心角色 | 出口商、银行/资金方、事实方、平台方、Agent、DeFi 智能合约。 |
| 决策来源 | 银行/资金方做放款、拒绝、补材料、人工复核决策。 |
| 合约作用 | 银行批准后检查资金池、规则、暂停、执行条件并执行或阻断。 |
| 人工参与 | 较高，尤其是授信、放款授权、高金额和高风险判断。 |

业务路径：

```text
出口商创建融资意图
→ Agent 整理材料和缺口
→ 事实方提供必要事实
→ 银行/资金方审查和授权
→ DeFi 智能合约检查执行条件
→ 执行、等待、阻断或失败
```

### 3.2 纯 DeFi 路径

| 项目 | 说明 |
|---|---|
| 适用场景 | 规则明确、资金池明确、抵押/保证或风险边界清楚，不需要银行审批的融资或结算。 |
| 核心角色 | 业务方、事实方、Agent、平台方、DeFi 智能合约。 |
| 决策来源 | 预设规则、资金池状态、抵押/保证、事实状态和授权边界。 |
| 合约作用 | 直接执行或阻断资金动作。 |
| 人工参与 | 低，主要处理异常、争议、规则调整和高风险情况。 |

业务路径：

```text
业务方创建融资/结算意图
→ Agent 整理材料和事实
→ 事实方确认必要事实
→ Agent 检查缺口和规则适配
→ DeFi 智能合约检查资金池、抵押/保证、规则和暂停状态
→ 自动执行或阻断
```

### 3.3 Agent + 智能合约低人工路径

| 项目 | 说明 |
|---|---|
| 适用场景 | 法律、结算、保证金、付款责任、运营规则清楚且可被合约约束的场景。 |
| 核心角色 | Agent、DeFi 智能合约、必要事实方、平台方。 |
| 决策来源 | 预先定义的法律/业务规则、授权边界、事实状态和异常升级规则。 |
| 合约作用 | 锁定、释放、结算、阻断资金或记录关键状态。 |
| 人工参与 | 最低，正常场景由 Agent + 合约推进，重大争议或高风险责任交给人。 |

业务路径：

```text
规则和授权边界预先定义
→ Agent 收集、整理和解释事实
→ Agent 生成待办和提醒
→ DeFi 智能合约执行锁定、释放、结算或阻断
→ 人只处理异常、争议、高风险授权或约定仲裁
```

## 4. 场景分类示例

| 场景 | 推荐路径 | 是否需要银行 | 人工参与 |
|---|---|---:|---|
| 传统应收账款融资 | 银行辅助路径 | 是 | 中到高 |
| 银行保理融资 | 银行辅助路径 | 是 | 中到高 |
| 基于合约资金池的自动小额融资 | 纯 DeFi 路径 | 否 | 低 |
| 抵押充足的自动放款 | 纯 DeFi 路径 | 否 | 低 |
| 规则明确的保证金释放 | Agent + 智能合约低人工路径 | 否 | 很低 |
| 付款责任到期自动提醒和结算 | Agent + 智能合约低人工路径 | 否 | 很低 |
| 法律条款明确的资金锁定/释放 | Agent + 智能合约低人工路径 | 否 | 很低，争议时升级 |
| 高金额或监管敏感融资 | 银行辅助路径 | 通常是 | 高 |
| 质检 dispute 或买方争议 | 任一路径升级人工 | 视场景而定 | 高 |

## 5. Agent 在不同路径中的作用

| Agent 能力 | 银行辅助路径 | 纯 DeFi 路径 | 低人工路径 |
|---|---|---|---|
| 整理材料 | 给银行和相关角色生成摘要。 | 给合约规则和参与方生成事实摘要。 | 给合约和少量人类监督者生成摘要。 |
| 检查缺口 | 找出银行审查缺口。 | 找出规则和事实缺口。 | 找出法律/运营规则缺口。 |
| 生成待办 | 分派给出口商、事实方、资金方。 | 分派给事实方、平台方或业务方。 | 分派给事实方或人工升级角色。 |
| 推进流程 | 推进到银行审查和合约执行。 | 推进到合约自动执行。 | 推进到合约锁定、释放、结算或阻断。 |
| 升级异常 | 高风险、争议、授权。 | 规则例外、事实冲突、资金池异常。 | 法律争议、责任冲突、高风险授权。 |

## 6. DeFi 智能合约在不同路径中的作用

| 合约职责 | 银行辅助路径 | 纯 DeFi 路径 | 低人工路径 |
|---|---|---|---|
| 检查授权 | 检查银行/资金方授权。 | 检查规则授权和用户授权。 | 检查规则授权和预设边界。 |
| 检查资金池 | 必须检查。 | 必须检查。 | 视资金动作而定。 |
| 检查抵押/保证 | 可选。 | 常见核心条件。 | 常见核心条件。 |
| 检查事实状态 | 使用事实方和 Agent 结果。 | 使用事实方和 Agent 结果。 | 使用事实方、Agent 和预设规则。 |
| 执行资金动作 | 银行批准后执行。 | 条件满足时直接执行。 | 条件满足时锁定、释放、结算或阻断。 |
| 阻断 | 条件不满足时阻断。 | 条件不满足时阻断。 | 规则或争议触发时阻断。 |

## 7. 人工参与边界

人必须处理：

| 场景 | 原因 |
|---|---|
| 高金额融资 | 资金风险高。 |
| 银行授信和放款授权 | 需要银行参与的场景中，决策权属于银行/资金方。 |
| 重大法律争议 | Agent 和合约不能自动裁决重大争议。 |
| 事实冲突 | 贸易、质检、物流、买方责任等事实冲突需要人处理。 |
| 监管敏感事项 | 涉及监管判断时不能完全自动化。 |
| 平台级暂停和恢复 | 影响系统级风险。 |

人可以少参与：

| 场景 | 原因 |
|---|---|
| 小额、规则明确、抵押充足融资 | DeFi 规则和资金池可以自动约束。 |
| 保证金按条件释放 | 条件明确，合约可执行。 |
| 到期提醒和自动结算 | Agent 可提醒，合约可执行。 |
| 材料完整且无异常的标准流程 | Agent 可整理，合约可检查。 |

## 8. MVP 范围

### P0 必须做

| 范围 | 说明 |
|---|---|
| 定义三类执行路径 | 银行辅助、纯 DeFi、Agent + 智能合约低人工。 |
| 明确银行不是必经角色 | 银行只在部分场景中参与。 |
| 明确 DeFi 合约核心性 | 合约是资金执行核心，不是外围证明工具。 |
| 明确 Agent 低人工能力 | Agent 负责整理、检查、待办、提醒、异常升级。 |
| 明确人工边界 | 异常、争议、高风险、监管和平台控制交给人。 |

### 暂不做

| 范围 | 原因 |
|---|---|
| Solidity 实现 | 当前只写业务需求。 |
| 数据库设计 | 当前不进入技术实现。 |
| UI 页面设计 | 当前不做组件和页面。 |
| hash / proof 细节 | 当前不进入底层证明产品。 |
| 法律条款自动执行代码 | 当前只定义业务场景和边界。 |

## 9. 本步结论

ChainTrace 的场景执行模型不是“所有融资都找银行审批”。

它的本质是：

> 根据场景选择银行辅助、纯 DeFi、或 Agent + 智能合约低人工路径；银行是部分场景角色，DeFi 智能合约是资金执行核心，Agent 是低人工运营核心，人主要处理异常、争议、高风险和监管判断。
