# 20 P0 需求完成度和下一阶段检查清单

## 1. 定位

本文件用于检查 ChainTrace P0 MVP 是否已经具备进入下一阶段的业务需求基础。

它不是测试用例，不是开发计划，也不是技术验收标准。

它的核心作用是：

> 在进入 UI、数据、合约或实现设计之前，确认 P0 业务闭环、角色边界、状态流转、Agent 模型和排除范围已经清楚。

本文件只写需求完成度检查，不写页面、组件、数据库、智能合约实现、钱包签名、hash 或 proof 细节。

## 2. 检查原则

| 原则 | 说明 |
|---|---|
| 先业务闭环 | 必须确认融资闭环跑得通。 |
| 先角色边界 | 每个角色能做什么、不能做什么必须明确。 |
| 先状态和异常 | 案件状态、卡点、补材料、阻断必须可解释。 |
| 先 Agent 操作模型 | Agent 能做什么、不能做什么、三种模式差异必须明确。 |
| 先排除范围 | 不做什么必须写清楚，防止提前跳到实现。 |
| 不以代码为验收 | 当前阶段只验收需求文档。 |

## 3. P0 业务闭环检查

| 检查项 | 是否应满足 | 说明 |
|---|---|---|
| 出口商可以创建融资意图 | 是 | 融资需求起点清楚。 |
| 出口商可以提交贸易材料 | 是 | 合同、订单、发票、装箱单等核心材料清楚。 |
| Agent 可以整理材料和缺口 | 是 | 摘要、缺口、建议动作清楚。 |
| 质检方可以提供独立事实 | 是 | pass / fail / dispute 清楚。 |
| 资金方可以审查和决策 | 是 | 允许放款、拒绝、补材料、人工复核清楚。 |
| 平台/规则/合约业务层可以检查执行条件 | 是 | 资金池、规则、暂停、执行输出清楚。 |
| 出口商可以看到结果或卡点 | 是 | 到账、等待、拒绝、补材料、阻断清楚。 |

结论要求：

```text
如果以上任一项不清楚，不应进入 UI 或技术实现阶段。
```

## 4. 角色边界检查

| 角色 | 必须清楚的问题 |
|---|---|
| 出口商 | 只创建和管理自己的融资案件，不能审批放款或修改他人事实。 |
| 资金方 | 做放款判断，不能伪造贸易、质检、物流、保险、买方事实。 |
| 质检方 | 提供货物事实，不能决定放款或修改出口商材料。 |
| 平台方 | 管资金池、规则、异常、暂停和执行秩序，不能替资金方审批。 |
| 物流方 | 提供运输事实，不能决定融资。 |
| 海关 / 报关方 | 提供通关和监管事实，不能替资金方审批。 |
| 保险公司 | 提供风险缓释判断，不能替资金方放款。 |
| 买方 | 确认订单、收货、争议、付款责任，不能修改融资申请。 |
| Agent | 整理、检查、提醒、推进低风险流程，不能伪造事实或裁决重大争议。 |
| 智能合约业务层 | 检查规则和资金执行条件，不能判断贸易真实性。 |

## 5. Agent Native 检查

| 检查项 | 是否应满足 | 说明 |
|---|---|---|
| Agent 不是单独用户 | 是 | Agent 是每个角色身边的业务助理。 |
| 每个角色有三种模式 | 是 | 全人工确认、半自动托管、全自动 Agent。 |
| Agent 可以整理材料 | 是 | 但不能伪造事实。 |
| Agent 可以检查缺口 | 是 | 缺什么、找谁补、下一步是什么。 |
| Agent 可以生成待办 | 是 | 待办必须有责任角色和完成标准。 |
| Agent 可以推进低风险流程 | 是 | 授权范围内推进。 |
| Agent 必须升级异常 | 是 | 高风险、争议、授权、平台控制交给人。 |
| 70/30 目标清楚 | 是 | 70% 正常案件自动推进，不超过 30% 人工处理。 |

## 6. 状态机和异常检查

| 检查项 | 是否应满足 | 说明 |
|---|---|---|
| 主状态已定义 | 是 | Draft 到 Funds arrived / Closed。 |
| 每个状态有业务解释 | 是 | 当前状态、原因、下一步。 |
| 补材料回路清楚 | 是 | Need materials 后回到原流程。 |
| 人工复核回路清楚 | 是 | Manual review 处理高风险和争议。 |
| 执行检查回路清楚 | 是 | Approved 后仍检查规则、资金池、平台状态。 |
| Blocked 不是黑盒 | 是 | 必须说明阻断原因和责任角色。 |
| 异常不会覆盖历史 | 是 | 补充材料形成补充记录。 |

## 7. 信息架构和工作区检查

| 检查项 | 是否应满足 | 说明 |
|---|---|---|
| 信息以融资案件为中心 | 是 | 不以技术模块为中心。 |
| 角色信息需求清楚 | 是 | 出口商、质检方、资金方、平台方分别回答自己的问题。 |
| Agent 摘要结构清楚 | 是 | 摘要、缺口、建议、状态解释。 |
| 卡点信息结构清楚 | 是 | 状态、原因、责任方、下一步。 |
| 工作区业务分区清楚 | 是 | 每个分区回答一个业务问题。 |
| 跨角色协作清楚 | 是 | 通过待办和状态变化推进。 |
| 权限可见性清楚 | 是 | 不同角色不能看无关案件或越权修改。 |

## 8. 智能合约业务定位检查

| 检查项 | 是否应满足 | 说明 |
|---|---|---|
| 智能合约不是页面 | 是 | 不承担 UI 职责。 |
| 智能合约不是数据库 | 是 | 不保存完整业务材料。 |
| 智能合约不是 Agent | 是 | 不整理和解释业务材料。 |
| 智能合约是执行约束层 | 是 | 检查规则、资金池、暂停、执行条件。 |
| 智能合约不判断贸易真实性 | 是 | 事实来自业务角色。 |
| 智能合约不替资金方审批 | 是 | 放款权属于资金方。 |
| 业务输出清楚 | 是 | 允许执行、不允许执行、等待条件、规则阻断、资金池不足等。 |

## 9. 排除范围检查

以下内容如果出现在当前阶段，应视为跑偏：

| 排除项 | 原因 |
|---|---|
| 登录前展示页 | 当前不做营销页。 |
| UI 组件 | 当前不做前端设计。 |
| 页面布局和导航细节 | 当前只做业务分区。 |
| 数据库表结构 | 当前不进入技术设计。 |
| 智能合约代码 | 当前只定义业务定位。 |
| 钱包签名细节 | 当前不进入实现。 |
| hash / proof / proof pack 细节 | 当前不跳到底层证明产品。 |
| 原旧版 app/components/lib/contracts/scripts 代码继承 | 当前分支是全新瀑布重建设计分支。 |

## 10. 进入下一阶段的建议标准

只有当以下标准满足后，才建议进入下一阶段：

| 标准 | 说明 |
|---|---|
| P0 角色需求已经写清楚 | 出口商、资金方、质检方、平台方。 |
| P1/P2 角色边界已经写清楚 | 物流、保险、买方、海关/报关。 |
| Agent Native 模型已经写清楚 | 三种模式、能做和不能做、70/30 目标。 |
| 智能合约业务角色已经写清楚 | 执行约束层，不做事实判断。 |
| P0 用户旅程已经写清楚 | 从融资意图到账或卡点。 |
| P0 状态机已经写清楚 | 状态、原因、下一步、责任角色。 |
| P0 异常和补材料已经写清楚 | 缺口、责任、回流、人工升级。 |
| P0 Agent 待办已经写清楚 | 待办结构、优先级、模式差异、回流。 |
| P0 信息架构和工作区分区已经写清楚 | 信息域、角色视角、业务分区。 |

## 11. 下一阶段建议

完成上述检查后，下一阶段建议进入：

> P0 业务原型说明和验收口径。

注意：这里的“业务原型说明”仍然不是 UI 组件设计。

建议先写：

| 文档 | 目的 |
|---|---|
| P0 业务原型说明 | 用文字定义各角色工作区应该表达的业务内容。 |
| P0 验收口径 | 定义每个角色在 MVP 中必须能完成哪些业务动作。 |
| P0 示例案件脚本 | 用一个虚拟贸易融资案件串起全流程。 |
| P0 阶段边界说明 | 再次确认不做登录前展示页、不做代码、不做合约实现。 |

## 12. 本步结论

P0 需求完成度检查的本质不是“准备开发清单”。

它的本质是：

> 在进入任何 UI、数据或技术实现之前，确认 ChainTrace 作为 Agent Native + Crypto Native 供应链金融前站系统的 P0 业务闭环已经完整、边界清楚、状态可解释、异常可处理、Agent 可协作。

下一步应进入：

> P0 业务原型说明、验收口径和示例案件脚本。
