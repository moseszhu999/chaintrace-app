# 40 P0 视觉原型准备说明

## 1. 定位

本文件定义 ChainTrace P0 阶段进入视觉原型前的业务准备说明。

视觉原型准备说明不是视觉设计稿，不是 UI 组件说明，不是页面布局规范，也不是前端实现计划。

它的作用是：

> 用业务语言说明视觉原型应该围绕哪些角色视角、控制塔层级、供应链节点、路径差异、Agent 输出、DeFi 执行结果和验收走读场景组织，确保视觉原型仍然服务 ChainTrace 的供应链金融业务闭环。

本文件不写组件、样式、颜色、布局尺寸、图表库、页面路由、接口路径、请求响应格式、数据库、Solidity、ABI、钱包签名、hash 或 proof 细节。

## 2. 视觉原型准备原则

| 原则 | 说明 |
|---|---|
| 先业务判断，后视觉表现 | 视觉原型首先要帮助用户判断案件状态、卡点、责任方和下一步。 |
| 先控制塔，后局部详情 | 第一层表达供应链全链路，再进入节点、事实、待办和异常。 |
| 先角色视角，后全局展示 | 不同角色只能看到自己的业务范围。 |
| 先路径差异，后通用模板 | 银行辅助、纯 DeFi、低人工路径必须有明显业务差异。 |
| 先 Agent 摘要，后原始材料 | Agent 先把信息整理成摘要、缺口、建议和待办。 |
| 先业务输出，后技术细节 | DeFi 合约输出必须是允许、等待、阻断、失败或完成，不展示实现细节。 |
| 不做登录前展示页 | 视觉原型从注册后业务工作区开始。 |

## 3. 视觉原型目标

视觉原型应让业务用户在第一轮走读中回答以下问题。

| 问题 | 视觉原型需要支撑的业务判断 |
|---|---|
| 这是什么案件？ | 融资/结算意图、贸易背景、金额、路径和当前状态清楚。 |
| 这笔业务从哪里到哪里？ | 供应链源头到终端的关键节点清楚。 |
| 当前卡在哪里？ | 卡点类型、原因、责任角色和解除条件清楚。 |
| 谁应该处理下一步？ | 待办、责任角色、完成标准清楚。 |
| 哪些事实已经确认？ | 质检、物流、通关、保险、买方等节点状态清楚。 |
| 当前走哪条路径？ | 银行辅助、纯 DeFi、低人工或人工升级清楚。 |
| DeFi 是否允许执行？ | 规则、资金池、授权、保证条件和暂停状态的业务结果清楚。 |
| 我能看到哪些内容？ | 角色可见范围和不可见范围清楚。 |

## 4. 视觉原型输入物

进入视觉原型前，应以以下文档作为业务输入。

| 输入物 | 视觉原型应使用的内容 |
|---|---|
| P0 PRD 总纲 | 产品定位、核心角色、路径、状态和范围。 |
| 信息架构业务需求 | 信息域、角色信息需求、状态和卡点结构。 |
| 工作区业务分区 | 各角色工作区的业务分区。 |
| 业务原型说明 | 每个角色应看到什么、完成什么。 |
| 控制塔式原型页面业务要求 | 大数据控制塔表达、节点、角色视角和路径差异。 |
| API 业务契约 | 业务动作、输入、输出、失败解释和控制塔输出。 |
| 原型验收脚本 | 视觉原型走读时必须通过的业务场景。 |
| 产品原型冻结清单 | 进入视觉原型前已冻结和不得新增的范围。 |

## 5. 控制塔层级准备

视觉原型应准备以下业务层级，但不在本阶段规定具体布局。

| 层级 | 要表达的业务内容 | 业务目的 |
|---|---|---|
| 全链路层 | 业务意图、贸易材料、事实确认、审查/规则、执行、资金结果。 | 让用户知道案件在哪里。 |
| 节点状态层 | Not started、Waiting、In progress、Completed、Warning、Dispute、Blocked、Failed、Skipped。 | 让用户知道每个节点是否正常。 |
| 角色视角层 | 出口商、资金方、事实方、平台方、Agent 各自可见范围。 | 避免全员看全量数据。 |
| Agent 层 | 摘要、缺口、待办、建议、提醒、升级。 | 降低业务噪音。 |
| 风险和异常层 | 事实冲突、争议、高风险、规则阻断、资金池不足、执行失败。 | 让人处理真正需要人的事项。 |
| DeFi 执行层 | 允许、等待、阻断、失败、完成。 | 解释资金执行是否能继续。 |
| 审计和历史层 | 关键动作、事实、授权、阻断、恢复和执行结果。 | 支撑可信和追踪。 |

## 6. 角色视角准备

视觉原型应分别准备以下角色视角。

| 角色视角 | 第一眼要回答 | 主要业务内容 |
|---|---|---|
| 出口商 / 资金需求方 | 我这笔业务能不能融资或结算，卡在哪里？ | 自己案件、材料、事实进度、Agent 缺口、资金结果。 |
| 资金方 / 银行 / 保理机构 | 哪些案件需要审查，风险是否可控？ | 相关案件摘要、材料完整性、事实状态、风险、授权、执行条件。 |
| 质检方 | 哪些任务需要质检，我要确认什么？ | 待质检任务、货物摘要、质检要求、结论和补充要求。 |
| 物流方 | 哪些运输节点需要确认，是否异常？ | 揽收、装柜、出港、到港、签收、延误、损坏、丢失。 |
| 海关 / 报关方 | 哪些通关事实需要确认，是否有监管异常？ | 报关、清关、监管状态、一致性提示。 |
| 保险公司 | 是否承保、加费、拒保或进入理赔？ | 风险摘要、承保结论、补材料、理赔状态。 |
| 买方 / 进口商 | 这笔订单和我是否有关，是否收货或有争议？ | 订单真实性、收货、争议、付款责任。 |
| 平台方 / 合约运营 | 系统是否安全，哪里需要介入？ | 资金池、规则、暂停、异常、执行失败、Agent 堆积。 |
| Agent | 当前角色最应该知道和处理什么？ | 摘要、缺口、待办、建议、提醒、升级原因。 |

## 7. 关键走读场景准备

视觉原型至少要准备以下业务走读场景。

| 场景 | 视觉原型要验证 |
|---|---|
| 出口商控制塔走读 | 出口商能从源头到资金结果看清自己案件。 |
| 银行辅助正常通过 | 资金方审查、授权、合约执行检查和到账可讲通。 |
| 纯 DeFi 自动执行 | 无银行路径、规则/资金池/保证条件检查和执行完成可讲通。 |
| 低人工结算或保证金释放 | Agent 自动推进、合约锁定/释放、异常升级可讲通。 |
| 质检 dispute | 争议原因、影响、人工复核和补材料路径可讲通。 |
| 资金池不足 | 用户知道不是材料问题，而是资金池等待或平台处理。 |
| 规则阻断 | 阻断规则、影响范围、解除条件和责任角色可讲通。 |
| 买方争议 | 自动执行停止，进入人工或约定仲裁可讲通。 |
| 平台暂停/恢复 | 暂停原因、影响范围、恢复条件和记录可讲通。 |
| 越权阻断 | 无关案件、事实篡改、越权审批被阻断并有解释。 |

## 8. 业务文案准备

视觉原型中的文字应使用业务语言，不使用技术错误码替代业务解释。

| 场景 | 推荐业务表达 |
|---|---|
| 材料不足 | 还缺少发票金额差异说明，请出口商补充。 |
| 事实等待 | 正在等待质检方确认货物事实。 |
| 资金方审查 | 该案件进入银行辅助路径，等待资金方审查。 |
| 纯 DeFi | 该案件不需要银行审批，正在检查规则和资金池条件。 |
| 规则阻断 | 当前规则暂停该品类资金执行，等待平台处理。 |
| 资金池不足 | 资金方已授权，但当前资金池余额不足。 |
| Agent 升级 | Agent 发现争议或高风险，已升级人工确认。 |
| 执行完成 | 资金动作已完成，案件进入资金结果节点。 |
| 执行失败 | 资金动作未完成，平台方需要处理失败原因。 |
| 权限不足 | 当前角色不能查看或处理该案件。 |

不应使用：

| 不应表达 | 原因 |
|---|---|
| 只有错误码 | 用户不知道业务原因。 |
| 只有交易失败 | 没有解释资金池、规则、授权或平台原因。 |
| 只有处理中 | 不知道谁在处理、处理什么。 |
| 只有已完成 | 不知道哪个节点完成、影响什么。 |

## 9. 视觉原型输出物准备

视觉原型阶段应产出以下业务可评审材料。

| 输出物 | 业务要求 |
|---|---|
| 控制塔主视角说明 | 说明全链路节点、状态、卡点和责任角色如何被表达。 |
| 角色视角说明 | 说明每个角色看到什么、看不到什么、能做什么。 |
| 路径差异说明 | 说明银行辅助、纯 DeFi、低人工路径如何区分。 |
| Agent 表达说明 | 说明摘要、缺口、待办、建议、升级如何进入原型。 |
| DeFi 执行说明 | 说明允许、等待、阻断、失败、完成如何被用户理解。 |
| 异常场景说明 | 说明补材料、争议、规则阻断、资金池不足、执行失败如何回流。 |
| 走读脚本映射 | 每个关键画面或状态要能对应 38 号原型验收脚本。 |
| 排除范围记录 | 说明哪些内容没有进入视觉原型，以及为什么后置。 |

## 10. 视觉原型不得进入的范围

| 不得进入 | 原因 |
|---|---|
| 登录前展示页 | 当前不是营销阶段。 |
| 具体 UI 组件规范 | 当前只准备视觉原型业务输入。 |
| 组件库选型 | 当前不做技术或设计系统选型。 |
| 页面路由和前端实现 | 当前不写实现。 |
| API endpoint 和请求响应格式 | 技术接口设计后置。 |
| 数据库表和字段 | 数据设计后置。 |
| Solidity / ABI / 部署 | 合约技术设计后置。 |
| 钱包签名流程细节 | 钱包实现后置。 |
| hash / proof / proof pack | 底层证明产品后置。 |
| 旧版 app/components/lib/contracts/scripts 代码 | 当前分支不继承旧代码。 |

## 11. 视觉原型评审问题

视觉原型评审时，应逐项回答以下问题。

| 评审问题 | 通过标准 |
|---|---|
| 是否一眼看出 ChainTrace 是供应链金融系统？ | 控制塔围绕供应链金融案件和资金执行，而不是营销首页。 |
| 是否能看到源头到终端流转？ | 业务意图、贸易、事实、审查/规则、执行、资金结果清楚。 |
| 角色视角是否隔离？ | 每个角色只看自己相关案件、节点、事实和待办。 |
| 银行是否只在相关路径出现？ | 纯 DeFi 和低人工路径不强制银行审批。 |
| DeFi 是否是执行核心？ | 合约输出能解释资金动作允许、等待、阻断、失败和完成。 |
| Agent 是否像业务助理？ | Agent 输出摘要、缺口、待办、建议和升级，不伪造事实。 |
| 异常是否可处理？ | 每个异常都有原因、责任角色、解除条件和下一步。 |
| 是否避免技术跑偏？ | 没有提前进入接口、数据库、Solidity、hash/proof 或旧代码。 |

## 12. MVP 范围

### 12.1 P0 视觉原型必须准备

| 范围 | 说明 |
|---|---|
| 控制塔主表达 | 供应链全链路节点和资金执行结果。 |
| 角色视角 | 出口商、资金方、事实方、平台方、Agent 的可见范围。 |
| 路径差异 | 银行辅助、纯 DeFi、低人工、人工升级。 |
| Agent 输出 | 摘要、缺口、待办、建议、提醒、升级。 |
| DeFi 输出 | 允许、等待、阻断、失败、完成。 |
| 异常回流 | 补材料、事实争议、规则阻断、资金池不足、平台暂停、执行失败。 |
| 走读脚本映射 | 对应 38 号原型验收脚本。 |

### 12.2 P0 视觉原型暂不准备

| 范围 | 原因 |
|---|---|
| 完整 P1/P2 深度流程 | P0 只保留节点入口或占位。 |
| 多组织多成员管理 | P0 保持单钱包单角色。 |
| 完整法律自动执行 | P0 保持低人工边界和人工升级。 |
| 技术接口、数据库和合约实现 | 当前仍是业务准备阶段。 |
| hash / proof 产品页 | 当前不进入底层证明产品。 |

## 13. 本步结论

P0 视觉原型准备说明的本质不是“开始设计组件”。

它的本质是：

> 把已冻结的控制塔表达、角色视角、路径差异、Agent 输出、DeFi 执行解释、异常回流和验收脚本转化为视觉原型团队可以使用的业务输入，确保下一步原型仍围绕供应链金融系统闭环展开。

下一步应进入：

> P0 视觉原型走读大纲，用业务语言定义每个角色和关键场景的走读顺序，不写具体 UI 组件。
