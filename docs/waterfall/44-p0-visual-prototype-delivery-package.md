# 44 P0 视觉原型交付包说明

## 1. 定位

本文件定义 ChainTrace P0 视觉原型阶段最终应交付的业务材料包。

视觉原型交付包说明不是 UI 组件规范，不是设计系统说明，不是前端开发清单，也不是技术验收文档。

它的作用是：

> 在视觉原型完成准备、走读、评审和变更管理后，用统一业务材料证明 P0 原型已经讲清 ChainTrace 的供应链金融系统定位、控制塔表达、角色视角、路径差异、Agent 边界、DeFi 执行、异常回流和后置范围。

本文件不写组件、样式、布局尺寸、图表库、页面路由、接口路径、请求响应格式、数据库、Solidity、ABI、钱包签名、hash 或 proof 细节。

## 2. 交付原则

| 原则 | 说明 |
|---|---|
| 交付业务证据 | 交付包证明业务闭环是否讲通，而不是证明页面是否好看。 |
| 交付角色视角 | 每个角色看到什么、不能看到什么、要完成什么必须有记录。 |
| 交付路径差异 | 银行辅助、纯 DeFi、低人工和人工升级必须可区分。 |
| 交付控制塔表达 | 必须能看清供应链源头到终端的流转、节点状态、卡点和资金结果。 |
| 交付 Agent 边界 | Agent 的摘要、缺口、待办、建议、升级和不可做事项必须清楚。 |
| 交付 DeFi 输出 | 合约业务输出必须以用户能理解的结果呈现。 |
| 交付后置清单 | P1/P2 和技术设计内容必须有后置记录。 |
| 不交付实现任务 | 当前交付包不包含代码、接口、数据库或合约实现任务。 |

## 3. 交付包输入依据

| 输入依据 | 用途 |
|---|---|
| P0 PRD 总纲 | 确认产品定位、核心闭环、角色和 P0 范围。 |
| 场景路径矩阵 | 确认银行辅助、纯 DeFi、低人工和人工升级路径。 |
| 权限矩阵 | 确认角色可见范围和不可做事项。 |
| 业务规则清单 | 确认状态、阻断、异常和人工升级规则。 |
| DeFi 合约业务规则清单 | 确认合约业务输出和资金执行条件。 |
| 控制塔原型业务要求 | 确认大数据控制塔式表达和全链路节点。 |
| 原型验收脚本 | 确认正常、异常、越权和后置场景是否走通。 |
| 视觉原型准备说明 | 确认视觉原型前置业务输入。 |
| 视觉原型走读大纲 | 确认走读顺序和覆盖场景。 |
| 视觉原型评审清单 | 确认每轮评审结论。 |
| 视觉原型变更管理说明 | 确认反馈如何分类、处理和后置。 |

## 4. 交付包总目录

P0 视觉原型交付包应至少包含以下材料。

| 交付物 | 必须包含 | 交付目的 |
|---|---|---|
| 交付包封面说明 | 版本、日期、范围、状态、责任人。 | 让评审方知道交付对象和边界。 |
| 业务定位说明 | 完整供应链金融系统、Agent Native、Crypto Native / DeFi。 | 防止回到前站或单纯银行系统。 |
| 控制塔表达说明 | 源头到终端、节点、状态、卡点、责任方、资金结果。 | 证明控制塔主表达成立。 |
| 角色视角说明 | 每个角色能看什么、不能看什么、要完成什么。 | 证明权限和职责清楚。 |
| 路径差异说明 | 银行辅助、纯 DeFi、低人工、人工升级。 | 证明不同场景不混淆。 |
| Agent 表达说明 | 摘要、缺口、待办、建议、提醒、升级和边界。 | 证明 Agent 是业务助理。 |
| DeFi 执行说明 | 允许、等待、阻断、资金池不足、暂停、失败、完成。 | 证明合约业务输出可理解。 |
| 异常回流说明 | 材料不足、事实冲突、dispute、规则阻断、资金池不足、执行失败。 | 证明异常有责任方和下一步。 |
| 走读记录 | 走读轮次、角色、路径、场景、通过项、问题项。 | 证明原型被业务走读过。 |
| 评审结论 | 通过、小幅调整、需补需求、后置、阻断。 | 证明评审结果可追踪。 |
| 变更记录 | A/B/C/D 分类、处理动作、状态、复核结论。 | 证明反馈没有绕过 waterfall。 |
| 后置清单 | P1/P2、技术设计和证明产品后置项。 | 证明 P0 没有范围膨胀。 |
| 冻结清单 | 已冻结的业务表达和不得再改的边界。 | 支撑进入下一阶段。 |

## 5. 控制塔表达交付要求

控制塔表达是交付包的核心，不是普通页面截图清单。

| 交付项 | 必须证明 | 不得只交付 |
|---|---|---|
| 全链路流转 | 能看到供应链源头、贸易材料、事实节点、审查/规则、DeFi 执行、资金结果。 | 单个孤立案件卡片。 |
| 节点状态 | 每个关键节点有业务状态和含义。 | 无解释的颜色或图标。 |
| 当前卡点 | 原因、责任角色、解除条件、下一步清楚。 | 只有“处理中”。 |
| 角色过滤 | 不同角色看到的链路范围不同。 | 所有人看到同一张全量图。 |
| 路径差异 | 银行辅助、纯 DeFi、低人工路径有不同业务节点。 | 一条通用审批链路。 |
| 异常回流 | 异常能回到补材料、复核、授权、规则处理或人工升级。 | 只有失败状态。 |
| 资金结果 | 放款、结算、锁定、释放、到账、失败或等待清楚。 | 只有“完成”。 |

## 6. 角色视角交付要求

| 角色 | 必须交付的视角证明 | 不得出现 |
|---|---|---|
| 出口商 / 资金需求方 | 自己案件、材料缺口、事实进度、资金结果、卡点解释。 | 查看其他出口商案件。 |
| 资金方 / 银行 / 保理机构 | 待审案件、风险摘要、授权条件、补材料请求、执行前状态。 | 修改事实方结论。 |
| 质检方 | 待质检任务、必要货物摘要、质检结论和 dispute。 | 决定融资或放款。 |
| 物流方 | 待确认运输节点、物流事实、异常说明。 | 修改贸易材料或质检事实。 |
| 海关 / 报关方 | 通关事实、监管状态、一致性异常。 | 替资金方审批。 |
| 保险公司 | 核保、拒保、加费、补材料、理赔入口。 | 绕过规则承保。 |
| 买方 / 进口商 | 订单真实性、收货、争议、付款责任。 | 修改出口商融资申请。 |
| 平台方 / 合约运营 | 规则、资金池、暂停、执行失败、Agent 堆积。 | 伪造贸易或事实方结论。 |
| Agent | 所属角色的摘要、缺口、待办、建议、升级。 | 超过所属角色权限。 |

## 7. 路径差异交付要求

| 路径 | 必须交付 | 交付判断 |
|---|---|---|
| 银行辅助路径 | 资金方审查、授权、补材料、授权后 DeFi 检查、资金结果。 | 银行不是唯一执行者，合约仍检查条件。 |
| 纯 DeFi 路径 | 无银行审批、规则检查、资金池检查、事实完整性、保证条件、执行结果。 | 不得强制等待银行。 |
| 低人工路径 | Agent 自动推进低风险节点、合约锁定/释放/结算、异常升级。 | 人主要处理争议、高风险和授权。 |
| 人工升级路径 | 争议、高风险、监管异常、法律责任、越权风险。 | 系统不得静默通过。 |

## 8. Agent 表达交付要求

| Agent 输出 | 交付要求 | 边界 |
|---|---|---|
| 摘要 | 能说明材料、事实、风险和当前状态。 | 必须来自可见材料和事实。 |
| 缺口 | 能说明缺什么、为什么缺、谁负责。 | 不替相关角色补材料。 |
| 待办 | 有责任角色、动作和完成标准。 | 不生成越权任务。 |
| 建议 | 提供业务建议和原因。 | 不替资金方做高风险决策。 |
| 自动推进 | 只在授权范围、低风险、规则明确、事实完整时推进。 | 不裁决重大争议。 |
| 异常升级 | 高风险、争议、执行失败、越权风险进入人工。 | 不静默放过异常。 |

## 9. DeFi 执行交付要求

| 业务输出 | 交付要求 |
|---|---|
| 允许执行 | 用户知道哪些条件已满足。 |
| 不允许执行 | 用户知道当前不能继续的业务原因。 |
| 等待条件 | 用户知道等待事实、授权、资金池、规则或保证条件。 |
| 规则阻断 | 用户知道阻断规则、影响范围和解除条件。 |
| 资金池不足 | 用户知道资金池、币种或余额问题。 |
| 平台暂停 | 用户知道暂停对象、原因和恢复条件。 |
| 执行失败 | 用户知道资金动作未完成、谁处理、下一步是什么。 |
| 执行完成 | 用户知道放款、结算、锁定、释放或到账完成。 |

DeFi 执行交付不得包含：

| 不得包含 | 原因 |
|---|---|
| 合约代码说明 | 当前不是合约实现阶段。 |
| ABI、部署、Gas 或交易技术细节 | 当前只交付业务输出。 |
| 钱包签名流程细节 | 钱包实现后置。 |
| hash / proof 产品页 | 当前不进入底层证明产品。 |

## 10. 走读和评审记录交付要求

| 记录类型 | 必须包含 | 交付目的 |
|---|---|---|
| 走读记录 | 日期、角色、路径、场景、通过项、未回答问题。 | 证明原型被业务路径验证过。 |
| 评审记录 | 评审轮次、范围、结论、问题、后置项、冻结项。 | 证明评审结论可追踪。 |
| 权限记录 | 越权场景、期望结果、是否通过。 | 证明角色隔离有效。 |
| 异常记录 | 异常类型、原因、责任方、下一步。 | 证明异常可处理。 |
| 变更记录 | 分类、依据、责任人、状态、复核结论。 | 证明反馈没有直接绕过需求。 |
| 后置记录 | 后置原因、后续阶段、P0 处理方式。 | 证明范围控制清楚。 |

## 11. 冻结和后置交付要求

### 11.1 必须冻结

| 冻结项 | 冻结含义 |
|---|---|
| 产品定位 | ChainTrace 是完整供应链金融系统，不是前站。 |
| 控制塔主表达 | 以供应链源头到终端和资金执行为主线。 |
| 角色边界 | 单钱包注册后永久绑定单一业务身份。 |
| 路径差异 | 银行辅助、纯 DeFi、低人工、人工升级不能混淆。 |
| Agent 边界 | Agent 是角色身边业务助理，不是独立审批人。 |
| DeFi 业务定位 | 智能合约是业务规则和资金执行约束层。 |
| 异常回流 | 异常必须有原因、责任方、解除条件和下一步。 |
| P0 范围 | 不做登录前展示页，不进入实现细节。 |

### 11.2 必须后置

| 后置项 | 后置说明 |
|---|---|
| 物流深度工作流 | P0 保留节点和异常入口。 |
| 保险完整理赔 | P0 保留承保和理赔状态。 |
| 买方完整付款系统 | P0 保留确认、争议和付款责任。 |
| 海关真实接入 | P0 保留通关事实节点。 |
| 多组织多成员权限 | P0 保持单钱包单角色。 |
| 复杂法律自动执行 | P0 只定义低人工边界和人工升级。 |
| API、数据库、合约实现 | 后续技术设计阶段处理。 |
| hash / proof 产品化 | 后续证明产品阶段处理。 |

## 12. 交付通过标准

| 通过项 | 通过标准 |
|---|---|
| 业务定位通过 | 评审方能明确 ChainTrace 是供应链金融系统。 |
| 控制塔通过 | 评审方能看清源头到终端、节点状态、卡点和资金结果。 |
| 角色视角通过 | 每个角色能看到自己的事，不能看到无关案件。 |
| 路径差异通过 | 银行辅助、纯 DeFi、低人工和人工升级均能讲通。 |
| Agent 表达通过 | Agent 输出有帮助但不越权。 |
| DeFi 输出通过 | 合约业务输出能被非技术用户理解。 |
| 异常回流通过 | 补材料、争议、阻断、暂停、失败都有责任方和下一步。 |
| 变更管理通过 | 每条反馈都有 A/B/C/D 分类和处理状态。 |
| 后置范围通过 | P1/P2 和技术内容没有挤入 P0。 |

## 13. 不得作为交付完成的情况

| 不得通过 | 原因 |
|---|---|
| 只有视觉画面，没有业务说明 | 无法证明业务闭环成立。 |
| 只有控制塔全局，没有角色视角 | 无法证明权限和职责成立。 |
| 所有路径都像银行审批 | 破坏纯 DeFi 和低人工路径。 |
| Agent 被描述成审批者 | 破坏 Agent Native 模型。 |
| 合约被描述成页面或数据库 | 破坏 DeFi 执行定位。 |
| 异常只有失败，没有下一步 | 无法运行业务。 |
| 后置项没有记录 | 范围不可控。 |
| 评审反馈没有分类 | 原型可能绕过需求变更。 |
| 进入接口、数据库、Solidity、钱包签名、hash/proof 细节 | 阶段跑偏。 |
| 继承旧版页面或旧代码口径 | 违背全新瀑布重建设计分支。 |

## 16. 本步结论

P0 视觉原型交付包的本质不是“交一组好看的页面”。

它的本质是：

> 用完整的业务材料证明 ChainTrace P0 视觉原型已经讲清供应链金融控制塔、角色权限、路径差异、Agent Native 协作、DeFi 资金执行、异常回流、评审结论、变更处理和后置范围，并且没有把实现细节提前带入当前阶段。

下一步应进入：

> P0 视觉原型阶段收口检查清单，用于判断视觉原型阶段是否可以正式冻结并进入下一阶段需求交接。
