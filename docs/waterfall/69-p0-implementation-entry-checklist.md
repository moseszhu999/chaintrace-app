# 69 P0 实现入口检查清单

## 1. 定位

本文件定义 ChainTrace P0 从规格准备阶段进入受控实现阶段前的入口检查清单。

它不是开发任务列表，不是 sprint plan，不是代码实现方案，不是接口详细文档，不是数据库表结构，不是 Solidity 实现，不是 UI 组件清单，也不是测试脚本。

它的作用是：

> 在真正进入实现前，检查 P0 的接口、数据、合约、Agent、原型和测试计划准备是否已经足够清楚，是否仍然遵守已冻结的业务边界、权限边界、路径差异、Agent 边界、合约边界和后置范围。

本文件只判断是否可以进入实现，不直接启动实现。

## 2. 入口检查原则

| 原则 | 说明 |
|---|---|
| 先检查边界 | 先确认 P0 主线、角色、Agent、合约和路径边界没有破坏。 |
| 再检查规格 | 接口、数据、合约、Agent、原型、测试计划要能互相支撑。 |
| 不带缺口进实现 | 阻断级回流项必须先解决。 |
| 不带 P1/P2 进实现 | 后置项不能偷偷变成 P0 实现任务。 |
| 不带旧 demo 主线进实现 | proof/hash/dashboard 旧叙事不能回流为 P0 主流程。 |
| 不以代码补需求 | 未定义动作、状态、异常、权限不能用代码临时补。 |
| 受控实现 | 可以进入实现时，也必须按规格逐步实现，而不是一次性扩张。 |

## 15. 允许进入实现的最低条件

P0 允许进入受控实现，至少必须满足：

```text
1. 接口候选已经能覆盖 P0 业务动作。
2. 概念数据对象已经能支撑 P0 信息域和事实来源。
3. 合约规格候选已经能表达执行条件和业务输出。
4. Agent 实现准备已经明确授权、自动推进和人工升级边界。
5. 原型范围已经能表达角色工作区、主链路、卡点和资金结果。
6. 测试计划候选已经覆盖正常闭环、路径差异、权限、异常和后置范围。
7. 阻断回流项已经解决。
8. P1/P2 和 proof/hash 后置项没有混入 P0 必做。
9. 无一票否决项。
```

## 16. 进入实现后的控制方式

即使允许进入实现，也必须按以下方式控制：

| 控制项 | 要求 |
|---|---|
| 分阶段实现 | 先实现身份/角色、案件主线、材料、Agent、质检、资金方、合约状态、控制塔。 |
| 保留回流 | 实现中发现新动作、新状态、新异常时回流，而不是直接写代码。 |
| 保持 P0 范围 | 不实现 P1/P2 完整能力。 |
| 不继承旧主线 | 不把旧 proof/hash/demo 作为主流程。 |
| 先业务闭环 | 先让出口商看到资金结果和卡点。 |
| 再优化体验 | UI 美化、BI、复杂工作流后置。 |
| 测试跟随业务 | 测试场景先验证业务闭环和边界。 |

## 17. 推荐的受控实现顺序

如果入口检查通过，建议实现顺序如下：

| 顺序 | 实现主题 | 目标 |
|---:|---|---|
| 1 | Identity and Role Foundation | 钱包连接、注册状态、单钱包单角色、角色工作区入口。 |
| 2 | Financing Case Core | 出口商创建融资意图、案件主状态、基础材料。 |
| 3 | Material and Gap Flow | 材料完整性、Agent 缺口、补材料待办。 |

## 19. 本步结论

P0 实现入口检查的本质不是“开始开发”。

它的本质是：

> 在进入实现之前，最后确认 ChainTrace P0 的接口、数据、合约、Agent、原型和测试计划准备都已经继承业务边界，能够支撑出口商融资闭环、DeFi/银行双路径、Agent Native、控制塔表达、异常处理和权限隔离，同时没有把 P1/P2、proof/hash 旧主线或实现细节提前混入。

如果本检查通过，下一步才可以进入：

> P0 受控实现阶段。

如果本检查不通过，必须先回流对应 waterfall 文档，不得用代码补需求缺口。
