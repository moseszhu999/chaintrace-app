# 53 P0 权限设计边界

## 1. 定位

本文件定义 ChainTrace P0 技术设计阶段的权限设计边界。

它不是鉴权实现方案，不是 token/session 设计，不是数据库权限表，不是 API 权限中间件，不是钱包签名流程，也不是前端菜单控制说明。

它的作用是：

> 用业务语言定义 P0 中每个角色能看什么、能做什么、不能做什么，以及 Agent 如何继承角色权限，确保技术设计不会放大权限、混淆角色或破坏单钱包单角色原则。

本文件只定义权限业务边界，不进入实现。

## 2. 权限设计原则

| 原则 | 说明 |
|---|---|
| 单钱包单角色 | 一个钱包注册后只能绑定一个永久业务角色。 |
| 角色不可变 | 角色一旦确认，P0 不允许修改。 |
| 角色专属工作区 | 每个角色进入不同业务空间，不是共用后台换菜单。 |
| 案件相关可见 | 用户只能看到与自己业务相关的案件、事实、任务或运营状态。 |
| 事实来源隔离 | 谁负责哪类事实，谁才能确认或补充该事实。 |
| 决策权隔离 | 资金方决定融资，质检方提供事实，平台维护系统秩序。 |
| Agent 权限继承 | Agent 继承所属角色权限，不是独立超级用户。 |
| 平台不是万能管理员 | 平台可以暂停、恢复、维护规则和资金池，但不能伪造事实或替资金方批准。 |
| 越权要有业务解释 | 被阻断时要告诉用户业务原因，而不是只返回技术错误。 |
| 权限缺口必须回流 | 如果发现某动作责任角色不清，回流 waterfall 文档。 |

## 3. 角色类型

| 角色 | P0/P1/P2 | 权限定位 |
|---|---|---|
| 出口商 / 资金需求方 | P0 | 创建融资意图、提交贸易材料、补材料、查看资金结果。 |
| 资金方 / 银行 / 保理机构 | P0 | 审查融资案件，做允许、拒绝、补材料、人工复核等资金决策。 |
| 质检方 / 第三方检验机构 | P0 | 提供货物质量、数量、包装、现场状态等质检事实。 |
| 平台方 / 合约运营 | P0 | 维护资金池、规则、合约、异常和 Agent 自动化秩序。 |
| 物流方 / 货代 / 承运方 | P1 | 提供运输、出港、到港、签收、延误、损坏等物流事实。 |
| 保险公司 / 承保方 | P1 | 提供核保、承保、拒保、理赔和风险缓释判断。 |
| 买方 / 进口商 / 付款方 | P1 | 确认订单、收货、争议和付款责任。 |
| 海关 / 报关 / 通关数据方 | P2 | 提供报关、清关、出口/进口记录和监管事实。 |
| Agent | 贯穿全部角色 | 作为所属角色的业务助理，继承角色权限。 |
| 智能合约 / 规则执行层 | 执行约束层 | 检查条件并输出业务执行结果，不是业务用户。 |

## 4. 钱包身份状态边界

| 钱包状态 | 可见范围 | 可做事项 | 不可做事项 |
|---|---|---|---|
| 未连接钱包 | 无业务可见。 | 只能连接钱包。 | 不能注册、不能进入业务工作区。 |
| 已连接但未注册 | 只能看到注册入口和角色选择。 | 可以选择唯一角色并提交注册信息。 | 不能创建融资案件、不能看案件、不能处理任务。 |
| 已注册已激活 | 进入固定角色工作区。 | 执行该角色允许的业务动作。 | 不能切换角色、不能越权查看或处理无关内容。 |
| 已注册但受限 | 只看到受限原因和允许的补充动作。 | 根据原因补资料或等待平台处理。 | 不能继续高风险动作或绕过限制。 |
| 已暂停 | 只看到暂停原因和处理路径。 | 可以查看原因、提交说明或等待恢复。 | 不能继续创建、审核、确认、执行关键动作。 |

## 5. 角色可见范围

### 5.1 出口商 / 资金需求方

| 可见对象 | 可见条件 | 不可见对象 |
|---|---|---|
| 自己创建的融资案件 | 钱包为该案件资金需求方。 | 其他出口商案件。 |
| 自己上传的贸易材料 | 属于自己的案件。 | 其他角色的内部判断记录。 |
| Agent 对自己案件的摘要和缺口 | 与自己案件相关。 | 资金方内部风险偏好。 |
| 资金方给自己的决策结果 | 该决策已返回给出口商。 | 资金方未公开的内部评审细节。 |
| 质检、物流、保险、买方等与本案有关的事实状态 | 与自己案件相关。 | 无关案件事实。 |
| 合约/规则输出的业务状态 | 与自己案件相关。 | 平台内部规则维护细节。 |

### 5.2 资金方 / 银行 / 保理机构

| 可见对象 | 可见条件 | 不可见对象 |
|---|---|---|
| 分配或匹配给自己的融资案件 | 该资金方有处理权。 | 其他资金方专属案件。 |
| 案件摘要、材料完整性、风险建议 | 用于贷前、贷中、贷后判断。 | 无关出口商全部业务历史。 |
| 出口商、买方、质检、物流、保险的案件相关事实 | 只限该案件需要的信息。 | 与本案无关的事实和隐私。 |
| 合约/规则执行条件和结果 | 用于判断是否可执行。 | 平台内部维护操作细节，除非影响本案。 |
| 贷后回款、逾期、争议状态 | 与自己放款或拟放款案件相关。 | 其他资金方贷后状态。 |

### 5.3 质检方 / 第三方检验机构

| 可见对象 | 可见条件 | 不可见对象 |
|---|---|---|
| 分配给自己的质检任务 | 自己是任务质检方。 | 无关融资案件。 |
| 完成质检所需的贸易信息 | 品类、数量、地点、包装、检查要求。 | 资金方内部决策、融资额度细节。 |
| 自己上传的质检报告、照片、说明 | 自己提交或补充。 | 其他质检机构记录。 |
| 质检补充请求 | 资金方、平台或 Agent 要求补充。 | 资金池、规则、合约内部状态。 |
| dispute 相关说明 | 与自己质检结论相关。 | 无关争议材料。 |

### 5.4 平台方 / 合约运营

| 可见对象 | 可见条件 | 不可见对象 |
|---|---|---|
| 平台运行总览 | 资金池、合约、规则、异常、Agent 状态。 | 不应查看无业务需要的敏感商业细节。 |
| 异常案件 | 案件触发平台级异常或风险。 | 正常案件的逐单业务细节，除非需要排障。 |
| 资金池状态 | 用于维护执行秩序。 | 资金方内部商业决策偏好，除非影响规则。 |
| 规则版本和影响范围 | 用于暂停、恢复、升级。 | 出口商无关私有材料。 |
| Agent 自动化异常 | 自动化失败、堆积、误判风险。 | Agent 所属角色无关私有上下文。 |
| 平台操作记录 | 暂停、恢复、升级、维护。 | 不能删除或篡改关键记录。 |

### 5.5 P1/P2 事实方

| 角色 | 可见对象 | 不可见对象 |
|---|---|---|
| 物流方 | 与自己运输任务相关的物流事实、补充请求。 | 融资审批、资金方内部风险、无关案件。 |
| 保险方 | 待核保案件的风险摘要、理赔材料、保险相关事实。 | 资金方放款决策内部细节、无关案件。 |
| 买方 | 与自己相关的订单、收货、争议、付款责任确认。 | 出口商其他融资申请、资金方内部审批。 |
| 海关/报关方 | 待确认通关事实、报关/清关状态、监管异常。 | 融资审批和资金池状态。 |

## 6. 角色动作边界

### 6.1 出口商 / 资金需求方动作边界

| 可以做 | 说明 |
|---|---|
| 创建融资意图 | 声明一笔贸易需要融资。 |
| 上传贸易材料 | 合同、订单、发票、装箱单、物流、质检、补充说明等。 |
| 查看 Agent 缺口提示 | 知道缺什么、找谁补、下一步。 |
| 提交融资申请 | 在材料达到最低要求后提交。 |
| 补充材料或解释异常 | 对金额不一致、材料缺口、买方信息不足等补充。 |
| 查看案件状态和资金结果 | 看能不能借、能借多少、是否到账、卡在哪里。 |
| 调整自己的 Agent 模式 | 全人工、半自动、全自动之间调整。 |

| 不可以做 | 原因 |
|---|---|
| 修改永久角色 | 单钱包单角色不可变。 |
| 查看其他出口商案件 | 数据隔离。 |
| 替资金方审批 | 资金决策权属于资金方。 |
| 替质检方出具 pass/fail/dispute | 质检事实必须来自质检角色。 |
| 修改平台规则或资金池状态 | 平台/合约运营权限。 |
| 直接绕过规则触发放款 | 合约和规则是执行边界。 |

### 6.2 资金方 / 银行 / 保理机构动作边界

| 可以做 | 说明 |
|---|---|
| 查看待处理融资案件 | 只看分配或匹配给自己的案件。 |
| 查看案件摘要和证据完整性 | 用于贷前判断。 |
| 查看 Agent 风险建议 | 作为辅助，不等于自动审批。 |
| 要求补材料 | 向出口商或相关事实方发出补充要求。 |
| 允许放款 | 在权限范围内做资金决策。 |
| 拒绝融资 | 给出业务原因。 |
| 降低融资金额 | 根据风险调整融资金额或比例。 |
| 转人工复核 | 高风险、争议、置信不足时升级。 |
| 查看贷后状态 | 回款、逾期、争议、保险等。 |

| 不可以做 | 原因 |
|---|---|
| 修改出口商原始材料 | 防止证据污染。 |
| 冒充质检方或物流方确认事实 | 事实来源必须独立。 |
| 修改平台规则 | 规则属于平台/合约运营。 |
| 越权查看其他资金方案件 | 商业隔离。 |
| 绕过规则强制放款 | 防止资金池和系统风险。 |
| 删除关键审批记录 | 审计要求。 |

### 6.3 质检方动作边界

| 可以做 | 说明 |
|---|---|
| 查看待质检任务 | 只看分配给自己的任务。 |
| 查看质检所需信息 | 品类、数量、地点、检查清单等。 |
| 上传质检报告、照片、说明 | 形成货物事实。 |
| 给出 pass/fail/dispute | 统一质检结论。 |
| 补充质检材料 | 响应资金方、平台或 Agent 请求。 |
| 解释质检争议 | 说明结论依据。 |

| 不可以做 | 原因 |
|---|---|
| 决定是否放款 | 放款权属于资金方。 |
| 修改出口商贸易材料 | 防止证据污染。 |
| 修改资金方决策 | 决策权隔离。 |
| 修改平台规则 | 平台权限。 |
| 查看无关融资案件 | 数据隔离。 |
| 伪造物流、海关、保险或买方事实 | 只能负责质检事实。 |

### 6.4 平台方 / 合约运营动作边界

| 可以做 | 说明 |
|---|---|
| 查看运行总览 | 资金池、合约、规则、异常、Agent 状态。 |
| 查看异常案件 | 只围绕异常和系统级风险处理。 |
| 暂停/恢复资金池 | 控制系统级资金执行风险。 |
| 暂停/恢复规则 | 管理规则风险和适用范围。 |
| 暂停/恢复合约执行能力 | 维护系统安全。 |
| 暂停/恢复角色账户 | 身份异常或疑似欺诈时处理。 |
| 分派异常处理 | 把异常交给对应责任角色。 |
| 查看和保留操作记录 | 支持审计。 |

| 不可以做 | 原因 |
|---|---|
| 替出口商伪造贸易材料 | 平台不是事实制造者。 |
| 替质检方出具 pass/fail/dispute | 质检事实必须独立。 |
| 替资金方批准放款 | 资金决策权属于资金方。 |
| 随意修改已提交案件事实 | 防止证据污染。 |
| 绕过规则强制放款 | 破坏系统可信边界。 |
| 删除关键操作记录 | 破坏审计能力。 |

## 7. Agent 权限继承边界

## 7.1 Agent 权限原则

| 原则 | 说明 |
|---|---|
| Agent 不是独立角色 | Agent 是所属用户或机构的业务助理。 |
| Agent 继承所属角色可见范围 | 出口商 Agent 看出口商范围，资金方 Agent 看资金方范围。 |
| Agent 继承所属角色动作边界 | 所属角色不能做的动作，Agent 也不能做。 |
| Agent 可以降低人工负担 | 整理、摘要、缺口、待办、提醒、请求补充。 |
| Agent 不能放大权限 | 不能因为自动化而看更多、做更多。 |
| Agent 必须可升级给人 | 高风险、争议、失败、置信不足时交给人。 |

### 7.2 Agent 可执行动作

| 动作 | 权限前提 |
|---|---|
| 整理材料 | 该材料属于所属角色可见范围。 |
| 生成摘要 | 摘要来源于可见材料和事实。 |
| 检查缺口 | 缺口与所属角色案件或任务相关。 |
| 生成待办 | 待办责任方明确。 |
| 请求补充 | 只能请求相关角色补充对应事实。 |
| 自动推进低风险流程 | 授权、低风险、事实完整、规则明确。 |
| 异常升级 | 争议、高风险、越权、失败或规则冲突。 |

### 7.3 Agent 禁止动作

| 禁止动作 | 原因 |
|---|---|
| 伪造事实 | 事实必须来自责任角色。 |
| 伪造质检结论 | pass/fail/dispute 由质检方确认。 |
| 高风险自动放款 | 资金方必须保留关键决策权。 |
| 自动裁决重大争议 | dispute 必须人工处理或明确责任方处理。 |
| 绕过平台暂停或规则阻断 | 规则和合约是执行边界。 |
| 查看所属角色无权查看的案件 | 防止数据越权。 |
| 删除或篡改关键记录 | 审计要求。 |

## 8. 合约 / 规则执行层权限边界

合约不是用户角色，不进入角色权限体系，但它影响业务动作能否执行。

| 合约可做 | 说明 |
|---|---|
| 检查执行条件 | 授权、事实、规则、资金池、暂停状态。 |
| 输出业务执行结果 | 允许、不允许、等待、阻断、不足、暂停、失败、完成。 |
| 阻断不满足条件的动作 | 防止越过规则或资金池状态。 |
| 记录关键状态变化 | 执行结果和关键状态需要可追踪。 |

| 合约不可以做 | 原因 |
|---|---|
| 判断贸易真实性 | 真实性来自业务事实方。 |
| 替资金方审批 | 决策权属于资金方或 DeFi 规则路径。 |
| 替质检方出结论 | 质检事实来自质检方。 |
| 替平台运营判断所有异常 | 平台负责系统级运营判断。 |
| 解释业务材料 | Agent 和业务角色负责解释。 |

## 9. 业务对象级权限边界

| 业务对象 | 可见/可操作原则 |
|---|---|
| 钱包身份 | 钱包本人和平台在必要情况下可查看状态；角色不可随意修改。 |
| 账户资料 | 用户可维护自己的非角色资料；平台可处理受限/暂停状态。 |
| 融资案件 | 出口商拥有创建和补充权；资金方拥有决策权；平台只在异常时介入。 |
| 贸易材料 | 出口商提交；相关角色在案件范围内查看；不得越权修改。 |
| 质检事实 | 质检方提交和补充；出口商/资金方可查看相关结果。 |
| 物流事实 | 物流方确认；出口商/资金方可查看相关状态。 |
| 保险事实 | 保险方确认；资金方和出口商按案件范围查看。 |
| 买方确认 | 买方确认订单、收货、争议、付款责任。 |
| 资金方决策 | 资金方创建；出口商查看返回结果；平台不能替代。 |
| 平台规则 | 平台维护；其他角色只能查看与自己案件相关的业务影响。 |
| 合约输出 | 相关角色可查看与自己案件相关的执行结果。 |
| Agent 输出 | 所属角色及相关案件可见；摘要不等于事实结论。 |
| 审计记录 | 关键记录不可删除；按角色和平台审计范围查看。 |

## 10. 关键业务场景权限判断

| 场景 | 权限判断 |
|---|---|
| 出口商创建融资案件 | 允许，但只能创建自己的案件。 |
| 出口商查看资金方内部风险模型 | 不允许，只能看返回给自己的业务结论。 |
| 资金方要求出口商补材料 | 允许，如果案件分配给该资金方。 |
| 资金方修改出口商上传的发票 | 不允许，只能要求出口商补充或修正。 |
| 质检方查看资金池状态 | 不允许，质检不需要资金池信息。 |
| 质检方提交 dispute | 允许，如果任务分配给该质检方。 |
| 平台暂停某资金池 | 允许，但必须记录业务原因。 |
| 平台替资金方批准放款 | 不允许。 |
| Agent 自动请求物流方补充运输事实 | 允许，如果案件相关且授权范围内。 |
| Agent 自动通过高风险放款 | 不允许，必须人工升级。 |
| 合约阻断资金池不足的放款 | 允许，这是执行约束。 |
| 合约判断货物是否真实 | 不允许，货物事实来自质检/物流/买方等角色。 |

## 11. 越权阻断业务解释

被阻断时，用户应看到业务解释，而不是只有技术错误。

| 越权类型 | 业务解释示例 |
|---|---|
| 角色不匹配 | 当前钱包注册为出口商，不能执行资金方审核动作。 |
| 无关案件 | 该案件与你的账户或机构无关，不能查看。 |
| 事实来源不匹配 | 该事实需要由质检方确认，你不能替质检方提交结论。 |
| 决策权不匹配 | 放款决策需要资金方完成。 |
| 平台权限不足 | 当前账户没有暂停资金池或规则的运营权限。 |
| Agent 越权 | Agent 只能在你的角色权限内操作，不能执行该动作。 |
| 合约阻断 | 当前资金池、规则或暂停状态不允许继续执行。 |
| 账户受限 | 当前账户存在限制，暂时不能继续该业务动作。 |

## 13. P0 权限范围

### 13.1 P0 必须定义

| 范围 | 说明 |
|---|---|
| 单钱包单角色 | 注册后角色永久绑定。 |
| P0 角色可见范围 | 出口商、资金方、质检方、平台方。 |
| P0 角色动作边界 | 创建、补材料、审核、质检、暂停、恢复等。 |
| Agent 权限继承 | Agent 不能越过所属角色。 |
| 平台运营边界 | 平台控制系统级风险，但不替代业务事实和资金决策。 |
| 合约执行阻断 | 合约可阻断执行，但不判断事实。 |
| 越权业务解释 | 阻断要能被用户理解。 |

### 13.2 P0 可占位，后续细化

| 范围 | 说明 |
|---|---|
| 物流方权限 | P1 完整工作区后续细化。 |
| 保险方权限 | P1 完整核保/理赔后续细化。 |
| 买方权限 | P1 买方确认工作区后续细化。 |
| 海关/报关方权限 | P2 接入后细化。 |
| 多资金方权限竞争 | 后续多资金方匹配设计。 |
| 多机构多操作员权限 | 后续组织权限设计。 |

## 14. 权限设计通过标准

| 检查项 | 通过标准 |
|---|---|
| 单钱包单角色清楚 | 注册后角色不可变，不能随意切换。 |
| 角色可见范围清楚 | 每个角色知道能看什么、不能看什么。 |
| 角色动作边界清楚 | 每个角色知道能做什么、不能做什么。 |
| 事实责任清楚 | 质检、物流、保险、买方、海关等事实来源不混淆。 |
| 决策权清楚 | 资金方决策、平台运营、合约执行不混淆。 |
| Agent 权限继承清楚 | Agent 不成为超级用户。 |
| 越权解释清楚 | 权限阻断能被业务用户理解。 |
| 实现内容后置 | 不出现 token、权限表、鉴权中间件、代码。 |

## 15. 一票否决项

出现以下任一情况，P0 权限设计不得通过：

| 一票否决项 | 原因 |
|---|---|
| 一个钱包可以随意切换多个业务角色 | 破坏身份基础。 |
| 出口商可以查看其他出口商案件 | 破坏数据隔离。 |
| 资金方可以修改出口商原始事实 | 破坏证据可信性。 |
| 质检方可以决定放款 | 破坏决策权隔离。 |
| 平台可以替资金方批准放款 | 破坏资金方责任边界。 |
| Agent 可以执行所属角色无权动作 | Agent 越权。 |
| 合约被设计成事实判断者 | 合约业务角色越界。 |
| 权限设计直接进入 token、session、权限表或代码 | 阶段越界。 |
| P1/P2 权限被强行塞入 P0 必做 | 破坏 MVP 范围。 |

## 16. 本步结论

P0 权限设计边界的本质不是“怎么写鉴权代码”。

它的本质是：

> 在业务层明确 ChainTrace 中每个钱包、角色、Agent、平台和合约执行层的可见范围、动作边界、责任边界和越权阻断解释，确保后续接口、数据、Agent、合约、控制塔和测试设计都不会放大权限或混淆角色。

> `54-p0-business-action-interface-boundary.md`，定义 P0 业务动作和接口边界，但仍不写 endpoint、请求响应、错误码或接口实现。
