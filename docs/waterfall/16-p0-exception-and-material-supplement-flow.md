# 16 P0 异常和补材料流程

## 1. 定位

本文件定义 ChainTrace P0 MVP 中异常和补材料的业务处理方式。

异常和补材料流程的核心目标是：

> 不让用户猜案件为什么卡住，而是明确告诉用户缺什么、谁负责、补完后回到哪里。

本文件只写业务流程，不写 UI、数据库、智能合约实现、钱包签名、hash 或 proof 细节。

## 2. 异常处理原则

| 原则 | 说明 |
|---|---|
| 明确责任角色 | 每个缺口必须指向出口商、质检方、资金方、平台方或 Agent。 |
| 明确业务原因 | 不能只说失败，必须说明缺什么或哪里不一致。 |
| 明确下一步 | 每个异常都要转成可执行待办。 |
| 正常异常自动推进 | 普通缺材料由 Agent 生成待办和提醒。 |
| 高风险异常人工处理 | 争议、高金额、规则阻断、疑似欺诈必须人处理。 |
| 不覆盖历史 | 补充材料形成补充记录，不应抹掉原始提交事实。 |

## 3. P0 异常分类

| 异常类型 | 典型场景 | 默认责任角色 |
|---|---|---|
| 基础材料缺失 | 缺合同、订单、发票、装箱单。 | 出口商 |
| 材料不一致 | 发票金额和订单金额不一致。 | 出口商 / 资金方 |
| 质检待处理 | 需要质检但质检方未提交。 | 质检方 |
| 质检材料不足 | 缺报告、缺照片、缺结论。 | 质检方 |
| 质检 fail | 货物不符合要求。 | 质检方 / 资金方 |
| 质检 dispute | 质检存在争议或无法判断。 | 质检方 / 资金方 / 平台方 |
| 资金方补材料请求 | 资金方认为材料不足。 | 出口商 / 对应事实方 |
| 高风险人工复核 | 高金额、异常国家、疑似欺诈等。 | 资金方 / 平台方 |
| 规则阻断 | 当前业务规则不允许继续。 | 平台方 / 资金方 |
| 资金池不足 | 已批准但资金池余额不足。 | 平台方 / 资金方 |
| 平台暂停 | 平台、规则或资金池暂停。 | 平台方 |
| 执行失败 | 放款执行未完成。 | 平台方 |

## 4. 补材料标准流程

| 步骤 | 业务动作 | 责任角色 | 输出 |
|---|---|---|---|
| 1 | 发现缺口 | Agent / 资金方 / 质检方 / 平台方 | 缺口说明。 |
| 2 | 生成补材料任务 | Agent | 待办、责任角色、截止时间。 |
| 3 | 通知责任角色 | Agent | 角色收到明确补充请求。 |
| 4 | 责任角色补充材料 | 出口商 / 质检方 / 资金方 / 平台方 | 新材料或说明。 |
| 5 | Agent 重新检查 | Agent | 完整 / 仍缺 / 需人工确认。 |
| 6 | 回到原流程 | 系统 / Agent | 继续提交、审查或执行检查。 |

补材料任务必须包含：

| 字段 | 业务含义 |
|---|---|
| 缺口名称 | 缺少什么。 |
| 缺口原因 | 为什么影响融资。 |
| 责任角色 | 谁来补。 |
| 建议材料 | 建议上传或确认什么。 |
| 截止时间 | 什么时候前需要处理。 |
| 回到状态 | 补完后回到哪个流程节点。 |

## 5. 典型异常处理

### 5.1 基础材料缺失

| 项目 | 说明 |
|---|---|
| 触发条件 | 缺合同、订单、发票、装箱单等核心材料。 |
| 责任角色 | 出口商。 |
| Agent 动作 | 生成缺口清单，说明每份材料为什么需要。 |
| 回到状态 | Preparing materials 或 Ready to submit。 |
| 用户解释 | 当前缺少基础贸易材料，补齐后才能提交融资。 |

### 5.2 材料金额不一致

| 项目 | 说明 |
|---|---|
| 触发条件 | 订单金额、发票金额、申请融资金额明显不一致。 |
| 责任角色 | 出口商，必要时资金方人工判断。 |
| Agent 动作 | 标记差异项，要求出口商解释或上传修正材料。 |
| 回到状态 | Need materials 或 Manual review。 |
| 用户解释 | 金额信息存在不一致，需要解释后才能继续审查。 |

### 5.3 质检材料不足

| 项目 | 说明 |
|---|---|
| 触发条件 | 质检报告、照片、结论或异常说明不足。 |
| 责任角色 | 质检方。 |
| Agent 动作 | 说明缺报告、缺照片、缺结论或结论冲突。 |
| 回到状态 | Inspection pending 或 Inspection submitted。 |
| 用户解释 | 质检材料还不够，质检方需要补充。 |

### 5.4 质检 fail

| 项目 | 说明 |
|---|---|
| 触发条件 | 质检方提交 fail。 |
| 责任角色 | 资金方判断是否拒绝，平台方必要时介入。 |
| Agent 动作 | 汇总 fail 原因和影响，不自动裁决。 |
| 回到状态 | Manual review、Rejected 或 Blocked。 |
| 用户解释 | 质检结果不通过，资金方需要判断是否拒绝或要求补充说明。 |

### 5.5 质检 dispute

| 项目 | 说明 |
|---|---|
| 触发条件 | 质检方提交 dispute，或出口商/资金方对质检结论有争议。 |
| 责任角色 | 质检方、资金方，必要时平台方。 |
| Agent 动作 | 汇总争议点、相关材料、建议补充项。 |
| 回到状态 | Manual review、Need materials、Under review 或 Blocked。 |
| 用户解释 | 质检存在争议，需要人工复核或补充材料。 |

### 5.6 资金方要求补材料

| 项目 | 说明 |
|---|---|
| 触发条件 | 资金方审查后认为材料不足或风险解释不充分。 |
| 责任角色 | 出口商或对应事实方。 |
| Agent 动作 | 将资金方要求拆成具体补充任务。 |
| 回到状态 | Under review。 |
| 用户解释 | 资金方需要补充材料，补齐后继续审查。 |

### 5.7 规则阻断

| 项目 | 说明 |
|---|---|
| 触发条件 | 当前规则不允许该案件继续执行。 |
| 责任角色 | 平台方 / 资金方。 |
| Agent 动作 | 说明规则阻断原因，不自动绕过。 |
| 回到状态 | Blocked、Manual review 或 Closed。 |
| 用户解释 | 当前规则不允许继续，需要平台或资金方处理。 |

### 5.8 资金池不足

| 项目 | 说明 |
|---|---|
| 触发条件 | 资金方已允许放款，但资金池余额不足。 |
| 责任角色 | 平台方 / 资金方。 |
| Agent 动作 | 提醒出口商等待，提醒平台方或资金方处理资金池。 |
| 回到状态 | Execution waiting 或 Disbursement pending。 |
| 用户解释 | 资金方已允许，但当前资金池不足，暂时不能执行放款。 |

### 5.9 平台暂停

| 项目 | 说明 |
|---|---|
| 触发条件 | 平台、资金池、规则或案件被暂停。 |
| 责任角色 | 平台方。 |
| Agent 动作 | 说明暂停对象、原因、影响范围。 |
| 回到状态 | Execution waiting、Blocked 或 Closed。 |
| 用户解释 | 平台当前暂停了相关执行能力，需要等待恢复。 |

### 5.10 执行失败

| 项目 | 说明 |
|---|---|
| 触发条件 | 放款执行未成功完成。 |
| 责任角色 | 平台方。 |
| Agent 动作 | 汇总失败原因，提醒平台方处理。 |
| 回到状态 | Execution waiting、Blocked 或 Disbursement pending。 |
| 用户解释 | 放款执行失败，需要平台处理后才能继续。 |

## 6. Agent 自动处理边界

Agent 可以自动处理：

| 场景 | 动作 |
|---|---|
| 明确缺材料 | 生成补材料待办。 |
| 明确责任角色 | 通知对应角色。 |
| 普通状态提醒 | 提醒截止时间和卡点。 |
| 补完后重新检查 | 判断完整、不完整或需要人工。 |
| 低风险流程恢复 | 条件满足后推进回原流程。 |

Agent 不可以自动处理：

| 场景 | 原因 |
|---|---|
| 伪造材料 | 事实必须来自对应角色。 |
| 修改质检结论 | 质检结论属于质检方。 |
| 替资金方批准 | 放款权属于资金方。 |
| 绕过规则阻断 | 规则和平台是执行边界。 |
| 自动裁决重大争议 | 重大争议必须人工处理。 |
| 删除关键历史 | 补充材料不能覆盖原始记录。 |

## 7. 用户可理解表达

所有异常必须避免技术语言。

| 不要说 | 要说 |
|---|---|
| Validation failed | 发票金额和订单金额不一致，需要出口商解释。 |
| Missing document | 缺少质检报告，资金方无法判断货物状态。 |
| Gate blocked | 当前规则不允许继续执行放款。 |
| Execution error | 放款执行失败，平台方需要处理。 |
| Pending dependency | 当前正在等待质检方补充照片。 |

## 8. MVP 范围

### P0 必须做

| 范围 | 说明 |
|---|---|
| 异常分类 | 覆盖材料、质检、资金方、规则、资金池、平台执行。 |
| 补材料任务 | 明确缺口、责任角色、下一步。 |
| Agent 重新检查 | 补充后判断是否回到原流程。 |
| 人工升级 | 高风险、争议、规则阻断进入人工处理。 |
| 用户可理解解释 | 每个异常都能解释给业务用户。 |

### 暂不做

| 范围 | 原因 |
|---|---|
| 数据库补材料表 | 当前不写实现。 |
| 通知系统实现 | 当前只定义业务动作。 |
| 合约异常代码 | 当前只写业务语义。 |
| UI 异常组件 | 当前不做页面设计。 |
| hash / proof 异常 | 当前不进入底层证明细节。 |

## 9. 本步结论

P0 异常和补材料流程的本质不是“报错处理”。

它的本质是：

> 把每个卡点变成清楚的业务原因、责任角色和下一步动作；Agent 负责生成待办和重新检查，人负责授权、争议和高风险判断。

下一步应进入：

> P0 Agent 待办模型。
