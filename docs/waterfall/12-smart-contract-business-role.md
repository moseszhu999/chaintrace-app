# 12 Smart Contract Business Role

## 1. 定位

智能合约在 ChainTrace 中不是页面，不是数据库，也不是 Agent。

智能合约的核心定位是：

> 业务规则和资金执行约束层。

它负责在关键执行动作前检查条件、规则、资金池和暂停状态，并对允许或阻断给出明确业务结果。

智能合约不负责理解所有业务材料，也不负责判断贸易真实性。贸易、质检、物流、通关、保险、买方确认等事实，仍然来自对应业务角色。

## 2. 进入条件

智能合约进入业务流程前，必须已经具备：

| 条件 | 要求 |
|---|---|
| 案件已形成 | 出口商已经创建融资意图并提交案件。 |
| 角色事实已汇总 | 贸易、质检、物流、保险、买方等必要事实已经形成业务状态。 |
| 资金方已有决策 | 资金方已经允许放款，或案件进入需要执行检查的节点。 |
| 规则版本明确 | 当前应适用哪套业务规则必须清楚。 |
| 资金池状态明确 | 可用余额、暂停状态、异常状态必须清楚。 |
| 平台状态明确 | 平台是否暂停、规则是否暂停、执行是否允许。 |

智能合约不处理登录前展示页。
智能合约不处理未注册用户的营销转化页面。
智能合约不替代任何角色工作区。

## 3. 智能合约最想完成的业务结果

| 优先级 | 业务结果 | 说明 |
|---|---|---|
| P0 | 判断是否允许执行 | 放款或关键状态变化前必须有明确结果。 |
| P0 | 阻断不满足条件的执行 | 材料、规则、资金池、暂停状态不满足时不能继续。 |
| P0 | 约束资金池使用 | 资金池不足、暂停或异常时不能执行。 |
| P0 | 约束规则版本 | 当前规则无效、暂停或不匹配时不能执行。 |
| P0 | 记录关键状态变化 | 关键执行结果必须可追溯。 |
| P1 | 给业务角色返回清晰原因 | 不要只说失败，要说明等待条件、规则阻断、资金池不足等。 |

## 4. 智能合约核心业务职责

### 4.1 检查放款条件

智能合约不判断贸易真实性，但可以检查业务状态是否满足执行条件。

| 检查项 | 说明 |
|---|---|
| 资金方是否已允许放款 | 没有资金方允许，不进入放款执行。 |
| 案件状态是否允许执行 | 草稿、补材料、争议等状态不能执行放款。 |
| 必要事实是否已完成 | 例如质检、物流、保险、买方确认等业务条件。 |
| 是否存在未关闭异常 | 重大争议、监管异常、平台阻断等不能忽略。 |
| 是否符合当前规则版本 | 规则要求必须满足。 |
| 是否超过授权金额 | 不能超出资金方允许范围。 |

输出应该是业务语言：

| 输出 | 含义 |
|---|---|
| 允许执行 | 条件满足，可以进入执行。 |
| 等待条件 | 条件未齐，但不是永久阻断。 |
| 规则阻断 | 当前规则不允许执行。 |
| 执行失败 | 执行尝试失败，需要处理原因。 |

### 4.2 检查资金池状态

智能合约是资金执行约束层，必须防止资金池在不安全状态下执行。

| 检查项 | 说明 |
|---|---|
| 资金池是否存在 | 案件必须对应有效资金池。 |
| 资金池余额是否足够 | 不足时不能执行。 |
| 资金池是否暂停 | 暂停时不能执行。 |
| 资金池是否异常 | 冻结、执行失败、维护中等状态不能执行。 |
| 待执行金额是否超过可用额度 | 防止超额占用。 |

业务输出：

| 输出 | 说明 |
|---|---|
| 资金池正常 | 可以继续检查其他条件。 |
| 资金池不足 | 已批准案件等待资金池。 |
| 资金池暂停 | 平台或规则暂停资金池执行。 |
| 资金池异常 | 需要平台方处理。 |

### 4.3 检查规则版本

ChainTrace 的规则必须版本化，智能合约负责执行前约束。

| 检查项 | 说明 |
|---|---|
| 当前规则版本是否有效 | 过期或废止规则不能执行。 |
| 案件是否适用该规则 | 国家、币种、品类、融资类型必须匹配。 |
| 规则是否暂停 | 暂停规则不能继续执行。 |
| 是否存在规则升级影响 | 已进入升级或迁移状态时需要明确处理。 |
| 是否存在规则阻断项 | 高风险国家、品类、金额、争议等可能阻断。 |

业务输出：

| 输出 | 说明 |
|---|---|
| 规则通过 | 当前规则允许继续。 |
| 规则阻断 | 当前规则不允许执行。 |
| 规则暂停 | 平台暂停了相关规则。 |
| 规则不匹配 | 案件不适用当前规则。 |
| 等待规则处理 | 需要平台方处理规则状态。 |

### 4.4 检查暂停状态

暂停状态是系统级风险控制。

| 暂停对象 | 说明 |
|---|---|
| 平台暂停 | 整个平台或核心执行能力暂停。 |
| 资金池暂停 | 某个资金池暂停执行。 |
| 规则暂停 | 某类案件或某条规则暂停。 |
| 案件暂停 | 单个案件因异常被暂停。 |
| 角色账户暂停 | 某个业务角色因风险被暂停。 |

只要相关暂停状态生效，智能合约应阻断执行。

### 4.5 执行或阻断放款流程

智能合约的业务动作不是“审批”，而是“执行约束”。

| 动作 | 说明 |
|---|---|
| 执行放款 | 条件满足时执行资金流转。 |
| 阻断放款 | 条件不满足时拒绝执行。 |
| 返回阻断原因 | 告诉业务系统为什么不能执行。 |
| 记录执行结果 | 执行完成、失败、等待、阻断都要形成记录。 |

智能合约输出不应该是技术错误堆栈，而应该转成业务可理解状态。

## 5. 智能合约不负责什么

智能合约的边界必须清楚。

| 不负责事项 | 原因 |
|---|---|
| 判断贸易真实性 | 贸易真实性来自出口商、买方、物流、通关等事实。 |
| 替资金方审批 | 放款决策权属于资金方。 |
| 替质检方出结论 | 质检事实属于质检方。 |
| 替物流方确认运输 | 物流事实属于物流方。 |
| 替保险公司承保 | 承保结论属于保险公司。 |
| 替买方确认付款责任 | 付款责任必须由买方确认。 |
| 替平台做所有运营判断 | 平台仍需要处理规则、暂停、异常和升级。 |
| 替 Agent 解释业务材料 | Agent 负责整理和解释业务上下文。 |
| 保存完整业务材料 | 业务文件和材料不是智能合约职责。 |
| 做 UI 页面 | 页面设计不是智能合约职责。 |

## 6. 智能合约业务输出

智能合约的输出必须能被其他角色理解。

| 输出 | 业务含义 | 后续动作 |
|---|---|---|
| 允许执行 | 条件满足，可以执行。 | 进入放款或关键状态执行。 |
| 不允许执行 | 条件不满足，不能执行。 | 查看具体原因。 |
| 等待条件 | 还缺某个条件。 | 由 Agent 生成补充待办。 |
| 规则阻断 | 当前规则不允许。 | 平台或资金方处理。 |
| 资金池不足 | 可用资金不够。 | 等待资金池或选择其他资金方。 |
| 平台暂停 | 平台暂停了相关执行能力。 | 等待平台恢复。 |
| 执行失败 | 执行尝试失败。 | 平台方排查。 |
| 执行完成 | 执行成功。 | 通知出口商、资金方和相关角色。 |

### 6.1 输出给不同角色的表达

| 角色 | 看到的业务表达 |
|---|---|
| 出口商 | 钱是否能到账、卡在哪里、下一步找谁。 |
| 资金方 | 已批准案件是否真的可以执行、为什么不能执行。 |
| 平台方 | 资金池、规则、暂停、执行失败等系统级问题。 |
| Agent | 根据阻断原因生成待办和提醒。 |

示例：

```text
输出：资金池不足
出口商理解：资金方已批准，但当前资金池余额不足，暂时不能放款。
资金方理解：案件批准有效，但需要等待资金池补足或切换资金池。
平台方理解：资金池出现执行瓶颈，需要运营处理。
```

## 7. 与 Agent 的关系

智能合约和 Agent 的关系必须分清。

| 对象 | 负责什么 |
|---|---|
| Agent | 整理材料、生成摘要、检查缺口、提醒、请求补充、推进低风险流程。 |
| 智能合约 | 检查执行条件、资金池、规则版本、暂停状态，执行或阻断关键动作。 |

Agent 可以读取智能合约业务输出并解释给用户。

Agent 不可以绕过智能合约结果。

智能合约不负责解释完整业务上下文。

## 8. 与平台方的关系

平台方负责维护规则、资金池、暂停、恢复、异常和升级。

智能合约负责在执行时遵守这些状态。

| 平台方动作 | 智能合约响应 |
|---|---|
| 暂停资金池 | 阻断相关资金池执行。 |
| 恢复资金池 | 允许重新检查执行条件。 |
| 暂停规则 | 阻断适用该规则的执行。 |
| 升级规则版本 | 按当前有效版本检查。 |
| 暂停案件 | 阻断该案件执行。 |
| 标记平台暂停 | 阻断相关执行能力。 |

## 9. MVP 范围

### P0 必须做

| 功能 | 说明 |
|---|---|
| 定义智能合约业务定位 | 规则和资金执行约束层。 |
| 定义放款条件检查 | 检查是否允许执行。 |
| 定义资金池状态检查 | 正常、不足、暂停、异常。 |
| 定义规则版本检查 | 有效、暂停、不匹配、阻断。 |
| 定义暂停状态检查 | 平台、资金池、规则、案件、角色暂停。 |
| 定义业务输出 | 允许执行、不允许、等待条件、规则阻断、资金池不足等。 |
| 定义角色边界 | 不判断贸易真实性，不替资金方审批。 |

### P1 后续做

| 功能 | 说明 |
|---|---|
| 多资金池业务规则 | 不同资金方、币种、规则对应不同资金池。 |
| 规则升级业务流程 | 版本迁移、影响案件、回滚策略。 |
| 更细执行状态 | 多阶段放款、部分执行、失败重试等。 |
| 与保险覆盖联动 | 保险不足时影响执行条件。 |

### 暂不做

| 功能 | 原因 |
|---|---|
| Solidity 实现 | 当前只写业务定位。 |
| 合约 ABI / 接口 | 当前不进入技术设计。 |
| 数据库设计 | 当前仍处于功能需求阶段。 |
| 页面组件设计 | 还没有进入 UI 阶段。 |
| 钱包签名细节 | 当前不进入技术实现。 |

## 10. 本步结论

智能合约不是 ChainTrace 的页面、数据库或 Agent。

它的本质是：

> 业务规则和资金执行约束层；它检查放款条件、资金池状态、规则版本和暂停状态，允许或阻断关键执行，并记录关键状态变化。

> MVP 范围和角色优先级。
