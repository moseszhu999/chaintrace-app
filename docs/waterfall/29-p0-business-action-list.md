# 29 P0 业务动作清单

## 1. 定位

本文件定义 ChainTrace P0 阶段的业务动作清单。

业务动作清单不是 API 列表，不是按钮设计，也不是后端任务拆解。

它的作用是：

> 明确每个角色在 P0 中可以触发哪些业务动作、动作前置条件、业务结果和权限边界。

本文件不写页面、组件、接口、数据库、Solidity、钱包签名、hash 或 proof 细节。

## 2. 动作设计原则

| 原则 | 说明 |
|---|---|
| 动作必须有业务结果 | 每个动作都要改变状态、生成待办、补充事实或形成决策。 |
| 动作必须有责任角色 | 不能出现没有责任人的流程推进。 |
| 动作必须有边界 | 明确能做、不能做、越权后果。 |
| 动作必须能解释 | 用户要知道动作为什么成功、等待、阻断或失败。 |
| Agent 动作从属角色 | Agent 只能在所属角色授权范围内代办或建议。 |
| 合约动作是资金执行动作 | DeFi 智能合约负责检查、执行、阻断和记录关键资金状态。 |

## 3. 出口商 / 资金需求方动作

| 动作 | 前置条件 | 业务结果 | 边界 |
|---|---|---|---|
| 创建融资/结算意图 | 钱包已绑定出口商角色。 | 形成案件 Draft。 | 只能创建自己的案件。 |
| 填写贸易基础信息 | 案件处于 Draft 或 Need materials。 | 形成贸易摘要基础。 | 不能伪造买方、质检、物流事实。 |
| 提交贸易材料 | 案件处于 Draft 或 Need materials。 | 合同、订单、发票、装箱单等进入案件材料。 | 只能提交自己的材料。 |
| 请求 Agent 检查材料 | 已有基础材料。 | Agent 输出缺口、摘要和建议。 | Agent 不替出口商编造材料。 |
| 确认提交案件 | 最低材料满足。 | 案件进入 Submitted 或 Material checking。 | 高风险或缺口未处理时可能被阻断。 |
| 回应补材料待办 | 存在 Need materials 待办。 | 补充材料或说明，回到检查或审查流程。 | 不能覆盖原始记录。 |
| 查看案件卡点 | 案件已创建。 | 看到当前状态、原因、责任方和下一步。 | 只能看自己的案件。 |
| 申请关闭案件 | 案件未完成或需终止。 | 进入关闭申请或 Closed。 | 已进入执行的案件可能需要平台或规则处理。 |

## 4. 资金方 / 银行 / 保理机构动作

资金方动作只适用于银行辅助路径。

| 动作 | 前置条件 | 业务结果 | 边界 |
|---|---|---|---|
| 查看待审案件 | 案件进入银行辅助路径并分配给该资金方。 | 看到待审列表。 | 不能查看无关案件。 |
| 查看 Agent 摘要 | 案件材料已被整理。 | 理解贸易、金额、货物、事实和风险摘要。 | 摘要不能替代原始事实来源。 |
| 请求补材料 | 材料或事实不足。 | 生成补材料待办并回流给责任角色。 | 不能要求无关角色补不相关材料。 |
| 标记人工复核 | 风险、金额、争议或规则异常。 | 案件进入 Manual review。 | 不能绕过平台规则直接执行。 |
| 允许放款 | 银行辅助场景中审查通过。 | 案件进入 Approved 或 Execution pending。 | 后续仍需 DeFi 合约检查执行条件。 |
| 拒绝融资 | 风险或材料不满足要求。 | 案件进入 Rejected 或 Closed。 | 必须给出业务原因。 |
| 查看执行条件结果 | 已允许放款或等待执行。 | 看到允许执行、资金池不足、规则阻断、平台暂停等结果。 | 不能修改合约执行结果。 |

## 5. 质检方动作

| 动作 | 前置条件 | 业务结果 | 边界 |
|---|---|---|---|
| 查看待质检任务 | 案件需要质检且任务分配给该质检方。 | 看到货物摘要和质检要求。 | 不能查看无关融资详情。 |
| 接受质检任务 | 任务可接受。 | 任务进入处理中。 | 不能替其他质检方接受。 |
| 请求补充质检信息 | 货物或现场信息不足。 | 生成补充待办。 | 不能要求修改贸易事实。 |
| 上传质检材料 | 质检任务处理中。 | 报告、照片、现场说明进入事实材料。 | 不能伪造或替出口商提交贸易材料。 |
| 提交 pass | 质检通过。 | 案件获得通过事实。 | 不能决定放款。 |
| 提交 fail | 质检不通过。 | 案件进入人工复核、拒绝或阻断判断。 | 必须说明原因。 |
| 提交 dispute | 事实存在争议。 | 案件进入争议或人工复核。 | 不能自行裁决融资结果。 |

## 6. 平台方 / 合约运营动作

| 动作 | 前置条件 | 业务结果 | 边界 |
|---|---|---|---|
| 查看运行总览 | 平台角色已注册。 | 看到资金池、规则、暂停、异常、Agent 堆积。 | 不等于查看所有业务细节。 |
| 维护业务规则状态 | 规则需要启用、暂停、恢复或版本调整。 | 规则状态变化。 | 不直接改业务事实。 |
| 维护资金池状态 | 资金池不足、暂停、异常或恢复。 | 资金池状态变化。 | 不能伪造资金可用。 |
| 暂停案件或路径 | 存在风险、异常或规则要求。 | 案件或路径进入 Blocked / Paused。 | 必须说明原因和影响范围。 |
| 恢复案件或路径 | 暂停原因解除。 | 案件回到可推进状态。 | 必须保留恢复原因。 |
| 处理执行失败 | 合约或资金执行返回失败。 | 标记失败原因和下一步。 | 不能删除失败记录。 |
| 处理 Agent 异常 | Agent 堆积、失败或越权风险。 | 升级人工或调整模式。 | 不替业务角色完成事实确认。 |

## 7. DeFi 智能合约业务动作

本节只定义业务动作，不写智能合约实现。

| 动作 | 前置条件 | 业务结果 | 边界 |
|---|---|---|---|
| 检查规则版本 | 案件进入执行检查。 | 返回规则有效、暂停、过期或不匹配。 | 不解释完整业务材料。 |
| 检查资金池状态 | 案件需要资金动作。 | 返回资金池可用、不足、暂停或异常。 | 不替平台补充资金。 |
| 检查授权边界 | 银行辅助或用户授权存在。 | 返回授权满足或不满足。 | 不绕过授权。 |
| 检查抵押/保证条件 | 纯 DeFi 或低人工场景需要。 | 返回满足、等待或不足。 | 不伪造抵押状态。 |
| 允许执行 | 条件满足。 | 进入资金执行。 | 必须基于规则和状态。 |
| 阻断执行 | 条件不满足。 | 返回规则阻断、资金池不足、平台暂停等原因。 | 不能静默失败。 |
| 执行资金动作 | 条件满足且未暂停。 | 放款、结算、锁定、释放或到账。 | 不判断贸易真实性。 |
| 记录关键状态变化 | 发生执行、阻断、失败或完成。 | 形成关键业务状态记录。 | 不保存完整业务材料。 |

## 8. Agent 动作

| 动作 | 前置条件 | 业务结果 | 边界 |
|---|---|---|---|
| 整理材料摘要 | 所属角色有可见材料。 | 形成角色可读摘要。 | 不改变原始材料。 |
| 检查缺口 | 案件存在最低材料要求。 | 输出缺口、责任角色、建议动作。 | 不自行补假材料。 |
| 生成待办 | 存在缺口、风险、确认或授权节点。 | 形成明确待办。 | 待办必须有责任角色。 |
| 提醒用户 | 状态变化、超时、风险或授权节点。 | 用户知道下一步。 | 不替用户做高风险判断。 |
| 请求相关角色补材料 | 缺口明确。 | 相关角色收到补充任务。 | 不能越权请求无关材料。 |
| 自动推进低风险流程 | 授权范围内、规则明确、事实完整。 | 案件进入下一状态或执行检查。 | 异常必须升级人。 |
| 生成风险摘要 | 材料、事实和规则已有。 | 给资金方、平台或业务方可读建议。 | 不替资金方做高风险放款决策。 |
| 升级异常 | 争议、高风险、规则例外、置信不足。 | 案件进入 Manual review 或平台处理。 | 不能压制异常。 |

## 9. P1/P2 事实方动作

| 角色 | 动作 | 业务结果 | 边界 |
|---|---|---|---|
| 物流方 | 确认揽收、装柜、出港、到港、签收、异常。 | 物流事实进入案件。 | 不决定融资。 |
| 物流方 | 上传提单、运单、签收记录、照片、异常说明。 | 物流材料完整性提高。 | 不修改贸易或质检事实。 |
| 海关 / 报关方 | 确认报关、清关、出口、进口、监管异常。 | 通关事实进入案件。 | 不替资金方审批。 |
| 保险公司 | 给出承保、拒保、加费、补材料、人工核保。 | 风险缓释状态明确。 | 不替资金方放款。 |
| 保险公司 | 处理理赔材料。 | 出险处理进入理赔路径。 | 不修改贸易或质检事实。 |
| 买方 / 进口商 | 确认订单、收货、争议、付款责任。 | 付款责任事实进入案件。 | 不修改出口商融资申请。 |

## 10. 动作结果类型

所有业务动作最终必须落到以下结果之一：

| 结果 | 含义 |
|---|---|
| 状态推进 | 案件进入下一业务状态。 |
| 生成待办 | 某个角色需要补充、确认、授权或处理。 |
| 形成事实 | 质检、物流、通关、保险或买方事实被确认。 |
| 形成决策 | 允许、拒绝、补材料、人工复核或暂停。 |
| 进入执行检查 | 合约检查规则、资金池、授权和条件。 |
| 执行完成 | 资金到账、结算完成、锁定或释放完成。 |
| 阻断或等待 | 因规则、资金池、暂停、事实缺口或争议无法继续。 |
| 进入人工 | 争议、高风险、监管、法律责任或异常需要人处理。 |

## 11. MVP 范围

### 11.1 P0 必须定义

| 范围 | 说明 |
|---|---|
| 角色动作 | 出口商、资金方、质检方、平台方、Agent、合约动作清楚。 |
| 路径动作 | 银行辅助、纯 DeFi、低人工路径动作差异清楚。 |
| 动作前置条件 | 什么情况下可以触发动作。 |
| 动作结果 | 动作触发后进入什么状态或产生什么待办。 |
| 动作边界 | 哪些动作明确越权。 |

### 11.2 暂不做

| 范围 | 原因 |
|---|---|
| 按钮名称和页面交互 | 当前不做 UI。 |
| API endpoint | 当前不写接口。 |
| 数据表字段 | 当前不做数据库。 |
| 合约函数签名 | 当前不写 Solidity。 |

## 12. 本步结论

P0 业务动作清单明确：

> ChainTrace 的每个业务动作都必须有责任角色、前置条件、业务结果和边界。Agent 可以整理、提醒、生成待办和推进低风险流程；DeFi 智能合约负责资金执行条件检查、执行和阻断；人负责授权、异常、争议、高风险和重大法律判断。
