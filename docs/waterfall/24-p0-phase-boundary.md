# 24 P0 阶段边界说明

## 1. 定位

本文件定义 ChainTrace P0 MVP 当前阶段的边界。

阶段边界说明不是开发计划，不是技术路线图，也不是 UI 设计说明。

它的核心作用是：

> 明确当前瀑布需求阶段已经完成什么、仍然不做什么、下一阶段允许开始讨论什么。

本文件不写页面、组件、数据库、智能合约实现、钱包签名、hash 或 proof 细节。

## 2. 当前阶段已经完成的业务范围

| 范围 | 状态 | 说明 |
|---|---|---|
| 产品定位 | 已定义 | Agent Native + Crypto Native 供应链金融前站系统。 |
| 注册和角色绑定 | 已定义 | 钱包注册后永久绑定单一业务身份。 |
| P0 角色需求 | 已定义 | 出口商、资金方、质检方、平台方。 |
| P1/P2 角色边界 | 已定义 | 物流、海关/报关、保险、买方。 |
| Agent Native 模型 | 已定义 | 三种模式、能做什么、不能做什么、70/30 目标。 |
| 智能合约业务角色 | 已定义 | 业务规则和资金执行约束层。 |
| MVP 角色优先级 | 已定义 | P0/P1/P2 分层。 |
| P0 用户旅程 | 已定义 | 从融资意图到到账或卡点。 |
| P0 状态机 | 已定义 | 主状态、解释、流转和回路。 |
| P0 异常和补材料 | 已定义 | 缺口、责任角色、回流、人工升级。 |
| P0 Agent 待办模型 | 已定义 | 待办结构、优先级、模式差异、回流。 |
| P0 信息架构 | 已定义 | 信息域、角色视角、卡点、可见性。 |
| P0 工作区业务分区 | 已定义 | 每个角色的业务分区和边界。 |
| P0 验收口径 | 已定义 | 按角色、Agent、状态、异常、边界验收。 |
| P0 示例案件脚本 | 已定义 | 用虚拟案件串起主流程和异常分支。 |

## 3. 当前阶段仍然明确不做

| 不做 | 原因 |
|---|---|
| 登录前展示页 | 当前不是营销和转化阶段。 |
| UI 页面设计 | 当前只做业务需求。 |
| UI 组件设计 | 当前不进入前端实现。 |
| 页面布局和导航细节 | 当前只定义业务分区。 |
| 数据库设计 | 当前不定义表、字段、关系。 |
| API 设计 | 当前不定义接口和请求响应。 |
| 智能合约实现 | 当前只定义业务角色和输出。 |
| Solidity / ABI / 部署 | 当前不进入合约技术实现。 |
| 钱包签名流程细节 | 当前不进入实现。 |
| hash / proof / proof pack | 当前不跳到底层证明产品。 |
| 旧版代码继承 | 当前分支是全新瀑布重建设计分支。 |

## 4. 下一阶段可以进入的范围

下一阶段可以进入：

> P0 业务蓝图到产品需求说明的过渡阶段。

这仍然不等于写代码。

可以开始写：

| 文档 | 目的 |
|---|---|
| P0 PRD 总纲 | 汇总 P0 业务目标、角色、流程、状态、验收。 |
| P0 角色工作区内容清单 | 更具体地列出每个工作区应承载的业务内容。 |
| P0 业务动作清单 | 列出每个角色可以触发的业务动作和结果。 |
| P0 权限矩阵 | 用表格定义角色可见、可操作、不可操作范围。 |
| P0 业务事件清单 | 用业务语言列出关键状态变化，不写技术事件实现。 |
| P0 非功能业务要求 | 审计、可解释、权限隔离、人工确认、异常升级等。 |

## 5. 下一阶段仍然不能进入的范围

即使进入 PRD 过渡阶段，仍然不能直接进入：

| 禁止提前进入 | 原因 |
|---|---|
| 写 app 代码 | 尚未进入实现阶段。 |
| 写 components | 尚未进入 UI 组件阶段。 |
| 写 lib | 尚未进入应用逻辑实现。 |
| 写 contracts | 尚未进入智能合约实现。 |
| 写 scripts | 尚未进入脚本和部署阶段。 |
| 写 database schema | 尚未进入数据设计阶段。 |
| 写 hash/proof/wallet 细节 | 当前产品定位先于底层证明细节。 |

## 6. 进入 UI 阶段前必须补齐

如果未来要进入 UI 阶段，至少还需要先补齐：

| 需要补齐 | 说明 |
|---|---|
| PRD 总纲 | 把瀑布需求汇总成产品需求。 |
| 角色工作区内容清单 | 明确每个工作区业务内容。 |
| 业务动作清单 | 明确每个动作的触发、结果和边界。 |
| 权限矩阵 | 明确谁能看、能做、不能做。 |
| 状态和异常口径 | 确保 UI 表达不会偏离业务状态。 |
| Agent 文案口径 | 确保 Agent 用业务语言解释，而不是技术语言。 |

## 7. 进入技术设计前必须补齐

如果未来要进入技术设计，至少还需要先补齐：

| 需要补齐 | 说明 |
|---|---|
| PRD 已冻结 | 角色、流程、状态、边界基本稳定。 |
| 数据需求说明 | 只在业务数据需求清楚后再设计数据库。 |
| API 业务契约 | 先定义业务契约，再定义技术接口。 |
| 合约业务规则清单 | 先定义哪些规则需要执行约束，再写合约。 |
| 审计和权限要求 | 先定义安全和可追溯要求。 |
| 异常和回滚口径 | 先定义业务处理，再定义技术回滚。 |

## 8. 阶段完成判断

当前瀑布需求阶段可以认为完成了 P0 业务基础，如果满足：

| 判断项 | 要求 |
|---|---|
| 角色齐全 | P0/P1/P2 角色定位和边界已写清楚。 |
| 闭环清楚 | P0 融资闭环从创建到到账或卡点已写清楚。 |
| Agent 清楚 | Agent 模式、能力、边界、待办已写清楚。 |
| 状态清楚 | 主状态、异常、补材料、阻断、执行等待已写清楚。 |
| 验收清楚 | 每个角色的业务验收口径已写清楚。 |
| 排除清楚 | 明确不做 UI、DB、合约实现、hash/proof、旧代码继承。 |

## 9. 本步结论

P0 阶段边界说明的本质不是“停止推进”。

它的本质是：

> 防止项目在业务需求尚未沉淀完整前跳到 UI、数据库、合约或底层证明细节；下一阶段可以进入 PRD 总纲、权限矩阵、业务动作清单和业务事件清单。

下一步应进入：

> P0 PRD 总纲和权限矩阵。
