# 32 P0 PRD 冻结检查清单

## 1. 定位

本文件定义 ChainTrace P0 PRD 冻结前的检查清单。

PRD 冻结检查清单不是开发计划，不是 UI 原型说明，不是数据库设计，也不是智能合约实现清单。

它的作用是：

> 在进入产品原型、业务契约或技术设计前，确认 P0 产品需求是否已经足够稳定、边界是否清楚、仍缺什么、冻结后什么不能随意改。

本文件不写页面、组件、API、数据库、Solidity、钱包签名、hash 或 proof 细节。

## 2. 冻结原则

| 原则 | 说明 |
|---|---|
| 冻结业务口径 | 冻结角色、场景、路径、状态、动作、权限、事件和验收口径。 |
| 不冻结技术实现 | 当前不冻结数据库、接口、合约代码或 UI 组件。 |
| 不新增大范围场景 | P0 只保留已定义的核心闭环和多路径模型。 |
| 不破坏角色边界 | 单钱包单角色、事实来源独立、Agent 权限从属角色。 |
| 不回退产品定位 | ChainTrace 是 Agent Native + Crypto Native 的供应链金融系统。 |
| 不把银行设为所有场景必经 | 银行只在银行辅助路径中参与。 |
| 不把合约降级为展示工具 | DeFi 智能合约是资金执行约束层。 |

## 3. 产品定位冻结检查

| 检查项 | 冻结要求 | 状态 |
|---|---|---|
| ChainTrace 定位 | Agent Native + Crypto Native 供应链金融系统。 | 可冻结 |
| 不是供应链金融前站 | 不再使用“前站系统”作为核心定位。 | 可冻结 |
| 不是单纯银行审批系统 | 银行是部分场景参与方，不是所有场景必经角色。 | 可冻结 |
| DeFi 智能合约定位 | 资金执行核心层和业务规则约束层。 | 可冻结 |
| Agent 定位 | 每个角色身边的业务助理，不是单独用户。 | 可冻结 |
| 旧代码继承 | 当前分支不继承旧 app/components/lib/contracts/scripts/demo 页面和代码。 | 可冻结 |

## 4. 角色范围冻结检查

| 角色 / 能力 | 冻结要求 | 状态 |
|---|---|---|
| 出口商 / 资金需求方 | 创建融资/结算意图、提交材料、查看结果和卡点。 | 可冻结 |
| 资金方 / 银行 / 保理机构 | 只在需要银行参与的场景中审查和授权。 | 可冻结 |
| 质检方 | 提供独立货物事实，不决定融资。 | 可冻结 |
| 平台方 / 合约运营 | 维护规则、资金池、暂停、异常和执行秩序。 | 可冻结 |
| DeFi 智能合约 | 检查条件、执行或阻断资金动作。 | 可冻结 |
| Agent | 整理、检查、待办、提醒、低风险推进、异常升级。 | 可冻结 |
| 物流方 | P1 事实增强角色，提供物流事实。 | 可冻结 |
| 保险公司 | P1 风险缓释角色，提供承保和理赔事实。 | 可冻结 |
| 买方 / 进口商 | P1 付款责任确认方。 | 可冻结 |
| 海关 / 报关方 | P2 通关和监管事实来源。 | 可冻结 |

## 5. 场景路径冻结检查

| 路径 | 冻结要求 | 状态 |
|---|---|---|
| 银行辅助路径 | 需要银行/资金方审查和授权的融资场景。 | 可冻结 |
| 纯 DeFi 路径 | 规则、资金池、抵押/保证、事实状态满足后由合约执行。 | 可冻结 |
| Agent + 智能合约低人工路径 | 规则清楚、授权边界明确、事实结构化的低人工结算、保证金、付款责任或法律/运营场景。 | 可冻结 |
| 人工升级路径 | 争议、高风险、事实冲突、规则例外、监管或重大法律责任进入人工。 | 可冻结 |

冻结后不得改成：

| 禁止改动 | 原因 |
|---|---|
| 所有场景都必须走银行审批 | 破坏多路径模型。 |
| 所有场景都自动走纯 DeFi | 忽略银行辅助和人工边界。 |
| Agent 自动裁决重大法律争议 | 违反 Agent 边界。 |
| 合约自行判断贸易真实性 | 违反事实来源边界。 |

## 6. 状态模型冻结检查

| 状态 | 冻结要求 |
|---|---|
| Draft | 案件草稿。 |
| Submitted | 案件已提交。 |
| Material checking | Agent 检查材料。 |
| Need materials | 需要补材料或补说明。 |
| Fact confirming | 事实方确认事实。 |
| Under review | 银行辅助场景中资金方审查。 |
| DeFi checking | 纯 DeFi 或低人工场景中合约检查执行条件。 |
| Manual review | 人工处理异常、争议、高风险或授权。 |
| Approved | 已允许进入执行检查。 |
| Rejected | 已拒绝继续。 |
| Execution pending | 等待执行条件或资金动作。 |
| Blocked | 被规则、资金池、平台、事实冲突或风险阻断。 |
| Funds arrived | 资金到账或结算完成。 |
| Closed | 案件关闭。 |

状态冻结要求：

| 要求 | 说明 |
|---|---|
| 状态必须可解释 | 每个状态有原因、责任角色和下一步。 |
| 阻断不是黑盒 | Blocked 必须说明阻断原因和解除条件。 |
| 补材料可回流 | Need materials 后必须能回到检查、事实确认、审查或执行检查。 |
| 执行失败可处理 | Execution failed 类结果必须进入平台处理或人工处理。 |

## 7. 权限和边界冻结检查

| 检查项 | 冻结要求 | 状态 |
|---|---|---|
| 钱包角色 | 一个钱包永久绑定一个业务身份。 | 可冻结 |
| 角色可见性 | 只能看相关案件、任务、事实和结果。 | 可冻结 |
| 事实来源 | 贸易、质检、物流、通关、保险、买方事实各自独立。 | 可冻结 |
| 资金方权限 | 只在银行辅助路径中做放款决策。 | 可冻结 |
| 平台权限 | 管规则、资金池、暂停、异常和执行秩序，不伪造事实。 | 可冻结 |
| Agent 权限 | 从属于所属角色和授权模式。 | 可冻结 |
| 合约权限 | 执行或阻断资金动作，不解释完整业务材料。 | 可冻结 |

必须保持阻断：

| 越权行为 | 冻结要求 |
|---|---|
| 出口商修改质检结论 | 必须阻断。 |
| 资金方修改事实方结论 | 必须阻断。 |
| 平台替资金方审批 | 必须阻断。 |
| Agent 伪造事实 | 必须阻断。 |
| 合约绕过银行辅助场景授权 | 必须阻断。 |
| 无关角色查看无关案件 | 必须阻断。 |

## 8. Agent Native 冻结检查

| 检查项 | 冻结要求 |
|---|---|
| Agent 身份 | Agent 不是单独用户，而是每个角色身边的业务助理。 |
| 三种模式 | 全人工确认、半自动托管、全自动 Agent。 |
| 默认建议 | P0 默认推荐半自动托管。 |
| Agent 可以做 | 整理、摘要、缺口、待办、提醒、请求补材料、低风险推进、异常升级。 |
| Agent 不可以做 | 伪造事实、裁决重大争议、替资金方高风险放款、绕过平台规则、删除关键记录。 |
| 自动化目标 | 70% 左右正常案件由 Agent + 规则推进，不超过 30% 人工处理。 |

## 9. DeFi 智能合约业务角色冻结检查

| 检查项 | 冻结要求 |
|---|---|
| 合约不是页面 | 不承担 UI 职责。 |
| 合约不是数据库 | 不保存完整业务材料。 |
| 合约不是 Agent | 不整理和解释业务材料。 |
| 合约是资金执行约束层 | 检查规则、资金池、授权、抵押/保证、暂停和执行条件。 |
| 银行辅助路径 | 银行/资金方授权是执行条件之一。 |
| 纯 DeFi 路径 | 规则和资金池满足时可以无银行执行。 |
| 低人工路径 | 可锁定、释放、结算或阻断资金。 |
| 业务输出 | 允许执行、不允许执行、等待条件、规则阻断、资金池不足、平台暂停、执行失败、执行完成。 |

## 10. 业务动作和事件冻结检查

| 范围 | 冻结要求 |
|---|---|
| 业务动作 | 每个动作有责任角色、前置条件、业务结果和边界。 |
| 业务事件 | 每个事件说明发生了什么、谁触发、影响什么、为什么、下一步是什么。 |
| 动作结果 | 状态推进、生成待办、形成事实、形成决策、执行检查、执行完成、阻断等待、进入人工。 |
| 事件分类 | 生命周期、材料、事实、Agent、银行辅助、DeFi 执行、平台运营、异常争议。 |
| 历史处理 | 补充和修正形成新事件，不覆盖历史。 |

## 11. 非功能业务要求冻结检查

| 范围 | 冻结要求 |
|---|---|
| 可解释性 | 状态、决策、卡点、下一步必须清楚。 |
| 审计 | 关键动作、事实、授权、阻断、执行结果必须可追踪。 |
| 权限隔离 | 单钱包单角色、案件相关可见、事实来源独立。 |
| 人工确认 | 授权、异常、争议、高风险、监管、重大法律责任交给人。 |
| 异常升级 | Agent、规则或合约不得静默吞掉异常。 |
| Agent 质量 | 摘要可追溯、缺口可操作、建议可解释、自动推进可控。 |
| DeFi 执行可信 | 执行前检查、阻断有原因、失败可处理。 |
| 业务连续性 | Agent、资金池、规则、事实方、资金方异常时必须可降级或等待。 |

## 12. 冻结前仍需确认的问题

以下问题不一定阻塞 PRD 冻结，但进入产品原型或技术业务契约前必须确认：

| 问题 | 是否阻塞 PRD 冻结 | 建议处理 |
|---|---:|---|
| P0 是否只做一个主样例案件，还是保留多个路径样例 | 否 | 产品原型前确认。 |
| P1 物流、保险、买方是否在首版原型中展示入口 | 否 | 原型阶段决定显示深度。 |
| P2 海关/报关是否只保留占位说明 | 否 | 后续接入阶段确认。 |
| 纯 DeFi 小额融资的业务额度边界 | 否 | 业务规则清单阶段定义。 |
| 低人工法律/运营场景的第一批样例 | 否 | 场景规则清单阶段定义。 |
| 平台暂停影响范围的分级 | 否 | 业务规则清单阶段定义。 |
| Agent 全自动授权上限 | 否 | Agent 策略规则阶段定义。 |

## 13. 冻结后仍然明确不做

| 不做 | 原因 |
|---|---|
| 登录前展示页 | 当前不是营销阶段。 |
| UI 组件设计 | PRD 冻结不等于组件设计。 |
| 页面布局和交互细节 | 后续产品原型阶段再处理。 |
| API 设计 | 业务契约明确后再定义技术接口。 |
| 数据库表结构 | 数据需求明确后再设计。 |
| Solidity / ABI / 部署 | 合约业务规则清单明确后再实现。 |
| 钱包签名流程细节 | 技术设计阶段再处理。 |
| hash / proof / proof pack | 当前不跳到底层证明产品。 |
| 继承旧版页面和代码 | 当前分支是全新瀑布重建设计分支。 |

## 14. 冻结后允许进入的下一阶段

PRD 冻结后，可以进入以下阶段之一：

| 下一阶段 | 目的 |
|---|---|
| P0 产品原型文字说明 | 把角色工作区、状态、动作、卡点转成原型说明。 |
| P0 业务规则清单 | 定义哪些规则影响路径选择、执行、阻断和升级。 |
| P0 数据需求说明 | 只从业务角度说明需要保存哪些业务信息。 |
| P0 API 业务契约 | 用业务语言定义接口前的契约，不写技术实现。 |
| P0 DeFi 合约业务规则清单 | 定义合约需要约束哪些资金执行规则，不写 Solidity。 |

## 15. 冻结判断

P0 PRD 可以冻结的条件：

| 条件 | 要求 |
|---|---|
| 产品定位稳定 | Agent Native + Crypto Native 供应链金融系统。 |
| 角色稳定 | P0/P1/P2 角色和边界清楚。 |
| 路径稳定 | 银行辅助、纯 DeFi、低人工、人工升级路径清楚。 |
| 状态稳定 | 主状态和异常回流清楚。 |
| 权限稳定 | 可见、可操作、不可操作和越权阻断清楚。 |
| Agent 稳定 | 模式、能力、边界、自动化目标清楚。 |
| 合约业务角色稳定 | 资金执行约束层定位清楚。 |
| 动作和事件稳定 | 谁能做什么、发生什么、影响什么清楚。 |
| 非功能业务要求稳定 | 可解释、审计、权限隔离、人工边界、异常升级清楚。 |
| 排除范围稳定 | 不提前进入 UI、DB、API、Solidity、hash/proof、旧代码继承。 |

## 16. 本步结论

P0 PRD 冻结检查清单的本质不是“停止需求演进”。

它的本质是：

> 把 ChainTrace P0 的业务边界固定下来，让后续产品原型、业务规则、数据需求、API 业务契约和 DeFi 合约业务规则清单都围绕同一个稳定业务模型展开。

下一步建议进入：

> P0 产品原型文字说明或 P0 业务规则清单。
