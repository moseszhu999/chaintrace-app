# 59 P0 业务验收设计

## 1. 定位

本文件定义 ChainTrace P0 技术设计阶段的业务验收设计。

它不是测试框架，不是测试代码，不是 API 自动化脚本，不是链上测试脚本，不是单元测试清单，也不是 CI/CD 计划。

它的作用是：

> 用业务语言定义 P0 什么时候算“业务上通过”：出口商是否完成融资闭环、资金方是否能决策、质检方是否能提供事实、平台是否能运营异常、Agent 是否能辅助推进、合约/规则是否能输出可理解结果，以及权限、路径和异常是否被正确保留。

本文件只定义业务验收口径，不进入测试实现。

## 2. 验收原则

| 原则 | 说明 |
|---|---|
| 验收业务结果 | 不验收页面是否好看，而验收业务结果是否达成。 |
| 验收角色边界 | 每个角色只能完成自己的业务职责。 |
| 验收路径差异 | 银行辅助、纯 DeFi、低人工、人工升级都要能成立。 |
| 验收 Agent 边界 | Agent 能辅助，但不能越权。 |
| 验收合约边界 | 合约能约束执行，但不判断贸易真实性。 |
| 验收异常可处理 | 卡点必须有原因、责任方和下一步。 |
| 验收权限隔离 | 无关案件不可见，越权动作必须阻断。 |
| 验收阶段边界 | 不以测试代码、接口实现或 UI 实现作为当前完成标准。 |

## 3. P0 总体验收目标

P0 业务验收通过，意味着 ChainTrace 的核心业务闭环已经在需求和设计层可验证：

```text
出口商创建融资意图
→ Agent 整理贸易材料并检查缺口
→ 质检方提供货物事实
→ 资金方或 DeFi 规则路径做融资判断
→ 合约/规则检查执行条件
→ 平台处理系统级异常
→ 出口商看到资金结果或明确卡点
```

P0 不要求完整实现所有 P1/P2 角色能力，也不要求 hash/proof 产品页进入主流程。

## 4. 验收场景总览

| 场景 | 验收目的 |
|---|---|
| 正常融资闭环 | 验证一笔案件能从融资意图走到资金结果。 |
| 银行辅助路径 | 验证资金方能审查、决策，并进入执行条件检查。 |
| 纯 DeFi 路径 | 验证无银行审批时，规则和资金池路径仍可表达。 |
| 低人工路径 | 验证 Agent + 规则能推进正常案件。 |
| 人工升级路径 | 验证高风险、争议、失败会交给人。 |
| 角色权限隔离 | 验证单钱包单角色和案件相关可见。 |
| 事实来源边界 | 验证质检、物流、买方、保险等事实不能被伪造。 |
| 合约业务输出 | 验证合约输出可被业务角色理解。 |
| 平台运营异常 | 验证平台能处理资金池、规则、暂停、执行失败。 |
| 后置范围控制 | 验证 P1/P2 和实现细节不进入 P0 必做。 |

## 5. 正常融资闭环验收

### 5.1 验收目标

验证出口商能够围绕一笔真实贸易，从融资意图走到可理解的资金结果。

### 5.2 业务前提

| 前提 | 要求 |
|---|---|
| 出口商账户 | 钱包已注册并绑定为出口商。 |
| 案件材料 | 至少存在合同、订单、发票等基础贸易材料。 |
| 质检事实 | P0 至少有质检任务和 pass/fail/dispute 结论路径。 |
| 资金方或规则路径 | 银行辅助或 DeFi 规则路径至少一个可用。 |
| Agent 模式 | 至少支持全人工和半自动业务表达。 |

### 5.3 验收步骤

| 步骤 | 业务动作 | 通过标准 |
|---:|---|---|
| 1 | 出口商创建融资意图。 | 案件草稿形成，能表达融资类型、金额、币种和期望到账。 |
| 2 | 出口商上传贸易材料。 | 材料进入案件，能表达完整/缺口/冲突。 |
| 3 | Agent 生成摘要和缺口。 | 出口商知道材料是否足够和下一步。 |
| 4 | 质检方提供质检事实。 | 质检状态能表达 pass/fail/dispute。 |
| 5 | 资金方或规则路径进入判断。 | 能表达审查中、补材料、允许、拒绝或规则路径。 |
| 6 | 合约/规则检查执行条件。 | 能表达 allowed、waiting、rule blocked、pool insufficient、platform paused。 |
| 7 | 出口商查看资金结果。 | 出口商知道能不能借、能借多少、是否到账、卡在哪里。 |

### 5.4 不通过表现

| 不通过 | 原因 |
|---|---|
| 出口商只看到文件上传完成，看不到融资结果。 | 业务闭环缺失。 |
| Agent 只输出技术错误，不告诉缺什么。 | Agent Native 价值缺失。 |
| 资金方批准后直接等同到账。 | 忽略合约/规则/资金池执行条件。 |
| 质检状态无法表达 dispute。 | 异常路径缺失。 |
| 卡点没有责任方和下一步。 | 业务不可处理。 |

## 6. 银行辅助路径验收

### 6.1 验收目标

验证资金方作为银行/保理机构，能围绕案件摘要、证据完整性、风险和执行条件做出业务决策。

### 6.2 验收步骤

| 步骤 | 业务动作 | 通过标准 |
|---:|---|---|
| 1 | 资金方查看待审案件。 | 只能看到分配或匹配给自己的案件。 |
| 2 | 资金方查看案件摘要。 | 能理解出口商、买方、金额、商品、融资类型。 |
| 3 | 资金方查看证据完整性。 | 能看到合同、订单、发票、质检、物流等是否完整。 |
| 4 | Agent 标记风险和缺口。 | 风险建议可读，但不等于最终审批。 |
| 5 | 资金方要求补材料。 | 出口商或事实方收到明确补充要求。 |
| 6 | 资金方允许/拒绝/降额/人工复核。 | 决策有业务原因。 |
| 7 | 合约执行条件检查。 | 资金方批准后仍可能等待、阻断或失败。 |

### 6.3 通过标准

| 验收项 | 通过标准 |
|---|---|
| 资金方边界 | 能决策融资，但不能修改原始事实。 |
| Agent 辅助 | 能生成摘要、风险和补材料建议。 |
| 决策结果 | 允许、拒绝、补材料、降额、人工复核都可表达。 |
| 执行检查 | 批准后还要检查规则、资金池和暂停状态。 |
| 出口商反馈 | 出口商能看到对自己有用的决策结果和原因。 |

### 6.4 不通过表现

| 不通过 | 原因 |
|---|---|
| 资金方可以修改出口商发票。 | 破坏证据可信。 |
| Agent 自动替资金方批准高风险案件。 | Agent 越权。 |
| 资金方批准后无视资金池不足。 | 合约/规则边界缺失。 |
| 资金方看不到材料缺口。 | 贷前审查不可用。 |

## 7. 纯 DeFi 路径验收

### 7.1 验收目标

验证系统不被设计成只能走银行审批；在纯 DeFi 路径下，规则、事实节点和资金池也能形成可解释执行路径。

### 7.2 验收步骤

| 步骤 | 业务动作 | 通过标准 |
|---:|---|---|
| 1 | 案件选择或进入 DeFi 规则路径。 | 控制塔能明确这是纯 DeFi 路径，不是银行辅助路径。 |
| 2 | Agent 检查事实是否满足规则。 | 能表达事实完整、缺口或需人工。 |
| 3 | 合约/规则检查执行条件。 | 能表达规则有效、资金池可用、平台未暂停。 |
| 4 | 输出业务执行结果。 | Allowed / Waiting / Rule blocked / Pool insufficient / Platform paused / Completed。 |
| 5 | 出口商看到结果。 | 知道不是银行拒绝，而是规则、资金池或执行条件的结果。 |

### 7.3 通过标准

| 验收项 | 通过标准 |
|---|---|
| 路径差异 | 纯 DeFi 不被强制合并到银行审批。 |
| 事实条件 | 必要事实节点仍需满足，不是无条件放款。 |
| 合约边界 | 合约不判断贸易真实性，只检查条件和执行。 |
| 风险升级 | dispute、高风险、规则冲突仍能升级人工。 |

### 7.4 不通过表现

| 不通过 | 原因 |
|---|---|
| 纯 DeFi 路径必须有银行批准才能继续。 | 双路径定位被破坏。 |
| DeFi 路径无视事实节点。 | 风险过高，事实边界缺失。 |
| 合约被描述为判断贸易真实。 | 合约越界。 |
| 规则阻断没有业务解释。 | 用户无法处理。 |

## 8. 低人工路径验收

### 8.1 验收目标

验证正常、低风险、事实完整、规则明确的案件，可以由 Agent + 规则自动推进大部分步骤，减少人工负担。

### 8.2 验收步骤

| 步骤 | 业务动作 | 通过标准 |
|---:|---|---|
| 1 | 用户选择半自动或全自动 Agent 模式。 | 授权范围清楚，不改变角色。 |
| 2 | Agent 自动整理材料。 | 摘要和缺口清楚。 |
| 3 | Agent 自动生成待办或请求补充。 | 责任方明确。 |
| 4 | Agent 在低风险条件下推进下一步。 | 推进动作可追踪。 |
| 5 | 规则/合约检查执行边界。 | Agent 不能绕过规则、资金池或平台暂停。 |
| 6 | 结果或异常反馈给用户。 | 用户知道 Agent 做了什么和还需要什么。 |

### 8.3 通过标准

| 验收项 | 通过标准 |
|---|---|
| 授权明确 | 用户知道 Agent 自动处理哪些事。 |
| 权限继承 | Agent 不越过所属角色权限。 |
| 自动条件 | 低风险、事实完整、规则明确时才推进。 |
| 可追踪 | Agent 自动动作有业务记录。 |
| 可升级 | 出现异常时能停止自动推进并交给人。 |

### 8.4 不通过表现

| 不通过 | 原因 |
|---|---|
| Agent 不经授权自动推进关键动作。 | 自动化越界。 |
| Agent 绕过规则阻断。 | 执行边界破坏。 |
| Agent 自动裁决 dispute。 | 重大争议处理越界。 |
| Agent 输出无法追踪。 | 审计缺失。 |

## 9. 人工升级路径验收

### 9.1 验收目标

验证系统遇到高风险、争议、置信不足、执行失败或权限不清时，能停止自动推进并升级给正确责任方。

### 9.2 人工升级触发条件

| 条件 | 必须升级给 |
|---|---|
| 高金额案件 | 资金方。 |
| 高风险国家/品类/买方 | 资金方、平台。 |
| 质检 Fail | 资金方、质检方。 |
| 质检 Dispute | 质检方、出口商、资金方。 |
| 金额不一致 | 出口商、资金方。 |
| 买方争议 | 买方、资金方。 |
| 保险不足 | 保险方、资金方。 |
| 资金池不足 | 平台、资金方。 |
| 规则阻断 | 平台。 |
| 平台暂停 | 平台。 |
| 执行失败 | 平台、资金方。 |
| Agent 置信不足 | 所属角色。 |
| 权限不清 | 回流 waterfall 文档。 |

### 9.3 通过标准

| 验收项 | 通过标准 |
|---|---|
| 升级原因清楚 | 用户知道为什么不能自动继续。 |
| 责任方清楚 | 谁要处理明确。 |
| 解除条件清楚 | 满足什么才能继续。 |
| Agent 停止越权 | 升级后不再自动裁决。 |
| 记录可追踪 | 升级记录保留。 |

### 9.4 不通过表现

| 不通过 | 原因 |
|---|---|
| dispute 被 Agent 自动裁决。 | 争议处理越界。 |
| 高风险案件自动放款。 | 资金风险过高。 |
| 执行失败没有责任方。 | 异常不可处理。 |
| 权限不清仍继续实现。 | 瀑布回流机制失效。 |

## 10. 权限隔离验收

### 10.1 验收目标

验证单钱包单角色、角色专属工作区、案件相关可见和越权阻断成立。

### 10.2 验收场景

| 场景 | 通过标准 |
|---|---|
| 出口商查看自己的案件 | 允许。 |
| 出口商查看其他出口商案件 | 阻断，并给业务解释。 |
| 出口商执行资金方审核动作 | 阻断，并说明角色不匹配。 |
| 资金方查看分配给自己的案件 | 允许。 |
| 资金方修改出口商原始发票 | 阻断。 |
| 质检方查看自己的质检任务 | 允许。 |
| 质检方决定放款 | 阻断。 |
| 平台暂停资金池 | 允许，但必须有业务原因和记录。 |
| 平台替资金方批准放款 | 阻断。 |
| Agent 执行所属角色无权动作 | 阻断。 |

### 10.3 通过标准

| 验收项 | 通过标准 |
|---|---|
| 单钱包单角色 | 注册后角色不可随意切换。 |
| 可见范围 | 只能看到相关案件和任务。 |
| 动作边界 | 每个动作责任角色明确。 |
| 越权解释 | 被阻断时有业务原因。 |
| Agent 继承权限 | Agent 不成为超级用户。 |

## 11. 事实来源边界验收

### 11.1 验收目标

验证每类关键事实都来自对应事实方，Agent、平台、资金方不能伪造事实。

### 11.2 验收场景

| 事实 | 责任方 | 通过标准 |
|---|---|---|
| 贸易材料 | 出口商 | 出口商提交，其他角色不能无痕修改。 |
| 质检结论 | 质检方 | pass/fail/dispute 由质检方确认。 |
| 物流状态 | 物流方/可接受事实来源 | 资金方和 Agent 不能伪造。 |
| 买方确认 | 买方 | 出口商不能替买方确认付款责任。 |
| 保险结论 | 保险方 | 资金方不能冒充保险方承保。 |
| 合约输出 | 合约/规则执行层 | 输出执行状态，不等于贸易真实性。 |
| Agent 摘要 | Agent | 摘要不等于事实。 |

### 11.3 不通过表现

| 不通过 | 原因 |
|---|---|
| Agent 自动生成质检 Pass。 | 事实伪造。 |
| 平台修改出口商发票。 | 证据污染。 |
| 合约被当成贸易真实性证明。 | 合约边界错误。 |
| 资金方修改物流状态。 | 事实来源错误。 |

## 12. 合约业务输出验收

### 12.1 验收目标

验证合约/规则执行层能输出业务可理解的状态，并且不越过业务边界。

### 12.2 验收场景

| 场景 | 期望业务输出 |
|---|---|
| 条件满足 | Allowed / Completed。 |
| 还缺事实或确认 | Waiting / Need materials。 |
| 质检 dispute 未解除 | Dispute / Need review。 |
| 规则暂停 | Rule blocked / Platform paused。 |
| 资金池不足 | Pool insufficient。 |
| 平台暂停案件或资金池 | Platform paused。 |
| 执行层失败 | Execution failed。 |

### 12.3 通过标准

| 验收项 | 通过标准 |
|---|---|
| 输出可理解 | 业务角色知道发生了什么。 |
| 下一步清楚 | 责任方和处理动作明确。 |
| 不判断事实 | 合约不输出“贸易真实/虚假”。 |
| 不替审批 | 合约不替银行辅助路径资金方审批。 |
| 尊重暂停 | 合约不绕过平台暂停。 |

## 13. 平台运营验收

### 13.1 验收目标

验证平台能够维护系统级秩序：资金池、规则、合约、异常和 Agent 自动化状态。

### 13.2 验收场景

| 场景 | 通过标准 |
|---|---|
| 查看平台运行总览 | 能看到资金池、规则、合约、异常、Agent 状态。 |
| 暂停资金池 | 能说明原因、影响范围和恢复条件。 |
| 恢复资金池 | 恢复前能看到暂停原因已解除。 |
| 暂停规则 | 相关案件显示规则阻断或等待。 |
| 查看执行失败 | 能看到失败原因、责任方和下一步。 |
| 查看 Agent 堆积 | 能看到哪些自动化失败或需要人工接管。 |
| 分派异常 | 异常能分派到责任角色。 |

### 13.3 不通过表现

| 不通过 | 原因 |
|---|---|
| 平台变成逐单审批员。 | 平台定位错误。 |
| 平台可以替资金方批准放款。 | 决策权越界。 |
| 平台暂停没有原因和恢复条件。 | 业务不可追踪。 |
| 平台可以删除审计记录。 | 信任基础破坏。 |

## 14. 异常处理验收

| 异常 | 验收口径 |
|---|---|
| 缺材料 | 说明缺什么、谁补、补完后如何继续。 |
| 金额不一致 | 说明冲突位置、责任方、解释或修正路径。 |
| 质检 Fail | 说明是否拒绝、复检、人工复核或补充说明。 |
| 质检 Dispute | 说明争议原因、责任方、解除条件。 |
| 物流异常 | 说明延误/损坏/丢失等原因和物流方责任。 |
| 买方争议 | 说明争议内容、付款责任影响和人工处理。 |
| 保险不足 | 说明是否需承保、加费、补材料或拒保。 |
| 资金池不足 | 说明等待、恢复、切换路径或人工处理。 |
| 规则阻断 | 说明哪类规则阻断、平台是否可处理。 |
| 平台暂停 | 说明暂停对象、原因、影响、恢复条件。 |
| 执行失败 | 说明失败原因、责任方、下一步。 |
| Agent 置信不足 | 升级给所属角色人工确认。 |

通过标准：

```text
每个异常都必须有原因、责任方、影响范围、解除条件和下一步。
```

## 15. 控制塔表达验收

| 验收项 | 通过标准 |
|---|---|
| 出口商视角 | 能看到资金结果、卡点和下一步。 |
| 资金方视角 | 能看到证据完整性、风险、决策和执行条件。 |
| 质检方视角 | 能看到任务、清单、结论和补充请求。 |
| 平台视角 | 能看到资金池、规则、合约、异常和 Agent 状态。 |
| Agent 表达 | 摘要、缺口、建议、待办、升级用业务语言。 |
| 合约表达 | Allowed、Waiting、Rule blocked、Pool insufficient 等能解释。 |
| 路径表达 | 银行辅助、纯 DeFi、低人工、人工升级可区分。 |

不通过表现：

| 不通过 | 原因 |
|---|---|
| 控制塔只是普通列表。 | 不能表达源头到终端。 |
| 卡点只显示技术失败。 | 用户不知道如何处理。 |
| 合约输出被表达成贸易真实性。 | 合约越界。 |
| Agent 建议被表达成审批结论。 | Agent 越界。 |

## 18. 一票否决项

| 一票否决项 | 原因 |
|---|---|
| 出口商无法看到资金结果或明确卡点 | P0 核心价值失败。 |
| 所有路径被合并成银行审批 | DeFi/低人工路径被破坏。 |
| 纯 DeFi 路径无视事实和规则 | 风险边界失败。 |
| Agent 可以越权审批、伪造事实或裁决 dispute | Agent 边界失败。 |
| 合约判断贸易真实性或替资金方审批 | 合约边界失败。 |
| 平台可以替资金方批准或删除审计记录 | 平台边界失败。 |
| 一个钱包可以随意切换角色 | 身份权限基础失败。 |
| 异常没有责任方和下一步 | 业务不可处理。 |
| P1/P2 被强行塞入 P0 必做 | MVP 范围失控。 |
| 验收文档直接进入测试框架、测试代码或自动化脚本 | 阶段越界。 |

## 20. 本步结论

P0 业务验收设计的本质不是“写测试代码”。

它的本质是：

> 在业务层定义 ChainTrace P0 如何证明融资闭环、角色边界、Agent Native、DeFi/银行双路径、合约执行、控制塔表达、异常处理和后置范围全部成立，并为后续测试设计和实现阶段提供清晰的验收口径。

下一步应进入：

> `60-p0-design-feedback-and-deferred-scope.md`，定义 P0 技术设计阶段发现缺口时如何回流，以及哪些内容必须后置到 P1/P2 或实现阶段。
