# 17 P0 Agent 待办模型

## 1. 定位

本文件定义 ChainTrace P0 MVP 的 Agent 待办模型。

Agent 待办模型不是任务数据库设计，不是 UI 列表组件，也不是 Agent 技术架构。

它的核心作用是：

> 把融资案件中的缺口、确认、异常、授权和提醒，转成不同业务角色可以执行的明确待办。

本文件只写业务需求，不写页面、组件、数据库、智能合约实现、钱包签名、hash 或 proof 细节。

## 2. 待办设计原则

| 原则 | 说明 |
|---|---|
| 一个待办只找一个责任角色 | 避免多人都以为别人会处理。 |
| 待办必须有业务原因 | 说明为什么需要做。 |
| 待办必须有完成标准 | 说明做到什么算完成。 |
| 待办必须回到流程 | 完成后知道案件回到哪个状态。 |
| Agent 可自动生成低风险待办 | 缺材料、提醒、普通补充可自动生成。 |
| 高风险待办必须人工确认 | 银行参与场景的放款授权、规则暂停、重大争议等必须人处理。 |

## 3. 待办基础结构

每个 Agent 待办必须包含：

| 项目 | 业务含义 |
|---|---|
| 待办名称 | 用户一眼知道要做什么。 |
| 责任角色 | 出口商、质检方、资金方、平台方等。 |
| 来源 | 出口商提交、Agent 检查、质检结果、资金方要求、平台规则等。 |
| 业务原因 | 为什么需要这个待办。 |
| 建议动作 | 用户应该上传、确认、审批、拒绝、补充还是复核。 |
| 完成标准 | 什么结果代表待办完成。 |
| 截止时间 | 可选，用于加急或关键节点。 |
| 影响范围 | 如果不处理，会影响提交、审查、执行还是到账。 |
| 回到状态 | 完成后案件回到哪个业务状态。 |

## 4. P0 待办类型

| 待办类型 | 说明 | 责任角色 |
|---|---|---|
| Material upload | 上传或补充材料。 | 出口商 / 质检方 |
| Material explanation | 解释材料差异或异常。 | 出口商 / 质检方 |
| Inspection task | 执行质检并提交结论。 | 质检方 |
| Inspection supplement | 补充质检报告、照片或说明。 | 质检方 |
| Financing review | 审查融资案件。 | 资金方 |
| Funding decision | 银行参与场景中允许放款 / 拒绝 / 补材料 / 人工复核。 | 资金方 |
| DeFi execution check | 纯 DeFi 场景中检查规则、资金池、抵押/保证和事实状态。 | DeFi 智能合约 / Agent |
| Manual review | 处理高风险或争议。 | 资金方 / 平台方 |
| Execution issue | 处理资金池、规则、暂停或执行失败。 | 平台方 |
| Status reminder | 提醒状态变化、截止时间、卡点。 | 对应角色 |
| Agent check | Agent 重新检查材料或条件。 | Agent |

## 5. 角色待办

### 5.1 出口商待办

| 待办 | 触发条件 | 完成标准 |
|---|---|---|
| 创建融资意图 | 出口商进入工作区并选择申请融资。 | 案件进入 Draft。 |
| 上传基础贸易材料 | 缺合同、订单、发票、装箱单等。 | Agent 检查基础材料完整。 |
| 解释金额差异 | 发票、订单、申请金额不一致。 | 提交解释或修正材料。 |
| 确认提交融资案件 | 案件 Ready to submit。 | 案件进入 Submitted。 |
| 补充资金方要求材料 | 资金方要求补材料。 | 资金方要求被逐项回应。 |
| 查看融资结果 | 资金方决策或执行状态变化。 | 出口商确认已知结果或下一步。 |

### 5.2 质检方待办

| 待办 | 触发条件 | 完成标准 |
|---|---|---|
| 接收质检任务 | 案件需要质检。 | 接受或拒绝任务并说明原因。 |
| 提交质检材料 | 质检任务已接受。 | 上传报告、照片、说明。 |
| 选择质检结论 | 质检材料准备完成。 | 提交 pass / fail / dispute。 |
| 补充质检照片 | Agent 或资金方认为照片不足。 | 上传指定照片并重新检查。 |
| 解释质检争议 | 结论为 dispute 或被资金方质疑。 | 提交争议说明或补充报告。 |

### 5.3 资金方待办

| 待办 | 触发条件 | 完成标准 |
|---|---|---|
| 查看待审案件 | 案件进入 Under review。 | 打开摘要并完成初审。 |
| 判断材料完整性 | Agent 标记材料可审或需人工确认。 | 确认完整、要求补材料或拒绝。 |
| 判断风险 | Agent 给出风险摘要。 | 选择正常、警示、高风险、阻断或人工复核。 |
| 做融资决策 | 案件达到资金方决策点。 | 允许放款 / 拒绝 / 补材料 / 人工复核。 |
| 处理执行等待 | 资金池不足、规则等待、平台暂停等。 | 等待、切换处理、取消或转平台处理。 |

### 5.4 平台方待办

| 待办 | 触发条件 | 完成标准 |
|---|---|---|
| 查看执行异常 | 执行失败或平台阻断。 | 处理原因并标记下一步。 |
| 处理资金池不足 | 已批准案件等待资金池。 | 补足、恢复、等待或说明无法处理。 |
| 处理规则阻断 | 当前规则不允许执行。 | 维持阻断、恢复、调整规则或关闭案件。 |
| 处理平台暂停 | 平台、规则或资金池暂停。 | 恢复或说明继续暂停原因。 |
| 处理 Agent 堆积 | Agent 待办长期未处理。 | 分派、升级或暂停自动化。 |

### 5.5 Agent 自身待办

Agent 自身也有业务动作，但不是单独用户。

| 待办 | 触发条件 | 完成标准 |
|---|---|---|
| 整理材料摘要 | 用户上传材料后。 | 形成角色视角摘要。 |
| 检查缺口 | 材料变化或状态变化后。 | 输出完整、不完整或需人工确认。 |
| 生成补材料任务 | 发现明确缺口。 | 待办分派给责任角色。 |
| 生成状态解释 | 案件状态变化后。 | 用户能理解当前状态和下一步。 |
| 升级人工处理 | 发现高风险、争议或授权动作。 | 创建人工复核待办。 |

## 6. 待办优先级

| 优先级 | 含义 | 示例 |
|---|---|---|
| P0 Critical | 不处理就无法继续融资或执行。 | 资金方决策、质检结论、规则阻断、资金池不足。 |
| P1 Important | 会影响审查质量或进度。 | 补充发票说明、补充质检照片、解释金额差异。 |
| P2 Reminder | 不一定阻断当前流程，但需要提醒。 | 截止时间、状态变化、结果通知。 |

优先级不等于角色优先级。

一个 P1 角色也可能收到 P0 Critical 待办，例如后续引入物流方后，货物丢失说明可能是关键阻断。

## 7. Agent 模式对待办的影响

| 待办行为 | 全人工确认 | 半自动托管 | 全自动 Agent |
|---|---|---|---|
| 创建待办 | Agent 建议，用户确认 | Agent 自动创建常规待办 | Agent 自动创建低风险待办 |
| 分派待办 | 用户确认分派 | Agent 自动分派明确责任待办 | Agent 自动分派并提醒 |
| 补材料提醒 | 用户主动查看 | Agent 主动提醒 | Agent 自动跟进直到完成或超时 |
| 低风险流程恢复 | 用户确认 | Agent 建议恢复，用户确认关键点 | Agent 自动恢复低风险流程 |
| 高风险人工复核 | 必须人处理 | 必须人处理 | 必须人处理 |
| 银行参与场景的放款授权 | 必须资金方确认 | 必须资金方确认 | 必须资金方确认 |
| 纯 DeFi 低风险执行 | 用户确认 | Agent 建议执行 | 授权范围内可自动推进到合约执行 |
| 平台暂停 / 恢复 | 必须平台方确认 | 必须平台方确认 | 必须平台方确认 |

## 8. 待办完成后的回流

待办完成后必须回到业务流程，而不是停在待办列表。

| 待办完成 | 回到状态 |
|---|---|
| 出口商补齐基础材料 | Preparing materials 或 Ready to submit。 |
| 出口商解释金额差异 | Ready to submit、Under review 或 Manual review。 |
| 质检方提交 pass | Inspection submitted，然后进入 Under review。 |
| 质检方提交 fail | Manual review、Rejected 或 Blocked。 |
| 质检方提交 dispute | Manual review 或 Need materials。 |
| 资金方要求补材料 | Need materials。 |
| 资金方允许放款 | 银行辅助路径进入 Approved，然后进入 Execution checking。 |
| DeFi 规则允许执行 | 纯 DeFi 路径进入 Approved 或 Disbursement pending。 |
| 平台恢复资金池 | Execution waiting 或 Disbursement pending。 |
| 平台解除规则阻断 | Execution checking 或 Under review。 |

## 9. 待办表达规范

待办必须是业务语言。

| 不要说 | 要说 |
|---|---|
| Missing required field | 请上传商业发票，资金方需要确认贸易金额。 |
| Validation failed | 发票金额和订单金额不一致，请补充说明。 |
| Review needed | 质检结论为 dispute，需要人工复核；银行辅助路径可由资金方处理，纯 DeFi 路径进入人工升级。 |
| Dependency pending | 当前等待质检方补充货物标签照片。 |
| Execution blocked | 当前资金池余额不足，已通知平台方处理。 |

## 10. MVP 范围

### P0 必须做

| 范围 | 说明 |
|---|---|
| 定义待办结构 | 名称、责任角色、原因、动作、完成标准、回到状态。 |
| 定义 P0 角色待办 | 出口商、质检方、资金方、平台方、Agent、DeFi 智能合约执行检查。 |
| 定义待办优先级 | Critical、Important、Reminder。 |
| 定义 Agent 模式差异 | 三种模式下待办创建、分派、提醒不同。 |
| 定义待办回流 | 完成后回到状态机。 |
| 定义业务表达规范 | 避免技术错误语言。 |

### 暂不做

| 范围 | 原因 |
|---|---|
| 任务表结构 | 当前不写数据库设计。 |
| 通知渠道设计 | 当前不写实现。 |
| 页面待办组件 | 当前不做 UI。 |
| Agent 调度技术 | 当前不写技术架构。 |
| hash / proof 任务 | 当前不进入底层证明细节。 |

## 11. 本步结论

P0 Agent 待办模型的本质不是“任务列表”。

它的本质是：

> 把融资案件的每个缺口、确认、授权、异常和提醒，转成明确责任角色、明确业务原因、明确完成标准、明确回流状态的可执行动作。

下一步应进入：

> P0 信息架构和工作区业务分区，仍然只写需求，不进入 UI 组件设计。
