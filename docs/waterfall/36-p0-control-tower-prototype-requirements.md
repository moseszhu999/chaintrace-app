# 36 P0 控制塔式原型页面业务要求

## 1. 定位

本文件定义 ChainTrace P0 阶段原型页面的业务呈现要求。

这里的“原型页面”不是 UI 组件设计，不是视觉稿，不是前端实现，也不是页面路由说明。

它的作用是：

> 明确后续产品原型应采用“大数据控制塔大屏”风格来表达供应链金融案件的全链路流转、节点状态、角色可见范围、Agent 待办、规则卡点和 DeFi 执行结果。

本文件不写组件、布局尺寸、图表库、前端代码、API、数据库、Solidity、钱包签名、hash 或 proof 细节。

## 2. 控制塔原型原则

| 原则 | 说明 |
|---|---|
| 以链路为主轴 | 原型应让用户看到供应链从源头到终端的业务流转。 |
| 以案件为中心 | 控制塔不是泛泛的数据大屏，而是围绕融资/结算案件展开。 |
| 以角色权限过滤 | 不同角色看到不同范围，不能因为大屏表达而越权展示。 |
| 以状态和卡点优先 | 第一眼应知道当前在哪里、为什么卡住、谁负责、下一步是什么。 |
| 以 Agent 摘要降噪 | Agent 把材料、事实、风险和待办整理成可理解摘要。 |
| 以 DeFi 执行可解释 | 合约执行、等待、阻断、失败、完成必须用业务语言表达。 |
| 不做营销首页 | 登录后进入业务控制塔，不做登录前展示页。 |

## 3. 控制塔应回答的业务问题

控制塔原型的第一目标不是“展示很多数据”，而是回答业务问题。

| 问题 | 业务表达 |
|---|---|
| 这笔业务从哪里来？ | 源头贸易、出口商、买方、货物和订单背景。 |
| 现在流转到哪里？ | 当前节点、当前状态、当前责任角色。 |
| 中间经过哪些节点？ | 贸易材料、质检、物流、通关、保险、买方确认、资金方、平台、合约执行。 |
| 哪些节点已经完成？ | 已确认事实、已完成动作、已满足条件。 |
| 哪些节点有异常？ | 缺材料、事实冲突、风险提示、争议、规则阻断、资金池不足。 |
| 下一步谁处理？ | 出口商、事实方、资金方、平台方、Agent 或合约等待条件。 |
| 是否可以执行资金动作？ | 银行辅助授权是否完成，DeFi 条件是否满足，资金池是否可用。 |
| 用户能看到哪些内容？ | 只显示该角色相关案件、节点、事实和结果。 |

## 4. 供应链全链路节点

P0 控制塔应以业务节点表达供应链金融案件，不应只表达文件列表或技术状态。

| 节点 | 业务含义 | 主要事实来源 | P0/P1/P2 |
|---|---|---|---|
| 业务意图 | 出口商或业务方创建融资/结算意图。 | 出口商 / 业务方 | P0 |
| 贸易材料 | 合同、订单、发票、装箱单、补充说明。 | 出口商 | P0 |
| Agent 整理 | 摘要、缺口、风险、待办、建议。 | Agent | P0 |
| 质检事实 | 货物是否 pass / fail / dispute。 | 质检方 | P0 |
| 物流事实 | 揽收、装柜、出港、到港、签收、延误、损坏、丢失。 | 物流方 | P1 |
| 通关事实 | 报关、清关、监管异常、品类/金额/国家一致性。 | 海关 / 报关方 | P2 |
| 保险事实 | 承保、拒保、加费、补材料、理赔。 | 保险公司 | P1 |
| 买方确认 | 订单真实性、收货、争议、付款责任。 | 买方 / 进口商 | P1 |
| 银行辅助审查 | 允许放款、拒绝、补材料、人工复核。 | 资金方 / 银行 / 保理机构 | 按场景 |
| DeFi 执行检查 | 规则、资金池、授权、暂停、执行条件。 | DeFi 智能合约 | P0 |
| 平台运营 | 规则、资金池、暂停、异常、Agent 堆积。 | 平台方 | P0 |
| 资金结果 | 到账、结算完成、等待、阻断、失败、关闭。 | 合约 / 平台 / 资金方 | P0 |

## 5. 控制塔信息层级

控制塔需要分层表达业务，不应把所有信息堆在一个列表里。

| 层级 | 应表达什么 | 不应表达什么 |
|---|---|---|
| 全局层 | 总体流转、关键节点、当前状态、卡点。 | 技术日志、底层交易细节。 |
| 节点层 | 每个节点是否完成、等待、异常、阻断。 | 与该角色无关的内部细节。 |
| 案件层 | 当前案件的材料、事实、决策、执行条件。 | 无关案件和无关角色材料。 |
| 待办层 | 谁要处理什么，完成标准是什么。 | 没有责任人的模糊提醒。 |
| 风险层 | 高风险、争议、规则阻断、资金池不足。 | 无解释的红色错误码。 |
| 执行层 | 允许执行、等待条件、阻断原因、执行结果。 | Solidity、ABI、hash、proof 等实现细节。 |

## 6. 角色视角差异

控制塔风格不等于每个角色都看全量数据。

| 角色 | 控制塔视角 | 可见重点 | 不可见内容 |
|---|---|---|---|
| 出口商 / 资金需求方 | 自己案件的端到端流转。 | 贸易材料、事实进度、Agent 缺口、资金结果、卡点。 | 其他出口商案件、资金方内部审查细节、平台全局运营数据。 |
| 资金方 / 银行 / 保理机构 | 与自己相关的审查案件和风险流。 | 案件摘要、材料完整性、事实状态、风险、授权结果、执行条件。 | 无关案件、其他资金方意见、平台敏感运营细节。 |
| 质检方 | 自己负责的质检节点控制塔。 | 待质检任务、货物摘要、质检要求、结论影响、补充要求。 | 资金审批、资金池、无关贸易全量材料。 |
| 物流方 | 自己负责的运输节点控制塔。 | 运输节点、物流状态、异常说明、签收和损坏事实。 | 放款决策、质检和保险内部事实、无关案件。 |
| 海关 / 报关方 | 自己负责的通关节点控制塔。 | 报关/清关状态、监管异常、一致性提示。 | 融资决策、资金池、无关案件。 |
| 保险公司 | 自己负责的核保/理赔节点控制塔。 | 风险摘要、承保结论、补材料、理赔进度。 | 资金方内部审批、平台规则维护入口、无关案件。 |
| 买方 / 进口商 | 自己相关订单和付款责任控制塔。 | 订单真实性、收货、争议、付款到期责任。 | 出口商其他融资申请、资金审批、质检内部过程。 |
| 平台方 / 合约运营 | 全局运行控制塔。 | 资金池、规则、暂停、异常、Agent 堆积、执行失败、关键审计。 | 不能伪造或修改业务事实。 |
| Agent | 所属角色的业务助理视角。 | 摘要、缺口、待办、建议、提醒、升级。 | 所属角色不可见的案件和事实。 |

## 7. 不同业务路径的控制塔差异

ChainTrace 不是单一路径产品。控制塔必须能表达银行辅助、纯 DeFi 和低人工路径的差异。

| 路径 | 控制塔应突出什么 | 不应误导用户认为 |
|---|---|---|
| 银行辅助路径 | 资金方审查、授权、补材料、拒绝、人工复核、合约执行条件。 | 所有资金动作都由合约无条件自动决定。 |
| 纯 DeFi 路径 | 规则、资金池、事实条件、抵押/保证、暂停状态、合约执行结果。 | 必须等待银行审批。 |
| Agent + 智能合约低人工路径 | Agent 自动推进、规则满足、事实完整、异常升级、合约锁定/释放/结算。 | Agent 可以自动裁决重大争议或伪造事实。 |

## 8. 节点状态表达

每个节点都应有业务可理解状态。

| 状态 | 业务含义 | 用户应理解 |
|---|---|---|
| Not started | 尚未开始。 | 该节点还未轮到或条件未满足。 |
| Waiting | 等待材料、确认、授权或外部事实。 | 需要知道等谁。 |
| In progress | 正在处理。 | 当前责任角色正在处理。 |
| Completed | 已完成。 | 该节点已满足当前路径要求。 |
| Need materials | 缺材料或缺说明。 | 需要哪个角色补什么。 |
| Warning | 有风险提示但未必阻断。 | 需要关注或人工确认。 |
| Dispute | 存在争议。 | 不能自动通过，需人工处理。 |
| Blocked | 被规则、事实冲突、资金池或平台状态阻断。 | 为什么被阻断，解除条件是什么。 |
| Failed | 执行或处理失败。 | 失败原因和下一步处理方。 |
| Skipped | 当前路径不需要该节点。 | 例如纯 DeFi 场景不需要银行审批。 |

## 9. 控制塔中的 Agent 表达

Agent 在控制塔中不是独立用户，而是每个角色身边的业务助理。

| Agent 输出 | 控制塔表达 |
|---|---|
| 摘要 | 把当前案件、节点和风险整理成角色可读说明。 |
| 缺口 | 在对应节点标出缺什么、谁负责、完成标准。 |
| 待办 | 把下一步动作分派给责任角色。 |
| 建议 | 给出提交、补材料、等待、人工复核、拒绝或执行建议。 |
| 自动推进 | 在授权范围内推进低风险、规则明确、事实完整的节点。 |
| 异常升级 | 将争议、高风险、越权风险、执行失败交给人。 |

Agent 不应在控制塔中被表达成：

| 禁止表达 | 原因 |
|---|---|
| 独立审批人 | Agent 权限来自所属角色。 |
| 事实制造者 | Agent 只能整理事实，不能伪造事实。 |
| 高风险裁决者 | 高风险、重大争议和法律责任需要人工处理。 |
| 绕过规则入口 | Agent 不能绕过平台规则或合约约束。 |

## 10. 控制塔中的 DeFi 执行表达

DeFi 智能合约是资金执行约束层，不是页面、数据库或 Agent。

控制塔需要用业务语言表达合约结果：

| 合约输出 | 控制塔表达 |
|---|---|
| 允许执行 | 当前条件满足，可以进入资金动作。 |
| 不允许执行 | 当前条件不满足，不能执行。 |
| 等待条件 | 还缺事实、授权、资金池或规则条件。 |
| 规则阻断 | 当前规则不允许继续，并说明业务原因。 |
| 资金池不足 | 资金池不可用、余额不足或被暂停。 |
| 平台暂停 | 平台、路径、规则、资金池或案件暂停。 |
| 执行失败 | 资金动作没有完成，需要平台或相关角色处理。 |
| 执行完成 | 放款、结算、锁定、释放或其他资金动作已完成。 |

控制塔不应展示：

| 不展示 | 原因 |
|---|---|
| 合约代码 | 当前不是实现阶段。 |
| ABI / 部署信息 | 当前不写技术方案。 |
| 钱包签名细节 | 当前不进入签名实现。 |
| hash / proof 页面 | 当前不进入底层证明产品。 |

## 11. 权限和可见性要求

控制塔必须继承 P0 权限矩阵。

| 要求 | 说明 |
|---|---|
| 单钱包单角色 | 登录后只能进入永久绑定角色的控制塔视角。 |
| 案件相关可见 | 只显示与当前角色相关的案件、节点、事实、待办和结果。 |
| 节点相关可见 | 事实方只看自己负责节点和必要上下文。 |
| 平台全局可见但不可越权 | 平台可看全局运行，不可伪造事实或替资金方审批。 |
| Agent 继承角色权限 | Agent 只能在所属角色可见范围内整理和建议。 |
| 路径决定可见性 | 纯 DeFi 场景默认不暴露银行审批视角；银行辅助场景才出现资金方审查节点。 |

## 12. MVP 范围

### 12.1 P0 必须定义

| 范围 | 说明 |
|---|---|
| 控制塔表达口径 | 后续原型页面采用大数据控制塔风格。 |
| 供应链全链路节点 | 从业务意图到资金结果的节点结构清楚。 |
| 角色视角差异 | 每个角色看到不同范围。 |
| 路径差异 | 银行辅助、纯 DeFi、低人工路径表达不同。 |
| 状态和卡点表达 | 每个节点知道状态、原因、责任方和下一步。 |
| Agent 表达 | Agent 摘要、缺口、待办、建议和升级清楚。 |
| DeFi 执行表达 | 合约输出用业务语言呈现。 |

### 12.2 P0 暂不做

| 范围 | 原因 |
|---|---|
| 视觉稿 | 当前不是 UI 设计阶段。 |
| 页面布局尺寸 | 当前只定义业务表达。 |
| 前端组件库 | 当前不写代码。 |
| 图表实现方案 | 当前不选择技术实现。 |
| 数据库设计 | 当前不写表结构。 |
| 智能合约实现 | 当前不写 Solidity。 |
| hash / proof 细节 | 当前不进入底层证明产品。 |

## 13. 本步结论

ChainTrace 的原型页面应采用大数据控制塔大屏风格，但它不是为了做“好看的数据展示”。

它的业务本质是：

> 让不同角色在自己的权限范围内，看清供应链从源头到终端的流转、每个节点的完成/等待/异常/阻断状态、Agent 建议、DeFi 执行条件和下一步责任人。

下一步应进入：

> P0 API 业务契约，用业务语言说明后续接口需要承载哪些动作、输入、输出和错误解释，不写接口实现。
