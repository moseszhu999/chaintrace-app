# 42 P0 视觉原型评审清单

## 1. 定位

本文件定义 ChainTrace P0 视觉原型的评审清单。

视觉原型评审清单不是 UI 验收标准，不是组件设计清单，不是页面测试脚本，也不是前端开发任务。

它的作用是：

> 在每轮视觉原型走读后，用业务语言记录哪些内容已经通过、哪些业务表达需要调整、哪些问题必须后置、哪些范围不能进入 P0，确保视觉原型持续围绕供应链金融系统闭环推进。

本文件不写组件、样式、布局尺寸、图表库、页面路由、接口路径、请求响应格式、数据库、Solidity、ABI、钱包签名、hash 或 proof 细节。

## 2. 评审原则

| 原则 | 说明 |
|---|---|
| 评审业务表达 | 先判断业务问题是否被回答，而不是评审视觉偏好。 |
| 评审角色边界 | 每个角色看见什么、不能看见什么必须清楚。 |
| 评审路径差异 | 银行辅助、纯 DeFi、低人工和人工升级不能混淆。 |
| 评审 Agent 边界 | Agent 是业务助理，不是事实制造者或高风险裁决者。 |
| 评审 DeFi 输出 | 合约输出必须可解释，不展示实现细节。 |
| 评审异常回流 | 等待、阻断、失败、争议必须有责任方和下一步。 |
| 评审排除范围 | 登录前展示页、旧代码、组件、接口、数据库、Solidity、hash/proof 不进入本阶段。 |

## 3. 每轮评审记录

| 记录项 | 说明 |
|---|---|
| 评审轮次 | 第几轮视觉原型评审。 |
| 评审日期 | 记录评审发生时间。 |
| 评审范围 | 控制塔、某个角色视角、某条路径或某个异常场景。 |
| 参与角色 | 出口商、资金方、事实方、平台方、产品、业务或技术观察者。 |
| 走读脚本 | 对应 41 号视觉原型走读大纲中的哪个阶段。 |
| 主要结论 | 通过、需调整、需补需求、后置、阻塞。 |
| 业务表达问题 | 哪些业务表达仍不清楚。 |
| 权限问题 | 是否出现越权可见或越权动作。 |
| 异常解释问题 | 是否缺原因、责任方、解除条件或下一步。 |
| 后置项 | 哪些内容不进 P0，但保留入口或记录。 |
| 冻结项 | 哪些表达在本轮后可以冻结。 |

## 4. 总体定位评审

| 检查项 | 通过标准 | 结论 |
|---|---|---|
| 是否是供应链金融系统 | 控制塔围绕融资/结算案件、供应链节点、资金执行展开。 | 待评审 |
| 是否避免前站定位 | 不再表达成供应链金融前站或营销导流产品。 | 待评审 |
| 是否避免单纯银行审批 | 银行只在银行辅助路径中出现。 | 待评审 |
| 是否体现 DeFi 执行核心 | 合约执行、阻断、等待、失败、完成是资金动作核心。 | 待评审 |
| 是否体现 Agent Native | Agent 摘要、缺口、待办、提醒、升级清楚。 | 待评审 |
| 是否从业务工作区开始 | 不做登录前展示页。 | 待评审 |

## 5. 控制塔表达评审

| 检查项 | 通过标准 | 结论 |
|---|---|---|
| 全链路节点 | 业务意图、贸易材料、事实确认、审查/规则、执行、资金结果清楚。 | 待评审 |
| 当前状态 | 用户能知道案件当前在哪个节点。 | 待评审 |
| 已完成节点 | 用户能知道哪些节点已经完成。 | 待评审 |
| 等待节点 | 用户能知道正在等待谁或什么条件。 | 待评审 |
| 异常节点 | Warning、Dispute、Blocked、Failed 能被业务理解。 | 待评审 |
| 卡点说明 | 原因、责任角色、解除条件和下一步清楚。 | 待评审 |
| 路径差异 | 银行辅助、纯 DeFi、低人工路径不混淆。 | 待评审 |
| 角色过滤 | 当前角色只看到可见范围。 | 待评审 |

## 6. 角色视角评审

| 角色 | 必须回答的问题 | 通过标准 | 结论 |
|---|---|---|---|
| 出口商 / 资金需求方 | 我这笔业务能不能融资/结算，卡在哪里？ | 自己案件、材料、事实进度、Agent 缺口、资金结果清楚。 | 待评审 |
| 资金方 / 银行 / 保理机构 | 哪些案件需要审查，风险是否可控？ | 相关案件、材料完整性、事实状态、风险、授权和执行条件清楚。 | 待评审 |
| 质检方 | 我要确认什么货物事实？ | 只看到自己的质检任务和必要上下文。 | 待评审 |
| 物流方 | 哪些运输事实需要确认？ | 只看到自己的物流节点和异常说明。 | 待评审 |
| 海关 / 报关方 | 哪些通关事实需要确认？ | 只看到自己的通关任务和监管状态。 | 待评审 |
| 保险公司 | 是否承保、加费、拒保或理赔？ | 只看到核保/理赔相关风险摘要。 | 待评审 |
| 买方 / 进口商 | 订单、收货、争议和付款责任是否确认？ | 只看到与自己相关的订单和责任。 | 待评审 |
| 平台方 / 合约运营 | 系统哪里需要介入？ | 资金池、规则、暂停、异常、Agent 堆积清楚。 | 待评审 |
| Agent | 当前角色最该知道和处理什么？ | 摘要、缺口、待办、建议、升级原因清楚。 | 待评审 |

## 7. 路径评审

| 路径 | 必须讲通 | 不得出现 | 结论 |
|---|---|---|---|
| 银行辅助路径 | 资金方审查和授权，授权后合约检查执行条件。 | 银行授权后无条件到账。 | 待评审 |
| 纯 DeFi 路径 | 无银行审批，规则、资金池、事实和保证条件驱动合约执行。 | 强制等待资金方审批。 | 待评审 |
| 低人工路径 | Agent 推进低风险节点，合约锁定/释放/结算，异常升级人工。 | Agent 自动裁决重大争议。 | 待评审 |
| 人工升级路径 | 争议、高风险、监管、法律责任、越权风险交给人。 | 系统静默失败或自动放过。 | 待评审 |

## 8. Agent 表达评审

| 检查项 | 通过标准 | 结论 |
|---|---|---|
| 摘要 | Agent 摘要可追溯到材料和事实。 | 待评审 |
| 缺口 | 缺什么、为什么缺、谁负责清楚。 | 待评审 |
| 待办 | 待办有责任角色、动作和完成标准。 | 待评审 |
| 建议 | 建议是业务建议，不替代高风险决策。 | 待评审 |
| 自动推进 | 只在授权范围、低风险、事实完整、规则明确时推进。 | 待评审 |
| 异常升级 | 高风险、争议、越权、置信不足、执行失败会升级人工。 | 待评审 |
| 边界 | Agent 不伪造事实、不绕过规则、不裁决重大争议。 | 待评审 |

## 9. DeFi 执行评审

| 合约输出 | 用户应理解 | 结论 |
|---|---|---|
| 允许执行 | 当前业务条件满足，可以进入资金动作。 | 待评审 |
| 不允许执行 | 当前条件不满足，不能继续。 | 待评审 |
| 等待条件 | 仍缺事实、授权、资金池、规则或保证条件。 | 待评审 |
| 规则阻断 | 当前规则不允许继续，并有解除条件。 | 待评审 |
| 资金池不足 | 资金池不可用、余额不足、币种不支持或暂停。 | 待评审 |
| 平台暂停 | 平台、路径、规则、资金池或案件暂停。 | 待评审 |
| 执行失败 | 资金动作未完成，需要平台或相关角色处理。 | 待评审 |
| 执行完成 | 放款、结算、锁定、释放或到账完成。 | 待评审 |

DeFi 执行不得表达成：

| 不得表达 | 原因 |
|---|---|
| 合约展示页 | 合约是资金执行约束层。 |
| 交易技术日志 | P0 视觉原型只表达业务结果。 |
| 贸易真实性判断者 | 事实来自业务角色。 |
| 银行授权绕过者 | 银行辅助路径必须先有资金方授权。 |

## 10. 异常回流评审

| 异常 | 必须表达 | 责任角色 | 结论 |
|---|---|---|---|
| 材料不足 | 缺什么、谁补、完成标准。 | 出口商 / 事实方 | 待评审 |
| 事实不足 | 缺哪个事实节点。 | 对应事实方 | 待评审 |
| 事实冲突 | 冲突内容和影响。 | 相关角色 / 人工复核 | 待评审 |
| 质检 dispute | 争议原因和影响。 | 质检方 / 出口商 / 资金方 | 待评审 |
| 买方 dispute | 订单、收货或付款责任争议。 | 买方 / 出口商 / 人工角色 | 待评审 |
| 银行授权不足 | 授权金额、期限、条件或状态不足。 | 资金方 | 待评审 |
| 规则阻断 | 阻断规则和解除条件。 | 平台方 / 责任角色 | 待评审 |
| 资金池不足 | 资金池、币种和影响范围。 | 平台方 / 资金池运营 | 待评审 |
| 平台暂停 | 暂停对象、原因和影响范围。 | 平台方 | 待评审 |
| Agent 越权风险 | Agent 为什么不能自动推进。 | 所属角色 / 平台方 | 待评审 |
| 执行失败 | 资金动作为什么未完成。 | 平台方 / 合约执行层 | 待评审 |

## 11. 权限隔离评审

| 越权场景 | 通过标准 | 结论 |
|---|---|---|
| 出口商查看其他出口商案件 | 阻断并说明无关案件不可见。 | 待评审 |
| 资金方查看无关案件 | 阻断并说明非相关案件不可见。 | 待评审 |
| 质检方查看资金方内部审批 | 阻断并说明质检方不是审批方。 | 待评审 |
| 出口商修改质检结论 | 阻断并说明质检事实属于质检方。 | 待评审 |
| 资金方修改事实方结论 | 阻断并说明事实来源不可篡改。 | 待评审 |
| 平台方替资金方审批 | 阻断并说明平台不替资金方决策。 | 待评审 |
| Agent 查看所属角色不可见案件 | 阻断并说明 Agent 权限继承角色。 | 待评审 |
| 合约绕过银行辅助授权 | 阻断并说明资金方授权是执行条件。 | 待评审 |

## 12. 后置范围评审

| 后置项 | P0 处理方式 | 结论 |
|---|---|---|
| P1 物流完整流程 | 保留节点和入口，深度流程后置。 | 待评审 |
| P1 保险完整核保/理赔 | 保留节点和结论入口，完整流程后置。 | 待评审 |
| P1 买方付款系统 | 保留确认和争议节点，完整付款系统后置。 | 待评审 |
| P2 海关/报关接入 | 保留通关节点或占位，真实接入后置。 | 待评审 |
| 多组织多成员权限 | P0 保持单钱包单角色。 | 待评审 |
| 复杂法律自动执行 | P0 只保留低人工边界和人工升级。 | 待评审 |
| 技术接口、数据库、合约实现 | 后续技术设计阶段处理。 | 待评审 |
| hash / proof 产品页 | 当前不进入底层证明产品。 | 待评审 |

## 13. 必须退回调整的情况

出现以下情况，本轮视觉原型不得通过。

| 退回原因 | 说明 |
|---|---|
| 出现登录前展示页作为主体验 | 当前不做营销首页。 |
| 看不出供应链全链路 | 控制塔表达失败。 |
| 银行被设为所有路径必经 | 破坏纯 DeFi 和低人工路径。 |
| 纯 DeFi 路径仍等待银行审批 | 场景路径错误。 |
| 合约被表达成展示组件 | DeFi 执行核心定位错误。 |
| Agent 被表达成独立审批人 | Agent 权限边界错误。 |
| 角色能看到无关案件 | 权限隔离失败。 |
| 事实方能决定融资 | 事实方边界错误。 |
| 平台方能伪造事实 | 平台边界错误。 |
| 异常没有原因和下一步 | 业务可解释性失败。 |
| 进入接口、数据库、Solidity、hash/proof 细节 | 阶段跑偏。 |
| 继承旧版页面或旧代码口径 | 当前分支是全新瀑布重建设计分支。 |

## 14. 评审结论分类

| 结论 | 含义 | 后续动作 |
|---|---|---|
| 通过 | 当前范围业务表达清楚。 | 可冻结或进入下一走读范围。 |
| 小幅调整 | 业务口径清楚，但表达需要微调。 | 调整文案或走读说明后复核。 |
| 需补需求 | 原型暴露出需求文档缺口。 | 回到 waterfall 文档补充。 |
| 后置 | 价值存在，但不属于 P0。 | 记录到 P1/P2 或技术设计后续范围。 |
| 阻塞 | 破坏核心定位、路径、权限或边界。 | 不得进入下一阶段，必须修正。 |

## 16. MVP 范围

### 16.1 P0 必须评审

| 范围 | 说明 |
|---|---|
| 产品定位 | 完整供应链金融系统，不是前站或单纯银行审批。 |
| 控制塔表达 | 全链路、节点状态、卡点、责任角色、下一步。 |
| 角色视角 | 各角色可见范围和不可见范围。 |
| 路径差异 | 银行辅助、纯 DeFi、低人工、人工升级。 |

## 17. 本步结论

P0 视觉原型评审清单的本质不是“评价页面好不好看”。

它的本质是：

> 用统一业务清单判断视觉原型是否讲清了 ChainTrace 的供应链金融控制塔、角色权限、路径差异、Agent 边界、DeFi 执行、异常回流和后置范围，并把每轮评审结论沉淀成可追踪记录。

下一步应进入：

> P0 视觉原型变更管理说明，用于定义视觉原型评审后哪些调整可以直接改、哪些必须回到 waterfall 需求文档补充。
