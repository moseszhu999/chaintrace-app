# 47 P0 技术设计准备说明

## 1. 定位

本文件定义 ChainTrace P0 从已冻结业务原型进入后续技术设计前，技术设计阶段应接收哪些业务输入、保持哪些业务边界，以及哪些内容仍然不得提前实现。

P0 技术设计准备说明不是技术架构文档，不是接口规范，不是数据库设计，不是智能合约设计，不是前端实现计划，也不是开发任务清单。

它的作用是：

> 在不写代码、不定义实现细节的前提下，把已冻结的 P0 业务口径整理成技术设计可接收的业务输入目录，确保后续接口、数据、合约、Agent、权限、原型实现和测试设计都继承 ChainTrace 的产品定位、角色边界、路径差异、DeFi 执行和异常回流规则。

本文件不写组件、样式、布局尺寸、图表库、页面路由、接口路径、请求响应格式、数据库、Solidity、ABI、钱包签名、hash 或 proof 细节。

## 2. 准备原则

| 原则 | 说明 |
|---|---|
| 技术设计接收业务输入 | 技术设计只能基于已冻结业务口径展开。 |
| 不反向重写业务 | 技术便利性不能改变产品定位、角色边界或路径差异。 |
| 分层准备 | 接口、数据、合约、Agent、权限、原型实现、测试分别接收不同业务输入。 |
| 先边界后方案 | 先明确不得做什么，再进入后续技术方案。 |
| 发现缺口回流 waterfall | 技术设计发现业务缺口，必须回到需求文档补充。 |
| 保持 DeFi 业务定位 | 智能合约是资金执行约束层，不是页面、数据库或 Agent。 |
| 保持 Agent 业务定位 | Agent 是角色身边业务助理，不是事实制造者或高风险裁决者。 |
| 保持控制塔主线 | 技术设计必须支撑源头到终端、节点状态、卡点和资金结果。 |

## 4. 技术设计输入总览

| 技术设计方向 | 接收的业务输入 | 设计前必须保持 | 当前不得定义 |
|---|---|---|---|
| 接口设计 | 业务动作、业务事件、角色权限、状态和错误解释。 | 业务动作责任方和业务结果。 | endpoint、请求响应、鉴权实现。 |
| 数据设计 | 信息域、事实来源、审计记录、可见范围。 | 事实来源独立和关键记录可追踪。 | 表结构、字段、索引、存储方案。 |
| 合约设计 | DeFi 输出、资金执行条件、规则阻断、暂停、资金池状态。 | 合约只做资金执行约束。 | Solidity、ABI、部署、Gas。 |
| Agent 设计 | 三种模式、摘要、缺口、待办、建议、升级边界。 | Agent 权限继承所属角色。 | 模型、提示词、工具调用实现。 |
| 权限设计 | 单钱包单角色、可见范围、不可做事项、越权阻断。 | 角色边界不可扩大。 | 鉴权中间件、权限表。 |
| 原型实现准备 | 控制塔表达、角色视角、路径差异、异常说明。 | 不继承旧页面和旧代码口径。 | 页面路由、组件、前端状态。 |
| 测试设计 | 正常路径、异常路径、越权路径、收口门禁。 | 测试业务闭环而非只测点击。 | 测试框架、测试代码。 |

## 5. 接口设计准备

接口设计准备阶段只接收业务动作和业务结果，不定义接口实现。

| 业务输入 | 接口设计应理解 | 当前不得定义 |
|---|---|---|
| 创建融资/结算意图 | 出口商发起业务案件。 | endpoint 名称。 |
| Agent 整理材料 | Agent 生成摘要、缺口、待办和建议。 | Agent 调用接口。 |
| 补材料 | 责任角色补充缺口材料或说明。 | 请求体结构。 |
| 确认事实 | 质检、物流、通关、保险、买方确认各自事实。 | 文件上传接口。 |
| 资金方授权 | 银行辅助路径中资金方给出授权、拒绝或补材料要求。 | 审批接口实现。 |
| DeFi 执行检查 | 检查规则、资金池、授权、事实和保证条件。 | 合约调用方式。 |
| 暂停/恢复 | 平台方对规则、资金池、路径或案件暂停和恢复。 | 管理接口实现。 |
| 异常升级 | dispute、高风险、越权、执行失败进入人工。 | 工单接口实现。 |

接口设计必须保留的业务输出：

| 输出 | 业务含义 |
|---|---|
| 成功 | 业务动作完成。 |
| 等待 | 仍缺事实、授权、规则、资金池或保证条件。 |
| 需补材料 | 某责任角色需要补充材料或说明。 |
| 规则阻断 | 当前规则不允许继续。 |
| 权限不足 | 当前角色不能查看或处理。 |
| 进入人工 | 争议、高风险、越权或置信不足。 |
| 执行失败 | 资金动作未完成，需要处理。 |

## 6. 数据设计准备

数据设计准备阶段只接收业务信息域和记录责任，不定义数据库实现。

| 信息域 | 业务含义 | 数据设计必须保持 |
|---|---|---|
| 案件 | 融资/结算案件、路径、状态、金额、责任角色。 | 案件相关可见。 |
| 贸易材料 | 合同、发票、装箱单、贸易背景、材料缺口。 | 材料来源可追踪。 |
| 事实节点 | 质检、物流、通关、保险、买方等事实。 | 事实来源独立。 |
| 资金方决策 | 授权、拒绝、补材料、人工复核。 | 不替代事实来源。 |
| Agent 输出 | 摘要、缺口、待办、建议、提醒、升级原因。 | 不等于事实结论。 |
| DeFi 执行结果 | 允许、等待、阻断、暂停、失败、完成。 | 不等于贸易真实性判断。 |
| 异常记录 | dispute、规则阻断、资金池不足、执行失败。 | 有责任方和下一步。 |
| 审计记录 | 关键动作、授权、事实确认、暂停、恢复、执行结果。 | 可追踪、不可随意删除。 |

数据设计不得提前定义：

| 不得定义 | 原因 |
|---|---|
| 表结构和字段 | 当前不是数据模型设计阶段。 |
| 存储方案 | 当前只准备业务信息域。 |
| 索引和性能方案 | 当前不做技术优化。 |
| 链上/链下拆分 | 当前只定义业务记录责任。 |
| 文件存储结构 | 当前只定义材料业务含义。 |

## 7. 合约设计准备

合约设计准备阶段只接收 DeFi 业务输出和资金执行约束，不写合约实现。

| 业务输入 | 合约设计应理解 | 当前不得定义 |
|---|---|---|
| 放款/结算/锁定/释放意图 | 资金动作需要被执行或阻断。 | 函数签名。 |
| 执行前条件 | 事实、授权、规则、资金池、暂停、保证条件。 | Solidity 逻辑。 |
| 规则版本 | 当前业务规则是否允许继续。 | 规则存储方式。 |
| 资金池状态 | 可用、不足、暂停、币种不支持。 | 资金池合约结构。 |
| 平台暂停 | 平台、路径、规则、资金池或案件暂停。 | 管理权限实现。 |
| 执行结果 | 允许、等待、阻断、失败、完成。 | 事件编码。 |

合约设计必须保持：

| 边界 | 说明 |
|---|---|
| 不判断贸易真实性 | 贸易真实性来自材料和事实方。 |
| 不替资金方审批 | 银行辅助路径保留资金方授权。 |
| 不替质检方出结论 | 质检事实来自质检方。 |
| 不替 Agent 解释材料 | Agent 负责整理和建议。 |
| 不绕过平台暂停 | 暂停是资金执行约束。 |
| 不自动裁决重大争议 | 重大争议进入人工或后续规则。 |

## 8. Agent 设计准备

Agent 设计准备阶段只接收 Agent 业务模式和边界，不定义模型或工具实现。

| 业务输入 | Agent 设计应理解 | 当前不得定义 |
|---|---|---|
| 全人工确认 | Agent 只整理、提醒和生成待办。 | 模型策略。 |
| 半自动托管 | Agent 在授权范围内推进低风险流程。 | 自动化执行框架。 |
| 全自动 Agent | Agent 在规则明确、事实完整、低风险场景自动推进。 | 工具调用实现。 |
| 摘要 | 来自材料和事实。 | 提示词模板。 |
| 缺口 | 缺什么、为什么缺、谁负责。 | 缺口算法。 |
| 待办 | 责任角色、动作和完成标准。 | 任务系统实现。 |
| 建议 | 业务建议，不替代高风险决策。 | 决策模型。 |
| 升级 | 争议、高风险、越权、置信不足、执行失败。 | 阈值和模型评分。 |

Agent 设计必须保持：

| 边界 | 说明 |
|---|---|
| 权限继承所属角色 | Agent 不能看角色不可见内容。 |
| 不伪造事实 | 事实来自业务角色。 |
| 不绕过规则 | 规则阻断必须尊重。 |
| 不替资金方高风险放款 | 高风险决策交给资金方或人工。 |
| 不裁决重大争议 | 重大争议进入人工或后续规则。 |

## 9. 权限设计准备

权限设计准备阶段接收角色边界和可见范围，不定义鉴权实现。

| 角色 | 权限设计应保持 | 不得放开 |
|---|---|---|
| 出口商 / 资金需求方 | 只能看自己的案件、材料、事实进度和资金结果。 | 其他出口商案件。 |
| 资金方 / 银行 / 保理机构 | 只能看相关案件、风险摘要、授权和执行前条件。 | 无关案件和事实篡改。 |
| 质检方 | 只能看自己的质检任务和必要上下文。 | 融资审批和其他事实。 |
| 物流方 | 只能看自己的运输节点和异常说明。 | 贸易材料修改。 |
| 海关 / 报关方 | 只能看自己的通关任务和监管状态。 | 资金审批。 |
| 保险公司 | 只能看核保/理赔相关风险摘要。 | 融资审批和质检结论修改。 |
| 买方 / 进口商 | 只能看相关订单、收货、争议和付款责任。 | 出口商融资申请修改。 |
| 平台方 / 合约运营 | 只能运营规则、资金池、暂停和执行异常。 | 伪造贸易或事实方结论。 |
| Agent | 继承所属角色权限。 | 独立越权访问。 |

权限设计阶段如果发现角色冲突，必须回流权限矩阵和角色需求文档。

## 10. 原型实现准备

原型实现准备接收业务表达，不开始页面或组件实现。

| 输入 | 必须表达 | 当前不得做 |
|---|---|---|
| 控制塔 | 源头到终端、节点状态、卡点、资金结果。 | 页面路由设计。 |
| 角色视角 | 每个角色看到不同范围。 | 组件拆分。 |
| 路径差异 | 银行辅助、纯 DeFi、低人工、人工升级。 | 前端状态设计。 |
| Agent 输出 | 摘要、缺口、待办、建议、升级。 | Agent UI 组件实现。 |
| DeFi 输出 | 允许、等待、阻断、资金池不足、暂停、失败、完成。 | 合约调用实现。 |
| 异常回流 | 原因、责任方、解除条件、下一步。 | 交互实现。 |
| 后置清单 | P1/P2 和技术细节不进入 P0。 | 旧页面复用。 |

原型实现准备必须避免：

| 不得做 | 原因 |
|---|---|
| 做登录前展示页 | 当前从注册后业务工作区开始。 |
| 继承旧版页面口径 | 当前分支是全新瀑布重建设计分支。 |
| 先画组件再讲业务 | 当前以控制塔和业务闭环为主。 |
| 先做 hash / proof 页面 | 当前不进入底层证明产品主流程。 |

## 11. 测试设计准备

测试设计准备接收业务验收口径，不写测试代码。

| 测试方向 | 必须覆盖的业务问题 | 当前不得定义 |
|---|---|---|
| 正常闭环 | 出口商意图、Agent 整理、事实确认、资金方/规则、DeFi 执行、资金结果。 | 测试框架。 |
| 银行辅助 | 资金方审查、补材料、授权、合约检查。 | API 测试代码。 |
| 纯 DeFi | 无银行审批、规则和资金池驱动执行。 | 链上测试脚本。 |
| 低人工 | Agent 自动推进低风险流程，异常升级人工。 | 自动化工具。 |
| 异常 | dispute、规则阻断、资金池不足、暂停、执行失败。 | 断言格式。 |
| 权限 | 无关案件不可见、越权动作阻断。 | 鉴权测试实现。 |
| 后置 | P1/P2 和技术内容不进入 P0。 | 后续版本测试计划。 |

## 12. 技术设计回流机制

| 发现的问题 | 回流文档 | 说明 |
|---|---|---|
| 业务动作缺责任角色 | 业务动作清单、权限矩阵。 | 先明确谁能做。 |
| 事件缺业务含义 | 业务事件清单。 | 先明确发生了什么。 |
| 状态缺解释 | 业务规则清单、控制塔要求。 | 先明确用户如何理解。 |
| 异常缺下一步 | 业务规则清单、异常回流说明。 | 先明确谁处理。 |
| 角色可见范围冲突 | 权限矩阵、角色需求文档。 | 先明确可见边界。 |
| 合约输出不够 | 智能合约业务角色、DeFi 规则清单。 | 先明确业务输出。 |
| Agent 权限不够或过大 | Agent Operating Model、权限矩阵。 | 先明确自动化边界。 |
| P1/P2 被技术需要提前 | MVP 范围和后置清单。 | 先重新评估范围。 |

| 钱包签名流程 | 后续钱包实现阶段处理。 |
| hash / proof 产品页 | 后续证明产品阶段处理。 |

## 16. 本步结论

P0 技术设计准备的本质不是“开始技术方案设计”。

它的本质是：

> 把 ChainTrace P0 已冻结的业务原型口径整理成技术设计可接收的输入目录，并在接口、数据、合约、Agent、权限、原型实现和测试设计之前，明确必须继承的业务边界、不得提前定义的技术细节，以及发现需求缺口时的 waterfall 回流机制。

下一步应进入：

> P0 技术设计阶段边界说明，用于进一步定义后续技术设计能展开到什么深度、哪些内容必须等待正式实现阶段。
