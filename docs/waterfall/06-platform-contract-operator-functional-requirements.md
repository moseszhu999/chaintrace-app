# 06 平台方 / 合约运营功能需求

## 1. 定位

平台方 / 合约运营进入 ChainTrace，不是为了逐单替银行审核，也不是为了替出口商补材料。

平台方的核心定位是：

> 维护 ChainTrace 的业务秩序、资金池安全、规则版本、合约状态和异常处理能力。

平台方不是融资案件的默认审批员。

正常案件应由出口商、质检方、物流方、资金方、保险方等角色协同完成；Agent 负责常规推进；规则和合约负责约束执行。平台方主要处理系统级风险、规则异常、资金池异常、合约异常、争议升级和运营暂停/恢复。

## 2. 进入条件

平台方进入本工作区前，必须已经完成：

| 条件 | 要求 |
|---|---|
| 钱包已连接 | 用户证明自己控制该钱包。 |
| 钱包已注册 | 该钱包已经完成 ChainTrace 注册。 |
| 角色已锁定 | 该钱包永久绑定为平台方 / 合约运营。 |
| 账户已激活 | 可以查看平台运行状态。 |
| 操作权限已分配 | 查看、维护、暂停、恢复、升级等权限必须明确。 |

不处理登录前展示页。
不处理未注册用户的营销转化页面。

## 3. 平台方最想完成的业务结果

| 优先级 | 业务结果 | 说明 |
|---|---|---|
| P0 | 知道资金池是否安全 | 资金池余额、占用、待放款、异常冻结必须清楚。 |
| P0 | 知道合约是否正常 | 合约状态、暂停状态、执行失败、规则版本必须清楚。 |
| P0 | 知道哪些案件异常 | 平台只关注异常案件，不逐单干预正常案件。 |
| P0 | 知道当前执行哪套规则 | 规则版本、适用范围、是否暂停必须明确。 |
| P0 | 能暂停 / 恢复 / 维护 | 在风险扩大前采取系统级动作。 |
| P1 | 知道 Agent 自动化是否正常 | Agent 是否误判、漏报、堆积任务。 |
| P1 | 形成平台级审计记录 | 谁在什么时候暂停、恢复、升级、修改规则，都要留下记录。 |

## 4. 平台方核心功能模块

### 4.1 平台运行总览

平台方登录后第一眼不应该看案件列表，而应该看运行状态。

功能要求：

| 总览项 | 说明 |
|---|---|
| 资金池状态 | 正常、余额不足、暂停、异常冻结。 |
| 合约状态 | 正常、暂停、升级中、执行异常。 |
| 规则状态 | 当前版本、待升级版本、已暂停规则。 |
| 案件异常数量 | 缺材料、质检争议、资金池不足、规则阻断、执行失败。 |
| Agent 状态 | 正常、异常堆积、自动化失败、等待人工。 |
| 高风险提醒 | 涉及高金额、高风险国家、重复异常、疑似欺诈等。 |
| 运营待办 | 需要平台处理的暂停、恢复、升级、异常复核。 |

平台方需要看到的是：

```text
现在系统是否安全？
哪里正在出问题？
我是否需要马上介入？
```

### 4.2 资金池监控

平台方要维护资金池安全，但不等于替资金方放款。

功能要求：

| 功能 | 说明 |
|---|---|
| 查看资金池余额 | 当前可用余额、已占用金额、待放款金额。 |
| 查看资金池状态 | 正常、余额不足、暂停、异常。 |
| 查看待执行放款 | 哪些案件已批准但等待资金池。 |
| 查看资金池异常 | 余额异常、执行失败、冻结、重复请求。 |
| 暂停资金池 | 出现系统级风险时暂停放款执行。 |
| 恢复资金池 | 异常解除后恢复执行。 |
| 查看资金池操作记录 | 谁暂停、谁恢复、原因是什么。 |

资金池状态应使用业务语言：

| 状态 | 平台方理解 |
|---|---|
| Normal | 资金池正常。 |
| Low balance | 可用资金不足，部分案件无法执行。 |
| Paused | 平台或规则已暂停资金池。 |
| Frozen | 出现异常，资金池被冻结。 |
| Execution backlog | 已批准案件堆积，等待执行。 |
| Failed execution | 放款执行失败，需要运营处理。 |

### 4.3 合约状态监控

平台方需要知道智能合约是否处于可执行状态。

功能要求：

| 功能 | 说明 |
|---|---|
| 查看合约状态 | 正常、暂停、升级中、异常。 |
| 查看合约版本 | 当前版本、部署时间、适用范围。 |
| 查看合约执行失败 | 哪些案件因为执行失败被阻断。 |
| 查看暂停原因 | 风险暂停、维护暂停、升级暂停、规则暂停。 |
| 暂停合约执行 | 系统级风险时暂停相关执行能力。 |
| 恢复合约执行 | 风险解除后恢复。 |
| 查看合约操作记录 | 暂停、恢复、升级、维护记录。 |

重要边界：

| 边界 | 说明 |
|---|---|
| 平台可以暂停执行 | 防止风险扩大。 |
| 平台不能伪造业务事实 | 不能替出口商、质检方、物流方造事实。 |
| 平台不能替资金方批准 | 放款决策仍属于资金方。 |
| 平台不能绕过规则放款 | 规则和合约是执行边界。 |

### 4.4 规则版本管理

ChainTrace 的规则必须版本化。平台不能让用户不知道系统按哪套规则执行。

功能要求：

| 功能 | 说明 |
|---|---|
| 查看当前规则版本 | 当前执行的规则版本。 |
| 查看规则适用范围 | 哪些国家、币种、品类、融资类型适用。 |
| 查看规则状态 | 生效、暂停、过期、待升级。 |
| 查看规则变更说明 | 为什么升级、影响什么。 |
| 暂停某条规则 | 风险出现时暂停某类案件。 |
| 恢复某条规则 | 风险解除后恢复。 |
| 查看规则影响案件 | 哪些案件被某个规则影响或阻断。 |

规则变更必须回答：

```text
改了什么？
为什么改？
影响哪些案件？
是否需要人工复核？
```

### 4.5 异常案件监控

平台不逐单审核正常案件，只看异常案件。

异常类型：

| 异常 | 说明 |
|---|---|
| 材料长期缺失 | 出口商未补材料，案件长期停滞。 |
| 质检 dispute | 质检存在争议，影响融资判断。 |
| 质检 fail | 货物不符合要求，可能阻断融资。 |
| 物流异常 | 延误、丢失、状态不一致。 |
| 买方争议 | 买方不确认收货或付款责任。 |
| 资金池不足 | 案件已批准但无法执行。 |
| 规则阻断 | 当前规则不允许继续。 |
| 合约执行失败 | 系统执行层出现失败。 |
| Agent 堆积 | 自动任务堆积或无法推进。 |
| 疑似欺诈 | 重复材料、金额异常、身份异常等。 |

平台方可执行动作：

| 动作 | 说明 |
|---|---|
| 查看异常原因 | 明确异常来自哪里。 |
| 分派处理方 | 出口商、质检方、物流方、资金方、保险方、平台运营。 |
| 暂停案件 | 防止异常案件继续推进。 |
| 恢复案件 | 异常解除后继续。 |
| 升级人工复核 | 高风险或争议案件进入人工处理。 |
| 标记风险类型 | 用于后续规则优化。 |

### 4.6 Agent 运营监控

ChainTrace 是 Agent Native，平台必须能看到 Agent 是否正常工作。

功能要求：

| 功能 | 说明 |
|---|---|
| 查看 Agent 任务数量 | 正常任务、待人工确认、失败任务、堆积任务。 |
| 查看 Agent 自动化结果 | 自动推进、补材料请求、风险提示、状态提醒。 |
| 查看 Agent 失败原因 | 无权限、缺材料、规则阻断、外部角色未响应等。 |
| 暂停某类 Agent 动作 | 如果发现误判，可以暂停相关自动化。 |
| 降级为人工确认 | 对高风险流程从自动切回人工。 |
| 查看 Agent 操作记录 | 记录 Agent 做了什么、为什么做。 |

平台要能回答：

```text
Agent 是否正常？
Agent 有没有误推进？
哪些事情需要人接管？
哪些自动化应该暂停？
```

### 4.7 平台级暂停、恢复、升级

平台方必须有系统级控制能力。

功能要求：

| 操作 | 说明 |
|---|---|
| 暂停某个资金池 | 资金池异常时阻断放款执行。 |
| 暂停某条规则 | 某类案件出现风险时暂停。 |
| 暂停某类案件 | 例如某国家、某品类、某融资类型。 |
| 暂停某个角色账户 | 用户身份异常、材料异常、疑似欺诈。 |
| 恢复暂停对象 | 风险解除后恢复。 |
| 升级规则版本 | 规则变更后切换版本。 |
| 标记维护状态 | 让其他角色知道当前系统状态。 |

每次操作必须要求填写业务原因。

```text
操作：暂停某资金池
原因：连续三笔放款执行失败，需排查资金池状态。
影响：已批准案件暂停执行，新增审批不受影响。
下一步：平台运营复核后恢复。
```

## 5. 平台方 Agent 模式差异

| 功能节点 | 全人工确认 | 半自动托管 | 全自动 Agent |
|---|---|---|---|
| 运行监控 | 用户主动查看 | Agent 汇总异常 | Agent 自动监控并推送异常 |
| 资金池异常 | 用户人工判断 | Agent 建议暂停/恢复 | 高危异常可自动建议暂停，但执行仍建议人工确认 |
| 合约异常 | 用户人工查看 | Agent 汇总失败原因 | Agent 自动聚合异常并触发告警 |
| 规则异常 | 用户人工发现 | Agent 标记影响案件 | Agent 自动生成规则影响报告 |
| 案件异常 | 用户筛选查看 | Agent 自动分组 | Agent 自动分派给对应角色 |
| Agent 失败 | 用户查看日志 | Agent 汇总原因 | Agent 自动降级为人工待办 |
| 平台暂停/恢复 | 必须人工确认 | 必须人工确认 | 高风险动作仍必须人工确认 |

重要边界：

即使是全自动 Agent 模式，以下平台动作也必须人工确认：

| 动作 | 原因 |
|---|---|
| 暂停资金池 | 影响放款和资金使用。 |
| 恢复资金池 | 可能重新释放风险。 |
| 暂停合约执行 | 系统级影响大。 |
| 升级规则版本 | 影响大量案件。 |
| 暂停角色账户 | 涉及用户业务权益。 |
| 标记欺诈 | 影响严重，必须人审。 |

## 6. 权限边界

平台方可以：

| 权限 | 说明 |
|---|---|
| 查看平台运行状态 | 资金池、规则、合约、异常、Agent 状态。 |
| 查看异常案件 | 只围绕异常和风险处理。 |
| 暂停 / 恢复资金池 | 控制系统级风险。 |
| 暂停 / 恢复规则 | 管理规则适用。 |
| 暂停 / 恢复合约执行能力 | 维护系统安全。 |
| 分派异常处理 | 把异常交给对应角色。 |
| 查看操作记录 | 支持平台审计。 |

平台方不可以：

| 禁止事项 | 原因 |
|---|---|
| 修改自己的永久角色 | 钱包角色不可变。 |
| 替出口商伪造贸易材料 | 平台不是事实制造者。 |
| 替质检方出具 pass/fail/dispute | 质检事实必须来自质检角色。 |
| 替资金方批准放款 | 资金决策权属于资金方。 |
| 随意修改已提交案件事实 | 防止证据污染。 |
| 绕过规则强制放款 | 破坏系统可信边界。 |
| 删除关键操作记录 | 破坏审计能力。 |

## 8. 本步结论

平台方工作区的本质不是“管理员后台”。

它的本质是：

> 平台方维护资金池、合约、规则、异常和 Agent 自动化秩序；正常案件不逐单干预，异常和系统级风险才介入。
