# 52 P0 业务输入到技术设计域映射

## 1. 定位

本文件定义 ChainTrace P0 已冻结业务输入如何映射到后续技术设计域。

它不是接口规格，不是数据库模型，不是智能合约设计，不是 Agent 实现方案，不是权限表结构，不是原型页面说明，也不是测试代码清单。

它的作用是：

> 在进入具体技术设计域之前，先证明每个设计域都从已冻结业务需求出发，并且每个冻结业务口径都有明确承接位置。

本文件只做映射，不做实现。

## 2. 设计域范围

P0 技术设计阶段暂分为七个主要设计域：

| 设计域 | 设计目标 | 不进入内容 |
|---|---|---|
| 权限设计 | 角色可见范围、动作边界、越权阻断。 | 鉴权中间件、token、权限表。 |
| 业务动作 / 接口边界设计 | 业务动作、业务结果、失败解释。 | endpoint、请求响应、错误码。 |
| 数据信息域设计 | 信息域、事实来源、记录责任、可见范围。 | 表结构、字段、索引。 |
| 合约业务设计 | 合约业务角色、执行条件、业务输出、暂停边界。 | Solidity、ABI、部署。 |
| Agent 业务设计 | Agent 模式、可做/不可做、自动推进、人工升级。 | 模型、提示词、工具调用。 |
| 控制塔表达设计 | 源头到终端、节点状态、卡点、资金结果表达。 | 页面路由、UI 组件、样式。 |
| 业务验收设计 | 正常闭环、路径差异、异常、权限验收口径。 | 测试框架、测试代码、脚本。 |

## 3. 总体映射原则

| 原则 | 说明 |
|---|---|
| 一个业务输入至少映射一个设计域 | 不允许冻结需求没有后续承接。 |
| 一个核心业务输入通常映射多个设计域 | 例如“单钱包单角色”同时影响权限、接口、数据、Agent、测试。 |
| 映射不是实现 | 只说明哪个设计域要承接，不说明如何编码。 |
| 缺口必须回流 | 如果设计域发现业务输入不够清楚，回流 waterfall 文档。 |
| 后置项必须标记 | P1/P2 或实现细节不进入 P0。 |

## 6. 核心业务口径映射

### 6.1 产品定位映射

| 冻结口径 | 承接设计域 | 设计要求 |
|---|---|---|
| ChainTrace 是 Agent Native + Crypto Native 供应链金融前站。 | 全部设计域 | 不能改成普通银行后台、普通 SaaS 或 proof demo。 |
| 正常案件由 Agent + 规则尽量自动推进。 | Agent、合约、接口、测试、控制塔 | 必须保留低人工路径。 |
| 人主要处理授权、异常、争议、高风险判断。 | 权限、Agent、接口、测试 | 必须保留人工升级条件。 |
| 业务从融资结果开始，而不是从页面开始。 | 控制塔、接口、测试 | 必须表达能不能借、能借多少、是否到账、卡在哪里。 |

### 6.2 注册和身份映射

| 冻结口径 | 承接设计域 | 设计要求 |
|---|---|---|
| 钱包先登录。 | 权限、接口、数据、测试 | 钱包连接和业务注册要区分。 |
| 注册后账户激活。 | 权限、接口、数据、控制塔、测试 | 未注册钱包不能进入业务工作区。 |
| 一个钱包只能绑定一个角色。 | 权限、数据、Agent、测试 | 不能出现多角色随意切换。 |
| 角色一旦选择不可修改。 | 权限、数据、控制塔、测试 | 角色不可变必须可解释。 |
| Agent 模式可变。 | Agent、权限、数据、控制塔、测试 | 全人工、半自动、全自动可以调整。 |

### 6.3 出口商映射

| 冻结口径 | 承接设计域 | 设计要求 |
|---|---|---|
| 出口商核心目标是融资。 | 控制塔、接口、测试 | 不能只设计成上传文件。 |
| 创建融资意图。 | 接口、数据、Agent、测试 | 需要形成案件草稿和业务状态。 |
| 准备贸易证据。 | 数据、Agent、接口、控制塔、测试 | Agent 帮助整理材料和缺口。 |
| 提交融资案件。 | 接口、权限、数据、测试 | 提交前后状态和责任清楚。 |
| 查看能否融资和资金到账。 | 控制塔、接口、合约、测试 | 必须表达资金结果和卡点。 |

### 6.4 资金方映射

| 冻结口径 | 承接设计域 | 设计要求 |
|---|---|---|
| 资金方关注贷前、贷中、贷后。 | 接口、数据、控制塔、测试 | 不能只做审核列表。 |
| 贷前看真实性、完整性、风险。 | Agent、数据、接口、控制塔、测试 | Agent 摘要不等于事实。 |
| 贷中决定允许、拒绝、补材料。 | 权限、接口、数据、合约、测试 | 决策权属于资金方。 |
| 贷中还要检查执行条件。 | 合约、接口、控制塔、测试 | 批准不等于一定可放款。 |
| 贷后关注回款、逾期、争议。 | 数据、接口、Agent、控制塔、测试 | P0 可保留核心状态，复杂贷后可后置。 |

### 6.5 事实方映射

| 角色 | 冻结口径 | 承接设计域 | 设计要求 |
|---|---|---|---|
| 质检方 | 提供 pass/fail/dispute 货物事实。 | 权限、接口、数据、Agent、控制塔、测试 | 不决定放款。 |
| 物流方 | 提供运输事实。 | 权限、接口、数据、Agent、控制塔、测试 | 不参与金融审批。 |
| 海关/报关方 | 提供通关和监管事实。 | 权限、接口、数据、Agent、控制塔、测试 | P2 接入难度高，可后置。 |
| 保险方 | 提供核保、承保、理赔和风险缓释。 | 权限、接口、数据、Agent、控制塔、测试 | 不替资金方放款。 |
| 买方 | 确认订单、收货、争议和付款责任。 | 权限、接口、数据、Agent、控制塔、测试 | 不修改出口商申请。 |

### 6.6 平台和合约运营映射

| 冻结口径 | 承接设计域 | 设计要求 |
|---|---|---|
| 平台不是逐单审批员。 | 权限、控制塔、测试 | 正常案件不由平台逐单干预。 |
| 平台维护资金池、规则、合约、异常和 Agent 秩序。 | 权限、接口、数据、合约、Agent、控制塔、测试 | 平台操作必须有业务原因和记录。 |
| 平台可以暂停/恢复/维护/升级。 | 权限、接口、数据、合约、测试 | 高影响动作必须保留人工确认。 |
| 平台不能替资金方批准或伪造事实。 | 权限、测试 | 必须阻断越权。 |

### 6.7 Agent 映射

| 冻结口径 | 承接设计域 | 设计要求 |
|---|---|---|
| Agent 是每个角色身边的业务助理。 | 权限、Agent、数据、控制塔、测试 | Agent 继承角色权限。 |
| Agent 可以整理材料、摘要、缺口、待办、提醒、请求补充。 | Agent、接口、数据、控制塔、测试 | 输出必须是业务语言。 |
| Agent 可以自动推进低风险流程。 | Agent、接口、合约、测试 | 只在授权、事实完整、规则明确时推进。 |
| Agent 不伪造事实、不审批高风险、不裁决重大争议。 | Agent、权限、测试 | 必须有人工升级条件。 |

### 6.8 合约映射

| 冻结口径 | 承接设计域 | 设计要求 |
|---|---|---|
| 合约是业务规则和资金执行约束层。 | 合约、接口、数据、控制塔、测试 | 不能变成事实判断者。 |
| 合约检查放款条件、资金池、规则版本、暂停状态。 | 合约、接口、数据、测试 | 执行前条件必须清楚。 |
| 合约输出允许、不允许、等待、阻断、资金池不足、平台暂停、失败、完成。 | 合约、控制塔、接口、测试 | 输出必须能被业务角色理解。 |
| 合约不替资金方审批、不判断贸易真实性、不裁决争议。 | 合约、权限、测试 | 必须防止合约越界。 |

|---|---|
| 设计域发现新角色 | 回流角色功能需求文档或 MVP 范围文档。 |
| 设计域发现新业务动作 | 回流对应角色功能需求或业务动作清单。 |
| 设计域发现新状态 | 回流状态与业务规则文档。 |
| 设计域发现新异常 | 回流异常处理文档。 |
| 设计域发现权限不清 | 回流权限矩阵或角色功能文档。 |
| 设计域发现 Agent 自动化边界不清 | 回流 Agent Native 模型文档。 |
| 设计域发现合约输出不足 | 回流合约业务角色文档。 |
| 设计域发现 P1/P2 被提前塞入 P0 | 回流 MVP 范围文档。 |
| 设计域需要 endpoint、表结构、Solidity、组件或测试代码 | 标记为后置，不在当前阶段处理。 |

## 10. 映射通过标准

| 检查项 | 通过标准 |
|---|---|
| 输入完整 | 所有冻结输入文档都有承接设计域。 |
| 设计域完整 | 七个设计域都有明确输入来源。 |
| 角色完整 | P0 角色全部进入权限、接口、数据、Agent、控制塔和验收。 |
| 路径完整 | 银行辅助、纯 DeFi、低人工、人工升级都有设计承接。 |
| 边界清楚 | Agent、合约、平台、事实方边界没有混淆。 |
| 后置清楚 | P1/P2 和实现细节没有进入 P0 设计实现。 |
| 回流清楚 | 发现缺口时知道回流到哪个文档。 |

## 11. 不得通过的映射情况

| 不得通过 | 原因 |
|---|---|
| 只映射到页面，不映射到业务结果 | 破坏业务优先。 |
| 只映射到接口，不映射到权限和数据责任 | 会导致越权和事实责任不清。 |
| 只映射银行路径，不映射纯 DeFi 路径 | 破坏双路径定位。 |
| Agent 输入只映射到自动化，不映射到边界和升级 | 容易导致 Agent 越权。 |
| 合约输入只映射到执行，不映射到不负责事项 | 容易导致合约越界。 |
| P1/P2 输入被当成 P0 必做 | 破坏 MVP 范围。 |
| 发现缺口但没有回流目标 | 破坏瀑布模式。 |

## 12. 本步结论

P0 业务输入到技术设计域映射的本质不是“开始设计接口或数据库”。

它的本质是：

> 先把冻结业务需求分配给权限、接口边界、数据信息域、合约业务、Agent 业务、控制塔表达和业务验收七个设计域，确保后续每一份技术设计都有业务来源、角色边界、路径差异、Agent 边界、合约边界和回流机制。

下一步应进入：

> `53-p0-permission-design-boundary.md`，开始定义 P0 权限设计边界，但仍不进入鉴权实现、token/session、权限表或代码。
