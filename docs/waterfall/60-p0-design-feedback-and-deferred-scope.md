# 60 P0 技术设计回流与后置范围

## 1. 定位

本文件定义 ChainTrace P0 技术设计阶段的回流机制和后置范围控制。

它不是新功能清单，不是开发排期，不是实现计划，不是测试计划，也不是 P1/P2 需求展开文档。

它的作用是：

> 当 P0 技术设计阶段发现需求缺口、边界不清、路径冲突、角色权限不明、Agent 自动化风险、合约输出不足或 P1/P2 范围被提前引入时，明确应该回流到哪类 waterfall 文档，哪些内容必须后置，防止 P0 技术设计阶段失控。

本文件只定义回流和后置规则，不进入实现。

## 2. 核心原则

| 原则 | 说明 |
|---|---|
| 缺口不直接实现 | 技术设计发现缺口时，不能直接用接口、数据库或代码补洞。 |
| 先回流再设计 | 需求不清先回流 waterfall 文档，再继续技术设计。 |
| P0 范围不可膨胀 | P1/P2 和证明产品阶段内容不能被塞进 P0 必做。 |
| 实现细节必须后置 | endpoint、表结构、Solidity、ABI、UI 组件、测试代码全部后置。 |
| 角色边界优先 | 角色权限不清必须先回流权限和角色需求。 |
| 路径差异优先 | 银行辅助、纯 DeFi、低人工、人工升级不能被合并。 |
| Agent 安全优先 | Agent 越权、自动裁决、高风险自动放款必须回流和阻断。 |
| 合约边界优先 | 合约判断事实或替资金方审批必须回流和阻断。 |
| 记录可追踪 | 每个回流项和后置项都要记录原因、影响和处理状态。 |

## 3. 回流与后置的区别

| 类型 | 含义 | 处理方式 |
|---|---|---|
| 回流 | P0 已有范围内的业务口径不清，必须补充 waterfall 需求或边界。 | 回到对应文档补清楚，再继续设计。 |
| 后置 | 当前内容有价值，但不属于 P0 或不属于当前阶段。 | 记录到后置清单，进入 P1/P2 或实现阶段。 |
| 阻断 | 不解决就会破坏 P0 主线、权限、Agent、合约或路径边界。 | 暂停相关设计域，先回流。 |
| 放弃 | 与 ChainTrace 定位冲突或旧版 demo 口径回流。 | 明确不纳入当前分支。 |

## 4. 回流类型总览

| 回流类型 | 触发条件 | 回流目标 |
|---|---|---|
| 新角色回流 | 技术设计发现 P0 需要未定义角色。 | 角色功能需求 / MVP 范围文档。 |
| 新业务动作回流 | 出现未定义的关键业务动作。 | 业务动作与接口边界 / 对应角色需求。 |
| 新状态回流 | 当前状态无法表达业务结果。 | 控制塔表达 / 业务规则 / 验收设计。 |
| 新异常回流 | 出现未定义异常或卡点。 | 异常处理 / 控制塔表达 / 验收设计。 |
| 权限不清回流 | 角色可见范围或动作责任不明确。 | 权限设计边界 / 角色需求。 |
| 事实来源不清回流 | 不知道某事实由谁确认。 | 数据信息域 / 角色需求。 |
| Agent 边界回流 | Agent 是否可自动推进不清。 | Agent 业务设计。 |
| 合约输出回流 | 合约业务输出不足或含义不清。 | 合约业务设计。 |
| 路径冲突回流 | 银行辅助、纯 DeFi、低人工、人工升级发生冲突。 | MVP 范围 / 验收设计 / 场景路径文档。 |
| P0/P1/P2 范围回流 | 某能力是否属于 P0 不清。 | MVP 范围和角色优先级文档。 |

## 5. 新角色回流规则

### 5.1 触发条件

| 触发情况 | 示例 |
|---|---|
| P0 闭环无法缺少某角色 | 没有某事实方，资金方无法判断。 |
| 某动作责任方不属于现有角色 | 谁处理争议仲裁不清。 |
| 一个角色被迫承担多个冲突职责 | 平台既要维护规则又要替资金方审批。 |
| P1/P2 角色被技术设计提前变成 P0 必做 | 海关/报关被强制进入 P0 主链路。 |

### 5.2 回流处理

| 情况 | 处理方式 |
|---|---|
| P0 必须角色遗漏 | 回流角色功能需求和 MVP 范围。 |
| P1/P2 角色需要占位 | 记录占位，不进入 P0 完整实现。 |
| 角色职责冲突 | 回流权限和角色功能需求。 |
| 临时运营角色需要定义 | 回流平台运营或权限设计。 |

## 6. 新业务动作回流规则

### 6.1 触发条件

| 触发情况 | 示例 |
|---|---|
| 设计中出现未定义动作 | “重新分配资金方”“替换质检方”“冻结案件”。 |
| 动作没有责任角色 | 谁可以重启执行检查不清。 |
| 动作结果不清 | 执行失败后是重试、关闭还是人工处理不清。 |
| 动作越过角色边界 | Agent 想替资金方批准。 |

### 6.2 回流处理

| 情况 | 回流目标 |
|---|---|
| P0 核心动作遗漏 | 业务动作与接口边界。 |
| 某角色动作边界不清 | 对应角色功能需求 + 权限设计。 |
| 动作涉及合约输出 | 合约业务设计。 |
| 动作涉及 Agent 自动化 | Agent 业务设计。 |
| 动作属于实现细节 | 后置到接口规格或实现阶段。 |

## 7. 新状态回流规则

### 7.1 触发条件

| 触发情况 | 示例 |
|---|---|
| 当前状态无法表达业务结果 | “已批准但未到账”不能只写 Approved。 |
| 状态混淆责任方 | Waiting 不知道等谁。 |
| 状态混淆路径 | 银行辅助和纯 DeFi 都叫 Under review。 |
| 状态混淆异常 | 质检 dispute 和质检 fail 混成 Failed。 |

### 7.2 回流处理

| 状态问题 | 回流目标 |
|---|---|
| 资金结果表达不足 | 控制塔表达设计。 |
| 合约输出不足 | 合约业务设计。 |
| Agent 状态不足 | Agent 业务设计。 |
| 异常状态不足 | 异常处理和业务验收设计。 |
| 状态是否 P0 不清 | MVP 范围文档。 |

## 8. 新异常回流规则

### 8.1 触发条件

| 触发情况 | 示例 |
|---|---|
| 异常没有责任方 | 规则阻断后谁处理不清。 |
| 异常没有解除条件 | 资金池不足何时恢复不清。 |
| 异常没有下一步 | 质检 dispute 之后做什么不清。 |
| 异常被技术错误替代 | 只显示 validation failed。 |
| 异常影响多个角色 | 买方争议同时影响出口商、资金方、保险方。 |

### 8.2 回流处理

| 异常问题 | 回流目标 |
|---|---|
| 材料异常 | 出口商需求、Agent 设计、业务动作边界。 |
| 事实异常 | 对应事实方需求、数据信息域。 |
| 资金异常 | 资金方需求、合约业务设计、平台运营。 |
| 平台异常 | 平台运营需求、控制塔表达。 |
| Agent 异常 | Agent 业务设计。 |
| 验收异常 | 业务验收设计。 |

## 9. 权限不清回流规则

| 触发情况 | 回流目标 | 必须回答 |
|---|---|---|
| 某角色能否看某信息不清 | 权限设计边界。 | 该信息是否与案件相关，是否必要。 |
| 某角色能否做某动作不清 | 角色功能需求 + 权限设计。 | 该动作的责任角色是谁。 |
| Agent 是否能自动触发动作不清 | Agent 业务设计 + 权限设计。 | 所属角色是否有权，是否已授权。 |
| 平台是否能介入案件不清 | 平台运营需求。 | 是否系统级异常，是否逐单审批。 |
| 合约是否能阻断动作不清 | 合约业务设计。 | 是否执行条件不满足。 |

一票回流情况：

- 一个钱包多角色切换。
- Agent 拥有超级权限。
- 平台替资金方批准。
- 资金方修改出口商原始材料。
- 合约判断贸易真实性。

## 10. Agent 边界回流规则

| 触发情况 | 回流目标 | 处理原则 |
|---|---|---|
| Agent 想自动放款 | Agent 业务设计 / 资金方需求 / 合约设计。 | 高风险必须人工，低风险也需授权和规则。 |
| Agent 想自动裁决 dispute | Agent 业务设计 / 异常处理。 | dispute 必须人工或责任方处理。 |
| Agent 想伪造事实 | Agent 业务设计 / 数据信息域。 | 直接否决。 |
| Agent 自动化范围不清 | Agent 业务设计。 | 明确授权、低风险、事实完整、规则明确。 |
| Agent 输出是否可作为事实不清 | 数据信息域 / Agent 设计。 | 摘要和建议不等于事实。 |
| Agent 行为记录不清 | 数据信息域 / 审计要求。 | 自动推进和升级必须可追踪。 |

## 11. 合约边界回流规则

| 触发情况 | 回流目标 | 处理原则 |
|---|---|---|
| 合约被要求判断贸易真实性 | 合约业务设计 / 数据信息域。 | 直接否决，事实来自业务角色。 |
| 合约被要求替资金方审批 | 合约业务设计 / 资金方需求。 | 银行辅助路径下资金方决策不可替代。 |
| 合约输出不足 | 合约业务设计。 | 补充业务输出和解释。 |
| 暂停边界不清 | 平台运营需求 / 合约业务设计。 | 明确暂停对象、影响、恢复条件。 |
| 资金池状态不清 | 合约业务设计 / 平台运营。 | 明确可用、不足、暂停、异常。 |
| 纯 DeFi 路径条件不清 | MVP 范围 / 合约业务设计 / 验收设计。 | 明确事实条件和规则路径。 |

## 12. 路径差异回流规则

| 路径问题 | 回流目标 | 处理原则 |
|---|---|---|
| 所有路径变成银行审批 | MVP 范围 / 验收设计。 | 必须恢复纯 DeFi 和低人工路径。 |
| 所有路径变成自动 DeFi | 资金方需求 / 合约设计 / 验收设计。 | 银行辅助路径必须保留。 |
| 低人工路径没有人工升级 | Agent 设计 / 验收设计。 | 必须补升级条件。 |
| 人工升级路径没有责任方 | 权限设计 / 异常处理。 | 必须明确谁处理。 |
| 纯 DeFi 路径无视事实节点 | 合约设计 / 数据信息域。 | 必须恢复事实条件。 |

## 13. 后置范围总览

| 后置类型 | 后置到哪里 | 当前 P0 处理方式 |
|---|---|---|
| P1 角色完整能力 | P1 需求阶段 | P0 可占位，不做完整工作区。 |
| P2 外部高难接入 | P2 需求阶段 | P0 只说明价值和后置原因。 |
| 接口规格 | 后续接口规格阶段 | P0 只定义业务动作和结果。 |
| 数据库模型 | 后续数据模型阶段 | P0 只定义信息域和记录责任。 |
| 合约实现 | 后续合约实现阶段 | P0 只定义业务输出和约束。 |
| 钱包签名 | 钱包实现阶段 | P0 只定义身份和授权口径。 |
| UI 实现 | 原型/前端实现阶段 | P0 只定义业务表达。 |
| 测试代码 | 测试实现阶段 | P0 只定义业务验收口径。 |
| hash/proof 产品页 | 后续证明产品阶段 | P0 不作为主流程。 |

## 14. P1 后置范围

P1 是增强贸易事实、风险缓释和协同能力，不应压进 P0 主线。

| P1 内容 | 当前 P0 处理 | 后置原因 |
|---|---|---|
| 物流方完整工作区 | P0 可表达物流事实占位。 | P0 主闭环先靠质检 + 基础贸易材料。 |
| 保险核保/理赔完整流程 | P0 可表达保险状态占位。 | 完整承保和理赔复杂度高。 |
| 买方完整确认工作区 | P0 可表达买方确认占位。 | 买方接入和协同后置。 |
| 多资金方匹配 | P0 保留单资金方或基础匹配口径。 | 资金市场复杂度高。 |
| 复杂贷后跟踪 | P0 保留基础贷后状态。 | 催收、逾期、理赔联动后置。 |
| Agent 跨组织协商 | P0 保留请求补充和升级。 | 跨组织自动谈判风险高。 |

## 15. P2 后置范围

P2 是高价值但接入难度较高的外部数据和监管事实能力。

| P2 内容 | 当前 P0 处理 | 后置原因 |
|---|---|---|
| 海关/报关/通关数据接入 | P0 只做业务占位。 | 数据接入、授权、合规复杂。 |
| 第三方物流 API 深度接入 | P0 不做实时接入。 | 外部 API 和多国物流差异复杂。 |
| 银行/保理机构系统集成 | P0 不做深度系统集成。 | 金融机构接入周期长。 |
| 保险公司系统集成 | P0 不做自动承保系统。 | 核保规则和理赔系统复杂。 |
| 跨链/多链资金执行 | P0 不做复杂多链。 | 风险和实现复杂。 |
| 高级合规和监管报送 | P0 不做正式监管报送。 | 合规要求需单独设计。 |

## 17. 证明产品后置范围

ChainTrace 早期文档和旧 demo 中存在 hash/proof/签名/证明展示口径。当前 P0 主线已明确不是证明产品页。

| 内容 | 当前处理 |
|---|---|
| browser-local hash | 后续证明产品阶段。 |
| proof 展示页 | 后续证明产品阶段。 |
| wallet signature demo | 钱包实现或证明产品阶段。 |
| on-chain registry proof | 后续证明产品阶段。 |
| PDF hash/proof 组件 | 不进入 P0 主流程。 |
| 旧 dashboard proof narrative | 不回流到当前 clean waterfall 主线。 |

当前 P0 主线仍然是：

```text
贸易融资意图
→ 材料和事实整理
→ 资金方/DeFi 规则路径
→ 合约/规则执行检查
→ 资金结果和卡点表达
```

## 20. 阻断级问题处理

以下问题属于阻断级，不能继续强行推进技术设计。

| 阻断级问题 | 处理方式 |
|---|---|
| P0 主线无法闭环 | 回流 P0 PRD 和验收设计。 |
| 出口商看不到资金结果 | 回流控制塔表达和出口商需求。 |
| 单钱包单角色被破坏 | 回流注册登录和权限设计。 |
| 资金方决策权被 Agent 或平台替代 | 回流资金方需求、Agent 和平台边界。 |
| 合约判断贸易真实性 | 回流合约设计和数据信息域。 |
| 事实来源不清 | 回流数据信息域和角色需求。 |
| 路径差异消失 | 回流 MVP 范围和验收设计。 |
| 异常无责任方和下一步 | 回流异常处理和控制塔表达。 |
| P1/P2 范围压入 P0 | 回流 MVP 范围。 |

## 21. 回流完成标准

| 检查项 | 完成标准 |
|---|---|
| 问题有编号 | 能追踪。 |
| 来源阶段清楚 | 知道哪个设计域发现。 |
| 影响范围清楚 | 知道影响角色、路径、权限、Agent、合约或验收。 |
| 回流目标清楚 | 知道补哪份 waterfall 文档。 |
| 处理结论清楚 | 已解决、后置、放弃或继续阻断。 |
| 后续设计更新 | 回流解决后，相关设计域同步更新。 |

## 22. 后置完成标准

| 检查项 | 完成标准 |
|---|---|
| 后置原因清楚 | 为什么不属于 P0。 |
| 后置阶段清楚 | 属于 P1、P2、实现、测试或证明产品。 |
| P0 占位方式清楚 | P0 如何表达但不实现。 |
| 不影响 P0 通过 | 不做该项也能完成 P0 主闭环。 |
| 后续触发条件清楚 | 什么时候重新打开该项。 |

## 23. P0 通过前必须确认

| 确认项 | 必须结果 |
|---|---|
| 所有阻断回流 | 已解决或明确后置且不影响 P0。 |
| 所有 P1/P2 请求 | 已记录后置，不进入 P0 必做。 |
| 所有实现细节 | 已后置到对应阶段。 |
| 所有路径差异 | 银行辅助、纯 DeFi、低人工、人工升级均保留。 |
| 所有 Agent 边界 | Agent 不越权、不伪造事实、不裁决争议。 |
| 所有合约边界 | 合约不判断事实、不替资金方审批。 |
| 所有权限边界 | 单钱包单角色和案件相关可见保留。 |
| 所有异常 | 有原因、责任方、解除条件和下一步。 |

## 24. 一票否决项

出现以下任一情况，P0 回流与后置范围控制不得通过：

| 一票否决项 | 原因 |
|---|---|
| 发现缺口后直接进入代码实现 | 破坏瀑布模式。 |
| P1/P2 被塞入 P0 必做且无后置记录 | MVP 范围失控。 |
| endpoint、表结构、Solidity、ABI、UI 组件、测试代码进入当前阶段 | 阶段越界。 |
| 旧 proof/hash/demo 口径回流为 P0 主线 | 破坏 clean waterfall 主线。 |
| Agent 越权问题未回流 | 破坏 Agent Native 边界。 |
| 合约越界问题未回流 | 破坏合约业务角色。 |
| 权限不清问题未回流 | 破坏数据隔离和角色责任。 |
| 路径差异消失但未回流 | 破坏 DeFi/银行双路径。 |
| 异常没有责任方和下一步但继续推进 | 业务不可处理。 |

## 25. 本步结论

P0 技术设计回流与后置范围的本质不是“增加更多功能”。

它的本质是：

> 保护 ChainTrace P0 技术设计阶段不失控：发现需求缺口先回流 waterfall，发现 P1/P2 或实现细节先后置，发现角色、权限、Agent、合约、路径或异常边界问题先阻断并修正，确保 P0 始终围绕融资闭环、Agent Native、DeFi/银行双路径、控制塔表达和业务验收推进。
