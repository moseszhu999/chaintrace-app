# 68 P0 测试计划准备

## 1. 定位

本文件定义 ChainTrace P0 规格准备阶段中的测试计划准备边界。

它不是测试框架，不是测试代码，不是 API 自动化脚本，不是 E2E 实现，不是链上测试脚本，不是 CI/CD 配置，也不是测试数据 fixture。

它的作用是：

> 从 `59-p0-business-acceptance-design.md` 已定义的业务验收口径出发，建立“业务验收场景 → 测试计划候选”的准备框架，明确后续测试计划应该覆盖哪些 P0 业务闭环、角色边界、Agent 行为、合约输出、控制塔表达、异常处理和后置范围。

本文件只做测试计划准备，不写测试实现。

## 3. 测试计划准备原则

| 原则 | 说明 |
|---|---|
| 从业务验收派生 | 测试计划只从 59 的业务验收口径派生，不凭空造测试。 |
| 先测闭环 | 优先验证出口商融资意图到资金结果的业务闭环。 |
| 先测边界 | 权限、Agent、合约、平台、事实来源边界必须测试。 |
| 先测路径差异 | 银行辅助、纯 DeFi、低人工、人工升级必须分别测试。 |
| 先测异常可处理 | 卡点必须有原因、责任方、解除条件和下一步。 |
| 不写测试代码 | 当前不写测试框架、脚本、断言代码或 fixture。 |
| 不写 API 自动化 | 当前只准备业务测试场景，不写 HTTP 调用。 |
| 不写 E2E 实现 | 当前只定义端到端业务场景和观察点。 |
| 缺口必须回流 | 测试计划发现需求或边界缺口时，回流对应 waterfall 文档。 |

## 4. 测试计划候选分组

| 测试组 | 业务目的 | 主要覆盖 |
|---|---|---|
| 正常融资闭环测试候选 | 验证出口商从融资意图走到资金结果。 | 出口商、材料、Agent、质检、资金方/DeFi、合约、资金结果。 |
| 银行辅助路径测试候选 | 验证资金方审查、决策和执行检查。 | 资金方、证据、风险、补材料、批准、合约检查。 |
| 纯 DeFi 路径测试候选 | 验证规则、事实、资金池和执行结果。 | DeFi 规则、事实节点、资金池、平台暂停。 |
| 低人工路径测试候选 | 验证 Agent + 规则低风险推进。 | Agent 授权、自动推进、合约边界、留痕。 |
| 人工升级路径测试候选 | 验证高风险、争议、失败会升级。 | dispute、高风险、资金池不足、规则阻断、执行失败。 |
| 权限隔离测试候选 | 验证单钱包单角色和越权阻断。 | 出口商、资金方、质检方、平台、Agent。 |
| 事实来源测试候选 | 验证事实不能被伪造或越权修改。 | 材料、质检、物流、买方、保险、Agent、合约。 |
| 合约输出测试候选 | 验证合约输出业务可解释且不越界。 | Allowed、Waiting、Rule blocked、Pool insufficient、Completed。 |
| 控制塔表达测试候选 | 验证状态、卡点、责任方、资金结果可见。 | 角色工作区、主链路、异常解释。 |
| 后置范围测试候选 | 验证 P1/P2 和实现细节未混入 P0。 | 范围控制、一票否决。 |

| 元信息 | 说明 |
|---|---|

| 业务目的 | 为什么需要测试该场景。 |
| 涉及角色 | 出口商、资金方、质检方、平台、Agent、合约/规则等。 |
| 业务前提 | 账户、角色、案件、材料、事实、资金池、规则、暂停等前置条件。 |
| 触发动作 | 触发哪个业务动作，不写 API 调用。 |
| 期望业务结果 | 业务上应看到什么结果。 |
| 观察点 | 在控制塔、角色工作区、Agent 输出、合约输出、异常卡点中观察什么。 |
| 权限断言 | 哪些角色可见，哪些角色不可见。 |
| 异常处理 | 失败时原因、责任方、解除条件和下一步。 |
| 回流条件 | 若无法测试，回流到哪个设计文档。 |
| 后置实现项 | 测试框架、脚本、fixture、自动化全部后置。 |

## 6. 正常融资闭环测试候选

### 6.1 测试目的

验证 P0 最核心价值：出口商能从融资意图走到资金结果；即使无法放款，也能看到明确卡点。

### 6.2 场景步骤

| 步骤 | 业务动作 | 期望结果 |
|---:|---|---|
| 1 | 出口商注册并进入工作区。 | 角色为出口商，只看到自己的案件。 |
| 2 | 出口商创建融资意图。 | Financing Case 草稿形成。 |
| 3 | 出口商提交合同、订单、发票、装箱单。 | 材料状态可表达完整性。 |
| 4 | Agent 生成案件摘要和材料缺口。 | 摘要可读，缺口有责任方和下一步。 |
| 5 | 质检方提交 Pass 结论。 | 质检事实状态为 Pass。 |
| 6 | 资金方或 DeFi 路径进入判断。 | 进入审查或规则路径。 |
| 7 | 合约/规则输出执行结果。 | Allowed / Waiting / Blocked / Failed / Completed 可解释。 |
| 8 | 出口商查看资金结果。 | 能看到能不能借、能借多少、是否到账或卡在哪里。 |

### 6.3 通过标准

| 检查项 | 通过标准 |
|---|---|
| 主线连续 | 案件从融资意图到资金结果可追踪。 |
| Agent 有价值 | 摘要、缺口、待办能帮助出口商推进。 |
| 事实来源清楚 | 质检结论来自质检方。 |
| 资金结果清楚 | 出口商知道结果和卡点。 |
| 异常可处理 | 任何等待或阻断都有责任方和下一步。 |

## 7. 银行辅助路径测试候选

### 7.1 测试目的

验证资金方能审查案件、要求补材料、允许/拒绝/降额/人工复核，并且批准后仍受合约、规则、资金池和平台暂停约束。

| 场景 | 前提 | 期望结果 |
|---|---|---|
| 资金方查看待审案件 | 案件已分配给资金方。 | 资金方可见案件摘要和证据完整性。 |
| 资金方要求补材料 | 材料缺口存在。 | 出口商收到明确补充请求。 |
| 资金方允许融资 | 证据足够且风险可接受。 | 决策为允许，但不等于到账。 |
| 资金方拒绝融资 | 风险不可接受。 | 出口商看到可理解拒绝原因。 |
| 资金方降额 | 金额或风险需要调整。 | 形成降额结果或待确认状态。 |
| 资金方人工复核 | 高风险或 dispute。 | 自动推进停止，进入人工。 |
| 资金方批准后资金池不足 | 资金池不可用。 | 合约输出 Pool insufficient。 |

一票失败：

```text
资金方批准直接等于到账，且不经过合约/规则/资金池检查。
```

## 8. 纯 DeFi 路径测试候选

### 8.1 测试目的

验证系统不是只能走银行审批；纯 DeFi 路径必须在事实、规则、资金池、暂停状态满足条件时形成执行结果。

| 场景 | 前提 | 期望结果 |
|---|---|---|
| DeFi 路径条件满足 | 事实完整、规则有效、资金池可用、平台未暂停。 | Allowed / Completed。 |
| 事实不足 | 缺少必要事实。 | Need materials / Waiting。 |
| 质检 dispute | 存在争议。 | Dispute / Need review。 |
| 规则暂停 | 规则不可用。 | Rule blocked / Platform paused。 |
| 资金池不足 | 资金池余额或币种不满足。 | Pool insufficient。 |
| 平台暂停 | 案件、资金池、规则或执行能力暂停。 | Platform paused。 |

一票失败：

```text
纯 DeFi 路径无视事实节点、规则状态或资金池状态直接完成。
```

## 9. 低人工路径测试候选

### 9.1 测试目的

验证 Agent 在授权、低风险、事实完整、规则明确且平台未暂停时，可以自动推进常规流程；异常时必须停止。

| 场景 | 前提 | 期望结果 |
|---|---|---|
| 半自动 Agent 摘要案件 | 用户授权半自动。 | Agent 自动摘要并留痕。 |
| Agent 自动生成缺口 | 材料缺口明确。 | 生成待办并指向责任方。 |
| Agent 请求补充材料 | 责任方明确。 | 请求发出，不能伪造材料。 |
| Agent 推进低风险流程 | 授权、低风险、事实完整。 | 推进记录可追踪。 |
| Agent 遇到平台暂停 | 平台暂停生效。 | 停止推进并解释暂停原因。 |
| Agent 遇到合约阻断 | Rule blocked 或 Pool insufficient。 | 不绕过，生成待办或升级。 |

通过标准：

```text
Agent 自动化必须同时满足授权、权限、低风险、事实完整、规则明确、未暂停、可追踪。
```

## 10. 人工升级路径测试候选

### 10.1 测试目的

验证系统遇到高风险、争议、失败、权限不清或置信不足时，不会继续自动推进，而是升级给正确责任方。

| 触发条件 | 期望升级对象 | 期望表达 |
|---|---|---|
| 高金额案件 | 资金方。 | 需要资金方人工复核。 |
| 高风险国家/品类/买方 | 资金方 / 平台。 | 风险高，停止自动推进。 |
| 质检 Fail | 资金方 / 质检方。 | 事实不支持自动执行。 |
| 质检 Dispute | 质检方 / 出口商 / 资金方。 | 争议原因、责任方、解除条件。 |
| 金额不一致 | 出口商 / 资金方。 | 冲突位置和解释要求。 |
| 资金池不足 | 平台 / 资金方。 | Pool insufficient 和下一步。 |
| 规则阻断 | 平台。 | Rule blocked 和规则原因。 |
| 执行失败 | 平台 / 执行责任方。 | Execution failed 和处理路径。 |
| Agent 置信不足 | 所属角色。 | 需要人工确认。 |

一票失败：

```text
Agent 自动裁决 dispute 或自动批准高风险融资。
```

## 11. 权限隔离测试候选

| 测试场景 | 期望结果 |
|---|---|
| 未注册钱包访问业务工作区 | 被阻断，提示注册。 |
| 已注册出口商访问资金方工作区 | 被阻断，角色不匹配。 |
| 出口商查看其他出口商案件 | 被阻断。 |
| 资金方查看分配给自己的案件 | 允许。 |
| 资金方修改出口商原始材料 | 被阻断。 |
| 质检方查看自己的质检任务 | 允许。 |
| 质检方决定放款 | 被阻断。 |
| 平台暂停资金池 | 允许，但必须有原因和记录。 |
| 平台替资金方批准放款 | 被阻断。 |
| Agent 执行所属角色无权动作 | 被阻断。 |

通过标准：

```text
每个越权阻断都必须有业务解释，不只是技术错误。
```

## 12. 事实来源测试候选

| 事实对象 | 正确来源 | 测试断言 |
|---|---|---|
| 贸易材料 | 出口商。 | 资金方、平台、Agent 不能无痕修改。 |
| 质检结论 | 质检方。 | Agent、资金方、平台不能生成 Pass/Fail/Dispute。 |
| 物流状态 | 物流方或可接受事实来源。 | P0 占位，不允许资金方伪造。 |
| 买方确认 | 买方。 | 出口商不能替买方确认付款责任。 |
| 保险结论 | 保险方。 | 资金方不能冒充保险方承保。 |
| 合约输出 | 合约/规则执行层。 | 输出执行状态，不等于贸易真实性。 |
| Agent 摘要 | Agent。 | 摘要不等于事实。 |

一票失败：

```text
任何设计允许 Agent、资金方或平台伪造事实来源。
```

## 13. 合约业务输出测试候选

| 场景 | 期望输出 | 业务解释 |
|---|---|---|
| 条件满足 | Allowed / Completed | 可以进入执行或执行完成。 |
| 条件未满足 | Waiting | 等待事实、资金方、规则或资金池。 |
| 缺材料 | Need materials | 出口商或事实方补充。 |
| 需要人工 | Need review | 资金方、平台或责任方处理。 |
| 规则阻断 | Rule blocked | 当前规则不允许继续。 |
| 资金池不足 | Pool insufficient | 批准不等于到账。 |
| 平台暂停 | Platform paused | 暂停对象、原因和恢复条件可见。 |
| dispute | Dispute | 进入人工争议处理。 |
| 执行失败 | Execution failed | 平台或执行责任方处理。 |

通过标准：

- 合约输出必须能被控制塔解释。
- 合约不能输出“贸易真实/虚假”。
- 合约不能替资金方审批。

## 14. 控制塔表达测试候选

| 角色视角 | 必须看到 | 不应看到 |
|---|---|---|
| 出口商 | 资金结果、卡点、补材料、合约/规则状态。 | 资金方内部风险模型、其他出口商案件。 |
| 资金方 | 证据完整性、风险建议、决策动作、执行条件。 | 其他资金方案件、修改原始材料能力。 |
| 质检方 | 质检任务、清单、结论、补充请求。 | 资金池内部状态、放款决策权。 |
| 平台 | 资金池、规则、合约、异常、Agent 状态。 | 伪造事实或替资金方审批。 |

控制塔通过标准：

```text
每个卡点都必须有原因、责任方、影响范围、解除条件和下一步。
```

## 15. 异常处理测试候选

| 异常 | 期望测试结果 |
|---|---|
| 缺材料 | 说明缺什么、谁补、补完后如何继续。 |
| 金额不一致 | 说明冲突点、责任方、解释或人工复核路径。 |
| 质检 Fail | 表达拒绝、复检、人工复核或补充说明路径。 |
| 质检 Dispute | 表达争议原因、责任方、解除条件。 |
| 物流异常 | P1 占位表达，不进入完整物流工作区。 |
| 买方争议 | P1 占位表达，不能自动裁决。 |
| 保险不足 | P1 占位表达，不能自动承保。 |
| 资金池不足 | 表达等待、恢复、切换路径或人工处理。 |
| 规则阻断 | 表达规则原因和平台处理路径。 |
| 平台暂停 | 表达暂停对象、原因、影响、恢复条件。 |
| 执行失败 | 表达失败原因、责任方、下一步。 |
| Agent 置信不足 | 升级给所属角色。 |

一票失败：

```text
异常只显示技术失败，没有责任方和下一步。
```

## 18. 测试计划与原型范围关系

## 22. 一票否决项

| 一票否决项 | 原因 |
|---|---|
| 测试计划不覆盖出口商资金结果 | P0 核心价值无法验收。 |
| 测试计划只测银行审批，不测 DeFi 路径 | 路径差异失败。 |
| 测试计划只测 happy path，不测异常和人工升级 | 风险控制失败。 |
| 测试计划不测权限隔离 | 单钱包单角色基础失败。 |
| 测试计划允许 Agent 越权、伪造事实或裁决 dispute | Agent 边界失败。 |
| 测试计划允许合约判断贸易真实性或替资金方审批 | 合约边界失败。 |
| 测试计划不验证卡点责任方和下一步 | 业务不可处理。 |
| P1/P2 被塞入 P0 必测范围 | MVP 范围失控。 |
| 旧 proof/hash demo 被作为 P0 主线测试 | 产品定位倒退。 |
| 直接写测试框架、测试代码、API 脚本或 E2E 实现 | 阶段越界。 |

## 24. 本步结论

P0 测试计划准备的本质不是“开始写测试代码”。

它的本质是：

> 从 P0 业务验收口径出发，准备后续测试计划必须覆盖的正常闭环、银行辅助、纯 DeFi、低人工、人工升级、权限隔离、事实来源、合约输出、控制塔表达、异常处理和后置范围，同时继续防止当前阶段提前进入测试框架、测试代码、API 自动化脚本、E2E 实现或链上测试脚本。

> `69-p0-implementation-entry-checklist.md`

即形成 P0 从规格准备进入实现阶段前的入口检查清单，判断接口、数据、合约、Agent、原型和测试计划准备是否足够进入受控实现。
