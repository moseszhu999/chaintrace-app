# 04 资金提供方 / 银行 / 保理机构功能需求

## 1. 定位

资金提供方进入 ChainTrace，不是为了看一个普通案件列表。

资金提供方的核心目的只有一个：

> 在贸易事实可信、证据完整、风险可控、执行条件正常的前提下，更快、更安全地决定是否放款。

ChainTrace 对资金方而言，是供应链金融贷前、贷中、贷后的一线业务控制台。

资金方不应该逐项翻材料、逐个文件人工比对。正常案件应由 Agent 做初审、汇总、风险提示和建议，人主要处理最终授权、异常、争议和高风险判断。

## 2. 进入条件

资金方进入本工作区前，必须已经完成：

| 条件 | 要求 |
|---|---|
| 钱包已连接 | 用户证明自己控制该钱包。 |
| 钱包已注册 | 该钱包已经完成 ChainTrace 注册。 |
| 角色已锁定 | 该钱包永久绑定为资金提供方 / 银行 / 保理机构。 |
| 账户已激活 | 可以查看和处理融资案件。 |
| 机构信息已完成 | 已填写机构名称、机构类型、国家、支持币种、支持融资类型等。 |

不处理登录前展示页。
不处理未注册用户的营销转化页面。

## 3. 资金方最想完成的业务结果

| 优先级 | 业务结果 | 说明 |
|---|---|---|
| P0 | 知道哪些案件值得看 | Agent 先筛掉明显不完整或明显不合规的案件。 |
| P0 | 知道证据是否完整 | 不需要人工一份份检查材料。 |
| P0 | 知道风险是否过线 | 直接看到风险结论、原因和异常点。 |
| P0 | 决定允许放款 / 拒绝 / 补材料 | 资金方要做明确业务决策。 |
| P0 | 知道是否可以执行放款 | 除了自己批准，还要知道资金池、规则、执行状态是否正常。 |
| P1 | 贷后知道钱是否按期回收 | 资金方不是只看放款前，也要看放款后状态。 |
| P1 | 让 Agent 替我跟踪异常 | 缺材料、质检争议、物流异常、买方风险，应自动提醒。 |

## 4. 资金方核心业务阶段

资金方业务不应只写成“审核申请”。

要按供应链金融真实流程拆成：

| 阶段 | 资金方关注点 |
|---|---|
| 贷前 | 这个融资申请是否真实、完整、可投、可贷。 |
| 贷中 | 是否允许放款，放款条件是否满足，执行是否正常。 |
| 贷后 | 货物、买方、付款、回款、逾期、争议是否正常。 |

## 5. 贷前功能需求

### 5.1 待审核案件接收

功能要求：

| 功能 | 说明 |
|---|---|
| 查看待审核融资申请 | 只显示与该资金方匹配或分配给该资金方的案件。 |
| 按状态筛选 | 待初审、待补材料、待人工复核、可放款建议、已拒绝等。 |
| 按融资类型筛选 | 订单融资、应收账款融资、发票融资、出货后融资等。 |
| 按币种筛选 | USD、EUR、CNY、JPY 等固定选项。 |
| 按风险等级筛选 | 正常、警示、高风险、阻断。 |
| 按金额区间筛选 | 资金方可快速找到符合偏好的案件。 |

Agent 责任：

| Agent 动作 | 说明 |
|---|---|
| 自动分组案件 | 把案件分成可快速处理、需补材料、需人工复核、建议拒绝。 |
| 自动生成案件摘要 | 不让资金方先打开一堆文件。 |
| 标记异常点 | 明确指出为什么需要看。 |

### 5.2 融资案件摘要

资金方打开案件时，第一眼不应该看到文件堆，而应该看到业务摘要。

功能要求：

| 摘要项 | 说明 |
|---|---|
| 出口商信息 | 名称、国家、历史情况、账户状态。 |
| 买方信息 | 名称、国家、付款责任、风险提示。 |
| 贸易金额 | 订单金额、发票金额、申请融资金额。 |
| 融资类型 | 应收账款、订单、发票、出货后等。 |
| 货物信息 | 品类、数量、包装、交付状态。 |
| 物流状态 | 未发货、已揽收、已出港、到港、签收等。 |
| 质检状态 | Pass、Fail、Dispute、Pending。 |
| 保险状态 | 有保险、无保险、保险不足、待核保。 |
| 材料完整性 | 完整、不完整、需人工确认。 |
| Agent 建议 | 可放款、拒绝、补材料、人工复核。 |

资金方最需要看到的是：

```text
这个案件是什么？
为什么可以看？
哪里有风险？
系统建议我怎么处理？
```

### 5.3 证据完整性审查

功能要求：

| 检查项 | 结果 |
|---|---|
| 合同是否存在 | 有 / 无 / 不清楚 |
| 订单是否存在 | 有 / 无 / 不清楚 |
| 发票是否存在 | 有 / 无 / 不清楚 |
| 发票金额是否与订单匹配 | 匹配 / 不匹配 / 无法判断 |
| 装箱单是否存在 | 有 / 无 / 不清楚 |
| 物流文件是否存在 | 有 / 无 / 不清楚 |
| 质检证据是否存在 | 有 / 无 / 不清楚 |
| 买方信息是否足够 | 足够 / 不足 / 需要确认 |
| 保险信息是否存在 | 有 / 无 / 不适用 |
| 异常说明是否充分 | 充分 / 不充分 / 不需要 |

Agent 输出必须是资金方能直接用来决策的语言：

| 不要说 | 要说 |
|---|---|
| Document mismatch | 发票金额比订单金额高 8%，需要出口商解释或上传修正文件。 |
| Missing evidence | 缺少物流文件，当前无法确认货物是否已出运。 |
| Inspection issue | 质检结论为 dispute，不建议自动放款。 |
| Buyer data weak | 买方信息不足，建议要求补充买方确认。 |

### 5.4 风险门槛审查

资金方不只看材料是否齐，还要看是否值得放款。

功能要求：

| 风险项 | 输出 |
|---|---|
| 出口商风险 | 正常 / 警示 / 高风险 |
| 买方风险 | 正常 / 警示 / 高风险 |
| 国家 / 地区风险 | 正常 / 警示 / 高风险 |
| 商品品类风险 | 正常 / 警示 / 高风险 |
| 金额异常风险 | 正常 / 警示 / 高风险 |
| 物流异常风险 | 正常 / 警示 / 高风险 |
| 质检异常风险 | 正常 / 警示 / 高风险 |
| 保险覆盖不足风险 | 正常 / 警示 / 高风险 |
| 材料一致性风险 | 正常 / 警示 / 高风险 |
| 综合风险建议 | 可放款 / 补材料 / 人工复核 / 拒绝 |

风险输出不能只是分数。必须给原因。

```text
综合建议：人工复核
原因：发票金额与订单金额不一致，且质检结论为 dispute。
建议动作：要求出口商补充金额差异说明，并等待质检方补充报告。
```

## 6. 贷中功能需求

贷中阶段的核心问题是：

> 资金方已经想放款，但系统是否真的允许执行？

### 6.1 决策动作

资金方可以做三类核心决策：

| 决策 | 含义 |
|---|---|
| 允许放款 | 资金方认可该案件，可以进入放款执行流程。 |
| 拒绝融资 | 当前案件不符合融资要求。 |
| 要求补材料 | 案件可能可做，但现在材料不足或异常未解释。 |

补充决策：

| 决策 | 含义 |
|---|---|
| 转人工复核 | Agent 建议不足以自动处理，需要资深人员判断。 |
| 降低可融资金额 | 不完全拒绝，但降低授信或融资比例。 |
| 要求质检复核 | 质检材料不充分或存在争议。 |
| 要求买方确认 | 需要付款方确认交易或付款责任。 |
| 要求保险补充 | 需要保险覆盖后才允许继续。 |

### 6.2 放款条件检查

资金方批准后，仍需要确认执行条件。

功能要求：

| 条件 | 结果 |
|---|---|
| 资金方是否已批准 | 是 / 否 |
| 案件材料是否锁定 | 是 / 否 |
| 质检是否通过 | Pass / Fail / Dispute / Pending |
| 风险门槛是否通过 | 通过 / 未通过 / 人工豁免 |
| 资金池是否有足够余额 | 足够 / 不足 / 暂停 |
| 当前规则版本是否有效 | 有效 / 过期 / 暂停 |
| 平台状态是否允许放款 | 允许 / 不允许 |
| 是否存在未关闭异常 | 无 / 有 |
| 是否允许进入放款执行 | 允许 / 不允许 |

资金方要看到的是：

```text
我已经批准了。
但现在是否真的能放款？
如果不能，卡在哪里？
```

### 6.3 放款执行状态

资金方不需要看底层执行细节，但要看业务状态。

| 状态 | 资金方理解 |
|---|---|
| Approved | 已批准，等待执行条件。 |
| Execution pending | 正在等待执行。 |
| Pool insufficient | 资金池不足。 |
| Rule blocked | 当前规则阻断。 |
| Operator paused | 平台暂停。 |
| Disbursement processing | 放款处理中。 |
| Disbursed | 已放款。 |
| Execution failed | 放款执行失败，需要处理。 |

## 7. 贷后功能需求

资金方不是放完款就结束。

贷后必须回答：

> 钱放出去以后，贸易是否继续正常？买方是否按期付款？是否出现争议、逾期或损失？

### 7.1 贷后状态跟踪

功能要求：

| 跟踪项 | 说明 |
|---|---|
| 货物状态 | 是否继续运输、是否签收、是否发生异常。 |
| 买方确认 | 买方是否确认收货、是否提出 dispute。 |
| 付款到期日 | 应收账款或付款义务什么时候到期。 |
| 回款状态 | 未到期、待付款、已付款、部分付款、逾期。 |
| 逾期风险 | 到期前提醒，逾期后升级。 |
| 保险状态 | 是否需要通知保险方或准备理赔材料。 |
| 争议状态 | 是否存在质量、数量、物流、付款争议。 |

### 7.2 贷后异常处理

| 异常 | 处理方式 |
|---|---|
| 买方未确认收货 | Agent 请求买方确认或提示出口商补充证明。 |
| 物流延误 | 请求物流方补充延误说明。 |
| 质检争议 | 请求质检方补充或复检。 |
| 买方拒付 | 进入争议/催收/保险流程。 |
| 到期未付款 | 标记逾期，通知资金方和平台。 |
| 部分付款 | 记录剩余应收，继续追踪。 |
| 保险可触发 | 请求保险方判断是否进入理赔。 |

### 7.3 回款结果

资金方最终要看到：

```text
这笔钱是否已经回收？
是否逾期？
是否有损失？
是否需要保险或人工介入？
```

## 8. 资金方 Agent 模式差异

| 功能节点 | 全人工确认 | 半自动托管 | 全自动 Agent |
|---|---|---|---|
| 接收案件 | 用户手动查看 | Agent 自动分组 | Agent 自动筛选并排队 |
| 证据检查 | 用户逐项确认 | Agent 检查，用户看异常 | Agent 自动检查并只提示异常 |
| 风险审查 | 用户阅读材料判断 | Agent 给建议，用户确认 | 低风险案件自动建议通过 |
| 要求补材料 | 用户手动发送 | Agent 生成补材料请求 | Agent 自动请求相关方补充 |
| 放款决策 | 必须人工确认 | 人工确认关键动作 | 授权范围内可自动批准低风险案件 |
| 放款执行前检查 | 用户查看 | Agent 检查并提示 | Agent 自动检查阻断条件 |
| 贷后跟踪 | 用户主动查看 | Agent 推送关键状态 | Agent 只推送异常、逾期、结果 |

重要边界：

即使是全自动 Agent 模式，也可以设置以下事项必须人工确认：

| 必须人工确认场景 | 原因 |
|---|---|
| 高金额案件 | 资金风险高。 |
| 高风险国家 / 地区 | 合规和回款风险高。 |
| 质检 dispute | 事实不确定。 |
| 保险不足 | 风险缓释不足。 |
| 买方信息不足 | 付款责任不清。 |
| 规则异常 | 不能绕过平台和规则约束。 |

## 9. 权限边界

资金方可以：

| 权限 | 说明 |
|---|---|
| 查看分配给自己的融资案件 | 只能看自己有权处理的案件。 |
| 查看案件摘要和证据完整性 | 用于贷前审查。 |
| 查看 Agent 风险建议 | 用于辅助决策。 |
| 要求补材料 | 向出口商、质检方、物流方、买方等请求补充。 |
| 允许放款 | 在权限范围内批准案件。 |
| 拒绝融资 | 给出拒绝原因。 |
| 降低融资金额 | 按风险调整融资额度。 |
| 查看贷后状态 | 跟踪回款、逾期、争议。 |

资金方不可以：

| 禁止事项 | 原因 |
|---|---|
| 修改自己的永久角色 | 钱包角色不可变。 |
| 修改出口商原始材料 | 防止证据污染。 |
| 冒充质检方给出 pass/fail/dispute | 质检结论必须来自质检角色。 |
| 修改平台规则 | 规则属于平台/合约运营。 |
| 越权查看其他资金方案件 | 商业隔离。 |
| 绕过规则强制放款 | 防止资金池和平台风险。 |

## 10. MVP 范围

### P0 必须做

| 功能 | 说明 |
|---|---|
| 资金方角色进入 | 已注册资金方钱包进入资金方工作区。 |
| 查看待审核融资申请 | 看到分配或匹配给自己的案件。 |
| 查看案件摘要 | 不先看文件堆，先看业务摘要。 |
| 查看证据完整性 | 合同、订单、发票、物流、质检等是否完整。 |
| 查看风险建议 | Agent 给出可放款、补材料、人工复核、拒绝建议。 |
| 做出决策 | 允许放款 / 拒绝 / 补材料。 |
| 查看执行条件 | 资金池、规则、异常、状态是否允许放款。 |
| 查看放款状态 | 待执行、处理中、已放款、失败。 |

### P1 后续做

| 功能 | 说明 |
|---|---|
| 贷后回款跟踪 | 付款到期、已付款、部分付款、逾期。 |
| 自动催收提醒 | 到期前和逾期后 Agent 提醒。 |
| 多资金方竞争或匹配 | 一个案件匹配多个资金方。 |
| 风险偏好规则配置 | 不同资金方有不同偏好。 |
| 保险联动 | 风险高时要求保险补充。 |

## 11. 本步结论

资金方工作区的本质不是“审核列表”。

它的本质是：

> Agent 把融资案件变成可判断的贷前、贷中、贷后业务对象；资金方只需要围绕真实性、完整性、风险、执行条件和回款结果做决策。
