# 21 P0 业务原型说明

## 1. 定位

本文件定义 ChainTrace P0 MVP 的业务原型说明。

业务原型不是 UI 原型，不是页面布局，也不是组件设计。

它的核心作用是：

> 用业务语言说明每个 P0 角色进入工作区后，应该看到哪些业务内容、回答哪些业务问题、完成哪些业务动作。

本文件不写页面、组件、路由、数据库、智能合约实现、钱包签名、hash 或 proof 细节。

## 2. 业务原型原则

| 原则 | 说明 |
|---|---|
| 先业务内容 | 先定义用户应该理解什么，再考虑未来如何呈现。 |
| 先角色结果 | 每个工作区围绕该角色最想完成的业务结果组织。 |
| 先 Agent 摘要 | Agent 先把材料、缺口、风险和建议整理成业务语言。 |
| 先卡点和下一步 | 用户必须知道现在卡在哪里、谁处理、下一步是什么。 |
| 不进入 UI | 不描述按钮样式、布局、组件、交互细节。 |

## 3. 出口商业务原型

出口商进入工作区后，第一眼应该回答：

```text
我这笔贸易能不能融资？
还缺什么？
资金方怎么判断？
钱有没有到账？
```

### 3.1 出口商应看到的业务内容

| 内容 | 业务含义 |
|---|---|
| 当前融资机会 | 可以融资 / 暂不建议 / 需要补材料 / 已拒绝。 |
| 建议融资金额 | Agent 或资金方给出的建议区间或批准金额。 |
| 当前状态 | Draft、Ready to submit、Under review、Approved、Blocked、Funds arrived 等。 |
| 当前卡点 | 材料、质检、资金方、规则、资金池、平台状态。 |
| Agent 摘要 | 贸易摘要、缺口、风险提示、下一步。 |
| 材料完整性 | 合同、订单、发票、装箱单、质检等是否足够。 |
| 待办 | 需要出口商上传、确认、解释或等待的事项。 |
| 融资结果 | 允许放款、拒绝、补材料、执行等待、到账。 |

### 3.2 出口商应完成的业务动作

| 动作 | 完成标准 |
|---|---|
| 创建融资意图 | 案件进入 Draft。 |
| 上传贸易材料 | Agent 能生成材料摘要和缺口检查。 |
| 确认提交 | 案件进入 Submitted。 |
| 补充材料 | 对每个缺口给出材料或说明。 |
| 查看资金方结果 | 明确知道允许、拒绝、补材料还是等待执行。 |
| 查看到账或卡点 | 明确知道钱是否到账，或卡在哪里。 |

### 3.3 出口商不应看到或操作

| 不应包含 | 原因 |
|---|---|
| 资金方内部审批过程 | 出口商只需要结果和业务原因。 |
| 平台规则维护 | 规则权属于平台方。 |
| 质检结论修改 | 质检结论属于质检方。 |
| 合约技术细节 | 当前不是实现阶段。 |

## 4. 质检方业务原型

质检方进入工作区后，第一眼应该回答：

```text
哪些案件需要我质检？
我要检查什么？
我的材料是否足够？
我的结论如何影响融资流程？
```

### 4.1 质检方应看到的业务内容

| 内容 | 业务含义 |
|---|---|
| 待质检任务 | 分配给该质检方的任务。 |
| 货物摘要 | 品类、数量、包装、地点、贸易背景。 |
| 质检清单 | 通用检查项和关键风险点。 |
| 材料要求 | 报告、照片、现场说明、异常说明。 |
| Agent 检查结果 | 缺照片、缺报告、缺结论或结论冲突。 |
| 质检结论 | pass / fail / dispute 和原因。 |
| 补充请求 | 资金方、出口商或 Agent 要求补充的内容。 |

### 4.2 质检方应完成的业务动作

| 动作 | 完成标准 |
|---|---|
| 接收或拒绝质检任务 | 明确接受，或说明拒绝原因。 |
| 提交质检材料 | 报告、照片、现场说明足够支持结论。 |
| 选择质检结论 | pass / fail / dispute 必须明确。 |
| 解释异常 | fail 或 dispute 必须带原因。 |
| 补充材料 | 回应资金方或 Agent 指出的缺口。 |

### 4.3 质检方不应看到或操作

| 不应包含 | 原因 |
|---|---|
| 放款审批动作 | 质检方不是资金决策方。 |
| 出口商融资申请修改 | 防止证据污染。 |
| 无关案件 | 数据隔离。 |
| 平台规则维护 | 与质检事实无关。 |

## 5. 资金方业务原型

资金方进入工作区后，第一眼应该回答：

```text
哪些案件值得看？
证据是否完整？
风险是否可控？
我应该允许放款、拒绝、补材料还是人工复核？
```

### 5.1 资金方应看到的业务内容

| 内容 | 业务含义 |
|---|---|
| 待审案件分组 | 可快速审查、需补材料、需人工复核、建议拒绝。 |
| 案件摘要 | 出口商、贸易金额、申请金额、货物、材料状态。 |
| 证据完整性 | 合同、订单、发票、装箱单、质检等是否足够。 |
| 质检事实 | pass / fail / dispute、原因、补充要求。 |
| 风险摘要 | 金额、主体、品类、质检、规则等风险提示。 |
| Agent 建议 | 允许放款、拒绝、补材料、人工复核。 |
| 执行条件 | 资金池、规则、平台状态是否允许执行。 |

### 5.2 资金方应完成的业务动作

| 动作 | 完成标准 |
|---|---|
| 查看案件摘要 | 能理解案件是什么和风险在哪里。 |
| 判断材料完整性 | 完整、缺材料或需要人工确认。 |
| 判断风险 | 正常、警示、高风险或阻断。 |
| 做融资决策 | 允许放款 / 拒绝 / 补材料 / 人工复核。 |
| 查看执行条件 | 批准后知道能否执行。 |
| 处理执行等待 | 对资金池不足、规则阻断等给出业务处理。 |

### 5.3 资金方不应看到或操作

| 不应包含 | 原因 |
|---|---|
| 其他资金方无关案件 | 数据隔离。 |
| 质检结论修改 | 质检事实属于质检方。 |
| 平台系统级控制替代入口 | 平台方负责系统级暂停和恢复。 |
| 绕过规则执行 | 规则和合约业务层是执行边界。 |

## 6. 平台方业务原型

平台方进入工作区后，第一眼应该回答：

```text
系统是否安全？
资金池、规则、执行和 Agent 是否正常？
哪些异常需要我介入？
```

### 6.1 平台方应看到的业务内容

| 内容 | 业务含义 |
|---|---|
| 运行总览 | 资金池、规则、执行异常、Agent 状态。 |
| 异常案件 | 需要平台处理的规则阻断、执行失败、资金池不足。 |
| 资金池状态 | 正常、不足、暂停、异常。 |
| 规则状态 | 当前版本、适用范围、暂停、影响案件。 |
| 平台暂停状态 | 平台、资金池、规则、案件是否暂停。 |
| Agent 运营状态 | 待办堆积、失败、人工升级、异常自动化。 |
| 操作记录 | 暂停、恢复、规则变更、异常处理原因。 |

### 6.2 平台方应完成的业务动作

| 动作 | 完成标准 |
|---|---|
| 查看运行总览 | 能判断系统是否需要介入。 |
| 处理执行异常 | 对执行失败、资金池不足、规则阻断给出处理。 |
| 暂停或恢复资金池 | 高风险时暂停，风险解除后恢复。 |
| 暂停或恢复规则 | 按业务原因处理规则风险。 |
| 分派异常 | 明确交给出口商、质检方、资金方或平台处理。 |
| 查看 Agent 异常 | 对堆积、误推进、失败任务进行处理。 |

### 6.3 平台方不应看到或操作

| 不应包含 | 原因 |
|---|---|
| 正常案件逐单审批 | 平台不是默认审批员。 |
| 出口商贸易事实伪造入口 | 平台不是事实制造者。 |
| 资金方放款决策替代入口 | 放款权属于资金方。 |
| 删除关键记录 | 审计能力必须保留。 |

## 7. Agent 业务原型

Agent 不是单独用户，但每个角色身边都应该有 Agent 助理能力。

Agent 应持续回答：

```text
当前最重要的事情是什么？
还缺什么？
谁要处理？
下一步是什么？
哪些事情必须交给人？
```

| Agent 输出 | 业务含义 |
|---|---|
| 摘要 | 把材料和状态整理成人能看懂的业务说明。 |
| 缺口 | 明确缺材料、缺确认、缺说明、缺决策。 |
| 待办 | 把缺口交给明确责任角色。 |
| 建议 | 建议提交、补材料、人工复核、等待或拒绝。 |
| 提醒 | 截止时间、状态变化、高风险、授权节点。 |
| 升级 | 争议、高风险、放款授权、平台控制交给人。 |

Agent 不可以：

| 禁止事项 | 原因 |
|---|---|
| 伪造事实 | 事实必须来自业务角色。 |
| 替资金方高风险放款 | 放款权属于资金方。 |
| 替质检方出真实结论 | 质检事实属于质检方。 |
| 替平台绕过规则 | 平台和规则是执行边界。 |
| 删除关键记录 | 审计必须保留。 |

## 8. MVP 范围

### P0 必须做

| 范围 | 说明 |
|---|---|
| 定义四个 P0 角色业务原型 | 出口商、质检方、资金方、平台方。 |
| 定义每个角色应看到的业务内容 | 只定义内容和业务问题。 |
| 定义每个角色应完成的业务动作 | 用完成标准描述。 |
| 定义 Agent 业务原型 | 摘要、缺口、待办、建议、提醒、升级。 |
| 定义不应出现的越界内容 | 防止 UI 或权限设计跑偏。 |

### 暂不做

| 范围 | 原因 |
|---|---|
| UI 原型图 | 当前不做视觉和交互。 |
| 页面布局 | 当前只写业务原型。 |
| 前端组件 | 当前不写代码。 |
| 数据库结构 | 当前不进入实现。 |
| 智能合约代码 | 当前只写业务约束。 |
| hash / proof 页面 | 当前不进入底层证明产品。 |

## 9. 本步结论

P0 业务原型说明的本质不是“画页面”。

它的本质是：

> 用业务语言定义每个角色工作区应该承载的内容、动作和边界，让后续 UI、数据和技术设计有明确的业务依据。

下一步应进入：

> P0 验收口径。
