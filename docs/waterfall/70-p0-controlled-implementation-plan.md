# 70 P0 受控实现计划

## 1. 定位

本文件定义 ChainTrace P0 从规格准备进入受控实现阶段后的推进计划。

它不是代码实现方案，不是 sprint 拆分，不是工时计划，不是 API 详细文档，不是数据库表结构，不是 Solidity 实现，不是 UI 组件清单，也不是测试脚本。

它的作用是：

> 在 `69-p0-implementation-entry-checklist.md` 允许进入实现后，定义 P0 实现应按什么顺序推进、每一步实现前必须满足什么条件、实现中发现缺口如何回流、哪些内容继续禁止越界。

本文件只定义受控实现计划，不直接写实现。

## 2. 受控实现原则

| 原则 | 说明 |
|---|---|
| 业务闭环优先 | 先实现出口商融资意图到资金结果的最小闭环。 |
| 身份权限先行 | 单钱包单角色、角色工作区和越权阻断先于业务功能。 |
| 主线优先 | 先做融资案件主线，不做展示页和高级 BI。 |
| Agent 辅助优先 | Agent 摘要、缺口、待办、状态解释先于复杂 Agent 自动化。 |
| 合约状态先行 | 先做合约/规则/资金池状态的业务表达，不急于链上实现。 |
| 控制塔投影优先 | 角色工作区要围绕主链路、卡点、责任方和资金结果。 |
| 异常可处理优先 | 每个阻断必须有原因、责任方、解除条件和下一步。 |
| 逐步实现 | 每个阶段只实现被前置文档允许的内容。 |
| 发现缺口即回流 | 新动作、新状态、新异常、新权限不直接写代码补。 |
| P1/P2 继续后置 | 完整物流、保险、买方、海关、proof/hash 继续不进 P0 主线。 |

## 4. P0 实现阶段总顺序

建议受控实现按以下顺序推进：

| 顺序 | 实现阶段 | 阶段目标 |
|---:|---|---|
| 1 | Identity and Role Foundation | 钱包入口、注册状态、单钱包单角色、角色工作区门禁。 |
| 2 | Financing Case Core | 出口商创建融资意图、案件主状态、融资目标。 |
| 3 | Material and Gap Flow | 贸易材料、完整性、Agent 缺口、补材料待办。 |
| 4 | Inspection Fact Node | 质检任务、材料、Pass/Fail/Dispute。 |
| 5 | Funding Review Path | 资金方审查、补材料、允许、拒绝、降额、人工复核。 |
| 6 | Contract State Facade | 合约/规则/资金池/暂停状态的业务表达。 |
| 7 | Agent Assist Layer | 摘要、待办、提醒、状态解释、低风险推进、人工升级。 |
| 8 | Control Tower Projection | 出口商、资金方、质检方、平台方的主链路和卡点视图。 |
| 9 | Platform Operation Basics | 资金池、规则、合约、Agent 异常的基础运营表达。 |
| 10 | P0 Acceptance Flow | 正常闭环、路径差异、异常和权限的手工验收流程。 |

该顺序不是开发排期，而是实现依赖顺序。

## 5. 阶段 1：Identity and Role Foundation

### 5.1 阶段目标

建立 P0 所有业务功能的入口基础：

```text
钱包连接
→ 注册状态
→ 唯一业务角色
→ 账户状态
→ 角色工作区入口
→ Agent 模式
```

### 5.2 必须实现的业务能力

| 能力 | 业务要求 |
|---|---|
| 钱包连接状态表达 | 区分未连接、已连接未注册、已注册。 |
| 注册信息提交 | 注册时选择唯一业务角色。 |
| 角色绑定 | 一个钱包只能绑定一个 P0 业务角色。 |
| 账户状态 | 激活、受限、暂停要可表达。 |
| 角色工作区入口 | 出口商、资金方、质检方、平台方进入不同工作区。 |
| Agent 模式状态 | 全人工、半自动、全自动只影响自动化，不改变角色。 |
| 越权阻断 | 角色不匹配时必须阻断并解释。 |

### 5.3 禁止越界

| 禁止项 | 原因 |
|---|---|
| 一个钱包多个角色切换 | 破坏 P0 权限基础。 |
| 钱包连接后直接进入所有工作区 | 未注册和角色边界被绕过。 |
| Agent 模式改变业务角色 | Agent 模式不是角色。 |
| 先做展示型首页 | P0 先做业务入口。 |

### 5.4 阶段出口条件

```text
用户能明确看到：自己是谁、角色是什么、账户是否可用、应该进入哪个工作区、哪些动作不能做。
```

## 6. 阶段 2：Financing Case Core

### 6.1 阶段目标

建立出口商融资案件主线的最小核心。

```text
出口商创建融资意图
→ 形成融资案件
→ 表达融资目标
→ 表达案件主状态
```

### 6.2 必须实现的业务能力

| 能力 | 业务要求 |
|---|---|
| 创建融资意图 | 出口商能创建自己的融资案件。 |
| 融资目标表达 | 金额、币种、期望到账、融资类型等业务目标可表达。 |
| 案件主状态 | 草稿、准备中、待提交、已提交、补材料、审查中、执行中、完成等。 |
| 案件参与关系 | 明确案件相关角色。 |
| 案件路径占位 | 银行辅助、纯 DeFi、低人工、人工升级可表达。 |
| 出口商查看案件 | 只能查看自己的案件。 |

### 6.3 禁止越界

| 禁止项 | 原因 |
|---|---|
| 案件只是文件夹 | P0 目标是融资案件，不是文件管理。 |
| 出口商查看他人案件 | 数据隔离失败。 |
| 资金结果后置到不可见 | 出口商核心价值缺失。 |
| 一开始实现复杂多资金方市场 | P0 先做基础路径。 |

### 6.4 阶段出口条件

```text
出口商能创建融资案件，并能看到该案件处于融资链路的哪个状态。
```

## 7. 阶段 3：Material and Gap Flow

### 7.1 阶段目标

让融资案件具备贸易材料基础，并让 Agent 能识别材料缺口。

```text
贸易材料提交
→ 材料完整性表达
→ Agent 缺口检查
→ 补材料待办
→ 材料版本和补充历史
```

### 7.2 必须实现的业务能力

| 能力 | 业务要求 |
|---|---|
| 合同材料表达 | 出口商提交合同材料。 |
| 订单材料表达 | 出口商提交订单材料。 |
| 发票材料表达 | 出口商提交发票材料。 |
| 装箱单材料表达 | 出口商提交装箱单材料。 |
| 补充说明 | 出口商解释缺口或冲突。 |
| 材料完整性状态 | 完整、缺口、冲突、需人工确认。 |
| Agent 缺口 | 缺什么、谁补、为什么影响融资。 |
| 补材料待办 | 责任方和下一步清楚。 |
| 历史不覆盖 | 原始提交和补充说明不能无痕覆盖。 |

### 7.3 禁止越界

| 禁止项 | 原因 |
|---|---|
| 资金方修改出口商原始材料 | 证据污染。 |
| Agent 自动生成贸易材料 | 伪造事实。 |
| 缺口直接等于拒绝 | 缺口是待处理事项。 |
| 直接做复杂文件存储架构 | 当前先完成业务链路。 |

### 7.4 阶段出口条件

```text
出口商能提交材料，Agent 能指出缺口和下一步，材料历史不能被无痕覆盖。
```

## 8. 阶段 4：Inspection Fact Node

### 8.1 阶段目标

实现 P0 核心事实节点：质检事实。

```text
质检任务
→ 质检材料
→ Pass / Fail / Dispute
→ 补充说明
→ 对融资链路产生影响
```

### 8.2 必须实现的业务能力

| 能力 | 业务要求 |
|---|---|
| 质检任务表达 | 质检方能看到自己的任务。 |
| 质检材料提交 | 报告、照片、说明等质检材料可表达。 |
| 质检结论 | Pass、Fail、Dispute 三类状态必须可表达。 |
| 补充质检说明 | 对 dispute 或补充请求做说明。 |
| 事实来源标识 | 质检结论来自质检方。 |
| 对案件状态影响 | Fail/Dispute 会影响资金方和合约路径。 |

### 8.3 禁止越界

| 禁止项 | 原因 |
|---|---|
| Agent 生成质检 Pass | 事实伪造。 |
| 资金方修改质检结论 | 事实来源越界。 |
| 质检方决定放款 | 资金方职责。 |
| P0 强行做完整物流/保险/买方工作区 | P1 后置。 |

### 8.4 阶段出口条件

```text
质检方能产生 Pass/Fail/Dispute，其他角色只能读取和处理影响，不能伪造结论。
```

## 9. 阶段 5：Funding Review Path

### 9.1 阶段目标

实现银行辅助路径中的资金方审查和决策能力。

```text
资金方查看待审案件
→ 查看证据完整性和 Agent 风险建议
→ 要求补材料 / 允许 / 拒绝 / 降额 / 人工复核
→ 进入执行条件检查
```

### 9.2 必须实现的业务能力

| 能力 | 业务要求 |
|---|---|
| 待审案件视图 | 资金方只能看分配/匹配给自己的案件。 |
| 案件摘要 | 出口商、买方、金额、商品、融资类型可读。 |
| 证据完整性 | 材料和质检事实是否完整可见。 |
| Agent 风险建议 | 建议不等于审批。 |
| 补材料请求 | 请求对象和原因明确。 |
| 允许融资 | 资金方决策，不等于到账。 |
| 拒绝融资 | 必须有可理解原因。 |
| 降额 | 金额调整可表达。 |
| 人工复核 | 高风险和 dispute 进入人工。 |

### 9.3 禁止越界

| 禁止项 | 原因 |
|---|---|
| Agent 自动批准高风险案件 | Agent 越权。 |
| 合约替资金方审批 | 合约越界。 |
| 平台替资金方批准 | 平台越界。 |
| 资金方批准直接等于到账 | 忽略执行约束。 |

### 9.4 阶段出口条件

```text
资金方能基于材料、事实和 Agent 建议作出融资决策，但批准后仍进入合约/规则/资金池检查。
```

## 10. 阶段 6：Contract State Facade

### 10.1 阶段目标

先实现合约/规则/资金池/暂停状态的业务表达层，而不是直接进入复杂链上合约实现。

```text
执行条件检查
→ 规则状态
→ 资金池状态
→ 平台暂停状态
→ 执行结果表达
```

### 10.2 必须实现的业务能力

| 能力 | 业务要求 |
|---|---|
| 执行条件检查状态 | Allowed、Waiting、Need materials、Need review。 |
| 规则状态表达 | Active、Paused、Expired、Not applicable、Conflicting。 |
| 资金池状态表达 | Available、Low balance、Paused、Failed。 |
| 平台暂停状态表达 | 案件、资金池、规则、执行能力、账户、Agent 自动化。 |
| 执行结果表达 | Completed、Execution failed、Rule blocked、Pool insufficient。 |
| 对控制塔输出 | 输出必须转成业务语言。 |

### 10.3 禁止越界

| 禁止项 | 原因 |
|---|---|
| 合约判断贸易真实性 | 合约边界错误。 |
| 合约生成质检结论 | 事实来源错误。 |
| 合约替资金方审批 | 资金方责任被替代。 |
| 直接上多链/多资金池复杂实现 | P0 范围失控。 |

### 10.4 阶段出口条件

```text
系统能用业务语言表达合约/规则/资金池/暂停对融资执行的影响。
```

## 11. 阶段 7：Agent Assist Layer

### 11.1 阶段目标

实现 P0 Agent Native 的基础辅助层。

```text
角色上下文
→ 摘要
→ 缺口
→ 待办
→ 请求补充
→ 状态解释
→ 低风险推进
→ 人工升级
```

### 11.2 必须实现的业务能力

| 能力 | 业务要求 |
|---|---|
| 角色上下文识别 | Agent 知道自己属于哪个角色。 |
| 案件摘要 | 摘要不替代原始材料。 |
| 材料缺口 | 缺什么、谁补、下一步清楚。 |
| 待办生成 | 待办有来源、责任方和状态。 |
| 请求补充 | 不越权、不伪造材料。 |
| 状态解释 | 用业务语言解释合约/规则/异常。 |
| 低风险推进 | 仅在授权、低风险、事实完整、规则明确时推进。 |
| 人工升级 | 高风险、dispute、失败、置信不足、权限不清时升级。 |
| 行为留痕 | 自动动作和升级可追踪。 |

### 11.3 禁止越界

| 禁止项 | 原因 |
|---|---|
| Agent 独立拥有超级权限 | 破坏权限模型。 |
| Agent 查看无权数据 | 数据越权。 |
| Agent 伪造事实 | 破坏可信基础。 |
| Agent 自动裁决 dispute | 争议处理越界。 |
| Agent 绕过平台暂停 | 运营边界失效。 |

### 11.4 阶段出口条件

```text
Agent 能帮助角色理解案件、发现缺口、生成待办、解释状态，并在风险边界内推进或升级。
```

## 12. 阶段 8：Control Tower Projection

### 12.1 阶段目标

把身份、案件、材料、事实、资金方决策、合约输出、Agent 输出、异常和平台状态投影为角色工作区。

### 12.2 必须实现的角色投影

| 角色 | 必须看到 |
|---|---|
| 出口商 | 自己的融资案件、材料缺口、事实状态、资金结果、卡点、下一步。 |
| 资金方 | 待审案件、证据完整性、风险建议、决策动作、执行条件。 |
| 质检方 | 质检任务、清单、报告、Pass/Fail/Dispute、补充请求。 |
| 平台方 | 资金池、规则、合约、异常队列、Agent 自动化状态。 |

### 12.3 必须表达的主链路

```text
融资意图
→ 材料准备
→ Agent 检查
→ 事实确认
→ 资金方审查 / DeFi 规则路径
→ 合约/规则执行检查
→ 资金结果
```

### 12.4 禁止越界

| 禁止项 | 原因 |
|---|---|
| 控制塔变成普通列表 | 业务状态不可见。 |
| 出口商看不到资金结果 | P0 核心价值失败。 |
| 所有角色同一工作区 | 权限模型失败。 |
| 卡点只显示 error | 业务不可处理。 |
| 旧 dashboard/proof 页面成为主线 | 产品定位倒退。 |

### 12.5 阶段出口条件

```text
不同角色进入系统后，能看到与自己相关的业务状态、卡点、责任方和下一步。
```

## 13. 阶段 9：Platform Operation Basics

### 13.1 阶段目标

实现平台基础运营能力，支撑资金池、规则、合约、Agent 自动化和异常处理。

### 13.2 必须实现的业务能力

| 能力 | 业务要求 |
|---|---|
| 运行总览 | 资金池、规则、合约、异常、Agent 状态。 |
| 资金池状态 | 可用、不足、暂停、冻结、失败。 |
| 规则状态 | Active、Paused、Expired、Pending upgrade、Conflict。 |
| 合约执行状态 | 正常、等待、失败、完成、暂停。 |
| Agent 自动化状态 | 正常、失败、堆积、需人工接管。 |
| 异常队列 | 缺材料、dispute、规则阻断、资金池不足、执行失败。 |
| 暂停/恢复记录 | 必须有原因、影响范围、恢复条件。 |

### 13.3 禁止越界

| 禁止项 | 原因 |
|---|---|
| 平台逐单审批所有案件 | 平台定位错误。 |
| 平台替资金方批准放款 | 决策权越界。 |
| 平台伪造事实 | 事实来源错误。 |
| 平台删除审计记录 | 可信基础破坏。 |

### 13.4 阶段出口条件

```text
平台能看到系统级异常并处理暂停、恢复、分派和 Agent 自动化问题，但不能变成逐单审批者。
```

## 14. 阶段 10：P0 Acceptance Flow

### 14.1 阶段目标

形成 P0 的手工业务验收流程。

### 14.2 必须验收的场景

| 场景 | 验收目标 |
|---|---|
| 正常融资闭环 | 出口商从融资意图走到资金结果。 |
| 银行辅助路径 | 资金方审查和决策后进入执行检查。 |
| 纯 DeFi 路径 | 规则、事实、资金池和暂停状态共同决定结果。 |
| 低人工路径 | Agent 在授权和低风险条件下推进。 |
| 人工升级路径 | 高风险、dispute、失败、置信不足升级给人。 |
| 权限隔离 | 单钱包单角色和越权阻断成立。 |
| 事实来源 | 材料、质检、Agent、合约边界成立。 |
| 异常处理 | 卡点有原因、责任方、解除条件和下一步。 |
| 后置范围 | P1/P2、proof/hash 没有进入 P0 主线。 |

### 14.3 阶段出口条件

```text
P0 可以被业务人员按场景验收，且每个失败点都能回溯到设计边界或实现缺陷。
```

## 15. 实现中回流规则

实现过程中出现以下情况，必须停止当前实现点并回流：

| 发现情况 | 回流目标 |
|---|---|
| 新业务动作 | 54 / 63。 |
| 新角色或新权限 | 53。 |
| 新信息域或数据对象 | 55 / 64。 |
| 新合约业务输出 | 56 / 65。 |
| 新 Agent 能力或自动化边界 | 57 / 66。 |
| 控制塔无法表达状态或卡点 | 58 / 67。 |
| 验收场景缺失 | 59 / 68。 |
| P1/P2 内容混入 | 60。 |
| proof/hash 旧主线回流 | 60，通常后置或拒绝。 |

不得用以下方式处理缺口：

- 直接加接口。
- 直接加字段。
- 直接写合约函数。
- 直接加 UI 按钮。
- 直接让 Agent 自动处理。
- 直接把异常吞掉。

## 18. 一票否决项

实现中出现以下任一项，必须停止相关实现并回流：

| 一票否决项 | 原因 |
|---|---|
| 一个钱包可以随意切换多个角色 | 权限基础失败。 |
| 出口商看不到资金结果和卡点 | P0 核心价值失败。 |
| 资金方可以修改出口商原始材料 | 事实可信失败。 |
| 质检方可以决定放款 | 角色职责越界。 |
| 平台可以替资金方批准 | 平台越界。 |
| Agent 可以查看无权数据 | 数据越权。 |
| Agent 可以伪造事实 | 可信基础失败。 |
| Agent 自动批准高风险融资 | 风险控制失败。 |
| Agent 自动裁决 dispute | 争议处理失败。 |
| 合约判断贸易真实性 | 合约越界。 |
| 合约替资金方审批 | 资金方责任失败。 |
| 合约绕过平台暂停 | 运营控制失败。 |
| 纯 DeFi 路径无事实和规则约束 | 风险边界失败。 |
| 路径差异消失 | 产品定位失败。 |
| 异常只显示技术错误 | 业务不可处理。 |
| P1/P2 被塞入 P0 | MVP 范围失控。 |
| 旧 proof/hash/demo 成为主线 | clean waterfall 主线失败。 |

## 19. 受控实现完成标准

P0 受控实现完成，不等于完整商业产品完成。

P0 受控实现完成的最低标准是：

```text
1. 钱包注册和单角色工作区成立。
2. 出口商可以创建融资案件。
3. 贸易材料和 Agent 缺口可表达。
4. 质检事实 pass/fail/dispute 可表达。
5. 资金方可以审查和决策。
6. DeFi/合约/规则/资金池/暂停状态可表达。
7. Agent 可以摘要、待办、状态解释和人工升级。
8. 控制塔能表达主链路、卡点、责任方和资金结果。
9. 平台能处理基础运营异常。
10. 正常闭环、路径差异、权限隔离和异常处理可验收。
```

## 20. 本步结论

P0 受控实现计划的本质不是“马上开写所有代码”。

它的本质是：

> 在已经完成技术设计、规格准备和实现入口检查之后，给 P0 实现阶段设置清晰顺序、阶段门禁、回流规则和禁止越界红线，确保实现先完成融资闭环、角色权限、Agent Native、Crypto Native、控制塔表达和异常处理，而不是回到旧 demo、卡片墙、proof/hash 工具页或无限扩张的后台系统。
