# 50 P0 技术设计评审清单

## 1. 定位

本文件用于评审 ChainTrace P0 技术设计交付物是否可以通过。

它不是新的需求文档，不是接口规格，不是数据库设计，不是智能合约设计，不是钱包签名流程，不是 hash/proof 方案，不是 UI 组件说明，也不是测试代码计划。

它的作用是：

> 在进入下一阶段之前，用统一检查清单确认技术设计交付物是否继承已冻结业务需求，是否保持 Agent Native 和 DeFi/银行双路径边界，是否没有提前进入实现。

如果评审发现设计缺口，不能直接用技术实现补洞，必须回流到对应 waterfall 文档。

## 2. 评审原则

| 原则 | 说明 |
|---|---|
| 先查业务继承 | 技术设计必须能追溯到已冻结业务需求。 |
| 先查边界再查方案 | 先确认没有越界，再评估设计表达是否完整。 |
| 先查角色再查流程 | 单钱包单角色和角色权限不能被技术设计放大。 |
| 先查路径差异 | 银行辅助、纯 DeFi、低人工和人工升级不能被合并成一个普通审批流。 |
| 先查 Agent 边界 | Agent 是角色助理，不是独立审批人。 |
| 先查合约边界 | 合约是执行约束层，不是贸易真实性判断者。 |
| 先查回流机制 | 发现缺口必须记录回流，不能直接进入实现。 |
| 先查后置范围 | P1/P2 和实现细节不能偷偷进入 P0。 |

## 3. 总体通过标准

| 检查项 | 通过标准 | 不通过表现 |
|---|---|---|
| 产品定位 | 仍然是 Agent Native + Crypto Native 的供应链金融前站。 | 被改成普通银行审批后台、普通 SaaS、普通 proof demo。 |
| P0 主线 | 保留出口商融资意图到资金结果的闭环。 | 只剩页面、列表、文件上传或证明展示。 |
| DeFi/银行双路径 | 银行辅助路径和纯 DeFi 路径都被保留。 | 所有路径都被合并成银行审批。 |
| Agent Native | Agent 负责整理、提醒、缺口、低风险推进、异常升级。 | Agent 被写成审批人、事实制造者或万能管理员。 |
| 单钱包单角色 | 每个钱包只对应一个永久角色。 | 一个钱包可以随意切换出口商、银行、质检方等角色。 |
| 角色专属工作区 | 不同角色只看到与自己业务相关的内容。 | 所有角色共用同一套后台视图。 |
| 阶段边界 | 没有 endpoint、表结构、Solidity、ABI、UI 组件、测试代码。 | 技术设计提前写成实现规格。 |

## 4. 业务输入继承检查

| 输入文档 | 必查问题 | 通过标准 |
|---|---|---|
| 01-business-requirements | 是否继承角色、目标、关键场景和结果导向？ | 技术设计从业务结果出发，而不是从页面或代码出发。 |
| 02-registration-login | 是否保留钱包先登录、注册后激活、角色永久绑定？ | 登录和注册边界清楚，角色不可变。 |
| 03-exporter-functional-requirements | 是否保留出口商融资意图、材料准备、Agent 检查、提交、状态和到账结果？ | 出口商不是只上传文件，而是完成融资结果。 |
| 04-funding-provider-functional-requirements | 是否保留贷前、贷中、贷后和资金方决策边界？ | 资金方看真实性、完整性、风险、执行条件和回款。 |
| 05-inspection-functional-requirements | 是否保留质检方事实来源定位？ | 质检方提供 pass/fail/dispute，不决定放款。 |
| 06-platform-contract-operator-functional-requirements | 是否保留平台系统级运营定位？ | 平台维护资金池、规则、合约、异常和 Agent 秩序，不逐单审批。 |
| 后续角色文档 | 是否保留物流、海关、保险、买方的事实/风险/付款责任边界？ | 每个事实方只处理自己的事实范围。 |
| P0 PRD 总纲 | 是否继承 P0 冻结范围？ | P0 不被 P1/P2 扩张。 |
| 技术设计边界文档 | 是否保持阶段边界？ | 不进入具体实现。 |

## 5. DeFi/银行双路径检查

| 检查项 | 通过标准 | 不通过表现 |
|---|---|---|
| 银行辅助路径 | 保留资金方查看摘要、完整性、风险建议、允许/拒绝/补材料。 | 银行路径缺失或只有合约路径。 |
| 纯 DeFi 路径 | 保留规则、资金池和执行条件驱动的路径。 | 纯 DeFi 被删除，必须银行审批才能推进。 |
| 低人工路径 | 保留 Agent + 规则自动推进正常案件。 | 所有节点都要求人工点按钮。 |
| 人工升级路径 | dispute、高风险、规则冲突、执行失败等能升级给人。 | 异常被 Agent 或合约自动吞掉。 |
| 路径差异表达 | 每条路径的触发条件、责任角色和输出结果不同。 | 不同路径只是在同一流程上换名称。 |
| 资金结果表达 | 出口商能看到到账、等待、阻断、失败或卡点。 | 只看到技术状态，不知道资金结果。 |

## 6. 角色权限检查

| 角色 | 必查权限边界 | 通过标准 |
|---|---|---|
| 出口商 / 资金需求方 | 只能创建和处理自己的融资案件。 | 不能看无关案件，不能修改资金方、质检方或平台结论。 |
| 资金方 / 银行 / 保理机构 | 只能处理分配或匹配给自己的案件。 | 能批准、拒绝、补材料，但不能改贸易事实。 |
| 质检方 | 只提供质检事实。 | 能出 pass/fail/dispute，但不能决定放款。 |
| 平台 / 合约运营 | 只处理系统级规则、资金池、合约和异常。 | 能暂停/恢复/升级，但不能替资金方批准或伪造事实。 |
| 物流方 | 只提供物流事实。 | 能确认运输状态，不能参与金融审批。 |
| 海关 / 报关方 | 只提供通关和监管事实。 | 能确认报关/清关状态，不能审批融资。 |
| 保险方 | 只处理承保、核保、理赔和风险缓释。 | 能承保/拒保/补材料，不能替资金方放款。 |
| 买方 / 进口商 | 只确认订单、收货、争议和付款责任。 | 不能修改出口商申请或资金方决策。 |
| Agent | 权限继承所属角色。 | 不能作为独立超级用户。 |

## 7. Agent Native 边界检查

### 7.1 Agent 可以做

| 检查项 | 通过标准 |
|---|---|
| 整理材料 | 能把文件、事实、状态整理成业务摘要。 |
| 检查缺口 | 能告诉用户缺什么、找谁补、下一步做什么。 |
| 生成待办 | 能把异常转成角色相关待办。 |
| 提醒用户 | 能提醒确认、补材料、异常、到账和逾期。 |
| 请求相关角色补充 | 能向物流、质检、买方、保险等角色请求补充事实。 |
| 自动推进低风险流程 | 在授权范围、事实完整、规则明确时推进。 |
| 异常升级 | 在争议、高风险、置信不足、规则冲突时交给人。 |

### 7.2 Agent 不可以做

| 检查项 | 不得出现 |
|---|---|
| 伪造事实 | Agent 不能替出口商、质检方、物流方、海关或买方制造事实。 |
| 高风险自动放款 | Agent 不能替资金方对高风险案件做最终放款决策。 |
| 自动裁决重大争议 | dispute 必须交给人或明确责任方处理。 |
| 绕过规则 | Agent 不能绕过平台规则、资金池状态或合约约束。 |
| 删除关键记录 | Agent 不能删除或篡改审计记录。 |
| 越权查看案件 | Agent 只能继承所属角色权限。 |

## 8. 合约 / DeFi 边界检查

| 检查项 | 通过标准 | 不通过表现 |
|---|---|---|
| 合约业务定位 | 合约是业务规则和资金执行约束层。 | 合约被设计成贸易真实性判断者。 |
| 执行前条件 | 授权、事实确认、规则、资金池、暂停状态等条件清楚。 | 只写“调用合约放款”，没有业务条件。 |
| 合约输出 | 允许、不允许、等待、规则阻断、资金池不足、平台暂停、执行失败、完成。 | 只写技术成功/失败。 |
| 不负责事项 | 不判断贸易真实性、不替资金方审批、不裁决争议。 | 合约负责所有判断。 |
| 暂停机制 | 平台、规则、资金池、案件暂停的业务含义清楚。 | 暂停只是技术开关，没有业务解释。 |
| 失败解释 | 执行失败有原因、责任方、下一步。 | 用户只看到失败，不知道怎么处理。 |

## 9. 异常与回流检查

| 异常类型 | 必须说明 | 通过标准 |
|---|---|---|
| 缺材料 | 缺什么、责任方、补充方式。 | 出口商或相关事实方知道下一步。 |
| 质检 fail/dispute | 原因、影响、是否复检或人工复核。 | 不被自动吞掉。 |
| 物流异常 | 延误、丢失、状态不一致、责任方。 | 可请求物流方补充说明。 |
| 买方争议 | 争议内容、付款责任影响、下一步。 | 可进入人工或补充确认。 |
| 保险不足 | 是否需要承保、加费、补材料或拒保。 | 风险缓释路径清楚。 |
| 资金池不足 | 影响哪些案件、等待还是切换路径。 | 出口商和资金方都能理解卡点。 |
| 规则阻断 | 哪条规则、为什么阻断、是否可解除。 | 平台或资金方知道处理方式。 |
| 平台暂停 | 暂停对象、原因、影响范围、恢复条件。 | 不是黑盒停机。 |
| Agent 失败 | 失败原因、是否降级人工。 | 自动化失败不会卡死。 |
| 新需求缺口 | 回流到哪份 waterfall 文档。 | 不直接进入实现补洞。 |

## 10. 技术设计域专项检查

### 10.1 接口设计评审

| 检查项 | 通过标准 | 不得出现 |
|---|---|---|
| 业务动作分组 | 创建意图、补材料、确认事实、授权、执行检查、暂停、恢复、异常升级等清楚。 | endpoint 名称。 |
| 业务结果 | 成功、等待、补材料、阻断、权限不足、人工升级、失败等清楚。 | 请求/响应格式。 |
| 角色动作边界 | 每个角色能触发什么动作清楚。 | 鉴权实现细节。 |
| 失败解释 | 失败原因、责任方、下一步清楚。 | 错误码格式。 |

### 10.2 数据设计评审

| 检查项 | 通过标准 | 不得出现 |
|---|---|---|
| 信息域 | 案件、贸易材料、事实节点、资金方决策、Agent 输出、DeFi 结果、异常、审计等清楚。 | 表结构。 |
| 事实来源 | 谁确认事实、谁负责事实清楚。 | 数据库关系模型。 |
| 可见范围 | 角色相关可见、无关案件不可见。 | 权限表。 |
| Agent 记录 | Agent 摘要不等于事实结论。 | Agent memory 实现。 |
| DeFi 记录 | 合约结果不等于贸易真实性判断。 | 链上/链下存储结构。 |

### 10.3 原型实现准备评审

| 检查项 | 通过标准 | 不得出现 |
|---|---|---|
| 控制塔表达 | 源头到终端、节点状态、卡点、资金结果清楚。 | 页面路由。 |
| 角色视角 | 不同角色看到不同业务范围。 | UI 组件结构。 |
| 路径表达 | 银行辅助、纯 DeFi、低人工、人工升级可区分。 | 前端状态实现。 |
| Agent 表达 | 摘要、缺口、待办、建议、升级清楚。 | Agent UI 组件。 |
| DeFi 表达 | 允许、等待、阻断、资金池不足、暂停、失败、完成清楚。 | 合约调用实现。 |

### 10.4 测试设计评审

| 检查项 | 通过标准 | 不得出现 |
|---|---|---|
| 正常闭环 | 出口商意图到资金结果可验收。 | 测试框架。 |
| 银行辅助路径 | 资金方审查、授权、补材料、执行检查可验收。 | API 测试代码。 |
| 纯 DeFi 路径 | 规则和资金池驱动路径可验收。 | 链上测试脚本。 |
| 低人工路径 | Agent 推进、异常升级可验收。 | 自动化工具。 |
| 权限验收 | 无关案件不可见、越权动作阻断可验收。 | 鉴权测试实现。 |

| 数据库字段 | 不进入 P0 技术设计评审。 |

| 钱包签名细节 | 不进入 P0 技术设计评审。 |
| hash / proof 细节 | 不进入 P0 技术设计评审。 |
| UI 组件和样式 | 不进入 P0 技术设计评审。 |
| 自动化测试代码 | 不进入 P0 技术设计评审。 |

## 12. 一票否决项

出现以下任一情况，P0 技术设计不得通过：

| 一票否决项 | 原因 |
|---|---|
| 技术设计把 ChainTrace 改回普通银行后台 | 破坏产品定位。 |
| 技术设计把所有路径合并成银行审批 | 破坏 DeFi 和低人工路径。 |
| 技术设计让 Agent 成为事实制造者或最终审批人 | 破坏 Agent Native 边界。 |
| 技术设计让合约判断贸易真实性 | 破坏合约业务角色。 |
| 技术设计允许一个钱包多角色切换 | 破坏身份和权限模型。 |
| 技术设计允许角色越权查看或修改无关案件 | 破坏数据隔离和信任。 |
| 技术设计直接写 endpoint、表结构、Solidity、ABI、组件或测试代码 | 阶段越界。 |
| 技术设计把 P1/P2 范围塞入 P0 必做 | 破坏 MVP 范围。 |
| 技术设计使用旧版 demo/page/hash/proof 口径作为主流程 | 违背 clean waterfall redesign。 |
| 技术设计发现需求缺口但不回流文档 | 破坏瀑布模式。 |

## 14. 本步结论

P0 技术设计评审清单的本质不是“检查技术实现是否写完”。

它的本质是：

> 检查技术设计是否忠实继承 ChainTrace 已冻结业务需求，是否保留 DeFi/银行双路径、Agent Native 边界、单钱包单角色、控制塔业务表达、异常回流和阶段边界，并阻止任何提前进入实现的内容。

下一步应进入：

> P0 技术设计阶段总入口或 P0 技术设计执行计划，用于按接口、数据、合约、Agent、权限、原型和测试设计域组织后续技术设计工作。
