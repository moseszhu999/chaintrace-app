# 78 P0 实现阶段 8：Control Tower Projection 计划

## 1. 定位

本文件定义 ChainTrace P0 受控实现阶段的第八个阶段：Control Tower Projection。

它不是 UI 视觉设计，不是页面路由表，不是前端组件拆分，不是 CSS 样式规范，不是图表库选型，不是 API 聚合实现，不是数据库查询优化，也不是 HTML 原型图本身。

它的作用是：

> 在 Phase 1–7 已经建立身份角色、融资案件、贸易材料、质检事实、资金方审查、合约状态和 Agent 辅助层之后，细化如何把这些业务对象和状态投影成不同角色可理解的 Control Tower 操作台。

本阶段的核心目标不是“做卡片墙”，而是：

```text
角色身份
→ 融资案件主链路
→ 当前节点状态
→ 材料/事实/资金/合约/Agent 输出
→ 卡点原因
→ 责任方
→ 下一步
→ 资金结果
→ 角色可见范围内的操作入口
```

写完本文件后，P0 的第一版 HTML 原型图即可开始。

## 3. Phase 8 阶段目标

Phase 8 必须让系统具备以下业务表达能力：

| 目标 | 说明 |
|---|---|
| 角色工作区投影 | 出口商、资金方、质检方、平台方看到不同控制塔视角。 |
| 融资主链路投影 | 融资意图、材料、Agent、质检、资金方/DeFi、合约、资金结果一条线表达。 |
| 当前节点可识别 | 用户知道案件当前卡在哪一步。 |
| 卡点可处理 | 每个卡点有原因、责任方、影响、解除条件和下一步。 |
| Agent 输出可见 | 摘要、缺口、待办、解释、自动推进和升级可被业务理解。 |
| 合约状态可解释 | Allowed、Rule blocked、Pool insufficient、Platform paused 等用业务语言展示。 |
| 资金结果可理解 | 出口商知道能不能借、能借多少、是否到账、失败或等待原因。 |
| 路径差异可见 | 银行辅助、纯 DeFi、低人工、人工升级不能被合并。 |
| 权限边界可见 | 无权内容不显示，无权操作被阻断并解释。 |

## 5. Control Tower Projection 的定义

Control Tower Projection 不是数据库表，也不是 UI 组件。

它是：

```text
把融资案件相关的身份、材料、事实、资金方决策、合约状态、Agent 输出、异常和平台运营状态，按照角色权限投影成业务可理解的工作区视图。
```

它必须回答五个问题：

```text
1. 这是什么案件？
2. 现在到了哪一步？
3. 卡在哪里？
4. 谁负责处理？
5. 下一步做什么？
```

## 6. P0 主链路投影

控制塔必须表达以下主链路。

```text
Wallet / Role
→ Financing Intent
→ Basic Trade Info
→ Materials
→ Agent Check
→ Inspection Fact
→ Funding Review / DeFi Rule Path
→ Contract / Rule / Pool Check
→ Funding Result
→ Exception / Next Step
```

| 节点 | 业务含义 | 主要来源 |
|---|---|---|
| Wallet / Role | 当前用户是谁、是什么角色、能看什么。 | Phase 1。 |
| Financing Intent | 出口商融资意图和案件主对象。 | Phase 2。 |
| Basic Trade Info | 买方、商品、金额、币种、贸易基础信息。 | Phase 2。 |
| Materials | 合同、订单、发票、装箱单、补充说明。 | Phase 3。 |
| Agent Check | 摘要、缺口、待办、风险提示。 | Phase 7。 |
| Inspection Fact | 质检任务、材料、Pass/Fail/Dispute。 | Phase 4。 |
| Funding Review / DeFi Rule | 资金方审查或纯 DeFi 规则路径。 | Phase 5 / Phase 6。 |
| Contract / Rule / Pool Check | 执行条件、规则、资金池、暂停状态。 | Phase 6。 |
| Funding Result | 能不能借、能借多少、是否到账或失败。 | Phase 5 / Phase 6。 |
| Exception / Next Step | 卡点、责任方、解除条件和下一步。 | 各阶段异常。 |

## 7. 节点状态表达

每个控制塔节点至少要表达以下状态维度。

| 状态维度 | 说明 |
|---|---|
| Node status | Not started、Waiting、In progress、Need materials、Need review、Blocked、Completed 等。 |
| Responsible party | 当前责任方。 |
| Blocking reason | 如果不能继续，原因是什么。 |
| Impact | 影响材料、质检、审查、执行还是资金结果。 |
| Next step | 下一步动作。 |
| Agent assist | Agent 对该节点的摘要、缺口、建议或升级。 |
| Action availability | 当前角色能不能操作。 |
| Audit hint | 关键状态是否有留痕语义。 |

禁止只显示绿色/红色状态点。

## 8. 出口商 Control Tower

出口商视角是 P0 最核心视角。

| 区域 | 出口商应看到 |
|---|---|
| 案件总览 | 自己的融资案件、融资目标、贸易基础信息、当前状态。 |
| 主链路 | 融资意图 → 材料 → Agent → 质检 → 资金方/DeFi → 合约 → 资金结果。 |
| 材料状态 | 缺什么、冲突在哪里、是否可进入下一步。 |
| Agent 待办 | 自己需要补什么、解释什么、确认什么。 |
| 质检状态 | Pass/Fail/Dispute、补充请求和下一步。 |
| 资金方状态 | 审查中、补材料、允许、拒绝、降额、人工复核。 |
| 合约/规则状态 | Allowed、Waiting、Rule blocked、Pool insufficient、Platform paused、Execution failed。 |
| 资金结果 | 能不能借、能借多少、是否执行、是否到账、失败原因。 |
| 当前卡点 | 原因、责任方、解除条件、下一步。 |

出口商不应看到：

- 其他出口商案件。
- 资金方内部全部风险模型。
- 其他资金方专属信息。
- 平台内部敏感运营细节。

## 9. 资金方 Control Tower

资金方视角围绕审查和执行前置。

| 区域 | 资金方应看到 |
|---|---|
| 待审案件 | 分配/匹配给自己的案件。 |
| 案件摘要 | 出口商、买方、商品、金额、币种、融资类型。 |
| 证据完整性 | 材料完整、缺口、冲突、补充说明。 |
| 质检事实 | Pass/Fail/Dispute、补充说明、争议状态。 |
| Agent 风险建议 | 风险提示、缺口、建议动作，但不等于审批。 |
| 决策区 | 补材料、允许、拒绝、降额、人工复核。 |
| 执行条件 | 资金方允许后是否被规则、资金池、暂停或 dispute 阻断。 |
| 贷后基础入口 | 结果完成、失败、等待、争议等基础状态占位。 |

资金方不应看到或做：

- 修改出口商原始材料。
- 修改质检结论。
- 查看其他资金方专属案件。
- 绕过资金池或平台暂停。

## 10. 质检方 Control Tower

质检方视角围绕任务和事实提交。

| 区域 | 质检方应看到 |
|---|---|
| 质检任务 | 分配给自己的任务。 |
| 任务上下文 | 商品、数量、包装、检查要求和相关基础信息。 |
| 材料要求 | 需要报告、照片、数量说明、包装说明、缺陷说明等。 |
| 质检结论 | Pass、Fail、Dispute。 |
| 补充请求 | 资金方、平台或 Agent 要求补什么。 |
| 争议状态 | Dispute 原因、责任方、解除条件和下一步。 |
| Agent 提醒 | 报告缺口、补充待办、升级原因。 |

质检方不应看到或做：

- 决定放款。
- 修改出口商贸易材料原件。
- 查看资金池内部状态。
- 查看无关质检任务或无关案件。

## 11. 平台 Control Tower

平台视角围绕运营异常，而不是逐单审批。

| 区域 | 平台应看到 |
|---|---|
| 运行总览 | 案件、资金池、规则、合约状态、Agent 自动化、异常队列。 |
| 异常队列 | 缺材料、dispute、规则阻断、资金池不足、平台暂停、执行失败。 |
| 资金池状态 | 可用、不足、暂停、冻结、执行失败。 |
| 规则状态 | Active、Paused、Expired、Pending upgrade、Conflicting。 |
| 合约执行状态 | Allowed、Waiting、Blocked、Failed、Completed。 |
| Agent 自动化状态 | 正常、停止、堆积、需人工接管。 |
| 暂停/恢复影响 | 暂停对象、原因、影响范围、恢复条件。 |

平台不应做：

- 替资金方批准融资。
- 伪造质检、物流、买方或保险事实。
- 删除关键审计记录。
- 把平台做成逐单审批中心。

## 12. Agent 输出投影

Agent 输出必须以辅助身份进入控制塔。

| Agent 输出 | 控制塔表达 | 边界 |
|---|---|---|
| Summary | 案件摘要或节点摘要。 | 不替代原始材料。 |
| Gap | 缺口或冲突提示。 | 不等于拒绝。 |
| Risk Hint | 风险提示。 | 不等于审批。 |
| Recommendation | 建议动作。 | 需要角色确认或受模式约束。 |
| Task | 待办。 | 必须有责任方。 |
| Explanation | 状态解释。 | 不能改写合约或事实含义。 |
| Automation Record | 自动推进记录。 | 必须显示授权和条件。 |
| Escalation Record | 人工升级记录。 | 不等于失败。 |

所有 Agent 输出必须有标识，例如业务语义上明确：

```text
Agent suggested / Agent found / Agent explained / Agent escalated
```

## 13. 合约/规则状态投影

| 合约/规则状态 | 控制塔表达 |
|---|---|
| Allowed | 当前满足执行条件，但不一定已到账。 |
| Waiting | 等待某个条件满足。 |
| Need materials | 缺材料或事实。 |
| Need review | 需要人工复核。 |
| Rule blocked | 当前规则不允许继续。 |
| Pool insufficient | 资金池不足或不可用。 |
| Platform paused | 平台暂停了案件、资金池、规则、执行能力或账户。 |
| Dispute | 有争议，不能自动继续。 |
| Execution failed | 执行失败，需要平台或执行责任方处理。 |
| Completed | 执行完成，资金结果可更新。 |

禁止表达：

- 合约判断贸易真实。
- 合约替资金方审批。
- 合约裁决 dispute。
- Pool insufficient 等于拒绝。
- Allowed 等于到账。

## 14. 异常卡点投影

控制塔中的每个异常必须可处理。

| 异常 | 控制塔必须表达 |
|---|---|
| 缺材料 | 缺什么、谁补、补完怎么继续。 |
| 材料冲突 | 冲突位置、责任方、解释或复核路径。 |
| 质检 Fail | 影响、复核/拒绝/补充路径。 |
| 质检 Dispute | 争议原因、责任方、解除条件。 |
| 资金方补材料 | 请求内容、责任方、下一步。 |
| 资金方拒绝 | 拒绝原因和是否可重新提交。 |
| 资金方降额 | 调整金额、原因、下一步。 |
| 规则阻断 | 规则原因、是否平台处理。 |
| 资金池不足 | 等待、恢复、切换或人工处理。 |
| 平台暂停 | 暂停对象、原因、影响、恢复条件。 |
| 执行失败 | 失败原因、责任方、下一步。 |
| Agent 置信不足 | 人工确认对象和原因。 |

一票否决：

```text
任何异常只显示“失败”或“error”，不允许通过。
```

## 15. 路径差异投影

控制塔必须保留四类路径差异。

| 路径 | 控制塔表达 |
|---|---|
| Bank-assisted path | 资金方审查、决策、批准后执行检查。 |
| Pure DeFi path | 无银行审批，但受事实、规则、资金池和平台暂停约束。 |
| Low-human path | Agent 在授权、低风险和条件满足时推进。 |
| Human-escalation path | 高风险、dispute、失败、权限不清时人工处理。 |

禁止：

- 全部合并成银行审批流。
- 全部合并成自动 DeFi 流。
- 全部显示成普通任务列表。

## 16. 资金结果投影

资金结果必须面向出口商清晰表达。

| 状态 | 出口商表达 |
|---|---|
| Not submitted | 尚未提交，暂无资金结果。 |
| Waiting for materials | 需要补材料。 |
| Waiting for inspection | 等待质检事实。 |
| Under review | 资金方或规则路径处理中。 |
| Need review | 需要人工复核。 |
| Allowed | 条件允许，但还未完成执行。 |
| Reduced | 可融资金额调整。 |
| Rejected | 融资被拒绝，有原因。 |
| Pool insufficient | 资金池不足，等待或处理。 |
| Rule blocked | 规则阻断。 |
| Platform paused | 平台暂停影响执行。 |
| Execution failed | 执行失败。 |
| Completed | 资金执行完成。 |

禁止：

- 案件创建成功 = 融资成功。
- 材料完整 = 融资批准。
- 资金方允许 = 已到账。
- 合约 Allowed = 已到账。

## 17. 控制塔操作入口范围

控制塔可以暴露业务操作入口，但入口必须来自已定义业务动作。

| 角色 | 可见操作入口 |
|---|---|
| 出口商 | 创建/查看案件、补材料、提交说明、查看资金结果、响应待办。 |
| 资金方 | 查看待审、请求补材料、允许、拒绝、降额、人工复核。 |
| 质检方 | 接收任务、提交质检材料、提交 Pass/Fail/Dispute、补充说明。 |
| 平台方 | 查看异常、处理暂停/恢复、处理资金池/规则/执行失败、接管 Agent 自动化。 |

禁止在控制塔里新增未设计业务动作。

## 21. 审计和留痕范围

Phase 8 应保留以下业务留痕要求。

| 留痕事件 | 原因 |
|---|---|
| 控制塔视图访问 | 证明角色可见范围。 |
| 资金结果查看 | 证明出口商查看结果。 |
| Agent 建议采纳/忽略 | 证明人机协作边界。 |
| 卡点处理动作 | 证明异常如何被处理。 |
| 越权查看阻断 | 证明投影权限有效。 |
| 状态解释来源 | 证明控制塔解释不是伪造事实。 |

## 22. Phase 8 验收场景

| 场景 | 前提 | 期望结果 |

## 24. 一票否决项

出现以下任一项，Phase 8 不得通过：

| 一票否决项 | 原因 |
|---|---|
| 控制塔变成普通列表或卡片墙 | 产品定位失败。 |
| 出口商看不到资金结果或卡点 | P0 核心价值失败。 |
| 所有角色看到同一个工作区 | 权限模型失败。 |
| Agent 输出被表达成事实或审批 | Agent 越界。 |
| 合约状态被表达成贸易真实性判断 | 合约越界。 |
| 资金方允许或合约 Allowed 被表达成到账 | 资金结果误导。 |
| 异常只显示技术错误 | 业务不可处理。 |
| 路径差异消失 | DeFi/银行/低人工/人工升级边界失败。 |
| P1/P2 完整工作区进入 P0 主控制塔 | 范围失控。 |
| 旧 proof/hash/dashboard 页面成为主线 | 产品定位倒退。 |
| 直接写 HTML、CSS、组件、路由或 API 聚合实现 | 当前文档阶段越界。 |

## 26. Phase 8 出口条件

Phase 8 完成后，系统应具备以下能力：

```text
1. 出口商、资金方、质检方、平台方都有清晰的 Control Tower 投影范围。
2. 融资案件主链路可以从融资意图看到资金结果。
3. 材料、质检、资金方审查、合约/规则状态和 Agent 输出能汇聚成业务状态。
4. 每个卡点都有原因、责任方、影响、解除条件和下一步。
5. Agent 输出被明确标识为辅助，不伪装成事实或审批。
6. 合约/规则输出被业务解释，但不判断贸易真实性。
7. 银行辅助、纯 DeFi、低人工、人工升级路径差异可见。
8. 出口商能理解资金结果：等待、允许、拒绝、降额、阻断、失败或完成。
9. 该文档可直接作为第一版 HTML 原型图的结构输入。
```

满足以上条件后，可以并行进入两条线：

```text
A. 开始第一版 HTML 原型图。
B. 继续写 79-p0-implementation-phase-9-platform-operation-basics-plan.md。
```

## 27. 本步结论

Phase 8 的本质不是“做 dashboard”。

它的本质是：

> 把 ChainTrace P0 的身份、融资案件、材料、质检、资金方、合约、Agent 和异常统一投影为角色可理解、可处理、可验收的 Control Tower 操作台，让每个角色都知道当前案件在哪里、卡在哪里、谁负责、下一步是什么，以及资金结果到底是什么状态。

从本文件完成开始，P0 已经具备第一版 HTML 原型图的结构基础。
