# 62 P0 技术设计阶段总结

## 1. 定位

本文件总结 ChainTrace P0 技术设计阶段的成果、边界、未进入内容和下一阶段建议。

它不是新的需求文档，不是开发计划，不是接口规格，不是数据模型，不是合约实现方案，不是 UI 原型说明，也不是测试计划。

它的作用是：

> 对 `49-61` 的 P0 技术设计阶段做阶段性收口，明确本阶段已经完成了哪些“设计边界”，哪些内容仍然不能进入实现，下一阶段应如何从“边界设计”进入更具体的“规格准备”。

## 2. 阶段目标回顾

P0 技术设计阶段的目标不是立刻写代码，而是把已经冻结的业务需求转成可进入规格准备的技术设计边界。

本阶段重点解决以下问题：

| 问题 | 结果 |
|---|---|
| ChainTrace P0 主线是什么？ | 出口商融资意图到资金结果的 Agent Native + Crypto Native 供应链金融前站。 |
| 谁能看什么、做什么？ | 单钱包单角色、角色工作区、案件相关可见、越权阻断。 |
| P0 有哪些业务动作？ | 身份、融资案件、材料、事实确认、Agent、资金方、合约、平台、异常。 |
| P0 需要哪些信息域？ | 身份、案件、材料、事实、资金方决策、合约输出、Agent 输出、异常、审计。 |
| 合约负责什么？ | 规则和资金执行约束，不判断贸易真实性，不替资金方审批。 |
| Agent 负责什么？ | 角色助理，整理、摘要、缺口、待办、提醒、低风险推进、人工升级。 |
| 控制塔表达什么？ | 源头到终端、节点状态、卡点、责任方、下一步、资金结果。 |
| 怎么验收 P0？ | 正常闭环、银行辅助、纯 DeFi、低人工、人工升级、权限和异常。 |
| 发现缺口怎么办？ | 回流 waterfall；P1/P2 和实现细节后置。 |

## 4. 本阶段形成的核心结论

### 4.1 产品定位结论

ChainTrace P0 不再是展示型 demo、hash/proof 工具页或普通银行审批后台。

P0 定位是：

```text
Agent Native + Crypto Native 供应链金融前站
```

核心业务主线是：

```text
出口商融资意图
→ 贸易材料和事实整理
→ Agent 缺口和待办
→ 质检等事实确认
→ 资金方或 DeFi 规则路径
→ 合约/规则执行检查
→ 资金结果和卡点表达
```

### 4.2 身份权限结论

P0 坚持单钱包单角色。

| 结论 | 说明 |
|---|---|
| 钱包先登录 | 钱包连接只是入口。 |
| 注册后激活 | 未注册钱包不能进入业务工作区。 |
| 一个钱包一个角色 | P0 不允许多角色随意切换。 |
| 角色不可变 | Agent 模式可变，业务角色不可变。 |
| 角色工作区分离 | 不同角色不是同一个后台换菜单。 |
| 案件相关可见 | 只能看到与自己业务相关的案件、事实、任务和状态。 |

### 4.3 Agent 结论

Agent 是角色助理，不是超级用户。

| Agent 可以做 | Agent 不可以做 |
|---|---|
| 整理材料 | 伪造事实。 |
| 生成摘要 | 替资金方审批。 |
| 检查缺口 | 替质检方出结论。 |
| 生成待办 | 裁决重大 dispute。 |
| 请求补充 | 绕过合约或平台暂停。 |
| 低风险自动推进 | 查看所属角色无权数据。 |
| 人工升级 | 删除或篡改关键记录。 |

### 4.4 合约结论

合约是业务规则和资金执行约束层。

| 合约负责 | 合约不负责 |
|---|---|
| 检查授权。 | 判断贸易真实性。 |
| 检查执行条件。 | 替资金方审批。 |
| 检查资金池。 | 替质检方出 pass/fail/dispute。 |
| 检查规则版本。 | 裁决重大争议。 |
| 检查平台暂停。 | 解释业务材料。 |
| 输出执行结果。 | 当数据库或 Agent。 |

### 4.5 路径结论

P0 必须保留四条路径：

| 路径 | 必须保留的原因 |
|---|---|
| 银行辅助路径 | 服务银行/保理等资金方的现实审批流程。 |
| 纯 DeFi 路径 | 保留 Crypto Native 的资金执行和规则路径。 |
| 低人工路径 | 体现 Agent Native 的效率价值。 |
| 人工升级路径 | 防止高风险、争议、失败和不确定性被自动化误处理。 |

任何设计把四条路径合并成单一银行审批流，或合并成无责任的纯自动 DeFi 流，都不得通过。
