# 22 P0 验收口径

## 1. 定位

本文件定义 ChainTrace P0 MVP 的业务验收口径。

验收口径不是自动化测试用例，不是开发任务，也不是 UI 验收。

它的核心作用是：

> 明确 P0 MVP 在业务上必须让每个角色完成哪些动作、看到哪些结果、理解哪些卡点，才算需求闭环可进入下一阶段。

本文件不写页面、组件、数据库、智能合约实现、钱包签名、hash 或 proof 细节。

## 2. 验收原则

| 原则 | 说明 |
|---|---|
| 按角色验收 | 每个角色都要能完成自己的业务目标。 |
| 按闭环验收 | 不只看单点功能，要看融资流程能否走到结果。 |
| 按异常验收 | 补材料、争议、阻断、等待都必须可解释。 |
| 按 Agent Native 验收 | Agent 必须能整理、检查、提醒、生成待办、升级异常。 |
| 按边界验收 | 不允许角色越权或提前进入技术实现细节。 |

## 3. 出口商验收口径

| 验收项 | 通过标准 |
|---|---|
| 创建融资意图 | 出口商能声明融资类型、金额、币种和期望到账时间。 |
| 上传贸易材料 | 出口商能提交合同、订单、发票、装箱单和说明。 |
| 查看 Agent 缺口 | 出口商能看到缺什么、为什么缺、找谁补。 |
| 确认提交案件 | 材料达到最低要求后，出口商能确认提交。 |
| 查看案件状态 | 出口商能理解 Draft、Submitted、Under review、Approved、Blocked、Funds arrived 等状态。 |
| 处理补材料 | 出口商能按待办补充材料或解释差异。 |
| 查看融资结果 | 出口商能知道允许、拒绝、补材料、等待执行或到账。 |

出口商验收失败示例：

| 失败情况 | 原因 |
|---|---|
| 只看到文件列表，不知道能否融资 | 没有业务闭环。 |
| 只看到错误码，不知道缺什么 | 不符合业务语言要求。 |
| 可以修改质检结论 | 越权。 |
| 可以进入平台规则维护 | 越权。 |

## 4. 质检方验收口径

| 验收项 | 通过标准 |
|---|---|
| 查看待质检任务 | 质检方能看到分配给自己的质检任务。 |
| 理解质检内容 | 质检方能看到货物摘要和质检清单。 |
| 提交质检材料 | 质检方能提交报告、照片、现场说明和异常说明。 |
| 给出统一结论 | 质检方能提交 pass / fail / dispute。 |
| 解释结论原因 | fail 或 dispute 必须带业务原因。 |
| 处理补充要求 | 质检方能看到缺什么并补充。 |

质检方验收失败示例：

| 失败情况 | 原因 |
|---|---|
| 质检方能决定是否放款 | 越权。 |
| 质检方能修改出口商贸易材料 | 证据污染。 |
| 质检结论没有原因 | 资金方无法判断。 |
| Agent 自动伪造质检事实 | 违反 Agent 边界。 |

## 5. 资金方验收口径

| 验收项 | 通过标准 |
|---|---|
| 查看待审案件 | 资金方能看到分配或匹配给自己的融资案件。 |
| 查看案件摘要 | 资金方能理解出口商、贸易、金额、货物、材料和状态。 |
| 查看材料完整性 | 资金方能知道证据完整、不完整或需要人工确认。 |
| 查看质检事实 | 资金方能看到 pass / fail / dispute 及原因。 |
| 查看风险摘要 | 资金方能看到风险提示和 Agent 建议。 |
| 做融资决策 | 需要银行参与的场景中，资金方能选择允许放款、拒绝、补材料、人工复核。 |
| 查看执行条件 | 资金方批准后能看到是否允许执行及卡点。 |

资金方验收失败示例：

| 失败情况 | 原因 |
|---|---|
| 资金方必须手工翻所有文件才能审查 | Agent 摘要缺失。 |
| 资金方批准后不知道为什么不能放款 | 执行条件解释缺失。 |
| 资金方能绕过规则强制执行 | 破坏执行边界。 |
| 资金方能修改质检结论 | 事实边界错误。 |

## 5A. 纯 DeFi 路径验收口径

| 验收项 | 通过标准 |
|---|---|
| 识别无银行场景 | 系统能说明该场景不需要银行/资金方审批。 |
| Agent 完成事实整理 | Agent 能整理材料、事实、缺口和规则适配结果。 |
| 合约检查执行条件 | DeFi 智能合约能检查规则、资金池、抵押/保证、事实状态和暂停状态。 |
| 自动执行或阻断 | 条件满足时执行资金动作，条件不满足时返回业务原因。 |
| 人工升级 | 争议、高风险、法律责任冲突或规则例外必须升级人处理。 |

纯 DeFi 路径验收失败示例：

| 失败情况 | 原因 |
|---|---|
| 无银行场景仍强制等待资金方审批 | 场景模型错误。 |
| 合约只做展示，不负责资金执行 | DeFi 核心定位错误。 |
| Agent 自动裁决重大法律争议 | 越权。 |

## 6. 平台方验收口径

| 验收项 | 通过标准 |
|---|---|
| 查看运行总览 | 平台方能知道资金池、规则、执行、Agent 是否正常。 |
| 查看异常案件 | 平台方能看到需要平台处理的异常。 |
| 处理资金池状态 | 平台方能对资金池不足、暂停、异常给出处理。 |
| 处理规则状态 | 平台方能查看规则版本、暂停、阻断和影响范围。 |
| 处理执行失败 | 平台方能看到失败原因并标记下一步。 |
| 管理 Agent 异常 | 平台方能看到 Agent 堆积、失败、人工升级。 |
| 保留操作记录 | 平台动作必须有原因和记录。 |

平台方验收失败示例：

| 失败情况 | 原因 |
|---|---|
| 平台方默认逐单审批正常案件 | 角色定位错误。 |
| 平台方能伪造贸易材料 | 事实边界错误。 |
| 平台暂停/恢复没有原因 | 审计能力不足。 |
| 平台可以删除关键记录 | 破坏可信边界。 |

## 7. Agent Native 验收口径

| 验收项 | 通过标准 |
|---|---|
| 三种模式清楚 | 全人工确认、半自动托管、全自动 Agent 差异明确。 |
| Agent 能整理摘要 | 不同角色能看到适合自己的业务摘要。 |
| Agent 能检查缺口 | 缺什么、为什么缺、谁处理清楚。 |
| Agent 能生成待办 | 待办有责任角色、建议动作、完成标准、回到状态。 |
| Agent 能提醒状态 | 状态变化、超时、风险、授权节点可提醒。 |
| Agent 能升级异常 | 高风险、争议、放款授权、平台控制交给人。 |
| Agent 能支持低人工路径 | 规则清楚时，Agent 能推进纯 DeFi、保证金释放、结算或法律条款明确的流程。 |
| Agent 不越权 | 不伪造事实、不裁决重大争议、不替资金方高风险放款。 |

## 8. 状态和异常验收口径

| 验收项 | 通过标准 |
|---|---|
| 状态可解释 | 每个状态有当前原因和下一步。 |
| 补材料可回流 | 补完后回到准备、审查或执行检查。 |
| 质检 fail/dispute 可处理 | 进入人工复核、拒绝、补材料或阻断。 |
| 规则阻断可解释 | 用户知道是哪类规则或平台状态影响。 |
| 资金池不足可解释 | 银行辅助或纯 DeFi 场景中，资金池不足时状态清楚。 |
| 执行失败可处理 | 平台方知道需要处理执行异常。 |
| 历史不被覆盖 | 补充材料不删除原始事实和关键记录。 |

## 9. 排除范围验收口径

如果出现以下内容，不视为 P0 需求验收通过：

| 跑偏项 | 原因 |
|---|---|
| 做登录前展示页 | 当前明确不做。 |
| 做 UI 组件或页面布局 | 当前仍是业务需求阶段。 |
| 写数据库设计 | 当前不进入技术设计。 |
| 写智能合约实现 | 当前只定义业务角色。 |
| 写钱包签名流程 | 当前不进入实现。 |
| 直接进入 hash / proof / proof pack | 当前不跳到底层证明细节。 |
| 继承旧版 app/components/lib/contracts/scripts 代码 | 当前分支是全新瀑布重建设计分支。 |

## 10. MVP 验收结论

P0 MVP 业务需求可以进入下一阶段的条件是：

| 条件 | 要求 |
|---|---|
| 角色能完成核心动作 | 出口商、质检方、资金方、平台方各自闭环；纯 DeFi 场景不强制资金方参与。 |
| Agent 能推进正常案件 | 整理、检查、待办、提醒、升级清楚。 |
| 纯 DeFi 能独立闭环 | 无银行场景能通过 Agent + DeFi 智能合约完成执行或阻断。 |
| 异常能解释和回流 | 缺口、责任角色、下一步、状态回流清楚。 |
| 执行约束能解释 | 资金池、规则、平台、合约业务输出清楚。 |
| 排除范围没有被突破 | 没有提前进入 UI、数据库、合约、hash/proof。 |

## 11. 本步结论

P0 验收口径的本质不是“测试列表”。

它的本质是：

> 确认每个角色能完成自己的业务结果，Agent 能推进和升级流程，异常能解释和回流，系统没有提前跳到实现细节。
