# 63 P0 接口规格准备

## 1. 定位

本文件定义 ChainTrace P0 规格准备阶段中的接口规格准备边界。

它不是完整 API 文档，不是 endpoint 清单，不是 HTTP 方法设计，不是请求/响应 JSON，不是错误码格式，不是鉴权 header 设计，也不是接口实现代码。

它的作用是：

> 从 `54-p0-business-action-interface-boundary.md` 已定义的 P0 业务动作出发，建立“业务动作 → 接口规格候选”的准备框架，明确后续接口规格应该覆盖哪些业务能力、每类接口必须保留哪些业务语义、哪些内容还不能进入实现。

本文件只做接口规格准备，不写具体接口实现。

本文件主要继承以下设计边界：

| 来源文档 | 继承内容 |
|---|---|
| 52-p0-business-input-to-design-domain-map | 接口规格必须能追溯到冻结业务输入。 |
| 53-p0-permission-design-boundary | 接口能力必须遵守单钱包单角色和角色权限边界。 |
| 54-p0-business-action-interface-boundary | 接口规格准备的直接输入，所有接口候选来自业务动作。 |
| 55-p0-data-information-domain-design | 接口涉及的信息必须来自已定义信息域。 |
| 56-p0-smart-contract-business-design | 合约相关接口只表达执行检查和业务输出，不写合约实现。 |
| 57-p0-agent-business-design | Agent 相关接口必须继承角色权限，不越权。 |
| 58-p0-control-tower-expression-design | 查询类接口必须支持控制塔业务表达。 |
| 59-p0-business-acceptance-design | 接口规格后续必须支持业务验收场景。 |
| 60-p0-design-feedback-and-deferred-scope | 新动作、新状态、新异常必须回流或后置。 |

## 3. 接口规格准备原则

| 原则 | 说明 |
|---|---|
| 从业务动作派生 | 不凭空设计接口，只从 54 的业务动作派生接口候选。 |
| 保留业务语义 | 接口不是 CRUD，而是业务动作、业务状态和业务结果。 |
| 不写 endpoint | 当前不定义路径、HTTP 方法、RPC 名称。 |
| 不写 JSON | 当前不定义请求体、响应体、字段类型。 |
| 不写错误码 | 当前只定义业务失败解释，不定义错误码格式。 |
| 不写鉴权实现 | 当前只继承权限边界，不写 token、header、middleware。 |
| 不写代码 | 当前不进入 controller、service、route handler 或 SDK。 |
| 每个接口候选必须有角色 | 谁可以触发、谁可以查看必须清楚。 |
| 每个接口候选必须有业务结果 | 不能只有技术成功/失败。 |
| 缺口必须回流 | 出现未定义业务动作或状态时回流 54/59/60。 |

## 4. 接口候选分组

P0 接口规格准备阶段暂按以下业务能力分组：

| 接口候选组 | 业务目的 | 主要继承来源 |
|---|---|---|
| 身份与注册接口候选 | 钱包连接后的注册状态、角色绑定、账户状态、Agent 模式。 | 53、54、55 |
| 融资案件接口候选 | 出口商创建融资意图、提交、补材料、查看结果。 | 54、55、58、59 |
| 贸易材料接口候选 | 上传、补充、锁定、摘要和缺口表达。 | 54、55、57 |
| 事实确认接口候选 | 质检、物流、保险、买方、通关等事实确认。 | 54、55 |
| Agent 辅助接口候选 | 摘要、缺口、待办、请求补充、提醒、升级。 | 57、58 |
| 资金方决策接口候选 | 查看案件、补材料、允许、拒绝、降额、人工复核。 | 53、54、55、59 |
| 合约/规则执行接口候选 | 执行条件检查、资金池、规则、暂停、执行结果。 | 56、58、59 |
| 平台运营接口候选 | 资金池、规则、合约、异常、Agent 状态的暂停/恢复和处理。 | 53、54、56、60 |
| 控制塔查询接口候选 | 不同角色查看主链路、节点状态、卡点和资金结果。 | 58、59 |
| 异常处理接口候选 | 缺材料、dispute、资金池不足、规则阻断、平台暂停、执行失败。 | 54、58、59、60 |

## 5. 通用接口规格元信息

后续每个接口规格应至少描述以下业务元信息。

| 元信息 | 说明 |
|---|---|
| 业务动作名称 | 该接口候选承接哪个业务动作。 |
| 业务目的 | 用户为什么需要这个能力。 |
| 触发角色 | 哪个角色可以触发。 |
| 可见角色 | 哪些角色可以看到结果。 |
| 业务前提 | 触发前必须满足什么。 |
| 业务输入摘要 | 业务上需要哪些信息，不写字段。 |
| 业务输出摘要 | 成功、等待、阻断、失败等业务结果。 |
| 权限边界 | 继承 53 的哪类权限限制。 |
| 信息域影响 | 影响 55 的哪些信息域。 |
| Agent 参与 | Agent 是否可触发、建议、提醒或升级。 |
| 合约参与 | 是否涉及合约/规则执行检查。 |
| 异常和失败解释 | 失败时业务原因、责任方、下一步。 |
| 后置实现项 | endpoint、JSON、错误码、鉴权、代码后置。 |

## 6. 身份与注册接口候选

| 业务动作 | 接口规格候选目的 | 触发角色 | 业务结果 |
|---|---|---|---|
| 检查钱包注册状态 | 判断钱包是否已注册、激活、受限或暂停。 | 已连接钱包用户 / 系统 | 未注册 / 已激活 / 受限 / 暂停。 |
| 提交注册信息 | 创建业务账户并绑定唯一角色。 | 未注册钱包用户 | 注册成功 / 需补信息 / 角色冲突。 |
| 激活账户 | 使账户进入可用状态。 | 系统 / 平台规则 | 已激活 / 受限 / 暂停。 |
| 查看账户状态 | 让用户知道自己能否进入业务工作区。 | 钱包用户 | 可进入 / 受限 / 暂停 / 需补资料。 |
| 设置 Agent 模式 | 设置全人工、半自动或全自动模式。 | 已激活用户 | 模式生效 / 权限不足 / 受限。 |

边界：

- 当前不写钱包签名流程。
- 当前不写 token/session/header。
- 当前不允许一个钱包多角色切换。
- Agent 模式接口不得改变永久角色。

## 7. 融资案件接口候选

| 业务动作 | 接口规格候选目的 | 触发角色 | 业务结果 |
|---|---|---|---|
| 创建融资意图 | 形成融资案件草稿。 | 出口商 / 出口商 Agent | 草稿创建 / 权限不足 / 需补资料。 |
| 更新融资目标 | 补充融资金额、币种、期望到账等业务目标。 | 出口商 / 授权 Agent | 已更新 / 冲突 / 权限不足。 |
| 添加贸易基本信息 | 记录出口商、买方、商品、金额等业务信息。 | 出口商 / 授权 Agent | 已记录 / 需补信息 / 冲突。 |
| 提交融资申请 | 把案件从草稿提交给资金方或规则路径。 | 出口商 / 授权 Agent | 已提交 / 需补材料 / 规则阻断。 |
| 撤回草稿 | 出口商撤回未提交或允许撤回的案件。 | 出口商 | 已撤回 / 不允许撤回。 |
| 查看案件状态 | 查看案件主链路当前状态。 | 案件相关角色 | 草稿 / 已提交 / 补材料 / 审查中 / 放款中 / 完成等。 |
| 查看资金结果 | 出口商查看能否融资、金额、到账和卡点。 | 出口商 | 可融资 / 不可融资 / 等待 / 失败 / 已到账。 |

边界：

- 融资案件接口不是普通 CRUD。
- 提交后不能无痕修改原始事实。
- 出口商不能通过接口直接触发高风险放款。
- 资金结果必须包含业务解释，而不是只返回技术状态。

## 8. 贸易材料接口候选

| 业务动作 | 接口规格候选目的 | 触发角色 | 业务结果 |
|---|---|---|---|
| 提交合同材料 | 为案件提供合同证据。 | 出口商 | 已接收 / 需补信息 / 格式或内容异常。 |
| 提交订单材料 | 为案件提供买方订单证据。 | 出口商 | 已接收 / 缺关键信息。 |
| 提交发票材料 | 为案件提供金额和付款义务证据。 | 出口商 | 已接收 / 金额冲突 / 需解释。 |
| 提交装箱单材料 | 为案件提供数量和包装证据。 | 出口商 | 已接收 / 数量冲突 / 需解释。 |
| 提交补充说明 | 解释材料缺口、金额冲突或特殊条款。 | 出口商 / 责任方 | 已提交 / 仍需补充。 |
| 锁定提交版本 | 案件正式提交后形成提交版本。 | 系统 | 已锁定 / 不可覆盖。 |
| 查看材料完整性 | 查看材料是否完整、缺什么、冲突在哪里。 | 案件相关角色 | 完整 / 缺口 / 冲突 / 需人工确认。 |

边界：

- 材料接口不能允许资金方或平台无痕修改出口商原始材料。
- Agent 可以摘要和检查缺口，但不能伪造材料。
- 后续接口规格需要区分原始提交、补充说明和版本历史，但当前不写字段。

## 9. 事实确认接口候选

### 9.1 质检事实接口候选

| 业务动作 | 接口规格候选目的 | 触发角色 | 业务结果 |
|---|---|---|---|
| 接收质检任务 | 质检方确认是否接收任务。 | 质检方 | 已接收 / 拒绝 / 需改派。 |
| 提交质检材料 | 上传报告、照片、说明等质检材料。 | 质检方 | 已提交 / 材料不足。 |
| 提交质检结论 | 给出 pass/fail/dispute。 | 质检方 | Pass / Fail / Dispute。 |
| 补充质检说明 | 响应资金方、平台或 Agent 的补充请求。 | 质检方 | 补充完成 / 仍需补充。 |
| 查看质检状态 | 查看本案质检节点状态。 | 案件相关角色 | 待质检 / Pass / Fail / Dispute / 需补充。 |

### 9.2 P1/P2 事实接口占位

| 事实类型 | P0 接口处理 | 后置原因 |
|---|---|---|
| 物流事实 | P0 可保留状态查询和材料占位。 | 完整物流工作区 P1。 |
| 保险事实 | P0 可保留承保/拒保/需补材料占位。 | 完整保险流程 P1。 |
| 买方事实 | P0 可保留买方确认/争议占位。 | 完整买方工作区 P1。 |
| 通关事实 | P0 只保留后置标记。 | 海关/报关接入 P2。 |

边界：

- 事实确认接口必须由对应事实方触发。
- Agent、平台、资金方不能通过接口伪造事实。
- 合约只能使用事实状态，不生成事实。

## 10. Agent 辅助接口候选

| 业务动作 | 接口规格候选目的 | 触发角色 | 业务结果 |
|---|---|---|---|
| 生成案件摘要 | 帮角色理解案件。 | 所属角色 / Agent | 摘要完成 / 置信不足。 |
| 检查材料缺口 | 找出缺少的材料和事实。 | 所属角色 / Agent | 缺口清单 / 无缺口 / 需人工确认。 |
| 生成角色待办 | 把缺口或异常转成下一步。 | Agent | 待办生成 / 责任方不清需回流。 |
| 请求补充材料 | 向责任方请求补充材料或事实。 | 授权 Agent / 角色 | 请求已发出 / 权限不足 / 责任方不清。 |
| 提醒状态变化 | 提醒提交、审查、放款、到账、异常。 | Agent | 提醒生成。 |
| 自动推进低风险流程 | 在授权和低风险条件下推进。 | 授权 Agent | 已推进 / 需人工 / 被规则阻断。 |
| 升级人工处理 | 高风险、争议、失败、置信不足时交给人。 | Agent | 已升级给责任角色。 |

边界：

- Agent 接口必须继承所属角色权限。
- Agent 建议不能通过接口被当成最终审批。
- Agent 摘要不能通过接口被当成原始事实。
- Agent 自动推进必须可追踪，并受合约/规则边界约束。

## 11. 资金方决策接口候选

| 业务动作 | 接口规格候选目的 | 触发角色 | 业务结果 |
|---|---|---|---|
| 查看待审案件 | 资金方查看自己可处理案件。 | 资金方 | 案件列表 / 无权限 / 无待审。 |
| 查看案件审查摘要 | 看材料、事实、风险和缺口。 | 资金方 / 资金方 Agent | 摘要可读 / 需补材料。 |
| 要求补材料 | 向出口商或事实方提出补充要求。 | 资金方 | 请求已发出。 |
| 允许融资 | 做出资金方批准。 | 资金方 | 已批准，进入执行检查。 |
| 拒绝融资 | 做出拒绝并说明原因。 | 资金方 | 已拒绝，返回原因。 |
| 降低融资金额 | 给出降额或调整融资比例。 | 资金方 | 降额建议 / 需确认。 |
| 转人工复核 | 高风险或争议进入人工复核。 | 资金方 / Agent | 已进入人工复核。 |
| 查看贷后状态 | 查看回款、逾期、争议、保险等。 | 资金方 | 正常 / 逾期 / 争议 / 后置。 |

边界：

- 资金方决策接口不能修改出口商原始材料。
- 资金方批准接口不等于合约执行完成。
- 资金方不能通过接口绕过资金池、规则或平台暂停。

## 12. 合约 / 规则执行接口候选

| 业务动作 | 接口规格候选目的 | 触发角色 | 业务结果 |
|---|---|---|---|
| 请求执行条件检查 | 检查案件是否可进入执行。 | 有权角色 / Agent / 系统 | Allowed / Waiting / Rule blocked / Need review。 |
| 查询资金池业务状态 | 查看资金池是否可用、足够、暂停或失败。 | 平台 / 相关资金方 / 受影响角色 | Available / Low balance / Paused / Failed。 |
| 查询规则状态 | 查看规则是否有效、暂停、过期或不适用。 | 平台 / 受影响角色 | Active / Paused / Expired / Not applicable。 |
| 查询平台暂停影响 | 说明案件、资金池、规则或执行能力是否暂停。 | 相关角色 | Platform paused / Available。 |
| 输出执行结果 | 返回执行完成、等待、失败或阻断。 | 合约/规则执行层 | Completed / Execution failed / Waiting / Blocked。 |

边界：

- 当前不写 Solidity、ABI、函数签名。
- 当前不写链上交易、gas、nonce、revert。
- 合约相关接口不能把合约输出表达成贸易真实性判断。
- 合约相关接口不能替资金方做银行辅助路径审批。

## 13. 平台运营接口候选

| 业务动作 | 接口规格候选目的 | 触发角色 | 业务结果 |
|---|---|---|---|
| 查看运行总览 | 看资金池、规则、合约、异常、Agent 状态。 | 平台方 | 总览可读。 |
| 暂停资金池 | 处理资金池不足、异常或风险。 | 平台方 | 资金池暂停，影响范围明确。 |
| 恢复资金池 | 暂停原因解除后恢复。 | 平台方 | 资金池恢复。 |
| 暂停规则 | 规则存在风险或维护。 | 平台方 | 规则暂停，相关案件等待或阻断。 |
| 恢复规则 | 规则复核后恢复。 | 平台方 | 规则恢复。 |
| 暂停案件 | 案件疑似欺诈或系统级异常。 | 平台方 | 案件暂停。 |
| 分派异常 | 将异常交给责任方处理。 | 平台方 / 平台 Agent | 异常已分派。 |
| 查看 Agent 异常 | 查看自动化失败、堆积、需人工接管。 | 平台方 | 异常可见。 |

边界：

- 平台运营接口不能变成逐单审批接口。
- 平台不能通过接口替资金方批准放款。
- 平台高影响动作必须保留业务原因、影响范围和恢复条件。

## 14. 控制塔查询接口候选

| 查询能力 | 接口规格候选目的 | 可见角色 | 业务结果 |
|---|---|---|---|
| 查询案件主链路 | 表达从融资意图到资金结果的节点状态。 | 案件相关角色 | 节点状态、卡点、下一步。 |
| 查询出口商资金结果 | 出口商看能不能借、能借多少、是否到账。 | 出口商 | 可融资 / 不可融资 / 等待 / 到账 / 失败。 |
| 查询资金方审查视图 | 资金方看证据、风险、决策和执行条件。 | 资金方 | 摘要、缺口、风险、决策状态。 |
| 查询质检任务视图 | 质检方看任务、清单、结论、补充请求。 | 质检方 | 待处理 / Pass / Fail / Dispute。 |
| 查询平台运营视图 | 平台看资金池、规则、合约、异常、Agent 状态。 | 平台方 | 正常 / 异常 / 暂停 / 堆积。 |
| 查询异常卡点 | 看到卡点原因、责任方、解除条件和下一步。 | 相关角色 | 可处理卡点说明。 |

边界：

- 控制塔查询接口必须继承角色可见范围。
- 不能把平台视图做成无边界全量案件查看。
- 查询结果必须服务业务表达，不是原始内部数据倾倒。

## 15. 异常处理接口候选

| 异常类型 | 接口规格候选目的 | 责任角色 | 业务结果 |
|---|---|---|---|
| 缺材料 | 请求补齐材料。 | 出口商 / 事实方 | 补充完成 / 仍缺材料。 |
| 金额不一致 | 请求解释或修正。 | 出口商 / 资金方 | 解释接受 / 需复核 / 拒绝。 |
| 质检 Fail | 进入拒绝、复检或人工复核。 | 质检方 / 资金方 | 拒绝 / 复检 / 人工。 |
| 质检 Dispute | 请求补充说明或复检。 | 质检方 / 出口商 / 资金方 | 争议解除 / 继续争议。 |
| 资金池不足 | 等待、恢复、切换路径或人工处理。 | 平台 / 资金方 | 等待 / 恢复 / 失败。 |
| 规则阻断 | 查看阻断原因并处理。 | 平台 / 责任方 | 恢复 / 继续阻断 / 后置。 |
| 平台暂停 | 等待恢复或人工处理。 | 平台方 | 恢复 / 继续暂停。 |
| 执行失败 | 平台或责任方处理。 | 平台 / 执行责任方 | 重试 / 关闭 / 人工处理。 |
| Agent 置信不足 | 升级给所属角色。 | 所属角色 | 人工确认 / 等待。 |

边界：

- 异常处理接口必须返回业务解释、责任方和下一步。
- 不能只返回技术失败。
- 权限不清的异常不能继续写接口，应回流 60。

## 16. 接口候选到验收场景映射

## 18. 接口规格准备通过标准

| 检查项 | 通过标准 |
|---|---|
| 来源清楚 | 每个接口候选能追溯到 54 的业务动作。 |
| 角色清楚 | 每个接口候选知道触发角色和可见角色。 |
| 前提清楚 | 每个接口候选有业务前提。 |
| 结果清楚 | 每个接口候选有业务结果。 |
| 失败解释清楚 | 失败时有原因、责任方和下一步。 |
| 权限继承清楚 | 未突破 53 的权限设计。 |
| 数据影响清楚 | 知道影响 55 的哪些信息域。 |
| Agent/合约边界清楚 | 不让 Agent 或合约越权。 |
| 路径差异保留 | 银行辅助、纯 DeFi、低人工、人工升级都被覆盖。 |
| 实现细节后置 | 未出现 endpoint、JSON、错误码、header 或代码。 |

## 19. 一票否决项

| 一票否决项 | 原因 |
|---|---|
| 直接把业务动作写成 endpoint 实现 | 阶段越界。 |
| 接口候选没有业务动作来源 | 可能凭空造功能。 |
| 接口候选只有 CRUD，没有业务语义 | 破坏业务动作边界。 |
| 接口允许一个钱包切换多个角色 | 破坏身份边界。 |
| 接口允许资金方修改出口商原始材料 | 破坏事实可信。 |
| 接口允许 Agent 伪造事实或自动裁决 dispute | Agent 越权。 |
| 接口允许合约判断贸易真实性 | 合约越界。 |
| 接口把所有路径合并成银行审批 | 破坏 DeFi/银行双路径。 |
| P1/P2 接口被塞入 P0 必做 | MVP 范围失控。 |
| 旧 proof/hash/demo 接口成为 P0 主线 | 破坏当前产品定位。 |

## 20. 回流规则

| 发现问题 | 处理方式 |
|---|---|
| 需要新增业务动作 | 回流 54。 |
| 需要新增角色或修改角色权限 | 回流 53。 |
| 需要新增信息域 | 回流 55。 |
| 需要新增合约业务输出 | 回流 56。 |
| Agent 自动化边界不清 | 回流 57。 |
| 控制塔查询结果表达不足 | 回流 58。 |

| P1/P2 或实现细节混入 | 回流 60 并后置。 |

## 21. 本步结论

P0 接口规格准备的本质不是“开始写 API”。

它的本质是：

> 从 P0 业务动作出发，建立接口规格的候选能力边界，确保后续真正写接口规格时，每个接口都服务清晰业务动作，继承角色权限，影响明确信息域，输出可理解业务结果，并保留 Agent、合约、控制塔、路径差异和异常处理边界。

下一步应进入：

> `64-p0-data-model-preparation.md`

即从 `55-p0-data-information-domain-design.md` 出发，准备数据模型，但仍先做信息域到概念模型的映射，不直接写数据库表结构、字段、索引或 ORM 实现。
