# 75 P0 实现阶段 5：Funding Review Path 计划

## 1. 定位

本文件定义 ChainTrace P0 受控实现阶段的第五个阶段：Funding Review Path。

它不是资金方接口实现，不是风控模型设计，不是授信系统，不是银行核心系统集成，不是数据库表结构，不是前端审批页面实现，也不是合约放款实现。

它的作用是：

> 在 Phase 2 融资案件、Phase 3 贸易材料、Phase 4 质检事实节点成立之后，细化 Phase 5 的资金方审查路径：待审案件、证据完整性、Agent 风险建议、补材料请求、允许融资、拒绝融资、降额、人工复核，以及资金方决策对后续合约/规则执行检查的影响。

本阶段的核心目标不是“放款”，而是：

```text
资金方工作区
→ 待审融资案件
→ 证据完整性和事实状态
→ Agent 风险建议
→ 补材料 / 允许 / 拒绝 / 降额 / 人工复核
→ 资金方决策留痕
→ 进入合约/规则/资金池执行检查前置
```

## 3. Phase 5 阶段目标

Phase 5 必须让系统具备以下基础能力：

| 目标 | 说明 |
|---|---|
| 资金方能看到待审案件 | 只看自己有权审查的融资案件。 |
| 证据完整性可表达 | 贸易材料、质检事实和缺口状态可见。 |
| Agent 风险建议可表达 | Agent 可整理风险、缺口、建议，但不替资金方决策。 |
| 补材料请求可表达 | 资金方可要求出口商补材料或解释冲突。 |
| 允许融资可表达 | 资金方可作出允许融资决策，但不等于放款完成。 |
| 拒绝融资可表达 | 资金方可拒绝，并给出业务原因。 |
| 降额可表达 | 资金方可基于风险或证据调整融资金额。 |
| 人工复核可表达 | 高风险、Dispute、置信不足等进入人工复核。 |
| 决策留痕可表达 | 决策、原因、状态变化必须可追踪。 |
| 执行前置可表达 | 批准后进入 Phase 6 合约/规则/资金池检查。 |

## 4. 本阶段非目标

Phase 5 明确不做以下内容：

| 非目标 | 后置阶段 |
|---|---|
| 银行核心系统集成 | 后续外部系统集成。 |
| 保理系统真实授信 | 后续金融机构集成。 |
| 自动风控模型 | 后续风险模型能力。 |
| 多资金方竞价/撮合 | 后续资金市场。 |
| 真实放款执行 | Phase 6 之后再接执行表达，真实资金执行后置。 |
| 合约/规则/资金池实现 | Phase 6。 |
| 复杂贷后催收 | 后续贷后模块。 |
| 保险赔付触发 | P1/P2 后续保险能力。 |
| 买方付款自动结算 | P1 买方路径成熟后。 |
| hash/proof 产品页 | 后续证明产品。 |

## 5. 资金方审查对象范围

Phase 5 的中心对象是 Funding Review。

| 概念对象 | 本阶段用途 | 边界 |
|---|---|---|
| Funding Review | 表达资金方审查过程。 | 只能由有权资金方处理。 |
| Evidence Completeness View | 表达证据是否完整。 | 不是新的事实来源。 |
| Risk Note | 表达风险提示和判断依据。 | 可来自 Agent 或人工备注，但不等于审批。 |
| Funding Decision | 表达允许、拒绝、降额、补材料、人工复核。 | 资金方确认。 |
| Decision Reason | 表达决策原因。 | 必须可理解和留痕。 |
| Funding Result Entry | 表达资金结果入口。 | Phase 5 不等于执行完成。 |

## 6. 资金方工作区范围

Phase 5 中资金方工作区应支持审查路径。

| 工作区区域 | Phase 5 表达 |
|---|---|
| 待审案件列表 | 只显示分配/匹配给该资金方的案件。 |
| 案件摘要 | 出口商、买方、商品、金额、融资类型、路径。 |
| 证据完整性 | 合同、订单、发票、装箱单、补充说明、质检状态。 |
| Agent 风险建议 | 缺口、冲突、质检状态、资金路径建议。 |
| 补材料请求 | 向出口商或责任方请求补充。 |
| 决策动作 | 允许、拒绝、降额、人工复核。 |
| 执行条件入口 | 批准后进入合约/规则/资金池检查。 |
| 决策历史 | 决策、原因、时间、责任方留痕语义。 |

禁止：

- 资金方修改出口商原始材料。
- 资金方伪造质检结论。
- 资金方批准后直接显示到账。
- 资金方查看其他资金方专属案件。

## 7. 证据完整性范围

资金方审查时，应看到证据完整性视图。

| 证据项 | 状态来源 | 资金方可做 |
|---|---|---|
| 合同材料 | Phase 3 贸易材料。 | 查看完整性，要求补充。 |
| 订单材料 | Phase 3 贸易材料。 | 查看完整性，要求补充。 |
| 发票材料 | Phase 3 贸易材料。 | 查看金额和冲突，要求解释。 |
| 装箱单材料 | Phase 3 贸易材料。 | 查看数量和包装基础。 |
| 补充说明 | Phase 3 补充说明。 | 查看并判断是否需要更多说明。 |
| 质检状态 | Phase 4 质检事实。 | 查看 Pass/Fail/Dispute，不能修改。 |
| Agent 缺口 | Phase 3/7 Agent 输出。 | 作为辅助参考，不是事实。 |

边界：

```text
Evidence Completeness View 是审查视图，不是新的事实来源。
```

## 8. Agent 风险建议范围

资金方 Agent 可辅助审查，但不能替资金方审批。

| Agent 输出 | 业务用途 | 边界 |
|---|---|---|
| Case summary | 总结案件基本信息。 | 不替代原始材料。 |
| Evidence gap summary | 汇总材料和质检缺口。 | 不等于拒绝结论。 |
| Risk note | 提醒金额冲突、质检 dispute、路径风险。 | 不等于资金方决策。 |
| Next action suggestion | 建议补材料、人工复核、等待或拒绝。 | 资金方人工确认。 |
| Escalation reason | 说明为什么不能自动继续。 | 不裁决 dispute。 |

一票否决：

```text
Agent 风险建议不得被表达成资金方最终审批。
```

## 9. 资金方决策类型

Phase 5 应支持以下资金方决策类型。

| 决策类型 | 业务含义 | 后续影响 |
|---|---|---|
| Need materials | 证据不足或冲突，需要补材料。 | 回到 Phase 3 补材料或补充说明。 |
| Allow financing | 资金方允许融资。 | 进入 Phase 6 执行条件检查。 |
| Reject financing | 资金方拒绝融资。 | 出口商看到拒绝原因。 |
| Reduce amount | 资金方降低可融资金额。 | 出口商看到调整结果，后续可进入执行检查或确认。 |
| Manual review | 需要人工复核。 | 停止自动推进，进入人工处理。 |
| Wait for fact | 等待质检或其他事实节点。 | 等待 Phase 4 或 P1/P2 占位事实。 |

边界：

- Allow financing 不等于 Completed。
- Reject financing 必须有原因。
- Reduce amount 不是资金到账。
- Manual review 不是失败，而是风险控制。

## 10. 补材料请求范围

| 请求类型 | 触发原因 | 责任方 | 下一步 |
|---|---|---|---|
| 补合同 | 合同缺失或不清楚。 | 出口商。 | 回到 Phase 3 补合同。 |
| 补订单 | 订单缺失或主体不清。 | 出口商。 | 回到 Phase 3 补订单。 |
| 补发票 | 发票缺失或金额冲突。 | 出口商。 | 回到 Phase 3 补发票或说明。 |
| 补装箱单 | 数量或包装信息不足。 | 出口商。 | 回到 Phase 3 补装箱单。 |
| 解释金额差异 | 金额冲突。 | 出口商。 | 提交补充说明。 |
| 质检补充 | 质检材料不足或 dispute。 | 质检方。 | 回到 Phase 4 补充质检说明。 |
| 人工复核 | 风险或证据复杂。 | 资金方。 | 资金方人工处理。 |

补材料请求必须形成卡点和待办，不得只显示“资料不足”。

## 11. 对 Phase 6 合约/规则的影响

Phase 5 不做执行，但必须为 Phase 6 提供决策前置。

| 资金方状态 | Phase 6 可能处理 |
|---|---|
| Allow financing | 可进入执行条件检查。 |
| Need materials | 输出 Need materials / Waiting。 |
| Reject financing | 不进入执行，出口商查看拒绝原因。 |
| Reduce amount | 按调整金额进入确认或执行前置。 |
| Manual review | 输出 Need review 或等待人工。 |
| Wait for fact | 输出 Waiting。 |

关键边界：

```text
资金方决策是执行条件之一，不是执行完成。
```

## 13. 控制塔表达范围

Phase 5 需要增强控制塔中的资金方审查节点。

| 控制塔节点 | Phase 5 表达 |
|---|---|
| Financing intent | 已创建。 |
| Materials | 完整 / 缺口 / 冲突 / 需补充。 |
| Inspection fact | Pass / Fail / Dispute / Need supplement。 |
| Funding review | Waiting / Under review / Need materials / Allowed / Rejected / Reduced / Manual review。 |
| Contract execution | Phase 6 接入。 |
| Funding result | 批准后仍等待执行检查；拒绝时显示拒绝原因。 |

控制塔必须明确：

```text
资金方审查结果是什么；是否还缺材料；是否批准；批准后为什么还没到账；下一步是谁处理。
```

## 14. 出口商可见表达

出口商在 Phase 5 应能看到与自己相关的资金方状态。

| 资金方状态 | 出口商看到 |
|---|---|
| Under review | 资金方正在审查。 |
| Need materials | 需要补哪些材料，为什么。 |
| Allow financing | 资金方已允许，等待执行条件检查。 |
| Reject financing | 被拒绝及原因。 |
| Reduce amount | 可融资金额调整和原因。 |
| Manual review | 进入人工复核，等待处理。 |
| Wait for fact | 等待质检或其他事实。 |

禁止：

- 出口商查看资金方内部全部风险模型。
- 出口商修改资金方决策。
- 出口商把 Allow financing 理解为已到账。

## 19. Phase 5 通过标准

| 检查项 | 通过标准 |
|---|---|
| 待审案件 | 资金方只能看到有权案件。 |
| 证据完整性 | 材料、质检、缺口和冲突可见。 |
| Agent 建议 | 风险建议不等于审批。 |
| 补材料请求 | 请求有原因、责任方和下一步。 |
| 允许融资 | 形成资金方决策，但不等于执行完成。 |
| 拒绝融资 | 有业务原因。 |
| 降额 | 金额调整和原因可表达。 |
| 人工复核 | 高风险、dispute、置信不足可进入人工。 |
| 权限边界 | 资金方不能修改原始材料或质检结论。 |
| 后置边界 | 未进入真实放款、合约执行、多资金方市场或银行系统集成。 |

## 20. 一票否决项

出现以下任一项，Phase 5 不得通过：

| 一票否决项 | 原因 |
|---|---|
| 资金方可以查看无关资金案件 | 数据隔离失败。 |
| 资金方可以修改出口商原始材料 | 事实可信失败。 |
| 资金方可以修改质检 Pass/Fail/Dispute | 事实来源失败。 |
| Agent 风险建议被当成审批结论 | Agent 越界。 |
| Agent 自动批准高风险融资 | 风险控制失败。 |
| 平台替资金方批准 | 平台越界。 |
| 允许融资被表达成已到账 | 资金结果误导。 |
| 拒绝融资没有原因 | 业务不可解释。 |
| 人工复核被表达成失败 | 业务语义错误。 |
| 银行辅助路径被删除 | 现实金融路径失败。 |
| Phase 5 混入真实放款、合约实现、多资金方竞价或银行核心集成 | 阶段越界。 |
| 直接写 endpoint、表结构、风控模型、UI 组件或代码 | 当前文档阶段越界。 |

## 22. Phase 5 出口条件

Phase 5 完成后，系统应具备以下能力：

```text
1. 资金方可以查看有权审查的融资案件。
2. 资金方可以看到证据完整性、材料缺口、质检状态和 Agent 风险建议。
3. 资金方可以要求补材料，并形成责任方待办。
4. 资金方可以作出允许、拒绝、降额或人工复核决策。
5. 资金方决策有原因和留痕语义。
6. Agent 可以辅助审查，但不能替资金方审批。
7. 资金方不能修改出口商原始材料或质检结论。
8. 允许融资不等于资金到账，必须进入 Phase 6 执行条件检查。
9. 出口商可以看到与自己案件相关的资金方结果和下一步。
```

满足以上条件后，才允许进入：

> `76-p0-implementation-phase-6-contract-state-facade-plan.md`

即细化 Phase 6：Contract State Facade，开始定义合约/规则/资金池/暂停状态的业务表达和执行条件检查。

## 23. 本步结论

Phase 5 的本质不是“做一个银行审批后台”。

它的本质是：

> 在融资案件、材料和质检事实基础上，让资金方在自己的工作区完成审查、补材料请求、允许、拒绝、降额和人工复核，并把决策作为后续合约/规则/资金池执行检查的前置条件，而不是直接等同于放款完成。

只有 Funding Review Path 成立，后续 Contract State Facade 才能正确表达“批准以后为什么还需要规则、资金池和平台暂停检查”。
