# 41 P0 视觉原型走读大纲

## 1. 定位

本文件定义 ChainTrace P0 视觉原型的业务走读大纲。

视觉原型走读大纲不是 UI 测试脚本，不是页面点击说明，不是组件验收清单，也不是前端实现计划。

它的作用是：

> 用业务语言定义视觉原型评审时应该按照什么顺序走读、每一步要回答什么问题、哪些角色视角必须覆盖、哪些路径和异常必须讲通，确保视觉原型服务供应链金融系统闭环。

本文件不写组件、样式、布局尺寸、图表库、页面路由、接口路径、请求响应格式、数据库、Solidity、ABI、钱包签名、hash 或 proof 细节。

## 2. 走读原则

| 原则 | 说明 |
|---|---|
| 先全局后角色 | 先确认 ChainTrace 作为供应链金融系统的控制塔表达，再进入角色视角。 |
| 先正常后异常 | 先走通正常闭环，再验证补材料、争议、阻断和执行失败。 |
| 先路径后细节 | 先确认银行辅助、纯 DeFi、低人工路径差异，再看节点细节。 |
| 先业务问题后视觉呈现 | 每一步先问业务问题是否被回答，而不是先讨论视觉样式。 |
| 先权限后操作 | 先确认当前角色能看什么、不能看什么，再看能做什么。 |
| 先 Agent 辅助后人工介入 | 确认 Agent 如何摘要、待办、推进，再确认何时交给人。 |
| 先 DeFi 输出后技术细节 | 只走读合约业务输出，不进入合约实现。 |

## 3. 走读总顺序

| 顺序 | 走读阶段 | 目标 |
|---:|---|---|
| 1 | 产品定位总览 | 确认不是登录前展示页，而是供应链金融业务控制塔。 |
| 2 | 控制塔全链路 | 确认源头到终端的节点流转清楚。 |
| 3 | 出口商视角 | 确认资金需求方能理解自己的案件、卡点和资金结果。 |
| 4 | 银行辅助路径 | 确认需要银行时，资金方审查和授权讲得通。 |
| 5 | 纯 DeFi 路径 | 确认无银行场景由规则、资金池和合约执行讲得通。 |
| 6 | 低人工路径 | 确认 Agent + 合约能推进低风险流程，异常交给人。 |
| 7 | 事实方节点 | 确认质检、物流、通关、保险、买方只处理自己的事实。 |
| 8 | 平台全局控制塔 | 确认平台能看规则、资金池、暂停、异常和 Agent 堆积。 |
| 9 | 异常和失败 | 确认补材料、争议、规则阻断、资金池不足、执行失败可解释。 |
| 10 | 权限隔离 | 确认无关案件、事实篡改、越权审批被阻断。 |
| 11 | 冻结和后置范围 | 确认没有进入登录页、旧代码、组件、接口、数据库、Solidity、hash/proof。 |

## 4. 走读阶段 1：产品定位总览

| 走读问题 | 通过标准 |
|---|---|
| 第一眼是否能看出 ChainTrace 是供应链金融系统？ | 控制塔围绕案件、供应链节点、资金执行和角色协作展开。 |
| 是否避免了登录前展示页？ | 原型从注册后业务工作区开始。 |
| 是否体现 Agent Native？ | Agent 摘要、缺口、待办、建议和升级出现在业务流中。 |
| 是否体现 Crypto Native / DeFi？ | DeFi 执行检查和资金结果是业务核心节点。 |
| 是否避免单纯银行审批系统？ | 原型能区分银行辅助、纯 DeFi 和低人工路径。 |

## 5. 走读阶段 2：控制塔全链路

| 走读节点 | 要回答的问题 | 通过标准 |
|---|---|---|
| 业务意图 | 这笔业务从哪里开始？ | 能看到融资/结算意图、金额、币种、路径。 |
| 贸易材料 | 业务依据是否齐全？ | 能看到材料完整、缺口或需补说明。 |
| Agent 整理 | 信息是否被整理成人能理解的摘要？ | 摘要、缺口、待办、建议清楚。 |
| 事实确认 | 哪些事实已确认？ | 质检、物流、通关、保险、买方节点状态清楚。 |
| 审查或规则 | 当前由资金方审查还是规则检查？ | 银行辅助和纯 DeFi 路径区分清楚。 |
| DeFi 执行 | 是否允许执行资金动作？ | 允许、等待、阻断、失败或完成可解释。 |
| 资金结果 | 钱是否到账、锁定、释放、结算或失败？ | 资金结果节点清楚。 |

## 6. 走读阶段 3：出口商视角

| 步骤 | 出口商应看到 | 通过标准 |
|---|---|---|
| 1 | 自己案件列表或当前案件入口。 | 只看到自己的案件。 |
| 2 | 案件全链路控制塔。 | 能看清从贸易到资金结果的节点。 |
| 3 | 材料完整性和缺口。 | 知道缺什么、为什么缺、如何补。 |
| 4 | Agent 摘要和待办。 | 待办有责任角色和完成标准。 |
| 5 | 事实确认状态。 | 知道质检等节点是否完成或异常。 |
| 6 | 资金方或 DeFi 执行状态。 | 知道当前是审查、执行检查、等待、阻断还是完成。 |
| 7 | 资金结果和卡点解释。 | 知道钱是否到账，或卡在哪里、谁处理。 |

出口商不应看到：

| 不应看到 | 原因 |
|---|---|
| 其他出口商案件 | 案件相关可见原则。 |
| 资金方内部审查细节 | 出口商只需要结果和业务原因。 |
| 平台全局规则维护入口 | 平台方职责。 |
| 修改事实方结论入口 | 事实来源独立。 |

## 7. 走读阶段 4：银行辅助路径

| 步骤 | 走读内容 | 通过标准 |
|---|---|---|
| 1 | 案件进入银行辅助路径。 | 明确该场景需要资金方参与。 |
| 2 | Agent 生成资金方审查摘要。 | 摘要包含贸易、材料、事实、风险和建议。 |
| 3 | 资金方查看待审案件。 | 只看到相关案件。 |
| 4 | 资金方请求补材料。 | 生成责任角色待办，出口商或事实方可回流。 |
| 5 | 资金方允许放款。 | 授权金额、条件和有效性进入执行检查。 |
| 6 | DeFi 合约检查执行条件。 | 授权后仍检查规则、资金池和暂停状态。 |
| 7 | 资金结果展示。 | 允许、等待、阻断、失败或完成都有原因。 |

银行辅助路径不得表达成：

| 不得表达 | 原因 |
|---|---|
| 银行授权后无条件到账 | 合约仍需检查执行条件。 |
| 合约绕过资金方授权 | 违反银行辅助路径授权边界。 |
| 资金方修改事实结论 | 事实来源不可篡改。 |

## 8. 走读阶段 5：纯 DeFi 路径

| 步骤 | 走读内容 | 通过标准 |
|---|---|---|
| 1 | 案件被识别为纯 DeFi。 | 银行审查节点显示不需要或跳过。 |
| 2 | Agent 整理材料和事实。 | 低风险、事实完整、规则清楚时可推进。 |
| 3 | 必要事实方确认事实。 | 事实节点完成或异常可见。 |
| 4 | 合约检查规则和资金池。 | 输出允许、等待、规则阻断或资金池不足。 |
| 5 | 合约检查抵押/保证条件。 | 满足、等待或不足可解释。 |
| 6 | 条件满足后执行资金动作。 | 放款、结算、锁定或释放完成。 |
| 7 | 出现争议或高风险。 | 自动执行停止，进入人工。 |

纯 DeFi 路径不得表达成：

| 不得表达 | 原因 |
|---|---|
| 必须等待银行审批 | 银行不是所有场景必经。 |
| Agent 伪造事实 | Agent 只能整理和提醒。 |
| 合约判断贸易真实性 | 事实来自业务角色。 |

## 9. 走读阶段 6：低人工路径

| 步骤 | 走读内容 | 通过标准 |
|---|---|---|
| 1 | 进入保证金释放、简单结算或付款责任场景。 | 路径不是银行辅助。 |
| 2 | Agent 检查事实完整性。 | 输出缺口或事实完整。 |
| 3 | Agent 检查规则是否清楚。 | 规则满足或阻断原因清楚。 |
| 4 | Agent 自动推进低风险节点。 | 自动推进必须在授权范围内。 |
| 5 | DeFi 合约锁定、释放或结算资金。 | 资金动作结果可解释。 |
| 6 | 出现重大争议。 | 进入人工或约定仲裁。 |

低人工路径走读重点：

| 重点 | 通过标准 |
|---|---|
| 正常案件少人参与 | Agent + 规则 + 合约可推进。 |
| 人处理关键判断 | 授权、异常、争议、高风险、重大法律责任交给人。 |
| 自动化有边界 | 不自动裁决重大争议。 |

## 10. 走读阶段 7：事实方节点

| 事实方 | 走读问题 | 通过标准 |
|---|---|---|
| 质检方 | 哪些案件需要质检？ | 只看到自己的质检任务和必要货物摘要。 |
| 物流方 | 哪些运输节点需要确认？ | 只看到自己的运输节点和异常说明。 |
| 海关 / 报关方 | 哪些通关事实需要确认？ | 只看到自己的通关任务和监管状态。 |
| 保险公司 | 哪些案件需要核保或理赔？ | 只看到核保/理赔相关风险摘要。 |
| 买方 / 进口商 | 哪些订单、收货或付款责任需要确认？ | 只看到与自己相关的订单和责任。 |

事实方共同边界：

| 边界 | 说明 |
|---|---|
| 不决定融资 | 事实方提供事实，不做资金决策。 |
| 不修改贸易材料 | 贸易材料由出口商提交。 |
| 不查看无关案件 | 节点相关可见。 |
| 不伪造其他事实 | 谁提供事实，谁负责事实。 |

## 11. 走读阶段 8：平台全局控制塔

| 步骤 | 平台方应看到 | 通过标准 |
|---|---|---|
| 1 | 系统运行总览。 | 正常案件、异常案件、Agent 状态清楚。 |
| 2 | 规则状态。 | 当前规则、暂停、阻断和影响范围清楚。 |
| 3 | 资金池状态。 | 可用、不足、暂停、异常和币种范围清楚。 |
| 4 | 执行失败。 | 失败原因、影响案件和下一步责任方清楚。 |
| 5 | Agent 异常。 | 堆积、失败、越权风险和人工升级清楚。 |
| 6 | 暂停或恢复。 | 必须有原因、影响范围和记录。 |

平台方不得表达成：

| 不得表达 | 原因 |
|---|---|
| 正常案件默认审批员 | 平台不是默认资金审批方。 |
| 业务事实制造者 | 平台不能伪造贸易、质检、物流、保险、买方事实。 |
| 删除关键记录入口 | 关键动作和执行结果必须可追踪。 |

## 12. 走读阶段 9：异常和失败

| 异常 | 走读问题 | 通过标准 |
|---|---|---|
| 材料不足 | 缺什么，谁补？ | 生成责任角色待办。 |
| 事实不足 | 缺哪个事实方？ | 生成事实方任务或等待。 |
| 事实冲突 | 哪些事实冲突？ | 进入人工复核或补说明。 |
| 质检 dispute | 争议影响什么？ | 审查或执行被暂停并说明原因。 |
| 买方 dispute | 是否影响付款责任？ | 停止自动执行，进入人工。 |
| 银行授权不足 | 授权哪里不足？ | 资金方处理。 |
| 规则阻断 | 哪类规则阻断？ | 平台或责任角色处理，解除条件清楚。 |
| 资金池不足 | 哪个资金池不足？ | 平台或资金池运营处理。 |
| 平台暂停 | 暂停什么，为什么？ | 等待恢复或处理。 |
| 执行失败 | 资金动作为什么失败？ | 平台处理并解释下一步。 |

## 13. 走读阶段 10：权限隔离

| 权限检查 | 期望结果 |
|---|---|
| 出口商查看其他出口商案件 | 阻断并说明无关案件不可见。 |
| 资金方查看其他资金方案件 | 阻断并说明非相关案件不可见。 |
| 质检方修改贸易材料 | 阻断并说明贸易材料由出口商负责。 |
| 出口商修改质检结论 | 阻断并说明质检事实属于质检方。 |
| 平台方替资金方审批 | 阻断并说明平台不替资金方决策。 |
| Agent 查看所属角色不可见案件 | 阻断并说明 Agent 权限继承所属角色。 |
| 纯 DeFi 场景强制银行审批 | 标记为路径设计错误。 |
| 合约绕过银行辅助授权 | 阻断并说明资金方授权是执行条件。 |

## 14. 走读阶段 11：冻结和后置范围

| 检查项 | 通过标准 |
|---|---|
| 是否出现登录前展示页 | 不应出现。 |
| 是否继承旧页面或旧代码口径 | 不应出现。 |
| 是否把 hash / proof 作为主流程 | 不应出现。 |
| 是否进入接口路径或请求响应 | 不应出现。 |
| 是否进入数据库表结构 | 不应出现。 |
| 是否进入 Solidity / ABI / 部署 | 不应出现。 |
| P1/P2 是否被过度展开 | P0 只保留节点入口或占位。 |
| 多组织多成员权限是否提前展开 | P0 保持单钱包单角色。 |

## 15. 走读输出记录

每次视觉原型走读应形成以下业务记录。

| 记录项 | 说明 |
|---|---|
| 走读日期 | 记录评审时间。 |
| 走读角色 | 出口商、资金方、事实方、平台方或 Agent 视角。 |
| 走读路径 | 银行辅助、纯 DeFi、低人工、人工升级或异常路径。 |
| 已回答问题 | 哪些业务问题已被原型回答。 |
| 未回答问题 | 哪些业务问题仍不清楚。 |
| 权限问题 | 是否出现越权展示或越权动作。 |
| 异常问题 | 异常是否能解释原因、责任方和下一步。 |
| 后置问题 | 哪些内容应放到 P1/P2 或技术设计阶段。 |
| 结论 | 通过、需调整业务表达、需补需求、暂不进入下一阶段。 |

## 16. MVP 范围

### 16.1 P0 必须走读

| 范围 | 说明 |
|---|---|
| 控制塔全链路 | 源头到终端的节点流转。 |
| 出口商视角 | 自己案件、卡点、资金结果。 |
| 银行辅助路径 | 资金方审查、授权、合约检查和执行结果。 |
| 纯 DeFi 路径 | 无银行、规则/资金池/保证条件、合约执行。 |
| 低人工路径 | Agent + 合约自动推进和异常升级。 |
| 事实方节点 | 事实方只处理自己的事实节点。 |
| 平台全局控制塔 | 规则、资金池、暂停、异常、Agent 堆积。 |
| 异常失败 | 补材料、争议、规则阻断、资金池不足、执行失败。 |
| 权限隔离 | 无关案件和越权动作被阻断。 |

### 16.2 P0 暂不走读

| 范围 | 原因 |
|---|---|
| 登录前展示页 | 当前不做。 |
| 具体 UI 组件交互 | 当前不做组件走读。 |
| 完整 P1/P2 深度流程 | 后续阶段增强。 |
| 多组织多成员权限 | P0 维持单钱包单角色。 |
| 技术接口、数据库、合约实现 | 当前是视觉原型业务走读。 |
| hash / proof 产品页 | 当前不进入底层证明产品。 |

## 17. 本步结论

P0 视觉原型走读大纲的本质不是“按页面点击一遍”。

它的本质是：

> 按业务顺序确认 ChainTrace 的控制塔、角色视角、银行辅助路径、纯 DeFi 路径、低人工路径、事实节点、平台运营、异常解释和权限隔离是否能被业务用户连续理解。

下一步应进入：

> P0 视觉原型评审清单，用于记录每轮走读是否通过、哪些业务表达需要调整、哪些内容必须后置。
