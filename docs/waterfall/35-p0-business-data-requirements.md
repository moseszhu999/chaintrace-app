# 35 P0 业务数据需求说明

## 1. 定位

本文件定义 ChainTrace P0 阶段的业务数据需求。

业务数据需求说明不是数据库设计，不是表结构，不是字段清单，不是索引设计，也不是数据存储方案。

它的作用是：

> 用业务语言说明后续产品原型、业务契约、数据设计和技术设计需要承载哪些业务信息、这些信息回答什么问题、由谁提供、谁可见、如何参与状态、规则、Agent 和 DeFi 资金执行。

本文件不写页面、组件、API、数据库表、Solidity、ABI、部署、钱包签名、hash 或 proof 细节。

## 2. 数据需求原则

| 原则 | 说明 |
|---|---|
| 数据围绕案件组织 | P0 的核心业务对象是融资/结算案件。 |
| 事实和判断分开 | 贸易、质检、物流等事实不能和资金方判断、规则判断混在一起。 |
| 原始材料和 Agent 摘要分开 | Agent 摘要不能替代原始材料。 |
| 角色可见性清楚 | 每类数据必须知道哪些角色可见、哪些角色不可见。 |
| 状态必须可解释 | 数据必须支撑当前状态、原因、责任角色、下一步。 |
| 规则结果必须可追踪 | 规则通过、等待、阻断和升级人工必须有业务原因。 |
| 资金执行必须可追踪 | DeFi 执行允许、等待、阻断、失败、完成必须可解释。 |
| 历史不覆盖 | 补充、修正、争议和关闭形成新业务记录，不覆盖关键历史。 |

## 3. P0 业务数据域总览

| 数据域 | 业务含义 | 主要来源 | 主要使用方 |
|---|---|---|---|
| 角色身份数据 | 钱包绑定后的永久业务身份。 | 注册流程 | 所有角色 |
| 案件基础数据 | 一笔融资/结算案件的业务基础。 | 出口商 / 业务方 | 所有相关角色 |
| 贸易材料数据 | 合同、订单、发票、装箱单、说明等。 | 出口商 | 出口商 / 资金方 / Agent |
| 事实确认数据 | 质检、物流、通关、保险、买方责任等事实。 | 事实方 | 资金方 / Agent / 合约规则 |
| Agent 数据 | 摘要、缺口、待办、建议、提醒、升级。 | Agent | 所属角色 / 平台方 |
| 决策数据 | 资金方允许、拒绝、补材料、人工复核。 | 资金方 | 出口商 / 平台方 / 合约 |
| 规则数据 | 路径选择、材料、事实、风险、执行规则状态。 | 平台方 / 规则 | Agent / 合约 / 平台方 |
| 资金池数据 | 资金池可用、暂停、不足、异常、占用。 | 平台方 / 合约业务层 | 平台方 / 合约 / 相关角色 |
| DeFi 执行数据 | 执行检查、允许、等待、阻断、失败、完成。 | DeFi 智能合约 | 出口商 / 资金方 / 平台 / Agent |
| 状态和卡点数据 | 当前状态、原因、责任方、下一步。 | Agent / 规则 / 角色动作 | 所有相关角色 |
| 事件和审计数据 | 关键动作、事实、决策、阻断、执行的历史记录。 | 所有关键动作 | 平台方 / 相关角色 |

## 4. 角色身份数据

| 数据需求 | 业务问题 | 来源 | 可见性 |
|---|---|---|---|
| 钱包业务身份 | 这个钱包是什么角色？ | 注册流程 | 当前用户 / 平台方 |
| 角色类型 | 出口商、资金方、质检方、平台方、物流方、保险方、买方等。 | 注册流程 | 相关业务流程 |
| 角色绑定状态 | 是否已经永久绑定。 | 注册流程 | 当前用户 / 平台方 |
| Agent 模式 | 全人工确认、半自动托管、全自动 Agent。 | 用户选择 / 平台调整 | 当前角色 / 平台方 |
| 角色暂停状态 | 该角色是否被暂停或限制。 | 平台方 | 当前角色 / 平台方 / 合约执行检查 |

业务要求：

```text
角色身份必须稳定；
Agent 权限必须继承角色身份；
角色暂停必须影响相关业务动作和合约执行。
```

## 5. 案件基础数据

| 数据需求 | 业务问题 | 来源 | 使用方 |
|---|---|---|---|
| 案件业务类型 | 是融资、结算、保证金、付款责任还是低人工运营场景？ | 出口商 / 业务方 | Agent / 平台 / 合约 |
| 场景路径 | 银行辅助、纯 DeFi、低人工或人工升级。 | 规则 / Agent | 所有相关角色 |
| 当前状态 | 案件处于哪个业务阶段？ | 角色动作 / 规则 / 合约 | 所有相关角色 |
| 申请金额和币种 | 想要融资或执行多少资金？ | 出口商 / 业务方 | 资金方 / 合约 |
| 期望到账或执行时间 | 用户希望什么时候完成？ | 出口商 / 业务方 | Agent / 资金方 |
| 责任角色 | 当前谁需要处理？ | Agent / 规则 | 所有相关角色 |
| 下一步动作 | 当前应该做什么？ | Agent / 规则 | 所有相关角色 |

案件基础数据必须支持：

| 能力 | 说明 |
|---|---|
| 路径选择 | 判断是否需要银行，是否可纯 DeFi，是否可低人工。 |
| 状态解释 | 解释当前状态和卡点。 |
| 权限过滤 | 只让相关角色看到相关案件。 |
| 执行检查 | 给 DeFi 合约提供业务状态结果。 |

## 6. 贸易材料数据

| 数据需求 | 业务问题 | 来源 | 使用方 |
|---|---|---|---|
| 合同材料 | 交易是否有合同依据？ | 出口商 | Agent / 资金方 |
| 订单材料 | 订单是否真实存在？ | 出口商 | Agent / 资金方 / 买方 |
| 发票材料 | 金额和币种是否有发票依据？ | 出口商 | Agent / 资金方 |
| 装箱单或货物说明 | 货物内容和数量是什么？ | 出口商 | Agent / 质检方 / 资金方 |
| 补充说明 | 差异、异常或特殊情况如何解释？ | 出口商 | Agent / 资金方 |
| 材料完整性状态 | 材料是否足够提交、审查或执行？ | Agent / 规则 | 出口商 / 资金方 |
| 材料冲突说明 | 金额、主体、品类、国家是否冲突？ | Agent / 资金方 | 出口商 / 资金方 / 平台 |

贸易材料数据边界：

| 边界 | 说明 |
|---|---|
| 出口商负责提交 | 其他角色不得伪造或替代提交贸易材料。 |
| Agent 负责整理 | Agent 不能改变原始材料。 |
| 资金方可审查 | 资金方不能修改原始贸易材料。 |
| 补充不覆盖历史 | 后续补充形成新记录。 |

## 7. 事实确认数据

| 事实类型 | 数据需求 | 来源 | 使用方 |
|---|---|---|---|
| 质检事实 | pass / fail / dispute、报告、照片、原因。 | 质检方 | 出口商 / 资金方 / Agent / 规则 |
| 物流事实 | 揽收、装柜、出港、到港、签收、延误、损坏、丢失、改港。 | 物流方 | Agent / 资金方 / 平台 |
| 通关事实 | 报关、清关、出口、进口、监管异常、不一致提示。 | 海关 / 报关方 | Agent / 资金方 / 平台 |
| 保险事实 | 承保、拒保、加费、补材料、理赔状态。 | 保险公司 | Agent / 资金方 / 平台 |
| 买方事实 | 订单真实性、收货状态、争议、付款责任。 | 买方 / 进口商 | Agent / 资金方 / 合约规则 |

事实确认数据必须满足：

| 要求 | 说明 |
|---|---|
| 来源明确 | 每个事实必须知道由哪个角色确认。 |
| 结论明确 | pass / fail / dispute 或对应业务状态必须清楚。 |
| 原因明确 | fail、dispute、异常、拒保必须说明原因。 |
| 不能被他人篡改 | 其他角色只能查看、质疑、请求补充。 |
| 可被规则使用 | 规则和合约只使用事实状态，不自行判断事实真假。 |

## 8. Agent 数据

| 数据需求 | 业务问题 | 来源 | 使用方 |
|---|---|---|---|
| 角色摘要 | 该角色现在需要知道什么？ | Agent | 所属角色 |
| 材料缺口 | 缺什么、为什么缺、谁补？ | Agent | 出口商 / 事实方 |
| 风险摘要 | 哪些风险需要人关注？ | Agent | 资金方 / 平台方 |
| 建议动作 | 下一步建议做什么？ | Agent | 所属角色 |
| 待办 | 谁需要完成什么，完成标准是什么？ | Agent / 角色动作 | 责任角色 |
| 提醒 | 哪些状态、超时、授权或异常需要提醒？ | Agent | 相关角色 |
| 自动推进记录 | Agent 在授权范围内推进了什么？ | Agent | 所属角色 / 平台方 |
| 异常升级记录 | Agent 为什么升级人工？ | Agent | 平台方 / 相关角色 |

Agent 数据边界：

| 边界 | 说明 |
|---|---|
| 摘要不等于事实 | Agent 摘要必须可追溯到材料或事实。 |
| 建议不等于决策 | 高风险放款、重大争议、平台暂停由人处理。 |
| 自动推进必须可解释 | 必须说明命中哪些低风险规则。 |
| 越权风险必须记录 | Agent 置信不足或越权风险需要升级。 |

## 9. 决策数据

| 数据需求 | 业务问题 | 来源 | 使用方 |
|---|---|---|---|
| 资金方审查状态 | 资金方是否开始审查？ | 资金方 / Agent | 出口商 / 平台 |
| 允许放款结果 | 资金方是否允许放款？ | 资金方 | 合约 / 出口商 / 平台 |
| 授权金额和条件 | 允许多少、什么条件下允许？ | 资金方 | 合约 / Agent |
| 拒绝原因 | 为什么拒绝？ | 资金方 | 出口商 / Agent |
| 补材料要求 | 需要补什么？ | 资金方 | 出口商 / 事实方 / Agent |
| 人工复核要求 | 为什么需要人工？ | 资金方 / 平台 / Agent | 相关角色 |

决策数据必须支持：

| 要求 | 说明 |
|---|---|
| 银行辅助路径授权 | 需要银行参与时，合约执行前必须看到资金方授权结果。 |
| 决策原因可解释 | 拒绝、补材料、人工复核不能只有状态名。 |
| 授权边界清楚 | 金额、期限、条件必须能被业务规则使用。 |
| 不影响纯 DeFi 必经性 | 纯 DeFi 场景不需要等待资金方决策。 |

## 10. 规则数据

| 数据需求 | 业务问题 | 来源 | 使用方 |
|---|---|---|---|
| 规则适用路径 | 规则适用于银行辅助、纯 DeFi 还是低人工？ | 平台方 / 规则 | Agent / 合约 |
| 规则适用场景 | 规则适用于融资、结算、保证金、付款责任还是托管？ | 平台方 / 规则 | Agent / 合约 |
| 规则状态 | 有效、暂停、过期、不匹配。 | 平台方 | Agent / 合约 |
| 规则版本 | 当前使用哪套规则口径？ | 平台方 | 平台 / 合约 |
| 规则阻断原因 | 为什么不允许继续？ | 规则 / 合约 | 所有相关角色 |
| 规则解除条件 | 如何从阻断或等待回到流程？ | 平台方 / 规则 | Agent / 责任角色 |

规则数据不等于规则引擎实现。

当前只要求后续设计能承载业务规则口径和状态。

## 11. 资金池数据

| 数据需求 | 业务问题 | 来源 | 使用方 |
|---|---|---|---|
| 资金池状态 | 正常、暂停、异常、维护中。 | 平台方 / 合约业务层 | 合约 / 平台 / 出口商 |
| 可用资金状态 | 是否足够支持执行？ | 平台方 / 合约业务层 | 合约 / 平台 |
| 占用状态 | 是否占用超限？ | 平台方 / 合约业务层 | 合约 / 平台 |
| 币种适配 | 资金池是否支持案件币种？ | 平台方 / 合约业务层 | 合约 |
| 适用范围 | 资金池适用于哪些路径、国家、品类或场景？ | 平台方 | 合约 / Agent |
| 暂停原因 | 为什么暂停？ | 平台方 | 平台 / Agent / 相关角色 |
| 恢复条件 | 何时可以重新执行？ | 平台方 | 平台 / Agent |

资金池数据必须支持：

| 要求 | 说明 |
|---|---|
| 资金池不足可解释 | 用户知道是资金池问题，不是材料问题。 |
| 平台可处理 | 平台方能知道受影响范围。 |
| 合约可检查 | DeFi 合约可基于业务状态执行或阻断。 |

## 12. DeFi 执行数据

| 数据需求 | 业务问题 | 来源 | 使用方 |
|---|---|---|---|
| 执行检查结果 | 是否允许执行？ | DeFi 智能合约 | Agent / 相关角色 |
| 授权检查结果 | 银行、用户、规则授权是否满足？ | DeFi 智能合约 | Agent / 资金方 / 平台 |
| 资金池检查结果 | 资金池是否足够、暂停或异常？ | DeFi 智能合约 | 平台 / 相关角色 |
| 规则检查结果 | 规则是否有效、适用、未暂停？ | DeFi 智能合约 | 平台 / Agent |
| 抵押/保证检查结果 | 抵押或保证条件是否满足？ | DeFi 智能合约 | Agent / 平台 |
| 暂停检查结果 | 平台、路径、规则、资金池、案件是否暂停？ | DeFi 智能合约 | 平台 / Agent |
| 执行动作结果 | 放款、结算、锁定、释放、阻断是否完成？ | DeFi 智能合约 | 相关角色 |
| 执行失败原因 | 为什么执行失败？ | DeFi 智能合约 / 平台方 | 平台 / Agent |

DeFi 执行数据必须输出业务状态：

| 输出 | 说明 |
|---|---|
| 允许执行 | 可以进入资金动作。 |
| 不允许执行 | 条件不满足。 |
| 等待条件 | 可补充或等待的条件未满足。 |
| 规则阻断 | 当前规则不允许。 |
| 资金池不足 | 资金池无法覆盖执行。 |
| 授权不足 | 授权金额、期限或范围不足。 |
| 平台暂停 | 平台、规则、路径、资金池或案件暂停。 |
| 执行失败 | 尝试执行但失败。 |
| 执行完成 | 资金动作完成。 |

## 13. 状态和卡点数据

| 数据需求 | 业务问题 | 来源 |
|---|---|---|
| 当前状态 | 案件现在在哪一步？ | 角色动作 / Agent / 规则 / 合约 |
| 状态原因 | 为什么在这一步？ | Agent / 规则 / 合约 |
| 卡点类型 | 缺材料、待事实、待审查、规则阻断、资金池不足、平台暂停、争议。 | Agent / 规则 |
| 责任角色 | 谁需要处理？ | Agent / 规则 |
| 下一步动作 | 下一步做什么？ | Agent |
| 解除条件 | 等待或阻断如何解除？ | 规则 / 平台 / 合约 |

状态和卡点数据必须让用户理解：

```text
当前状态是什么？
为什么不能继续？
责任方是谁？
下一步是什么？
解除条件是什么？
```

## 14. 事件和审计数据

| 数据需求 | 业务问题 | 来源 |
|---|---|---|
| 事件类型 | 发生了什么业务变化？ | 业务动作 / Agent / 规则 / 合约 |
| 事件来源 | 谁触发的？ | 角色 / Agent / 平台 / 合约 |
| 事件影响对象 | 影响哪个案件、任务、事实、规则、资金池或执行状态？ | 事件触发方 |
| 事件原因 | 为什么发生？ | 事件触发方 |
| 后续动作 | 下一步谁处理？ | Agent / 规则 |
| 历史顺序 | 事件发生顺序是什么？ | 系统记录 |
| 是否可见 | 哪些角色可以看到？ | 权限规则 |

审计数据必须覆盖：

| 范围 | 说明 |
|---|---|
| 关键业务动作 | 创建、提交、补材料、确认、授权、暂停、恢复、执行。 |
| 事实形成 | 质检、物流、通关、保险、买方事实。 |
| Agent 自动化 | 摘要、待办、自动推进、异常升级。 |
| 资金方决策 | 允许、拒绝、补材料、人工复核。 |
| 合约执行 | 允许、等待、阻断、失败、完成。 |
| 平台运营 | 规则、资金池、暂停、恢复、执行失败处理。 |
| 权限异常 | 越权尝试、Agent 越权风险。 |

## 15. 权限和可见性数据

| 数据需求 | 业务问题 |
|---|---|
| 数据所属案件 | 这条信息属于哪个案件？ |
| 数据所属角色 | 这条信息由哪个角色产生？ |
| 数据可见角色 | 哪些角色可以看？ |
| 数据可操作角色 | 哪些角色可以补充、确认、决策或处理？ |
| 数据不可修改边界 | 哪些角色不能修改？ |
| Agent 可用范围 | Agent 是否可读取和使用该数据？ |
| 合约可用状态 | 合约是否只使用该数据的业务状态结果？ |

权限数据必须防止：

| 风险 | 要求 |
|---|---|
| 无关角色看到无关案件 | 必须隔离。 |
| 非事实方修改事实 | 必须阻断。 |
| Agent 读取越权数据 | 必须阻断并升级。 |
| 平台伪造业务事实 | 必须阻断。 |
| 合约读取完整业务材料 | 合约只使用业务状态，不处理完整材料。 |

## 16. 数据质量要求

| 要求 | 说明 |
|---|---|
| 完整性 | 必要材料、事实、决策、规则、执行结果不能缺关键业务解释。 |
| 来源明确 | 每个事实、决策、规则和执行结果必须知道来源。 |
| 状态一致 | 案件状态、待办、事件和合约输出不能互相矛盾。 |
| 历史保留 | 补充、修正、争议、失败不能覆盖原记录。 |
| 可解释 | 业务用户能理解原因和下一步。 |
| 可审计 | 平台方和相关角色能回看关键历史。 |
| 可降级 | Agent 或合约无法继续时可以进入人工处理。 |

## 17. MVP 范围

### 17.1 P0 必须定义

| 范围 | 说明 |
|---|---|
| 角色身份数据 | 钱包角色、Agent 模式、角色暂停。 |
| 案件基础数据 | 案件类型、路径、状态、金额、责任角色、下一步。 |
| 贸易材料数据 | 合同、订单、发票、装箱单、补充说明和完整性状态。 |
| 事实确认数据 | 质检、物流、通关、保险、买方事实状态。 |
| Agent 数据 | 摘要、缺口、待办、建议、提醒、自动推进、异常升级。 |
| 决策数据 | 资金方允许、拒绝、补材料、人工复核。 |
| 规则和资金池数据 | 规则状态、资金池状态、暂停和解除条件。 |
| DeFi 执行数据 | 检查、允许、等待、阻断、失败、完成。 |
| 事件和审计数据 | 关键业务变化、来源、影响、原因、下一步。 |
| 权限和可见性数据 | 谁可见、谁可操作、谁不可修改。 |

## 18. 本步结论

P0 业务数据需求说明明确：

> ChainTrace 后续技术设计需要承载的不是抽象数据表，而是围绕融资/结算案件形成的业务信息：角色身份、案件、贸易材料、事实确认、Agent 摘要和待办、资金方决策、规则、资金池、DeFi 执行、状态卡点、事件审计和权限可见性。

下一步建议进入：

> P0 API 业务契约，用业务语言说明后续接口需要承载哪些动作、输入、输出和错误解释。
