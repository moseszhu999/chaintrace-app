# 45 P0 视觉原型阶段收口检查清单

## 1. 定位

本文件定义 ChainTrace P0 视觉原型阶段是否可以正式收口、冻结并进入下一阶段需求交接的检查清单。

视觉原型阶段收口检查清单不是 UI 验收报告，不是组件清单，不是前端开发入口，也不是技术实现计划。

它的作用是：

> 在 P0 视觉原型交付包完成后，用业务语言逐项确认产品定位、控制塔表达、角色视角、路径差异、Agent 边界、DeFi 执行、异常回流、变更处理、后置范围和交接材料是否已经满足冻结条件。

本文件不写组件、样式、布局尺寸、图表库、页面路由、接口路径、请求响应格式、数据库、Solidity、ABI、钱包签名、hash 或 proof 细节。

## 2. 收口原则

| 原则 | 说明 |
|---|---|
| 先业务闭环，后阶段冻结 | 只有业务闭环讲通，视觉原型阶段才能冻结。 |
| 先角色边界，后交接 | 每个角色看到什么、不能看到什么必须清楚。 |
| 先路径差异，后统一表达 | 银行辅助、纯 DeFi、低人工和人工升级不能被合并成单一路径。 |
| 先异常可处理，后通过 | 异常必须有原因、责任方、解除条件和下一步。 |
| 先变更归档，后收口 | 每条评审反馈都必须分类并有处理状态。 |
| 先后置清楚，后冻结 | P1/P2 和技术设计内容必须有后置说明。 |
| 只冻结业务表达 | 当前冻结的是业务原型口径，不冻结 UI 组件或技术实现。 |

## 3. 收口状态定义

| 状态 | 含义 | 后续动作 |
|---|---|---|
| Ready to freeze | 所有阻断项关闭，核心业务表达可冻结。 | 进入下一阶段需求交接。 |
| Minor adjustment | 核心业务成立，仅有不影响口径的小调整。 | 调整记录后复核。 |
| Need requirement update | 原型暴露出 waterfall 需求缺口。 | 先补需求文档，再重新评审。 |
| Deferred | 有价值但不属于 P0。 | 写入后置清单。 |
| Blocked | 产品定位、权限、路径、Agent、DeFi 或异常规则存在重大问题。 | 不得冻结，必须修正。 |

## 5. 产品定位收口检查

| 检查项 | 通过标准 | 结论 |
|---|---|---|
| 完整系统定位 | 原型表达 ChainTrace 是供应链金融系统，不是供应链金融前站。 | 待评审 |
| Agent Native | Agent 作为每个角色身边业务助理出现。 | 待评审 |
| Crypto Native / DeFi | 智能合约作为业务规则和资金执行约束层出现。 | 待评审 |
| 多路径场景 | 银行辅助、纯 DeFi、低人工和人工升级均能被理解。 | 待评审 |
| 注册后工作区 | 不以登录前展示页或营销页作为主体验。 | 待评审 |
| 不继承旧口径 | 不使用旧版 app、components、lib、contracts、scripts 页面和代码口径。 | 待评审 |

## 6. 控制塔收口检查

| 检查项 | 通过标准 | 结论 |
|---|---|---|
| 全链路主线 | 能看到供应链源头、贸易、事实、审查/规则、DeFi 执行和资金结果。 | 待评审 |
| 节点状态 | Not started、Waiting、In progress、Completed、Warning、Dispute、Blocked、Failed、Skipped 等状态含义清楚。 | 待评审 |
| 当前卡点 | 卡在哪里、为什么卡、谁处理、如何解除清楚。 | 待评审 |
| 角色过滤 | 不同角色看到的控制塔范围符合权限边界。 | 待评审 |
| 资金结果 | 放款、结算、锁定、释放、到账、等待或失败清楚。 | 待评审 |
| 大数据控制塔风格 | 表达能支持源头到终端的流转和中间节点观察。 | 待评审 |

## 7. 角色视角收口检查

| 角色 | 必须通过 | 结论 |
|---|---|---|
| 出口商 / 资金需求方 | 能看清自己案件、材料缺口、事实进度、资金结果和下一步。 | 待评审 |
| 资金方 / 银行 / 保理机构 | 能看清相关案件、风险摘要、授权、补材料和执行前条件。 | 待评审 |
| 质检方 | 只处理自己的质检任务和质检事实。 | 待评审 |
| 物流方 | 只确认自己的运输节点、物流事实和异常说明。 | 待评审 |
| 海关 / 报关方 | 只确认自己的通关事实、监管状态和一致性异常。 | 待评审 |
| 保险公司 | 只处理核保、承保、拒保、加费、补材料和理赔相关事实。 | 待评审 |
| 买方 / 进口商 | 只确认订单、收货、争议和付款责任。 | 待评审 |
| 平台方 / 合约运营 | 只运营规则、资金池、暂停、执行失败和 Agent 堆积。 | 待评审 |
| Agent | 权限继承所属角色，不越权查看或处理。 | 待评审 |

## 8. 路径差异收口检查

| 路径 | 必须讲通 | 不得出现 | 结论 |
|---|---|---|---|
| 银行辅助路径 | 资金方审查、授权、补材料、授权后 DeFi 检查、资金结果。 | 银行授权后无条件到账。 | 待评审 |
| 纯 DeFi 路径 | 无银行审批，由规则、资金池、事实和保证条件驱动执行。 | 强制等待银行审批。 | 待评审 |
| 低人工路径 | Agent + 规则 + 合约推进低风险流程，异常升级人工。 | Agent 自动裁决重大争议。 | 待评审 |
| 人工升级路径 | 争议、高风险、监管异常、法律责任和越权风险交给人。 | 系统静默通过或自动放过。 | 待评审 |

## 9. Agent Native 收口检查

| 检查项 | 通过标准 | 结论 |
|---|---|---|
| Agent 不是独立用户 | Agent 作为角色身边业务助理，不拥有独立业务身份。 | 待评审 |
| 三种模式可解释 | 全人工确认、半自动托管、全自动 Agent 的差异清楚。 | 待评审 |
| 摘要可追溯 | Agent 摘要来自材料和事实。 | 待评审 |
| 缺口可处理 | 缺什么、为什么缺、谁负责清楚。 | 待评审 |
| 待办有边界 | 待办不越过角色权限。 | 待评审 |
| 自动推进有限制 | 只在低风险、事实完整、规则明确、授权范围内推进。 | 待评审 |
| 异常会升级 | 争议、高风险、越权、置信不足和执行失败进入人工。 | 待评审 |

## 10. DeFi 执行收口检查

| 合约业务输出 | 通过标准 | 结论 |
|---|---|---|
| 允许执行 | 用户知道当前条件满足，可以进入资金动作。 | 待评审 |
| 不允许执行 | 用户知道当前不能继续的业务原因。 | 待评审 |
| 等待条件 | 用户知道等待事实、授权、资金池、规则或保证条件。 | 待评审 |
| 规则阻断 | 用户知道阻断规则、影响范围和解除条件。 | 待评审 |
| 资金池不足 | 用户知道资金池、币种、余额或暂停问题。 | 待评审 |
| 平台暂停 | 用户知道暂停对象、原因、影响范围和恢复条件。 | 待评审 |
| 执行失败 | 用户知道资金动作未完成、责任方和下一步。 | 待评审 |

| 不得要求 | 原因 |
|---|---|

| 展示 ABI 或部署信息 | 当前只冻结业务输出口径。 |
| 展示钱包签名流程 | 钱包实现后置。 |
| 展示 hash / proof 产品页 | 当前不进入底层证明产品。 |

## 15. 收口阻断条件

出现以下任一情况，P0 视觉原型阶段不得冻结。

| 阻断条件 | 原因 |
|---|---|
| 仍被表达成供应链金融前站 | 产品定位错误。 |
| 仍从登录前展示页开始 | 阶段范围错误。 |
| 看不出供应链源头到终端流转 | 控制塔表达失败。 |
| 银行仍是所有路径必经 | 破坏纯 DeFi 和低人工路径。 |
| 纯 DeFi 仍等待银行审批 | 路径逻辑错误。 |
| Agent 被表达成审批人或事实制造者 | Agent 边界错误。 |
| 合约被表达成页面、数据库或 Agent | DeFi 业务定位错误。 |
| 角色能看到无关案件 | 权限隔离失败。 |
| 事实方能决定融资 | 事实方边界错误。 |
| 平台方能伪造事实 | 平台方边界错误。 |
| 异常没有责任方和下一步 | 业务不可运行。 |
| D 类变更未关闭 | 变更管理未收口。 |
| B 类需求缺口未补文档 | waterfall 需求未闭合。 |
| P1/P2 或技术内容抢进 P0 | 范围失控。 |
| 进入接口、数据库、Solidity、钱包签名、hash/proof 细节 | 阶段跑偏。 |
| 继承旧版页面或旧代码口径 | 违背全新瀑布重建设计分支。 |

## 18. 本步结论

P0 视觉原型阶段收口检查的本质不是“确认页面已经画完”。

它的本质是：

> 判断 ChainTrace P0 视觉原型是否已经可以作为业务冻结物交接给下一阶段：它必须讲清完整供应链金融系统定位、控制塔流转、角色边界、路径差异、Agent Native 协作、DeFi 执行约束、异常回流、变更处理和后置范围。

下一步应进入：

> P0 下一阶段需求交接说明，用于定义视觉原型阶段冻结后，应如何把业务原型交接给后续产品、技术接口、数据设计、合约技术设计和实现阶段。
