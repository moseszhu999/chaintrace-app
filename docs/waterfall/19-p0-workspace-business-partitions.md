# 19 P0 工作区业务分区

## 1. 定位

本文件定义 ChainTrace P0 MVP 的工作区业务分区。

工作区业务分区不是页面布局，不是 UI 组件，也不是路由设计。

它的核心作用是：

> 说明每个 P0 角色进入工作区后，业务信息应该按哪些工作分区组织，以支持角色完成自己的业务结果。

本文件只写业务分区和业务职责，不写页面、组件、数据库、智能合约实现、钱包签名、hash 或 proof 细节。

## 2. 工作区分区原则

| 原则 | 说明 |
|---|---|
| 每个角色一个业务工作区 | 钱包绑定单一角色后进入对应角色工作区。 |
| 每个分区回答一个业务问题 | 不按技术模块拆分，而按用户要完成的事情拆分。 |
| Agent 贯穿所有分区 | Agent 帮每个分区整理摘要、缺口、待办和提醒。 |
| 正常案件减少人工入口 | 正常案件由 Agent + 规则推进，人主要看关键确认。 |
| 异常和授权必须可见 | 高风险、争议、补材料、放款授权、平台暂停必须突出。 |
| 不做 UI 设计 | 当前只定义业务分区，不定义视觉和交互。 |

## 3. 出口商工作区分区

出口商工作区围绕一个问题：

> 我这笔贸易能不能融资、缺什么、什么时候到账？

| 分区 | 业务问题 | 主要内容 |
|---|---|---|
| 融资意图 | 我要基于哪笔贸易申请融资？ | 融资类型、金额、币种、到账时间、Agent 模式。 |
| 贸易材料 | 我的材料是否足够？ | 合同、订单、发票、装箱单、补充说明。 |
| Agent 检查 | 现在能不能提交？ | 完整性、缺口、金额一致性、建议动作。 |
| 质检状态 | 货物事实是否已确认？ | 待质检、pass、fail、dispute、补充要求。 |
| 案件状态 | 当前卡在哪里？ | 当前状态、原因、责任方、下一步。 |
| 融资结果 | 资金方怎么决定？ | 允许放款、拒绝、补材料、人工复核、批准金额。 |
| 执行结果 | 钱是否到账？ | 等待执行、资金池不足、规则阻断、到账。 |

出口商工作区不应包含：

| 不包含 | 原因 |
|---|---|
| 资金方内部审批后台 | 出口商只看结果和原因。 |
| 平台规则维护入口 | 规则权属于平台方。 |
| 合约技术控制台 | 当前不是技术实现阶段。 |

## 4. 质检方工作区分区

质检方工作区围绕一个问题：

> 哪些货物需要我检查，我的质检事实是否足够进入融资流程？

| 分区 | 业务问题 | 主要内容 |
|---|---|---|
| 待质检任务 | 哪些案件需要我处理？ | 任务摘要、货物信息、截止时间、优先级。 |
| 质检清单 | 我要检查什么？ | 通用检查项、关键风险点、材料要求。 |
| 质检材料 | 我要提交什么证据？ | 报告、照片、现场说明、异常说明。 |
| 质检结论 | 结果是什么？ | pass / fail / dispute 和原因。 |
| Agent 检查 | 我的材料够不够？ | 缺报告、缺照片、缺结论、结论冲突。 |
| 补充和争议 | 为什么还要补？ | 资金方要求、出口商争议、平台异常。 |

质检方工作区不应包含：

| 不包含 | 原因 |
|---|---|
| 放款审批入口 | 质检方不是资金决策方。 |
| 出口商贸易材料修改入口 | 质检方不能修改原始贸易事实。 |
| 无关融资案件 | 数据隔离。 |

## 5. 资金方工作区分区

资金方工作区围绕一个问题：

> 这笔融资是否真实、完整、风险可控，并且是否可以放款？

| 分区 | 业务问题 | 主要内容 |
|---|---|---|
| 待审案件 | 哪些案件值得我看？ | Agent 分组、状态、风险、金额、优先级。 |
| 案件摘要 | 这是什么融资申请？ | 出口商、贸易、金额、货物、质检、状态。 |
| 材料完整性 | 证据是否足够？ | 合同、订单、发票、质检、缺口和一致性。 |
| 风险判断 | 风险是否过线？ | 出口商、买方、品类、金额、质检、规则提示。 |
| 决策动作 | 我应该怎么处理？ | 允许放款、拒绝、补材料、人工复核、降额。 |
| 执行条件 | 批准后是否能执行？ | 资金池、规则、平台状态、等待条件。 |
| 贷后入口 | 钱放出去后要看什么？ | MVP 只保留结果和后续跟踪入口。 |

资金方工作区不应包含：

| 不包含 | 原因 |
|---|---|
| 其他资金方案件 | 数据隔离。 |
| 平台系统级暂停/恢复控制 | 平台方负责。 |
| 质检结论修改入口 | 质检事实属于质检方。 |

## 6. 平台方工作区分区

平台方工作区围绕一个问题：

> 系统是否安全，哪些异常需要平台处理？

| 分区 | 业务问题 | 主要内容 |
|---|---|---|
| 运行总览 | 当前系统是否安全？ | 资金池、规则、执行、异常、Agent 状态。 |
| 资金池状态 | 资金是否足够和可执行？ | 正常、不足、暂停、异常、待执行案件。 |
| 规则状态 | 当前按哪套规则执行？ | 规则版本、适用范围、暂停、影响案件。 |
| 执行异常 | 哪些案件执行失败或等待？ | 资金池不足、规则阻断、平台暂停、执行失败。 |
| Agent 运营 | Agent 是否正常推进？ | 任务堆积、失败、误推进、人工升级。 |
| 平台控制 | 是否需要暂停、恢复或分派？ | 暂停资金池、暂停规则、恢复、分派异常。 |
| 操作记录 | 谁做了什么？ | 暂停、恢复、规则变更、异常处理原因。 |

平台方工作区不应包含：

| 不包含 | 原因 |
|---|---|
| 正常案件逐单审批入口 | 平台不是默认审批员。 |
| 伪造或修改业务事实入口 | 平台不是事实制造者。 |
| 资金方授信替代入口 | 放款权属于资金方。 |

## 7. Agent 工作分区

Agent 不是单独用户，但每个角色工作区都需要 Agent 工作分区。

| 分区 | 业务问题 | 主要内容 |
|---|---|---|
| 摘要 | 当前最重要的信息是什么？ | 角色视角业务摘要。 |
| 缺口 | 还缺什么？ | 缺材料、缺确认、缺说明、缺决策。 |
| 待办 | 谁要做什么？ | 责任角色、动作、完成标准、截止时间。 |
| 提醒 | 什么需要马上处理？ | 超时、高风险、状态变化、授权节点。 |
| 建议 | 下一步推荐动作是什么？ | 建议提交、补材料、人工复核、等待。 |
| 升级 | 哪些必须交给人？ | 争议、高风险、放款授权、平台暂停。 |

Agent 工作分区必须遵守：

| 边界 | 说明 |
|---|---|
| 不伪造事实 | 只能整理已有材料和角色确认。 |
| 不替代高风险决策 | 放款、重大争议、平台控制必须人处理。 |
| 不越权查看信息 | 只能在当前角色授权范围内工作。 |
| 不删除关键记录 | 关键操作必须可追溯。 |

## 8. 跨角色协作分区

P0 MVP 至少需要表达以下跨角色协作关系：

| 协作 | 发起方 | 处理方 | 业务目的 |
|---|---|---|---|
| 出口商提交融资案件 | 出口商 | 资金方 | 请求融资审查。 |
| Agent 生成质检任务 | Agent / 出口商 | 质检方 | 获取独立货物事实。 |
| 质检方提交结论 | 质检方 | 出口商 / 资金方 | 支持融资判断。 |
| 资金方要求补材料 | 资金方 | 出口商 / 质检方 | 补足审查证据。 |
| 资金方允许放款 | 资金方 | 平台 / 执行业务层 | 进入执行检查。 |
| 平台处理执行异常 | 平台方 | 资金方 / 出口商 | 解释和处理执行卡点。 |

跨角色协作必须通过待办和状态变化表达，不通过私聊式自由流程替代。

## 9. MVP 范围

### P0 必须做

| 范围 | 说明 |
|---|---|
| 定义 P0 角色工作区分区 | 出口商、质检方、资金方、平台方。 |
| 定义 Agent 工作分区 | 摘要、缺口、待办、提醒、建议、升级。 |
| 定义跨角色协作分区 | 提交、质检、补材料、审批、执行异常。 |
| 定义每个分区回答的问题 | 每个分区服务一个业务问题。 |
| 定义每个分区边界 | 明确不该做什么。 |

### 暂不做

| 范围 | 原因 |
|---|---|
| 页面布局 | 当前不做 UI。 |
| 导航菜单设计 | 当前只写业务分区。 |
| 前端组件拆分 | 当前不写代码。 |
| 数据库权限模型 | 当前不写实现。 |
| 智能合约调用流程 | 当前不进入技术设计。 |
| hash / proof 工作区 | 当前不进入底层证明细节。 |

## 10. 本步结论

P0 工作区业务分区的本质不是“设计几个页面”。

它的本质是：

> 每个钱包绑定角色进入自己的业务工作区，围绕自己的核心问题处理融资案件；Agent 贯穿摘要、缺口、待办、提醒和升级；跨角色协作通过状态和待办推进。
