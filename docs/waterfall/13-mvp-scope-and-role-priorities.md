# 13 MVP Scope and Role Priorities

## 1. 定位

本文件用于总结 ChainTrace MVP 阶段的角色优先级和业务闭环。

ChainTrace 当前产品定位是：

> Agent Native + Crypto Native 的供应链金融前站系统。

MVP 不追求一次性覆盖所有供应链金融场景。

MVP 的核心目标是先跑通一个清楚的融资闭环：

```text
出口商创建融资意图
→ Agent 整理贸易材料
→ 质检方提供质检事实
→ 资金方看摘要、完整性、风险建议
→ 资金方决定允许放款 / 拒绝 / 补材料
→ 平台/规则/合约检查是否允许执行
→ 出口商看到到账或卡点
```

## 2. MVP 范围原则

| 原则 | 说明 |
|---|---|
| 先闭环，后扩展 | 先完成融资最小业务闭环，再扩展更多角色。 |
| 先业务需求，后 UI 和实现 | 当前不写页面、组件、数据库、合约实现。 |
| 先角色边界，后技术边界 | 先明确每个角色能做什么、不能做什么。 |
| 先 Agent Native 操作模型 | 正常案件尽量由 Agent + 规则推进。 |
| 先融资判断必需事实 | 贸易、质检、资金方决策、平台规则优先。 |
| 外部接入后置 | 海关、报关、通关等高价值但接入复杂的能力后置。 |

MVP 阶段不做：

| 暂不做 | 原因 |
|---|---|
| 登录前展示页 | 当前不做营销和转化页面。 |
| UI 组件设计 | 当前仍处于需求阶段。 |
| 数据库设计 | 当前不进入技术实现。 |
| 智能合约实现 | 当前只定义业务定位。 |
| 钱包签名细节 | 当前不进入实现细节。 |
| hash / proof 细节 | 当前不直接跳到底层证明细节。 |

## 3. 角色优先级

### 3.1 P0：融资闭环必须角色

| 角色 | 优先级 | 原因 |
|---|---|---|
| 出口商 / 资金需求方 | P0 | 发起融资意图、提交贸易材料、查看融资结果，是需求起点。 |
| 资金方 / 银行 / 保理机构 | P0 | 决定允许放款、拒绝或要求补材料，是资金决策核心。 |
| 质检方 / 第三方检验机构 | P0 | 提供独立货物事实，是贸易真实性和风险判断关键依据。 |
| 平台方 / 合约运营 | P0 | 维护资金池、规则、合约状态、异常和执行秩序。 |

P0 是融资闭环必须角色。

没有出口商，就没有融资需求。
没有资金方，就没有放款决策。
没有质检方，货物事实缺少独立确认。
没有平台方，规则、资金池和执行状态缺少运营控制。

### 3.2 P1：增强贸易事实和风险缓释

| 角色 | 优先级 | 原因 |
|---|---|---|
| 物流方 / 货代 / 承运方 | P1 | 提供揽收、出港、到港、签收、异常等物流事实，增强贸易真实性。 |
| 保险公司 / 承保方 | P1 | 提供承保、加费、拒保、理赔等风险缓释能力。 |
| 买方 / 进口商 / 付款方 | P1 | 确认订单、收货、争议和付款责任，增强回款可信度。 |

P1 不是最小闭环必须条件，但会显著增强融资判断质量。

物流方能证明货物是否真实运输。
保险公司能缓释运输、买方、国家、品类等风险。
买方能确认付款责任和争议状态。

### 3.3 P2：高价值但接入难度高

| 角色 | 优先级 | 原因 |
|---|---|---|
| 海关 / 报关 / 通关数据方 | P2 | 价值高，但国家、数据源、权限、接入方式差异大，适合后续阶段。 |

P2 适合在核心融资闭环稳定后再做。

通关数据可以显著增强贸易真实性和监管可信度，但不应该阻塞 MVP 第一阶段。

## 4. MVP 核心闭环

### 4.1 标准业务路径

| 步骤 | 业务动作 | 主要角色 |
|---|---|---|
| 1 | 出口商创建融资意图 | 出口商 |
| 2 | Agent 整理贸易材料 | 出口商 Agent |
| 3 | Agent 检查材料缺口 | 出口商 Agent |
| 4 | 质检方接收质检任务 | 质检方 |
| 5 | 质检方提交 pass / fail / dispute | 质检方 |
| 6 | Agent 生成融资案件摘要 | 出口商 Agent / 资金方 Agent |
| 7 | 资金方查看完整性和风险建议 | 资金方 |
| 8 | 资金方决定允许放款 / 拒绝 / 补材料 | 资金方 |
| 9 | 平台 / 规则 / 合约检查是否允许执行 | 平台方 / 智能合约 |
| 10 | 出口商看到到账或卡点 | 出口商 |

### 4.2 出口商最终必须看到

| 信息 | 说明 |
|---|---|
| 能不能借 | 可以 / 不可以 / 需要补材料 / 等待条件。 |
| 能借多少 | 建议融资金额或资金方批准金额。 |
| 为什么不能继续 | 缺材料、质检 dispute、资金方拒绝、规则阻断、资金池不足等。 |
| 卡在哪里 | 出口商、质检方、资金方、平台、规则、资金池。 |
| 下一步找谁 | Agent 转成明确补材料或确认任务。 |
| 钱是否到账 | 已到账、放款中、等待执行、执行失败。 |

### 4.3 资金方最终必须看到

| 信息 | 说明 |
|---|---|
| 案件是否值得看 | Agent 分组和摘要。 |
| 材料是否完整 | 合同、订单、发票、质检、物流等核心证据状态。 |
| 风险是否过线 | 正常、警示、高风险、阻断。 |
| 建议动作 | 允许放款、拒绝、补材料、人工复核。 |
| 执行是否允许 | 资金池、规则、平台、合约状态是否通过。 |

## 5. MVP 中 Agent 的角色

MVP 不是把所有事情交给人，也不是让 Agent 自动做所有决定。

MVP Agent 目标：

| 目标 | 说明 |
|---|---|
| 整理材料 | 把贸易材料整理成角色可读摘要。 |
| 检查缺口 | 告诉用户缺什么、找谁补。 |
| 生成待办 | 把缺口变成明确动作。 |
| 推进常规流程 | 正常案件尽量自动推进。 |
| 升级异常 | 争议、高风险、授权、平台控制交给人。 |

MVP 阶段推荐默认：

| 模式 | 建议 |
|---|---|
| 全人工确认 | 适合新用户、高风险案件。 |
| 半自动托管 | MVP 默认推荐。 |
| 全自动 Agent | 限于低风险、授权明确、材料完整、规则清楚的流程。 |

## 6. MVP 中智能合约的角色

MVP 阶段只定义智能合约业务定位，不写实现。

| 职责 | 说明 |
|---|---|
| 检查放款条件 | 资金方是否已批准、案件是否满足执行条件。 |
| 检查资金池状态 | 正常、不足、暂停、异常。 |
| 检查规则版本 | 当前规则是否有效、暂停、匹配。 |
| 检查平台暂停状态 | 平台、规则、资金池、案件是否暂停。 |
| 返回业务输出 | 允许执行、不允许执行、等待条件、规则阻断、资金池不足等。 |

智能合约不负责：

| 不负责 | 原因 |
|---|---|
| 判断贸易真实性 | 事实来自业务角色。 |
| 替资金方审批 | 放款权属于资金方。 |
| 替 Agent 整理材料 | Agent 负责业务解释。 |
| 替平台做所有运营判断 | 平台仍负责规则、异常和系统级控制。 |

## 7. MVP 阶段排除范围

为了防止 MVP 失焦，以下内容不进入当前阶段：

| 排除项 | 说明 |
|---|---|
| 登录前展示页 | 不做营销官网或产品首页。 |
| UI 组件和页面设计 | 当前仅写业务需求和功能需求。 |
| 数据库表结构 | 后续技术设计再处理。 |
| 智能合约代码 | 本阶段只写业务定位和输出。 |
| 钱包签名流程细节 | 后续实现阶段再处理。 |
| hash / proof / proof pack 细节 | 当前不跳到底层证明产品。 |
| 完整保险核心系统 | 保险先作为风险缓释入口。 |
| 完整物流管理系统 | 物流先作为运输事实来源。 |
| 真实海关系统集成 | 通关数据作为 P2 后续接入。 |

## 8. 阶段推进建议

### 8.1 当前阶段完成标准

| 标准 | 说明 |
|---|---|
| 角色业务边界清楚 | 每个角色知道自己能做什么、不能做什么。 |
| Agent 操作模型清楚 | 每个角色都有三种 Agent 模式。 |
| 智能合约业务定位清楚 | 只作为规则和资金执行约束层。 |
| MVP 优先级清楚 | P0、P1、P2 分层明确。 |
| 排除范围清楚 | 不写 UI、数据库、合约实现、hash/proof 细节。 |

### 8.2 下一阶段建议

完成本轮需求文档后，下一阶段建议进入：

> P0 MVP 用户旅程和业务流程文档。

建议先写：

| 文档 | 目的 |
|---|---|
| P0 融资闭环流程 | 从出口商创建融资意图到资金方决策和执行检查。 |
| P0 状态机业务定义 | Draft、Submitted、Need materials、Approved、Blocked、Funds arrived 等状态。 |
| P0 异常和补材料流程 | 缺材料、质检 dispute、资金池不足、规则阻断等怎么流转。 |
| P0 Agent 待办模型 | Agent 如何把缺口转成不同角色待办。 |

## 9. 本步结论

ChainTrace MVP 不应该从页面、组件、合约或 hash/proof 细节开始。

它应该先完成一个可解释、可分工、可自动推进的供应链金融前站闭环：

> 出口商提出融资需求，Agent 整理材料和缺口，质检方提供独立事实，资金方做放款判断，平台和智能合约负责执行约束，出口商最终看到到账或卡点。

下一步应进入：

> P0 MVP 用户旅程、状态流转、异常处理和 Agent 待办模型。
