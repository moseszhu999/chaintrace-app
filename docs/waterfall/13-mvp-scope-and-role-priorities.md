# 13 MVP Scope and Role Priorities

## 1. 定位

本文件用于总结 ChainTrace MVP 阶段的角色优先级和业务闭环。

ChainTrace 当前产品定位是：

> Agent Native + Crypto Native 的供应链金融系统。

它不是供应链金融前站，而是包含出口商、银行/资金方、事实提供方、平台运营和 DeFi 智能合约资金执行层的完整供应链金融系统。

MVP 不追求一次性覆盖所有供应链金融场景。

MVP 的核心目标是先跑通一个清楚的融资闭环，同时承认不同场景可以有不同执行路径：

```text
出口商创建融资意图
→ Agent 整理贸易材料
→ 质检方提供质检事实
→ 根据场景进入银行辅助路径或纯 DeFi 路径
→ 银行/资金方决策，或 DeFi 智能合约按规则自动检查
→ 平台/规则/DeFi 智能合约检查是否允许执行
→ 出口商看到到账或卡点
```

## 2. MVP 范围原则

| 原则 | 说明 |
|---|---|
| 先闭环，后扩展 | 先完成融资最小业务闭环，再扩展更多角色。 |
| 先业务需求，后 UI 和实现 | 当前不写页面、组件、数据库、合约实现。 |
| 先角色边界，后技术边界 | 先明确每个角色能做什么、不能做什么。 |
| 先 Agent Native 操作模型 | 正常案件尽量由 Agent + 规则推进。 |
| 先融资判断必需事实 | 贸易、质检、资金方决策或 DeFi 规则、平台规则优先。 |
| 外部接入后置 | 海关、报关、通关等高价值但接入复杂的能力后置。 |

MVP 阶段不做：

| 暂不做 | 原因 |
|---|---|
| 登录前展示页 | 当前不做营销和转化页面。 |
| UI 组件设计 | 当前仍处于需求阶段。 |
| 数据库设计 | 当前不进入技术实现。 |
| 智能合约实现 | 当前只定义业务定位。 |
| 钱包签名细节 | 当前不进入实现细节。 |
| hash / proof 细节 | 当前不直接跳到底层证明细节。 |

## 3. 角色优先级

### 3.1 P0：融资闭环核心能力

| 角色 / 能力 | 优先级 | 原因 |
|---|---|---|
| 出口商 / 资金需求方 | P0 | 发起融资意图、提交贸易材料、查看融资结果，是需求起点。 |
| DeFi 智能合约资金执行层 | P0 | 负责资金池、规则、执行、结算和阻断，是 Crypto Native 场景的资金执行核心。 |
| Agent 自动化能力 | P0 | 整理材料、检查缺口、生成待办、推进低风险流程，是低人工参与的核心。 |
| 资金方 / 银行 / 保理机构 | P0 场景角色 | 需要银行参与的场景中，决定允许放款、拒绝或要求补材料。 |
| 质检方 / 第三方检验机构 | P0 | 提供独立货物事实，是贸易真实性和风险判断关键依据。 |
| 平台方 / 合约运营 | P0 | 维护资金池、规则、合约状态、异常和执行秩序。 |

P0 不是“每个场景必须有同一组人”。

P0 是融资系统必须具备的核心能力组合。

没有出口商，就没有融资需求。
没有 DeFi 智能合约资金执行层，就不是 Crypto Native 的供应链金融系统。
没有 Agent 自动化，就不能实现低人工参与。
没有银行/资金方，并不代表所有场景不能融资；纯 DeFi 场景可以由智能合约根据规则和资金池条件执行。
没有质检方，货物事实缺少独立确认。
没有平台方，规则、资金池和执行状态缺少运营控制。

### 3.2 P1：增强贸易事实和风险缓释

| 角色 | 优先级 | 原因 |
|---|---|---|
| 物流方 / 货代 / 承运方 | P1 | 提供揽收、出港、到港、签收、异常等物流事实，增强贸易真实性。 |
| 保险公司 / 承保方 | P1 | 提供承保、加费、拒保、理赔等风险缓释能力。 |
| 买方 / 进口商 / 付款方 | P1 | 确认订单、收货、争议和付款责任，增强回款可信度。 |

P1 不是最小闭环必须条件，但会显著增强融资判断质量。

物流方能证明货物是否真实运输。
保险公司能缓释运输、买方、国家、品类等风险。
买方能确认付款责任和争议状态。

### 3.3 P2：高价值但接入难度高

| 角色 | 优先级 | 原因 |
|---|---|---|
| 海关 / 报关 / 通关数据方 | P2 | 价值高，但国家、数据源、权限、接入方式差异大，适合后续阶段。 |

P2 适合在核心融资闭环稳定后再做。

通关数据可以显著增强贸易真实性和监管可信度，但不应该阻塞 MVP 第一阶段。

## 4. MVP 核心闭环

### 4.1 银行辅助业务路径

| 步骤 | 业务动作 | 主要角色 |
|---|---|---|
| 1 | 出口商创建融资意图 | 出口商 |
| 2 | Agent 整理贸易材料 | 出口商 Agent |
| 3 | Agent 检查材料缺口 | 出口商 Agent |
| 4 | 质检方接收质检任务 | 质检方 |
| 5 | 质检方提交 pass / fail / dispute | 质检方 |
| 6 | Agent 生成融资案件摘要 | 出口商 Agent / 资金方 Agent |
| 7 | 资金方查看完整性和风险建议 | 资金方 |
| 8 | 资金方决定允许放款 / 拒绝 / 补材料 | 资金方 |
| 9 | 平台 / 规则 / DeFi 智能合约检查是否允许资金执行 | 平台方 / DeFi 智能合约 |
| 10 | 出口商看到到账或卡点 | 出口商 |

### 4.2 纯 DeFi 业务路径

| 步骤 | 业务动作 | 主要角色 |
|---|---|---|
| 1 | 出口商或业务方创建融资/结算意图 | 出口商 / 业务方 |
| 2 | Agent 整理材料和事实 | Agent |
| 3 | 事实方提供必要事实 | 质检方 / 物流方 / 买方 / 其他事实方 |
| 4 | Agent 检查缺口和规则适配 | Agent |
| 5 | DeFi 智能合约检查资金池、抵押/保证、规则和暂停状态 | DeFi 智能合约 |
| 6 | 条件满足时自动执行资金动作 | DeFi 智能合约 |
| 7 | 条件不满足时返回阻断或等待原因 | DeFi 智能合约 / Agent |
| 8 | 人只处理异常、争议、高风险授权 | 相关角色 |

### 4.3 Agent + 智能合约低人工路径

| 步骤 | 业务动作 | 主要角色 |
|---|---|---|
| 1 | 业务规则和授权边界预先明确 | 平台方 / 业务方 |
| 2 | Agent 收集和整理事实 | Agent |
| 3 | Agent 生成待办、提醒和缺口检查 | Agent |
| 4 | 智能合约根据规则锁定、释放、结算或阻断资金 | DeFi 智能合约 |
| 5 | 法律或责任争议进入人工或约定仲裁 | 人 / 仲裁机制 |
| 6 | 正常低风险事项尽量自动完成 | Agent + DeFi 智能合约 |

### 4.4 出口商最终必须看到

| 信息 | 说明 |
|---|---|
| 能不能借 | 可以 / 不可以 / 需要补材料 / 等待条件。 |
| 能借多少 | 建议融资金额、资金方批准金额或 DeFi 规则可执行金额。 |
| 为什么不能继续 | 缺材料、质检 dispute、资金方拒绝、DeFi 规则阻断、资金池不足等。 |
| 卡在哪里 | 出口商、质检方、资金方、平台、规则、资金池。 |
| 下一步找谁 | Agent 转成明确补材料或确认任务。 |
| 钱是否到账 | 已到账、放款中、等待执行、执行失败。 |

### 4.5 资金方最终必须看到

资金方只在需要银行/机构参与的场景中出现。

| 信息 | 说明 |
|---|---|
| 案件是否值得看 | Agent 分组和摘要。 |
| 材料是否完整 | 合同、订单、发票、质检、物流等核心证据状态。 |
| 风险是否过线 | 正常、警示、高风险、阻断。 |
| 建议动作 | 允许放款、拒绝、补材料、人工复核。 |
| 执行是否允许 | 资金池、规则、平台、DeFi 合约状态是否通过。 |

## 5. MVP 中 Agent 的角色

MVP 不是把所有事情交给人，也不是让 Agent 自动做所有决定。

MVP Agent 目标：

| 目标 | 说明 |
|---|---|
| 整理材料 | 把贸易材料整理成角色可读摘要。 |
| 检查缺口 | 告诉用户缺什么、找谁补。 |
| 生成待办 | 把缺口变成明确动作。 |
| 推进常规流程 | 正常案件尽量自动推进。 |
| 升级异常 | 争议、高风险、授权、平台控制交给人。 |

MVP 阶段推荐默认：

| 模式 | 建议 |
|---|---|
| 全人工确认 | 适合新用户、高风险案件。 |
| 半自动托管 | MVP 默认推荐。 |
| 全自动 Agent | 限于低风险、授权明确、材料完整、规则清楚的流程。 |

## 6. MVP 中 DeFi 智能合约的角色

MVP 阶段只定义 DeFi 智能合约的业务定位，不写 Solidity 或部署实现。

在 ChainTrace 中，智能合约不是外围工具，而是资金执行核心层。

| 职责 | 说明 |
|---|---|
| 检查放款条件 | 银行辅助场景检查银行 / 资金方是否已批准；纯 DeFi 场景检查规则、资金池、抵押/保证和事实状态。 |
| 检查 DeFi 资金池状态 | 正常、不足、占用超限、暂停、异常。 |
| 检查规则版本 | 当前规则是否有效、暂停、匹配。 |
| 检查平台暂停状态 | 平台、规则、资金池、案件是否暂停。 |
| 返回业务输出 | 允许执行、不允许执行、等待条件、规则阻断、资金池不足等。 |

智能合约不负责：

| 不负责 | 原因 |
|---|---|
| 判断贸易真实性 | 事实来自业务角色。 |
| 替银行 / 资金方审批 | 需要银行参与的场景中，放款审批和授权权属于银行 / 资金方。 |
| 替人类裁决重大法律争议 | 法律或责任争议可由 Agent 整理、合约约束资金，但重大争议仍需要人或约定仲裁机制。 |
| 替 Agent 整理材料 | Agent 负责业务解释。 |
| 替平台做所有运营判断 | 平台仍负责规则、异常和系统级控制。 |

## 7. MVP 阶段排除范围

为了防止 MVP 失焦，以下内容不进入当前阶段：

| 排除项 | 说明 |
|---|---|
| 登录前展示页 | 不做营销官网或产品首页。 |
| UI 组件和页面设计 | 当前仅写业务需求和功能需求。 |
| 数据库表结构 | 后续技术设计再处理。 |
| 智能合约代码 | 本阶段只写业务定位和输出。 |
| 钱包签名流程细节 | 后续实现阶段再处理。 |
| hash / proof / proof pack 细节 | 当前不跳到底层证明产品。 |
| 完整保险核心系统 | 保险先作为风险缓释入口。 |
| 完整物流管理系统 | 物流先作为运输事实来源。 |
| 真实海关系统集成 | 通关数据作为 P2 后续接入。 |

| 标准 | 说明 |

| Agent 操作模型清楚 | 每个角色都有三种 Agent 模式。 |
| DeFi 智能合约业务定位清楚 | 作为资金执行核心层和规则约束层。 |
| MVP 优先级清楚 | P0、P1、P2 分层明确。 |
| 排除范围清楚 | 不写 UI、数据库、合约实现、hash/proof 细节。 |

### 8.2 下一阶段建议

完成本轮需求文档后，下一阶段建议进入：

建议先写：

## 9. 本步结论

ChainTrace MVP 不应该从页面、组件、Solidity 实现或 hash/proof 细节开始。

它应该先完成一个可解释、可分工、可自动推进的供应链金融系统闭环：

> 出口商或业务方提出融资/结算需求，Agent 整理材料和缺口，事实方提供独立事实；需要银行的场景由银行/资金方做放款判断，不需要银行的场景由 DeFi 智能合约按规则和资金池条件执行；平台维护规则和异常，出口商或业务方最终看到到账、结算完成或卡点。

下一步应进入：

> P0 MVP 用户旅程、状态流转、异常处理和 Agent 待办模型。
