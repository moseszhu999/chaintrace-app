# 43 P0 视觉原型变更管理说明

## 1. 定位

本文件定义 ChainTrace P0 视觉原型评审后的变更管理规则。

视觉原型变更管理说明不是 UI 修改清单，不是组件设计规范，不是前端任务拆分，也不是技术变更流程。

它的作用是：

> 在视觉原型走读和评审后，用业务语言判断每一条反馈应该直接调整、回到 waterfall 需求文档补充、后置到 P1/P2，还是阻断当前原型继续推进。

本文件不写组件、样式、布局尺寸、图表库、页面路由、接口路径、请求响应格式、数据库、Solidity、ABI、钱包签名、hash 或 proof 细节。

## 2. 变更管理原则

| 原则 | 说明 |
|---|---|
| 业务优先 | 所有变更先判断是否影响供应链金融系统闭环。 |
| 需求先于原型 | 需求文档没有定义的核心业务变更，不应直接进入视觉原型。 |
| 小调整不返工 | 不改变业务含义的表达优化可以直接调整。 |
| 大变化回 waterfall | 影响角色、路径、权限、Agent 边界、DeFi 输出或异常规则的变更必须回到需求文档。 |
| 后置有记录 | 有价值但不属于 P0 的内容必须记录原因和后续阶段。 |
| 阻断立即停 | 破坏产品定位、权限边界或路径逻辑的变更必须先修正。 |
| 不进入实现 | 本阶段只管理业务变更，不拆技术任务。 |

## 3. 变更来源

| 来源 | 示例 | 处理重点 |
|---|---|---|
| 视觉原型走读 | 某角色看不懂当前卡点。 | 判断是否是表达问题还是需求缺口。 |
| 评审清单 | 某条路径未通过评审。 | 对照 42 号评审清单分类。 |
| 角色反馈 | 资金方认为风险摘要不够清楚。 | 判断是否影响业务判断。 |
| 路径反馈 | 纯 DeFi 路径被误表达成银行审批。 | 优先修正路径定位。 |
| 权限反馈 | 事实方看到无关案件。 | 作为阻断项处理。 |
| 异常反馈 | 资金池不足没有责任方或下一步。 | 补充异常回流表达。 |
| P1/P2 发现 | 物流、保险、海关或买方提出深度流程需求。 | 判断是否只保留节点入口。 |

## 4. 变更分类

| 分类 | 含义 | 处理方式 |
|---|---|---|
| A 类：可直接调整 | 不改变业务规则、角色权限、路径或范围的表达优化。 | 原型说明直接调整，并记录。 |
| B 类：需补需求 | 暴露出 waterfall 文档未定义或定义不清的核心业务问题。 | 先补需求文档，再调整原型。 |
| C 类：后置 | 有价值，但不属于 P0 或当前阶段。 | 记录到后置清单，不进入 P0 原型。 |
| D 类：阻断 | 破坏产品定位、路径、权限、Agent 边界或 DeFi 执行逻辑。 | 暂停该部分原型推进，必须修正。 |

## 5. A 类：可直接调整

以下反馈可以直接进入视觉原型调整，不需要回到 waterfall 补需求。

| 可直接调整项 | 条件 | 记录要求 |
|---|---|---|
| 业务文案更清楚 | 不改变业务含义。 | 记录原问题和调整后表达。 |
| 卡点说明更清楚 | 原需求已有责任方、原因和下一步。 | 记录引用的需求来源。 |
| 节点命名更贴近业务 | 不新增节点、不改变状态含义。 | 记录命名变化。 |
| 角色提示更明确 | 不扩大可见范围和操作范围。 | 记录适用角色。 |
| Agent 摘要表达优化 | 不增加 Agent 权限。 | 记录摘要口径。 |
| DeFi 输出解释优化 | 不改变合约业务输出类别。 | 记录用户应理解的业务含义。 |
| 异常提示优化 | 不新增异常规则。 | 记录异常对应责任方。 |
| 后置项提示优化 | 不把后置范围提前进 P0。 | 记录后置阶段。 |

## 6. B 类：必须回到 waterfall 补需求

以下反馈不能直接改原型，必须先补充或修订 waterfall 需求文档。

| 需补需求项 | 原因 | 补充位置 |
|---|---|---|
| 新增业务角色 | 影响角色权限、工作区和流程。 | 角色需求或 P0 范围文档。 |
| 新增融资/结算路径 | 影响路径矩阵和验收脚本。 | 场景路径矩阵、PRD 总纲。 |
| 改变银行辅助路径 | 影响资金方授权和 DeFi 执行条件。 | 资金方需求、路径矩阵、业务规则。 |
| 改变纯 DeFi 路径 | 影响无银行场景的规则和资金执行。 | DeFi 业务规则、路径矩阵。 |
| 改变低人工路径 | 影响 Agent 自动推进和人工升级边界。 | Agent Operating Model、业务规则。 |
| 改变角色可见范围 | 影响权限边界。 | 权限矩阵、角色需求。 |
| 新增关键业务状态 | 影响控制塔、事件和验收。 | 业务事件、控制塔要求、验收脚本。 |
| 新增高风险判断规则 | 影响人工升级和资金执行。 | 业务规则、非功能业务要求。 |
| 新增事实来源 | 影响事实方边界和材料完整性。 | 对应角色需求、业务数据要求。 |
| 改变合约业务输出 | 影响 DeFi 执行解释和阻断逻辑。 | 智能合约业务角色、DeFi 规则清单。 |

## 7. C 类：后置处理

以下反馈可以记录，但不进入 P0 视觉原型深度展开。

| 后置项 | P0 处理方式 | 后续阶段 |
|---|---|---|
| 物流完整协同工作台 | P0 保留物流节点、状态和异常入口。 | P1 |
| 保险完整核保和理赔流程 | P0 保留承保/拒保/加费/理赔状态。 | P1 |
| 买方完整付款系统 | P0 保留确认、争议和付款责任节点。 | P1 |
| 海关/报关真实接入 | P0 保留通关事实节点或占位。 | P2 |
| 多组织多成员权限 | P0 保持单钱包单角色。 | P1/P2 |
| 复杂法律自动执行 | P0 只保留低人工边界和人工升级。 | P2 |
| 技术接口和数据模型 | P0 不做技术设计。 | 技术设计阶段 |
| 合约实现、部署和 ABI | P0 只保留业务输出。 | 合约技术设计阶段 |
| hash / proof 产品化页面 | 当前不进入底层证明产品。 | 后续证明产品阶段 |

## 8. D 类：阻断项

以下情况必须阻断当前视觉原型推进。

| 阻断项 | 阻断原因 | 修正要求 |
|---|---|---|
| 产品被表达成供应链金融前站 | 违背完整供应链金融系统定位。 | 回到产品定位和控制塔表达。 |
| 以登录前展示页作为主体验 | 违背当前阶段范围。 | 改为注册后业务工作区。 |
| 银行成为所有路径必经 | 破坏纯 DeFi 和低人工场景。 | 区分银行辅助、纯 DeFi、低人工路径。 |
| 纯 DeFi 路径等待银行审批 | 路径逻辑错误。 | 改为规则、资金池、事实和合约执行驱动。 |
| Agent 被表达成审批人 | 破坏 Agent 业务助理边界。 | 改为摘要、缺口、待办、建议和升级。 |
| 合约被表达成页面或数据库 | 破坏智能合约业务定位。 | 改为资金执行约束层输出。 |
| 事实方能决定融资 | 破坏事实来源边界。 | 事实方只确认事实。 |
| 平台方能伪造业务事实 | 破坏可信边界。 | 平台只做规则、资金池、暂停和异常运营。 |
| 角色能查看无关案件 | 权限隔离失败。 | 按角色相关性重新限定可见范围。 |
| 异常没有责任方和下一步 | 业务不可处理。 | 补充原因、责任角色、解除条件和下一步。 |
| 原型进入接口、数据库、Solidity 或 hash/proof 细节 | 阶段跑偏。 | 回到业务表达。 |
| 继承旧版页面或旧代码口径 | 违背全新瀑布重建设计分支要求。 | 重新按当前需求表达。 |

## 9. 变更处理流程

| 顺序 | 动作 | 输出 |
|---:|---|---|
| 1 | 记录评审反馈 | 反馈编号、来源、角色、场景和原始描述。 |
| 2 | 判断影响范围 | 产品定位、角色、路径、权限、Agent、DeFi、异常、后置。 |
| 3 | 归类变更 | A 类、B 类、C 类或 D 类。 |
| 4 | 指定责任动作 | 直接调整、补需求、后置记录或阻断修正。 |
| 5 | 记录依据 | 引用对应 waterfall 文档和评审条目。 |
| 6 | 更新状态 | Open、In review、Accepted、Deferred、Blocked、Closed。 |
| 7 | 复核结果 | 确认调整后没有引入新越权、新路径错误或新范围漂移。 |

## 10. 变更责任边界

| 责任方 | 可以做 | 不可以做 |
|---|---|---|
| 产品负责人 | 判断变更分类、决定是否回到需求文档。 | 直接绕过核心需求修改路径或权限。 |
| 业务评审人 | 提出业务表达、角色、路径和异常问题。 | 要求直接进入实现细节。 |
| 视觉原型负责人 | 根据已确认需求调整业务表达。 | 自行新增业务规则或角色权限。 |
| 平台方 / 合约运营 | 说明规则、资金池、暂停和执行失败口径。 | 替资金方审批或替事实方制造事实。 |
| 资金方 | 确认银行辅助路径中的审查和授权表达。 | 要求纯 DeFi 路径强制银行审批。 |
| 事实方 | 确认自己的事实节点表达是否准确。 | 修改其他角色事实或融资决策。 |
| Agent 设计负责人 | 确认摘要、缺口、待办、建议和升级边界。 | 把 Agent 设计成事实制造者或高风险裁决者。 |

## 12. 典型判定示例

| 反馈 | 分类 | 理由 |
|---|---|---|
| “资金池不足”提示不清楚，不知道谁处理。 | A 类 | 异常规则已有，只需要补清责任方和下一步表达。 |
| 原型希望新增“仲裁机构”角色。 | B 类 | 新角色影响权限、路径和争议处理，必须先补需求。 |
| 保险公司希望展开完整理赔材料管理。 | C 类 | 有价值，但 P0 只保留保险节点和结论。 |
| 纯 DeFi 路径仍要求银行审批。 | D 类 | 破坏产品路径定位，必须阻断修正。 |
| Agent 文案从“建议放款”改成“建议人工复核”。 | A 类 | 更符合 Agent 边界，不改变规则。 |
| 资金方希望看到所有出口商案件。 | D 类 | 违反案件相关可见原则。 |
| 海关真实系统接入细节被要求进入原型。 | C 类 | P2 接入价值高，但 P0 后置。 |
| 合约输出新增“规则阻断原因分类”。 | B 类 | 影响智能合约业务输出和业务规则清单。 |

## 14. MVP 范围

### 14.1 P0 必须管理

| 范围 | 说明 |
|---|---|
| 产品定位变更 | 是否仍是完整供应链金融系统。 |
| 控制塔表达变更 | 是否仍能看清源头到终端和资金结果。 |
| 角色视角变更 | 是否改变可见范围和责任边界。 |
| 路径变更 | 是否影响银行辅助、纯 DeFi、低人工和人工升级。 |
| Agent 表达变更 | 是否越过业务助理边界。 |
| DeFi 输出变更 | 是否改变允许、等待、阻断、失败、完成等业务输出。 |
| 异常回流变更 | 是否影响原因、责任方、解除条件和下一步。 |
| 后置范围变更 | 是否把 P1/P2 或技术设计提前塞进 P0。 |

### 14.2 P0 暂不管理

| 范围 | 原因 |
|---|---|
| 具体 UI 组件调整 | 当前不做组件规范。 |
| 视觉美术偏好 | 当前只管理业务表达变化。 |
| 页面路由和前端实现 | 当前不写代码。 |
| API endpoint 和请求响应 | 技术接口设计后置。 |
| 数据库表结构 | 数据设计后置。 |
| Solidity / ABI / 部署 | 合约实现后置。 |
| 钱包签名流程 | 钱包实现后置。 |
| hash / proof 产品页 | 当前不进入底层证明产品。 |

## 15. 本步结论

P0 视觉原型变更管理的本质不是“收集修改意见”。

它的本质是：

> 把视觉原型评审反馈纳入瀑布需求管理，区分可直接调整、必须补需求、必须后置和必须阻断的情况，防止原型阶段绕过 ChainTrace 的产品定位、角色权限、路径差异、Agent 边界、DeFi 执行和异常回流规则。

下一步应进入：

> P0 视觉原型交付包说明，用于定义视觉原型阶段最终应交付哪些业务材料、评审记录和后置清单，仍不进入代码实现。
