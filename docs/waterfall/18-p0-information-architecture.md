# 18 P0 信息架构业务需求

## 1. 定位

本文件定义 ChainTrace P0 MVP 的信息架构。

信息架构不是 UI 页面设计，不是组件设计，也不是数据库结构。

它的核心作用是：

> 说明融资案件中哪些业务信息必须被组织清楚、哪些角色需要看到哪些信息、每类信息要回答什么业务问题。

本文件只写业务信息需求，不写页面、组件、路由、数据库、智能合约实现、钱包签名、hash 或 proof 细节。

## 2. 信息架构原则

| 原则 | 说明 |
|---|---|
| 以融资案件为中心 | 所有信息都围绕一笔融资案件组织。 |
| 以角色问题为入口 | 每个角色先看到自己要解决的业务问题。 |
| 以 Agent 摘要减少噪音 | Agent 先整理摘要、缺口、风险和待办。 |
| 事实、判断、执行分开 | 贸易事实、质检事实、资金方判断、平台执行条件不能混在一起。 |
| 卡点必须可见 | 用户必须知道案件卡在哪里、谁负责、下一步是什么。 |
| 不提前设计 UI | 本阶段只定义信息结构和业务含义。 |

## 3. P0 信息域

P0 MVP 至少包含以下信息域：

| 信息域 | 业务含义 | 主要使用角色 |
|---|---|---|
| 角色身份 | 钱包绑定后的永久业务身份。 | 所有角色 |
| 融资案件 | 出口商围绕一笔贸易发起的融资申请。 | 出口商 / 资金方 / 平台方 |
| 贸易材料 | 合同、订单、发票、装箱单、补充说明等。 | 出口商 / 资金方 |
| 质检事实 | pass / fail / dispute、报告、照片、异常说明。 | 质检方 / 资金方 / 出口商 |
| Agent 摘要 | 材料摘要、缺口、风险提示、建议动作。 | 所有 P0 角色 |
| 待办 | 由 Agent 生成或角色发起的业务动作。 | 所有 P0 角色 |
| 资金方决策 | 允许放款、拒绝、补材料、人工复核。 | 资金方 / 出口商 / 平台方 |
| 执行条件 | 资金池、规则、暂停、平台状态、执行结果。 | 平台方 / 资金方 / 出口商 |
| 状态和卡点 | 当前状态、原因、下一步、责任角色。 | 所有 P0 角色 |
| 操作记录 | 关键业务动作和原因。 | 平台方 / 资金方 / 审计相关角色 |

## 4. 融资案件信息结构

融资案件是 P0 信息架构的中心。

| 信息组 | 要回答的问题 |
|---|---|
| 案件基本信息 | 这是一笔什么融资申请？ |
| 出口商信息 | 谁在申请融资？ |
| 贸易信息 | 基于哪笔贸易申请融资？ |
| 申请融资信息 | 想借多少、什么币种、什么时候到账？ |
| 材料完整性 | 材料是否足够提交或审查？ |
| 质检状态 | 货物事实是否已被独立确认？ |
| 风险摘要 | 资金方需要关注哪些风险？ |
| 资金方决策 | 资金方是否允许放款？ |
| 执行条件 | 已批准案件是否真的能执行？ |
| 当前卡点 | 案件现在卡在哪里？ |

融资案件不应被表达成“文件列表”。

融资案件应该被表达成：

```text
一笔贸易融资申请现在处于什么状态、证据是否完整、风险在哪里、谁需要做什么、是否可以执行。
```

## 5. 角色信息需求

### 5.1 出口商需要的信息

| 信息 | 业务问题 |
|---|---|
| 当前融资机会 | 我这笔贸易能不能融资？ |
| 建议融资金额 | 大概能借多少？ |
| 材料缺口 | 还缺什么？ |
| 提交前摘要 | 提交给资金方前会看到什么？ |
| 案件状态 | 当前是在准备、质检、审查、执行还是到账？ |
| 卡点责任方 | 是我、质检方、资金方还是平台方要处理？ |
| 融资结果 | 通过、拒绝、补材料、等待执行还是到账？ |

出口商不需要优先看到：

| 不优先展示 | 原因 |
|---|---|
| 平台规则细节 | 只需要知道影响结果的业务解释。 |
| 合约技术细节 | 当前不是实现阶段。 |
| 无关资金方内部审查 | 数据边界和角色边界。 |

### 5.2 质检方需要的信息

| 信息 | 业务问题 |
|---|---|
| 待质检任务 | 哪些案件需要我处理？ |
| 货物摘要 | 要检查什么货物？ |
| 质检清单 | 要检查哪些项目？ |
| 材料完整性 | 报告、照片、结论是否足够？ |
| 质检结论 | pass / fail / dispute 及原因。 |
| 补充要求 | 为什么要补充、补什么？ |

质检方不需要看到：

| 不应看到 | 原因 |
|---|---|
| 无关融资案件 | 数据隔离。 |
| 资金方内部授信判断 | 质检方不是审批方。 |
| 平台规则维护信息 | 与质检事实无关。 |

### 5.3 资金方需要的信息

| 信息 | 业务问题 |
|---|---|
| 待审案件分组 | 哪些案件值得先看？ |
| 融资案件摘要 | 这是什么贸易、谁申请、金额多少？ |
| 材料完整性 | 证据是否足够？ |
| 质检事实 | 货物是否 pass / fail / dispute？ |
| 风险摘要 | 哪些风险需要判断？ |
| 建议动作 | 允许放款、拒绝、补材料还是人工复核？ |
| 执行条件 | 批准后是否可以执行？ |

资金方不需要优先看到：

| 不优先展示 | 原因 |
|---|---|
| 原始文件堆 | Agent 应先生成可审摘要。 |
| 平台底层技术状态 | 只需要业务层执行结果和原因。 |
| 其他资金方案件 | 数据隔离。 |

### 5.4 平台方需要的信息

| 信息 | 业务问题 |
|---|---|
| 运行总览 | 系统是否安全？ |
| 异常案件 | 哪些案件需要平台介入？ |
| 资金池状态 | 是否足够、暂停、异常或执行失败？ |
| 规则状态 | 当前按哪套规则执行？是否暂停？ |
| 执行卡点 | 资金池不足、规则阻断、平台暂停还是执行失败？ |
| Agent 状态 | 是否有任务堆积、误推进或需要降级人工？ |
| 操作记录 | 谁做了什么平台级动作，原因是什么？ |

平台方不应该默认看到：

| 不应默认处理 | 原因 |
|---|---|
| 正常案件逐单审批 | 平台不是默认审批员。 |
| 出口商材料修改入口 | 平台不是事实制造者。 |
| 资金方最终放款判断 | 放款权属于资金方。 |

## 6. Agent 摘要信息结构

Agent 摘要是 P0 信息架构的关键入口。

| 摘要类型 | 面向角色 | 内容 |
|---|---|---|
| 出口商摘要 | 出口商 | 能不能提交、缺什么、下一步找谁。 |
| 质检任务摘要 | 质检方 | 要检查什么、关键风险点、截止时间。 |
| 资金方审查摘要 | 资金方 | 案件概况、材料完整性、质检状态、风险建议。 |
| 平台异常摘要 | 平台方 | 异常类型、影响范围、责任角色、建议处理。 |
| 状态解释摘要 | 所有角色 | 当前状态、原因、下一步。 |

Agent 摘要必须满足：

| 要求 | 说明 |
|---|---|
| 有事实来源 | 摘要不能编造事实。 |
| 有缺口列表 | 明确缺什么。 |
| 有建议动作 | 说明下一步做什么。 |
| 有责任角色 | 说明谁处理。 |
| 有风险提示 | 高风险、争议和阻断必须显眼。 |

## 7. 状态和卡点信息结构

每个案件必须能回答：

| 问题 | 说明 |
|---|---|
| 当前状态是什么？ | Draft、Under review、Approved、Blocked 等。 |
| 为什么在这里？ | 例如等待质检、资金方补材料、资金池不足。 |
| 谁负责处理？ | 出口商、质检方、资金方、平台方或 Agent。 |
| 下一步是什么？ | 上传、确认、审批、补充、等待或人工复核。 |
| 是否影响融资结果？ | 影响提交、审查、放款执行或到账。 |

状态信息不应只有一个英文状态码。

必须有业务解释：

```text
当前状态：Execution waiting
原因：资金方已允许放款，但资金池余额不足。
下一步：平台方或资金方处理资金池，出口商等待结果。
```

## 8. 权限和可见性

信息架构必须遵守角色边界。

| 角色 | 可以看到 | 不可以看到 |
|---|---|---|
| 出口商 | 自己的案件、材料、状态、卡点、融资结果。 | 其他出口商案件、资金方内部审查细节。 |
| 质检方 | 分配给自己的质检任务和必要贸易信息。 | 无关融资案件、资金方审批结论细节。 |
| 资金方 | 分配或匹配给自己的融资案件和审查所需材料。 | 其他资金方无关案件、平台内部敏感操作。 |
| 平台方 | 平台状态、异常案件、规则、资金池、Agent 状态。 | 伪造或随意修改业务事实。 |
| Agent | 当前角色授权范围内的信息。 | 超出角色权限的信息和关键记录篡改。 |

## 9. MVP 范围

### P0 必须做

| 范围 | 说明 |
|---|---|
| 定义 P0 信息域 | 融资案件、贸易材料、质检事实、Agent 摘要、待办、决策、执行条件。 |
| 定义角色信息需求 | 出口商、质检方、资金方、平台方各自看什么。 |
| 定义 Agent 摘要结构 | 面向不同角色的业务摘要。 |
| 定义状态和卡点结构 | 当前状态、原因、责任角色、下一步。 |
| 定义权限和可见性 | 不同角色的信息边界。 |

### 暂不做

| 范围 | 原因 |
|---|---|
| 页面导航设计 | 当前不做 UI。 |
| 组件层级 | 当前不做前端组件。 |
| 数据库表结构 | 当前不写实现。 |
| 智能合约接口 | 当前不写技术设计。 |
| hash / proof 信息模型 | 当前不进入底层证明细节。 |

## 10. 本步结论

P0 信息架构的本质不是“菜单结构”。

它的本质是：

> 围绕一笔融资案件，把贸易事实、质检事实、Agent 摘要、资金方决策、执行条件、状态卡点和角色权限组织清楚，让每个角色先回答自己的业务问题。

下一步应进入：

> P0 工作区业务分区。
