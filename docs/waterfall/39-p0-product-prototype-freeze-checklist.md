# 39 P0 产品原型冻结清单

## 1. 定位

本文件定义 ChainTrace P0 阶段进入视觉原型或技术设计前的产品原型冻结清单。

产品原型冻结清单不是 UI 设计稿，不是组件库规范，不是开发计划，也不是技术实现清单。

它的作用是：

> 在进入视觉原型、交互原型、技术接口设计、数据设计或合约设计前，确认控制塔表达、角色视角、路径差异、Agent 边界、DeFi 执行输出、异常解释和排除范围已经冻结。

本文件不写页面布局、组件样式、接口路径、请求响应格式、数据库、Solidity、ABI、钱包签名、hash 或 proof 细节。

## 2. 冻结原则

| 原则 | 说明 |
|---|---|
| 冻结业务表达，不冻结视觉细节 | 固定原型要表达什么，不规定具体图形、颜色、组件和布局。 |
| 冻结角色视角，不冻结页面实现 | 固定每个角色能看什么、不能看什么，不写页面路由。 |
| 冻结路径差异，不冻结技术编排 | 固定银行辅助、纯 DeFi、低人工路径的业务差异。 |
| 冻结 Agent 边界，不冻结模型实现 | 固定 Agent 能做和不能做，不写模型、提示词或任务队列。 |
| 冻结 DeFi 输出，不冻结合约代码 | 固定允许、等待、阻断、失败、完成等业务输出。 |
| 冻结异常解释，不冻结错误码 | 固定用户要看到的原因、责任方和下一步。 |
| 冻结排除范围 | 不做登录前展示页，不继承旧页面和代码，不提前进入 hash/proof。 |

## 3. 原型定位冻结

| 检查项 | 冻结要求 | 状态 |
|---|---|---|
| 产品定位 | Agent Native + Crypto Native 的供应链金融系统。 | 可冻结 |
| 原型入口 | 钱包注册并绑定角色后进入业务工作区。 | 可冻结 |
| 原型主表达 | 大数据控制塔大屏风格。 | 可冻结 |
| 核心对象 | 融资/结算案件，而不是文件列表或营销线索。 | 可冻结 |
| 业务主轴 | 供应链从源头到终端的节点流转。 | 可冻结 |
| 执行核心 | DeFi 智能合约负责资金执行约束。 | 可冻结 |
| 银行定位 | 银行/资金方只在银行辅助路径参与。 | 可冻结 |
| Agent 定位 | 每个角色身边的业务助理。 | 可冻结 |

冻结后不得改成：

| 禁止改动 | 原因 |
|---|---|
| 登录前营销首页 | 当前明确不做。 |
| 供应链金融前站 | 产品定位已更正为完整供应链金融系统。 |
| 单纯银行审批系统 | 破坏纯 DeFi 和低人工路径。 |
| 只展示链上数据的大屏 | ChainTrace 的核心是供应链金融业务流转和资金执行约束。 |

## 4. 控制塔表达冻结

| 控制塔要求 | 冻结口径 |
|---|---|
| 全链路流转 | 必须表达从业务意图、贸易材料、事实确认、审查/规则、执行到资金结果。 |
| 节点状态 | 每个节点可表达 Not started、Waiting、In progress、Completed、Warning、Dispute、Blocked、Failed、Skipped。 |
| 卡点解释 | 当前卡在哪里、为什么、谁负责、下一步是什么。 |
| Agent 摘要 | 摘要、缺口、待办、建议、提醒和升级进入控制塔表达。 |
| DeFi 执行 | 允许执行、等待条件、规则阻断、资金池不足、平台暂停、执行失败、执行完成。 |
| 路径差异 | 银行辅助、纯 DeFi、低人工路径在控制塔上不能混淆。 |
| 角色过滤 | 大屏不等于全员看全量数据。 |

控制塔不得表达成：

| 不允许 | 原因 |
|---|---|
| 纯统计看板 | 缺少案件流转和责任角色。 |
| 文件管理器 | 不能只展示材料列表。 |
| 技术监控页 | 当前不是日志、节点、合约交易监控。 |
| 全角色全量大屏 | 违反权限隔离。 |

## 5. 角色视角冻结

| 角色 | 冻结可见范围 | 冻结不可见范围 |
|---|---|---|
| 出口商 / 资金需求方 | 自己案件全链路、材料、事实进度、Agent 缺口、资金结果和卡点。 | 其他出口商案件、资金方内部审查、平台全局运营数据。 |
| 资金方 / 银行 / 保理机构 | 银行辅助路径中的相关案件、摘要、风险、授权和执行条件。 | 无关案件、其他资金方意见、事实方内部过程、平台敏感维护入口。 |
| 质检方 | 自己质检任务、必要货物摘要、质检要求、结论影响。 | 放款决策、资金池、无关贸易全量材料。 |
| 物流方 | 自己物流任务、运输节点、异常说明、签收或损坏事实。 | 放款决策、质检/保险内部事实、无关案件。 |
| 海关 / 报关方 | 自己通关任务、监管状态、一致性提示。 | 融资审批、资金池、无关案件。 |
| 保险公司 | 核保/理赔任务、风险摘要、承保结论、补材料或理赔状态。 | 资金方内部审批、平台规则修改入口、无关案件。 |
| 买方 / 进口商 | 自己订单、收货、争议、付款责任。 | 出口商其他融资申请、资金审批、质检内部过程。 |
| 平台方 / 合约运营 | 全局运行、资金池、规则、暂停、异常、Agent 堆积、执行失败。 | 不可伪造业务事实，不可替资金方审批。 |
| Agent | 所属角色可见范围内的摘要、缺口、待办、建议和升级。 | 所属角色不可见的案件和事实。 |

## 6. 路径表达冻结

| 路径 | 冻结表达 | 不能表达成 |
|---|---|---|
| 银行辅助路径 | 资金方审查和授权是关键节点，授权后仍需 DeFi 执行检查。 | 银行授权后无条件到账。 |
| 纯 DeFi 路径 | 不需要银行，规则、资金池、事实和保证条件满足即可执行或阻断。 | 必须等待资金方审批。 |
| Agent + 智能合约低人工路径 | Agent 推进低风险节点，合约锁定/释放/结算，异常升级人工。 | Agent 自动裁决重大争议。 |
| 人工升级路径 | 争议、高风险、监管、重大法律责任、越权风险进入人工。 | 系统静默失败或自动放过。 |

## 7. Agent 表达冻结

| Agent 能力 | 原型冻结口径 |
|---|---|
| 整理材料 | 把材料和事实整理成角色可读摘要。 |
| 检查缺口 | 明确缺什么、为什么缺、谁负责。 |
| 生成待办 | 待办必须有责任角色和完成标准。 |
| 提醒用户 | 状态变化、超时、风险、授权、异常要可提醒。 |
| 自动推进 | 只在授权范围、低风险、事实完整、规则明确时推进。 |
| 异常升级 | 争议、高风险、越权、置信不足、执行失败必须升级。 |

Agent 不得表达成：

| 禁止表达 | 原因 |
|---|---|
| 独立业务角色 | Agent 权限从属于所属角色。 |
| 事实制造者 | Agent 不能伪造贸易、质检、物流、保险、买方事实。 |
| 高风险放款决策者 | 高风险授权由资金方或人工角色处理。 |
| 法律争议裁决者 | 重大争议进入人工或约定仲裁。 |
| 规则绕过入口 | Agent 不能绕过平台规则和合约约束。 |

## 8. DeFi 执行表达冻结

| 合约业务输出 | 原型冻结表达 |
|---|---|
| 允许执行 | 当前业务条件满足，可以进入资金动作。 |
| 不允许执行 | 当前条件不满足，不能继续。 |
| 等待条件 | 仍缺事实、授权、资金池、规则或保证条件。 |
| 规则阻断 | 当前规则不允许继续，并说明解除条件。 |
| 资金池不足 | 资金池不可用、余额不足、币种不支持或暂停。 |
| 平台暂停 | 平台、路径、规则、资金池或案件被暂停。 |
| 执行失败 | 资金动作尝试失败，需要平台或相关角色处理。 |
| 执行完成 | 放款、结算、锁定、释放或到账完成。 |

DeFi 合约不得在原型中表达成：

| 禁止表达 | 原因 |
|---|---|
| 页面或数据库 | 合约不是页面，也不保存完整业务材料。 |
| Agent | 合约不整理材料，不生成业务摘要。 |
| 贸易真实性判断者 | 事实来自业务角色。 |
| 银行辅助授权绕过者 | 需要银行参与时，资金方授权是执行条件。 |

## 9. 异常解释冻结

| 异常类型 | 冻结解释口径 | 责任方 |
|---|---|---|
| 材料不足 | 缺什么、为什么缺、谁补、完成标准。 | 出口商 / 相关事实方 |
| 事实不足 | 缺哪个事实节点和确认角色。 | 对应事实方 |
| 事实冲突 | 哪些事实冲突，影响哪个路径或节点。 | 相关角色 / 人工复核 |
| 质检 dispute | 争议原因和对审查/执行的影响。 | 质检方 / 资金方 / 出口商 |
| 买方 dispute | 收货、订单或付款责任争议。 | 买方 / 出口商 / 人工角色 |
| 银行授权不足 | 授权金额、期限、条件或状态不足。 | 资金方 |
| 规则阻断 | 哪类规则阻断和解除条件。 | 平台方 / 责任角色 |
| 资金池不足 | 资金池、币种、影响范围。 | 平台方 / 资金池运营 |
| 平台暂停 | 暂停对象、原因、影响范围。 | 平台方 |
| Agent 越权风险 | Agent 为什么不能自动推进。 | 所属角色 / 平台方 |
| 执行失败 | 资金动作为什么未完成。 | 平台方 / 合约执行层 |

## 10. 原型验收冻结

进入视觉原型或技术设计前，以下业务验收脚本必须可走读。

| 验收脚本 | 冻结要求 |
|---|---|
| 出口商控制塔走读 | 出口商能看清自己案件全链路、卡点和资金结果。 |
| 银行辅助路径走读 | 资金方审查和授权、合约执行检查、执行结果可解释。 |
| 纯 DeFi 路径走读 | 不需要银行，规则和资金池满足时合约执行或阻断。 |
| 低人工路径走读 | Agent + 合约推进低风险流程，异常升级人工。 |
| 事实方节点视角 | 事实方只看自己节点和必要上下文。 |
| 平台全局控制塔 | 平台看到全局运行、规则、资金池、异常和 Agent 状态。 |
| 权限隔离 | 越权行为被阻断并有业务解释。 |
| 异常失败解释 | 等待、阻断、失败、争议都有责任方和下一步。 |

## 11. 可后置但必须保留入口的问题

| 后置项 | 原型冻结口径 |
|---|---|
| P1 物流完整流程 | P0 控制塔可显示物流节点，但深度后置。 |
| P1 保险完整核保/理赔 | P0 可显示保险节点和结论入口，完整流程后置。 |
| P1 买方付款系统 | P0 可显示买方确认和争议节点，完整付款系统后置。 |
| P2 海关/报关接入 | P0 可显示通关节点或占位，真实接入后置。 |
| 多组织多成员权限 | P0 保持单钱包单角色，组织权限后置。 |
| 复杂法律自动执行 | P0 只保留低人工边界和人工升级，不自动裁决重大争议。 |

## 12. 进入下一阶段前不得新增

| 不得新增 | 原因 |
|---|---|
| 登录前展示页 | 当前不做营销转化。 |
| 与控制塔无关的首页 | 业务工作区才是第一屏。 |
| 旧版 hash / proof / wallet 页面 | 当前不继承旧页面和旧代码。 |
| 旧版 app/components/lib/contracts/scripts 代码 | 当前分支是全新瀑布重建设计分支。 |
| 独立 UI 组件清单 | 视觉原型前只冻结业务表达。 |
| 数据库表和字段 | 数据库设计后置。 |
| API endpoint 和 JSON schema | 技术接口设计后置。 |
| Solidity / ABI / 部署方案 | 合约实现后置。 |
| 钱包签名流程细节 | 钱包签名实现后置。 |
| hash / proof / proof pack 细节 | 底层证明产品后置。 |

## 13. 冻结判断

P0 产品原型可以冻结的条件：

| 条件 | 通过要求 |
|---|---|
| 控制塔表达稳定 | 全链路节点、状态、卡点、下一步清楚。 |
| 角色视角稳定 | 每个角色看到什么、看不到什么清楚。 |
| 路径差异稳定 | 银行辅助、纯 DeFi、低人工、人工升级不混淆。 |
| Agent 表达稳定 | 摘要、缺口、待办、建议、自动推进、异常升级清楚。 |
| DeFi 输出稳定 | 允许、等待、阻断、失败、完成可解释。 |
| 异常解释稳定 | 责任方、原因、解除条件和下一步清楚。 |
| 权限隔离稳定 | 越权场景被阻断并有业务解释。 |
| 后置范围稳定 | P1/P2、组织权限、法律复杂场景、技术实现不抢进 P0。 |
| 排除范围稳定 | 不做登录前展示页、旧代码、组件、数据库、接口、Solidity、hash/proof。 |

## 14. 冻结后允许进入的下一阶段

| 下一阶段 | 进入条件 |
|---|---|
| 视觉原型 | 以控制塔式大屏为主表达，按角色视角和路径差异设计。 |
| 交互原型 | 只围绕业务动作、待办、卡点、异常回流设计交互。 |
| 技术接口设计 | 基于 API 业务契约再定义接口路径和请求响应。 |
| 数据库设计 | 基于业务数据需求再定义表、字段和存储。 |
| 合约技术设计 | 基于 DeFi 合约业务规则再定义 Solidity、ABI 和部署。 |
| Agent 任务设计 | 基于 Agent 业务边界再定义任务、提示词和自动化策略。 |

## 15. 本步结论

P0 产品原型冻结清单的本质不是“开始画页面”。

它的本质是：

> 在进入视觉原型或技术设计前，冻结 ChainTrace 的控制塔表达、角色视角、路径差异、Agent 边界、DeFi 执行输出、异常解释、后置范围和排除范围，确保下一阶段不会偏回登录页、组件库、旧代码或底层证明细节。

下一步应进入：

> P0 视觉原型准备说明，用业务语言说明进入视觉原型时应如何组织控制塔、角色视角和关键走读场景，不写具体 UI 组件。
