# 73 P0 实现阶段 3：Material and Gap Flow 计划

## 1. 定位

本文件定义 ChainTrace P0 受控实现阶段的第三个阶段：Material and Gap Flow。

它不是文件存储方案，不是上传接口设计，不是数据库字段清单，不是 OCR/解析算法设计，不是前端上传组件，不是 hash/proof 方案，也不是 Agent 代码实现。

它的作用是：

> 在 Phase 2 融资案件核心成立之后，细化 Phase 3 的贸易材料范围、材料完整性、材料版本语义、Agent 缺口检查、补材料待办、材料冲突和阶段验收口径，让融资案件从“有融资意图”推进到“可准备审查材料”。

本阶段的核心目标不是“做文件上传”，而是：

```text
融资案件
→ 贸易材料集合
→ 材料完整性状态
→ Agent 缺口检查
→ 责任方待办
→ 补充说明和版本历史
→ 可进入事实确认 / 资金审查前置
```

## 3. Phase 3 阶段目标

Phase 3 必须让系统具备以下基础能力：

| 目标 | 说明 |
|---|---|
| 贸易材料集合可表达 | 合同、订单、发票、装箱单、补充说明成为案件的一部分。 |
| 材料来源清楚 | P0 中贸易材料由出口商提交。 |
| 材料完整性可表达 | 完整、缺口、冲突、需人工确认等状态可见。 |
| Agent 缺口检查可表达 | Agent 能指出缺什么、为什么影响融资、谁来补。 |
| 补材料待办可表达 | 待办必须有责任方、原因、下一步。 |
| 补充说明可表达 | 对金额、数量、主体、日期等冲突进行说明。 |
| 材料历史不可无痕覆盖 | 原始提交和后续补充需要留痕语义。 |
| 融资案件可进入下一阶段 | 材料准备完成后可进入质检或资金审查前置。 |

## 5. 贸易材料范围

Phase 3 必须支持以下 P0 贸易材料类型的业务表达。

| 材料类型 | 业务用途 | 提交责任方 | Phase 3 要求 |
|---|---|---|---|
| Contract Material | 证明贸易合同关系。 | 出口商。 | 可提交、可查看状态、可补充说明。 |
| Purchase Order Material | 证明买方订单或采购请求。 | 出口商。 | 可提交、可查看状态、可用于完整性检查。 |
| Invoice Material | 证明金额、付款义务或应收基础。 | 出口商。 | 可提交、可检查金额冲突。 |
| Packing List Material | 证明数量、包装、货物明细。 | 出口商。 | 可提交、可检查数量/货物冲突。 |
| Supplemental Explanation | 解释缺口、冲突或特殊条款。 | 出口商 / 责任方。 | 可追加，不能覆盖原始材料。 |

注意：

```text
Phase 3 的“材料”是业务证据对象，不是 proof/hash 工具页。
```

## 6. 材料完整性状态

Phase 3 应表达以下材料完整性状态。

| 状态 | 业务含义 | 下一步 |
|---|---|---|
| Not started | 尚未开始提交材料。 | 出口商提交材料。 |
| Incomplete | 至少缺少一类必要材料。 | Agent 生成缺口和补材料待办。 |
| Submitted | 材料已提交，但尚未完成检查。 | Agent 或人工检查。 |
| Complete | 材料最低要求满足。 | 进入质检或资金审查前置。 |
| Conflict detected | 材料之间存在金额、数量、主体、日期等冲突。 | 出口商解释或资金方/人工复核。 |
| Need explanation | 材料有特殊情况需要补充说明。 | 出口商提交补充说明。 |
| Locked for review | 材料版本进入审查态，不能无痕覆盖。 | 后续只能追加补充。 |
| Reopened for supplement | 审查中要求补材料。 | 出口商追加材料或说明。 |

Phase 3 不允许只用 uploaded / not uploaded 表达材料状态。

## 7. 材料版本和补充历史范围

| 版本语义 | 业务要求 |
|---|---|
| 原始提交 | 记录出口商首次提交材料的业务语义。 |
| 补充提交 | 对缺口、冲突或资金方要求进行补充。 |
| 补充说明 | 对无法直接补齐的差异进行解释。 |
| 审查锁定 | 进入审查后原始材料不得无痕覆盖。 |
| 重新打开 | 需要补充时重新进入补材料状态。 |
| 历史可追踪 | 谁提交、何时提交、为什么补充需要留痕语义。 |

禁止：

- 覆盖原始合同。
- 删除原始发票痕迹。
- 用补充说明替代原始材料。
- 资金方直接改出口商材料。
- Agent 自动生成材料文件。

## 8. Agent 缺口检查范围

Phase 3 的 Agent 缺口检查是业务辅助，不是事实裁决。

| 缺口类型 | Agent 应表达 | 责任方 |
|---|---|---|
| Missing contract | 缺合同材料，无法证明贸易合同关系。 | 出口商。 |
| Missing purchase order | 缺订单材料，无法证明买方采购意图。 | 出口商。 |
| Missing invoice | 缺发票材料，无法确认金额或应收基础。 | 出口商。 |
| Missing packing list | 缺装箱单，无法确认数量和包装信息。 | 出口商。 |
| Amount conflict | 合同、订单、发票金额不一致。 | 出口商 / 后续资金方。 |
| Quantity conflict | 订单、装箱单、质检前置数量不一致。 | 出口商。 |
| Party mismatch | 合同、订单、发票主体不一致。 | 出口商。 |
| Date inconsistency | 日期顺序、付款期限或交货期异常。 | 出口商。 |
| Need explanation | 存在特殊条款或无法自动判断的材料情况。 | 出口商 / 人工复核。 |

Agent 输出必须包含：

| 输出项 | 说明 |
|---|---|
| 缺口名称 | 缺什么或冲突在哪里。 |
| 缺口原因 | 为什么影响融资。 |
| 责任方 | 谁需要处理。 |
| 下一步 | 应该补材料、解释还是人工复核。 |
| 是否阻断提交 | 是否影响进入下一阶段。 |

## 9. 补材料待办范围

| 待办类型 | 来源 | 责任方 | 完成条件 |
|---|---|---|---|
| 补合同 | Agent 缺口 / 人工检查。 | 出口商。 | 合同材料补充。 |
| 补订单 | Agent 缺口 / 人工检查。 | 出口商。 | 订单材料补充。 |
| 补发票 | Agent 缺口 / 人工检查。 | 出口商。 | 发票材料补充。 |
| 补装箱单 | Agent 缺口 / 人工检查。 | 出口商。 | 装箱单补充。 |
| 解释金额不一致 | 材料冲突。 | 出口商。 | 提交补充说明或进入人工复核。 |
| 解释数量不一致 | 材料冲突。 | 出口商。 | 提交说明或等待后续质检。 |
| 解释主体不一致 | 材料冲突。 | 出口商。 | 提交说明或人工复核。 |
| 重新提交补充材料 | 资金方或后续检查要求。 | 出口商。 | 新补充形成历史记录。 |

待办必须是业务动作，不是技术任务。

## 10. 出口商工作区范围

Phase 3 中出口商工作区应新增材料准备表达。

| 区域 | Phase 3 表达 |
|---|---|
| 材料总览 | 合同、订单、发票、装箱单、补充说明的状态。 |
| 缺口提示 | Agent 告诉出口商缺什么、为什么缺。 |
| 冲突提示 | 金额、数量、主体、日期冲突。 |
| 补材料待办 | 责任方为出口商的下一步动作。 |
| 材料历史 | 原始提交和补充提交的业务历史。 |
| 提交前检查 | 是否具备进入下一阶段的最低条件。 |
| 进入下一步 | 完整后进入质检或资金审查前置。 |

禁止：

- 把材料区做成纯文件列表。
- 没有缺口说明。
- 没有下一步。
- 没有历史语义。

## 11. 资金方和平台在本阶段的边界

Phase 3 以出口商材料准备为主，资金方和平台不应越界。

| 角色 | Phase 3 可做 | Phase 3 不可做 |
|---|---|---|
| 资金方 | 后续可查看材料完整性占位。 | 修改出口商原始材料。 |
| 平台方 | 后续可查看异常占位。 | 替出口商补材料或删除材料历史。 |
| 质检方 | Phase 4 再进入。 | 在 Phase 3 决定材料是否融资通过。 |
| Agent | 检查缺口、生成待办、解释状态。 | 伪造材料或替用户提交事实。 |

```text

```

## 14. 资金结果关系

Phase 3 仍不产生资金结果。

| 当前情况 | 资金结果表达 |
|---|---|
| 材料未开始 | 暂无资金结果，需要先准备材料。 |
| 材料缺口 | 暂无资金结果，需补材料。 |
| 材料冲突 | 暂无资金结果，需解释或人工复核。 |
| 材料完整 | 可进入后续质检/资金审查/规则路径，资金结果仍待定。 |

禁止：

- 材料完整 = 融资批准。
- Agent 缺口通过 = 放款允许。
- 补材料完成 = 资金到账。

## 19. Phase 3 通过标准

| 检查项 | 通过标准 |

| 材料集合 | 合同、订单、发票、装箱单、补充说明可表达。 |

| 完整性状态 | 缺口、完整、冲突、需解释可表达。 |

| 越权阻断 | 资金方、平台、Agent 不能修改或伪造材料。 |
| 资金结果边界 | 材料完整不等于融资成功。 |
| 后置边界 | 未进入 OCR、hash/proof、质检、资金方审批、合约执行。 |

## 20. 一票否决项

出现以下任一项，Phase 3 不得通过：

| 一票否决项 | 原因 |
|---|---|
| 资金方可以修改出口商原始合同/发票 | 事实可信失败。 |
| Agent 可以自动生成或伪造材料 | Agent 越界。 |
| 材料历史可以被无痕覆盖 | 审计失败。 |
| 材料完整被表达为融资批准 | 资金结果误导。 |
| 缺口没有责任方和下一步 | 业务不可处理。 |
| 材料区只是文件列表，没有业务状态 | 控制塔价值失败。 |
| Phase 3 混入完整质检、资金方审批或合约执行 | 阶段越界。 |
| proof/hash 页面成为材料主线 | 产品定位倒退。 |
| 直接写文件存储、endpoint、字段、UI 组件或代码 | 当前文档阶段越界。 |

| 材料权限边界不清 | 53。 |
| Agent 缺口边界不清 | 57 / 66。 |
| 资金结果表达被误导 | 58 / 59。 |
| hash/proof 或 P1/P2 混入 | 60 后置。 |

不得通过临时代码补材料业务语义。

## 22. Phase 3 出口条件

Phase 3 完成后，系统应具备以下能力：

```text
1. 出口商可以在自己的融资案件下提交合同、订单、发票、装箱单和补充说明。
2. 材料状态能表达未开始、缺口、已提交、完整、冲突、需解释、锁定等业务状态。
3. Agent 可以指出材料缺口和冲突，并生成补材料待办。
4. 每个缺口和待办都有责任方、原因和下一步。
5. 原始材料和补充说明不能无痕覆盖。
6. 资金方、平台和 Agent 不能修改或伪造出口商原始材料。
7. 材料完整只代表可以进入下一阶段，不代表融资成功或到账。
8. 材料准备完成后，可以进入 Phase 4 质检事实节点。
```

满足以上条件后，才允许进入：

> `74-p0-implementation-phase-4-inspection-fact-node-plan.md`

即细化 Phase 4：Inspection Fact Node，开始定义质检任务、质检材料、Pass/Fail/Dispute 和事实来源边界。

## 23. 本步结论

Phase 3 的本质不是“实现上传文件”。

它的本质是：

> 让融资案件拥有可信的贸易材料基础：出口商提交材料，系统表达完整性和冲突，Agent 发现缺口并生成待办，材料历史不能被无痕覆盖，任何角色都不能伪造事实，材料完整也不能被误解为融资成功。

只有 Material and Gap Flow 成立，后续质检、资金方审查、合约执行和控制塔资金结果才有可信的业务输入。
