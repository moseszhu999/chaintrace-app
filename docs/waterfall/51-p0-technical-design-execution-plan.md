# 51 P0 技术设计执行计划

## 1. 定位

本文件定义 ChainTrace P0 技术设计阶段如何推进。

它不是技术实现计划，不是开发排期，不是接口文档，不是数据库建模，不是智能合约实现，不是钱包签名流程，不是 hash/proof 方案，不是 UI 组件拆分，也不是测试代码计划。

它的作用是：

> 把已经冻结的 P0 业务需求和技术设计边界，分配到接口、数据、合约业务、Agent、权限、原型表达和测试验收七个设计域，形成可执行、可评审、可回流的技术设计工作顺序。

本阶段结束时，只允许产出业务对齐的技术设计材料，不允许产出具体实现规格。

## 2. 执行原则

| 原则 | 说明 |
|---|---|
| 业务冻结先行 | 所有技术设计必须从已冻结 waterfall 文档继承。 |
| 设计域分离 | 接口、数据、合约、Agent、权限、原型、测试分别设计，不互相越界。 |
| 边界优先 | 每个设计域先写能做什么、不能做什么，再写输出。 |
| 路径保留 | 银行辅助、纯 DeFi、低人工、人工升级必须分别保留。 |
| 角色隔离 | 单钱包单角色和角色可见范围必须贯穿全部设计域。 |
| Agent 不越权 | Agent 继承角色权限，不成为事实制造者或审批人。 |
| 合约不判断事实 | 合约只做业务规则和资金执行约束，不判断贸易真实性。 |
| 缺口必须回流 | 发现需求缺口时，回流 waterfall 文档，不用实现补洞。 |
| 实现细节后置 | endpoint、表结构、Solidity、ABI、组件、测试代码全部后置。 |

## 3. 设计域总览

| 顺序 | 设计域 | 主要产出 | 评审重点 |
|---:|---|---|---|
| 1 | 业务输入映射 | 冻结需求到技术设计域的映射。 | 是否继承业务口径。 |
| 2 | 权限设计 | 单钱包单角色、角色可见范围、动作边界。 | 是否防止越权。 |
| 3 | 业务动作 / 接口边界设计 | 业务动作分组、业务结果、失败解释。 | 是否未进入 endpoint。 |
| 4 | 数据信息域设计 | 信息域、事实来源、记录责任、可见范围。 | 是否未进入表结构。 |
| 5 | 合约业务设计 | 合约业务角色、执行条件、业务输出、暂停边界。 | 是否未进入 Solidity/ABI。 |
| 6 | Agent 业务设计 | Agent 模式、可做/不可做、自动推进、人工升级。 | 是否未越权。 |
| 7 | 控制塔原型表达设计 | 源头到终端、节点状态、卡点、资金结果表达。 | 是否未进入 UI 组件。 |
| 8 | 业务验收设计 | 正常闭环、路径差异、异常、权限验收口径。 | 是否未进入测试代码。 |
| 9 | 回流与后置清单 | 需求缺口、P1/P2、实现后置项。 | 是否范围受控。 |
| 10 | 设计评审 | 使用 50 号评审清单判断是否通过。 | 是否允许进入下一阶段。 |

## 5. 阶段 2：权限设计

### 5.1 目标

把单钱包单角色、角色专属工作区、动作边界、可见范围和越权阻断设计清楚。

### 5.2 必须回答

| 问题 | 说明 |
|---|---|
| 钱包注册后是什么角色？ | 角色永久绑定，不允许随意切换。 |
| 角色能看什么？ | 只能看与自己相关的案件、任务、事实或运营状态。 |
| 角色能做什么？ | 每个业务动作必须有责任角色。 |
| 角色不能做什么？ | 不能越权审批、造事实、改规则、看无关案件。 |
| Agent 的权限从哪里来？ | Agent 继承所属角色权限。 |
| 平台权限边界在哪里？ | 平台能暂停、恢复、维护规则和资金池，但不能替资金方批准。 |

### 5.3 输出

| 输出项 | 内容 |
|---|---|
| 角色可见范围表 | 每个角色能看到的案件、事实、状态。 |
| 角色动作边界表 | 每个角色可触发和不可触发的业务动作。 |
| 越权场景清单 | 无关案件、事实篡改、越权审批、Agent 越权等。 |
| 越权业务解释 | 用户看到的业务原因，不是技术错误。 |
| Agent 权限继承说明 | Agent 不是超级用户。 |

### 5.4 不得输出

| 禁止项 | 原因 |
|---|---|
| 鉴权中间件实现 | 后续实现阶段处理。 |
| token/session 结构 | 当前不进入实现。 |
| 权限表结构 | 当前只记录业务权限。 |

## 6. 阶段 3：业务动作 / 接口边界设计

### 6.1 目标

把 P0 需要支持的业务动作和业务结果设计清楚，但不写 endpoint。

### 6.2 业务动作分组

| 动作组 | 示例业务动作 |
|---|---|
| 身份与注册 | 钱包连接、角色注册、账户激活、Agent 模式选择。 |
| 融资案件 | 创建融资意图、补充材料、提交申请、查看状态。 |
| 事实确认 | 物流确认、质检结论、买方确认、保险核保。 |
| 资金方决策 | 允许放款、拒绝、补材料、人工复核、降低金额。 |
| 合约执行检查 | 检查资金池、规则、暂停状态、执行条件。 |
| 平台运营 | 暂停、恢复、规则维护、异常分派。 |
| Agent 动作 | 摘要、缺口、待办、提醒、请求补充、异常升级。 |
| 异常处理 | dispute、资金池不足、规则阻断、执行失败。 |

### 6.3 业务结果类型

| 结果 | 业务含义 |
|---|---|
| 成功 | 业务动作已完成。 |
| 等待 | 还需要其他角色或条件。 |
| 需要补材料 | 缺少事实或材料。 |
| 人工复核 | Agent 或规则不能自动处理。 |
| 权限不足 | 该角色不能做这个动作。 |
| 规则阻断 | 当前规则不允许继续。 |
| 资金池不足 | 放款执行条件不足。 |
| 平台暂停 | 平台或规则暂停执行。 |
| 执行失败 | 执行层失败，需要处理。 |
| 完成 | 资金结果或业务闭环完成。 |

### 6.4 输出

| 输出项 | 内容 |
|---|---|
| 业务动作清单 | 每个动作的触发角色、输入业务前提、输出结果。 |
| 业务结果清单 | 每种结果的用户解释和下一步。 |
| 失败解释清单 | 失败原因、责任方、解除条件。 |
| 回流记录 | 业务动作不清楚时回流需求文档。 |

### 6.5 不得输出

| 禁止项 | 原因 |
|---|---|
| endpoint 路径 | 后续接口规格阶段处理。 |
| 请求响应 JSON | 当前只设计业务动作和结果。 |
| 错误码格式 | 当前只写业务解释。 |

## 7. 阶段 4：数据信息域设计

### 7.1 目标

把系统需要记录的业务信息域、事实来源、记录责任和可见范围设计清楚，但不写数据库模型。

### 7.2 信息域

| 信息域 | 说明 |
|---|---|
| 身份与角色 | 钱包、角色、账户状态、Agent 模式。 |
| 融资案件 | 融资意图、融资类型、金额、状态、结果。 |
| 贸易材料 | 合同、订单、发票、装箱单、补充说明。 |
| 事实节点 | 质检、物流、买方确认、保险、通关等事实。 |
| 资金方决策 | 允许、拒绝、补材料、人工复核、降额。 |
| 合约业务输出 | 允许、不允许、等待、阻断、资金池不足、暂停、失败、完成。 |
| Agent 输出 | 摘要、缺口、待办、提醒、建议、升级。 |
| 异常 | dispute、缺材料、规则阻断、执行失败等。 |
| 审计记录 | 授权、暂停、恢复、升级、关键状态变化。 |

### 7.3 必须回答

| 问题 | 说明 |
|---|---|
| 谁产生事实？ | 出口商、质检方、物流方、买方、保险方、海关等。 |
| 谁确认事实？ | 事实来源角色必须清楚。 |
| 谁可以看事实？ | 按案件相关性和角色范围可见。 |
| Agent 输出算不算事实？ | 不算，只是摘要、建议和辅助判断。 |
| 合约输出算不算贸易真实性？ | 不算，只是执行状态和规则结果。 |
| 哪些记录不可删除？ | 关键授权、决策、暂停、执行结果和异常记录。 |

### 7.4 输出

| 输出项 | 内容 |
|---|---|
| 信息域清单 | 记录系统需要哪些业务信息域。 |
| 事实来源矩阵 | 每类事实由哪个角色产生或确认。 |
| 记录责任表 | 谁负责记录、谁负责确认、谁可见。 |
| 审计记录要求 | 哪些动作必须可追踪。 |
| 数据边界说明 | 不进入表结构、字段、索引。 |

### 7.5 不得输出

| 禁止项 | 原因 |
|---|---|
| 数据库表结构 | 后续数据模型阶段处理。 |
| 字段类型 | 当前只做信息域。 |
| 索引和关系 | 当前不做数据库设计。 |
| 链上/链下存储结构 | 当前不进入实现。 |

## 8. 阶段 5：合约业务设计

### 8.1 目标

把智能合约作为业务规则和资金执行约束层的角色设计清楚，但不写 Solidity、ABI 或部署方案。

### 8.2 必须回答

| 问题 | 说明 |
|---|---|
| 合约负责什么？ | 检查条件、资金池、规则版本、暂停状态、执行或阻断放款。 |
| 合约不负责什么？ | 不判断贸易真实性、不替资金方审批、不裁决争议。 |
| 合约输出是什么？ | 允许、不允许、等待、规则阻断、资金池不足、平台暂停、执行失败、完成。 |
| 谁触发执行检查？ | 资金方、Agent、平台或规则路径，但必须符合权限。 |
| 执行失败谁处理？ | 平台、资金方或相关责任角色，按失败原因处理。 |
| 暂停如何表达？ | 平台暂停、规则暂停、资金池暂停、案件暂停要有业务含义。 |

### 8.3 输出

| 输出项 | 内容 |
|---|---|
| 合约业务角色说明 | 合约不是数据库、不是 Agent、不是审批人。 |
| 执行条件清单 | 授权、事实、规则、资金池、暂停状态。 |
| 合约业务输出清单 | 用户和角色能理解的业务结果。 |
| 暂停边界说明 | 哪些暂停对象、影响范围和恢复条件。 |
| 不负责事项记录 | 防止合约越界。 |

### 8.4 不得输出

| 禁止项 | 原因 |
|---|---|
| Solidity 代码 | 后续实现阶段处理。 |
| ABI | 当前不进入合约规格。 |
| 函数签名 | 当前只做业务设计。 |
| 部署网络和地址 | 当前不做部署设计。 |

## 9. 阶段 6：Agent 业务设计

### 9.1 目标

把 Agent Native 的全系统工作方式设计清楚：Agent 是每个角色身边的业务助理，不是独立用户或超级审批人。

### 9.2 Agent 模式

| 模式 | 说明 |
|---|---|
| 全人工确认 | 每一步问用户，用户确认后继续。 |
| 半自动托管 | Agent 自动处理常规事项，关键节点问用户。 |
| 全自动 Agent | 在授权范围内自动推进，异常时提醒或升级。 |

### 9.3 Agent 可做事项

| 可做事项 | 说明 |
|---|---|
| 整理材料 | 从业务材料中整理摘要。 |
| 检查缺口 | 告诉角色缺什么、找谁补。 |
| 生成待办 | 把异常转成具体任务。 |
| 提醒用户 | 提醒确认、补材料、异常、到账、逾期。 |
| 请求相关角色补充 | 向物流、质检、买方、保险等请求事实。 |
| 自动推进低风险流程 | 在授权范围和规则明确时推进。 |
| 异常升级 | 高风险、争议、置信不足、执行失败时交给人。 |

### 9.4 Agent 不可做事项

| 不可做事项 | 说明 |
|---|---|
| 伪造贸易事实 | 不能替角色制造事实。 |
| 伪造质检结论 | 不能替质检方出 pass/fail/dispute。 |
| 替资金方高风险放款 | 不能越权审批。 |
| 绕过规则或合约 | 不能绕过平台暂停、规则阻断或资金池不足。 |
| 自动裁决重大争议 | dispute 必须升级。 |
| 删除关键记录 | 审计和关键状态不能被篡改。 |

### 9.5 输出

| 输出项 | 内容 |
|---|---|
| Agent 模式说明 | 三种模式的业务差异。 |
| 自动推进条件 | 授权、低风险、事实完整、规则明确。 |
| 人工升级条件 | 高风险、争议、越权、失败、规则冲突。 |
| Agent 权限继承说明 | Agent 继承所属角色权限。 |
| Agent 输出边界 | 摘要和建议不等于事实或审批。 |

### 9.6 不得输出

| 禁止项 | 原因 |
|---|---|
| 模型选择 | 当前不选 LLM 或工具。 |
| 提示词模板 | 当前不进入实现。 |
| 工具调用实现 | 当前只做业务边界。 |
| Agent memory 结构 | 当前不做数据实现。 |

### 10.2 必须表达

## 11. 阶段 8：业务验收设计

### 11.1 目标

把 P0 技术设计后续如何验收写成业务口径，但不写测试代码。

### 11.2 验收场景

| 场景 | 必须验收 |
|---|---|
| 正常闭环 | 出口商创建融资意图，Agent 整理，事实确认，资金方/规则处理，资金结果返回。 |
| 银行辅助路径 | 资金方查看摘要、补材料、允许/拒绝、执行条件检查。 |
| 纯 DeFi 路径 | 无银行审批时，规则和资金池驱动执行边界清楚。 |
| 低人工路径 | Agent 能自动推进正常案件，并在异常时升级。 |
| 人工升级路径 | dispute、高风险、规则冲突、执行失败进入人工。 |
| 权限隔离 | 无关案件不可见，越权动作被业务解释阻断。 |
| 异常处理 | 缺材料、质检争议、物流异常、资金池不足、平台暂停等可处理。 |

### 11.3 输出

| 输出项 | 内容 |
|---|---|
| 业务验收场景清单 | P0 需要验收的主场景和异常场景。 |
| 通过口径 | 用户业务结果是否达成。 |
| 失败口径 | 未达成时原因、责任方、回流。 |
| 后置测试说明 | 测试代码后续处理。 |

### 11.4 不得输出

| 禁止项 | 原因 |
|---|---|
| 测试框架 | 当前不做测试实现。 |
| API 测试脚本 | 当前不写代码。 |
| 链上测试脚本 | 当前不进入合约实现。 |
| 断言格式 | 当前只写业务验收。 |

## 12. 阶段 9：回流与后置清单

### 12.1 目标

把技术设计过程中发现的需求缺口、P1/P2 扩展和实现后置项记录下来，防止技术设计阶段失控。

### 12.2 回流类型

| 回流类型 | 说明 |
|---|---|
| 新角色 | P0 文档未定义但技术设计发现需要。 |
| 新路径 | 银行辅助、纯 DeFi、低人工、人工升级之外的新路径。 |
| 新状态 | 现有状态不能表达业务结果。 |
| 新异常 | 现有异常不能解释卡点。 |
| 新权限 | 角色需要新动作或可见范围调整。 |
| 新合约输出 | 当前合约业务输出不足。 |
| Agent 边界调整 | 自动化范围需要扩大或收缩。 |
| P1/P2 提前请求 | 有人想把后续范围塞入 P0。 |

### 12.3 输出

| 输出项 | 内容 |
|---|---|
| 回流事项表 | 问题、来源、影响、回流文档、状态。 |
| 后置清单 | endpoint、数据库、Solidity、UI、测试等后续处理。 |
| P1/P2 清单 | 明确不进入 P0 的内容。 |
| 阻断事项 | 不解决就不能进入下一阶段的问题。 |

## 13. 阶段 10：设计评审

### 13.1 目标

使用 `50-p0-technical-design-review-checklist.md` 对全部技术设计交付物进行评审。

### 13.2 评审输入

| 输入 | 说明 |
|---|---|
| 业务输入映射 | 是否继承冻结需求。 |
| 权限设计 | 是否保持单钱包单角色。 |
| 接口边界设计 | 是否只写业务动作和结果。 |
| 数据信息域设计 | 是否只写信息域和记录责任。 |
| 合约业务设计 | 是否只写合约业务角色和输出。 |
| Agent 业务设计 | 是否保持 Agent 边界。 |
| 控制塔表达设计 | 是否保持业务表达，不进入 UI 实现。 |
| 业务验收设计 | 是否只写业务验收口径。 |
| 回流与后置清单 | 是否范围受控。 |

### 13.3 评审结论

| 结论 | 含义 |
|---|---|
| 通过 | 可以进入下一阶段。 |
| 需回流 | 需要补 waterfall 文档。 |
| 需拆分 | 某些设计域混在一起，需要拆开。 |
| 需后置 | 某些内容属于 P1/P2 或实现阶段。 |
| 不通过 | 出现一票否决项。 |

## 15. 本步结论

P0 技术设计执行计划的本质不是“开始写代码”。

它的本质是：

> 按接口、数据、合约、Agent、权限、原型表达和测试验收等设计域，把已冻结业务需求转成可评审的技术设计材料，同时持续阻止阶段越界、角色越权、路径合并、Agent 越权和合约越界。
