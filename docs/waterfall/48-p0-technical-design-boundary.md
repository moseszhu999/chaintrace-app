# 48 P0 技术设计阶段边界说明

## 1. 定位

本文件定义 ChainTrace P0 后续技术设计阶段可以展开到什么深度、必须继承哪些业务边界，以及哪些内容必须等待正式实现阶段。

P0 技术设计阶段边界说明不是技术架构方案，不是接口文档，不是数据库模型，不是合约规格，不是前端设计稿，也不是开发排期。

它的作用是：

> 在 P0 业务需求和视觉原型已经冻结后，为后续技术设计设置清晰边界：技术设计可以把业务输入拆成设计域、职责、约束、输入输出和验收口径，但不能直接写实现细节或绕过 waterfall 需求口径。

本文件不写组件、样式、布局尺寸、图表库、页面路由、接口路径、请求响应格式、数据库、Solidity、ABI、钱包签名、hash 或 proof 细节。

## 2. 边界原则

| 原则 | 说明 |
|---|---|
| 业务冻结优先 | 技术设计必须继承已冻结的 P0 业务口径。 |
| 设计可以分层 | 可以定义接口设计、数据设计、合约设计、Agent 设计、权限设计、测试设计的边界。 |
| 不进入实现 | 技术设计阶段不写代码、不定义具体实现格式。 |
| 不改变角色边界 | 单钱包单角色和角色可见范围不得被技术设计扩大。 |
| 不改变路径差异 | 银行辅助、纯 DeFi、低人工和人工升级不得被技术设计合并。 |
| 不改变 Agent 定位 | Agent 仍是角色身边业务助理，不是独立审批人。 |
| 不改变合约定位 | 智能合约仍是资金执行约束层，不是事实判断者。 |
| 缺口必须回流 | 发现业务缺口必须回到 waterfall 文档，不得直接补到实现里。 |

## 3. 技术设计阶段允许做什么

| 允许事项 | 说明 | 产出形式 |
|---|---|---|
| 划分设计域 | 明确接口、数据、合约、Agent、权限、原型、测试等设计域。 | 设计域说明。 |
| 梳理业务输入 | 把业务动作、事件、状态、异常、角色和规则整理成技术设计输入。 | 输入清单。 |
| 定义职责边界 | 说明每个设计域负责什么、不负责什么。 | 职责边界表。 |
| 定义业务输出 | 说明每个设计域需要输出哪些业务结果。 | 输出清单。 |
| 定义失败解释 | 说明失败、阻断、等待、权限不足等如何保持业务可解释。 | 失败解释表。 |
| 定义回流规则 | 说明发现需求缺口时回到哪些 waterfall 文档。 | 回流矩阵。 |
| 定义验收口径 | 说明后续实现如何证明业务闭环成立。 | 验收口径说明。 |
| 定义后置范围 | 明确哪些 P1/P2 或技术细节不得进入 P0。 | 后置清单。 |

## 4. 技术设计阶段不得做什么

| 不得事项 | 原因 |
|---|---|
| 编写前端页面或组件 | 当前不是实现阶段。 |
| 定义页面路由 | 当前不进入页面结构实现。 |
| 定义 API endpoint | 当前只允许整理业务动作和业务结果。 |
| 定义请求响应格式 | 当前不进入接口结构。 |
| 定义数据库表、字段、索引 | 当前只允许整理业务信息域。 |
| 定义 Solidity、ABI、部署、Gas | 当前只允许整理合约业务输出和约束。 |
| 定义钱包签名流程 | 当前只保留身份和授权业务口径。 |
| 定义 hash / proof 产品页 | 当前不进入底层证明产品主流程。 |
| 迁移旧版 app/components/lib/contracts/scripts | 当前分支是全新瀑布重建设计分支。 |
| 以技术便利性改业务规则 | 技术设计不得反向重写业务。 |

## 5. 技术设计深度分级

| 深度 | 允许程度 | 示例 | P0 当前是否允许 |
|---|---|---|---|
| L0 业务输入整理 | 把 waterfall 文档整理成技术设计输入。 | 业务动作、事件、状态、角色、异常清单。 | 允许 |
| L1 设计域划分 | 明确接口、数据、合约、Agent、权限等边界。 | 哪个设计域负责什么业务结果。 | 允许 |
| L2 业务约束设计 | 定义业务约束、失败解释、权限边界和回流规则。 | 权限不足如何解释，规则阻断如何回流。 | 允许 |
| L3 抽象技术设计 | 描述技术模块职责，不落到具体实现格式。 | “接口层接收业务动作并返回业务结果”。 | 有条件允许 |
| L4 具体技术规格 | 定义 endpoint、字段、合约函数、组件结构。 | `/api/cases`、表字段、Solidity 函数。 | 不允许 |
| L5 实现代码 | 编写代码、脚本、合约、组件、测试。 | app、components、lib、contracts、scripts。 | 不允许 |

L3 有条件允许的前提：

| 条件 | 说明 |
|---|---|
| 不出现具体实现格式 | 不出现 endpoint、字段、函数签名、组件结构。 |
| 不改变业务口径 | 不改变角色、路径、Agent、DeFi 和异常边界。 |
| 不替代后续正式设计 | 只作为技术设计准备，不作为最终技术规格。 |
| 可回流 | 发现业务缺口必须回到 waterfall 文档。 |

## 6. 接口设计边界

| 可以展开 | 不得展开 |
|---|---|
| 业务动作分类 | endpoint 名称 |
| 业务事件分类 | 事件协议 |
| 状态含义 | 状态字段结构 |
| 角色可执行动作 | 鉴权实现 |
| 成功、等待、阻断、权限不足、执行失败等业务输出 | 错误码格式 |
| 责任角色和下一步 | 请求响应格式 |

接口设计必须继承：

| 业务边界 | 说明 |
|---|---|
| 出口商不能查看无关案件 | 接口设计不能假设全量可见。 |
| 事实方不做融资审批 | 事实确认动作不能变成资金决策。 |
| 资金方不修改事实结论 | 审查动作不能篡改事实来源。 |
| Agent 权限继承所属角色 | Agent 相关动作不能越权。 |
| DeFi 输出必须可解释 | 接口输出不能只有技术失败。 |

## 7. 数据设计边界

| 可以展开 | 不得展开 |
|---|---|
| 业务信息域 | 数据库表结构 |
| 事实来源和责任方 | 字段名和字段类型 |
| 关键记录类型 | 索引和性能方案 |
| 审计和追踪要求 | 存储引擎和部署方案 |
| 可见范围和敏感边界 | 权限表实现 |
| 链上/链下责任描述 | 链上/链下存储结构 |

数据设计必须继承：

| 业务边界 | 说明 |
|---|---|
| 事实来源独立 | 质检、物流、通关、保险、买方各自负责事实。 |
| Agent 输出不是事实 | Agent 摘要不能替代原始事实。 |
| 合约结果不是贸易判断 | 合约只表达执行约束和资金动作结果。 |
| 关键记录可追踪 | 授权、阻断、暂停、恢复、执行结果必须可追踪。 |
| 角色相关可见 | 无关案件不可见。 |

## 8. 合约设计边界

| 可以展开 | 不得展开 |
|---|---|
| 合约业务角色 | Solidity 代码 |
| 执行前业务条件 | 函数签名 |
| 允许、等待、阻断、失败、完成等业务输出 | ABI |
| 资金池状态业务含义 | 资金池合约结构 |
| 暂停和恢复的业务约束 | 权限实现 |
| 关键状态变化的业务记录要求 | 事件编码 |

合约设计必须继承：

| 业务边界 | 说明 |
|---|---|
| 不判断贸易真实性 | 真实性来自材料和事实方。 |
| 不替资金方审批 | 银行辅助路径需要资金方授权。 |
| 不替质检方出结论 | 质检事实来自质检方。 |
| 不替平台做全部运营判断 | 平台暂停和恢复仍是运营职责。 |
| 不替 Agent 解释材料 | Agent 负责摘要和建议。 |
| 不自动裁决重大争议 | 争议进入人工或后续规则。 |

## 9. Agent 设计边界

| 可以展开 | 不得展开 |
|---|---|
| Agent 三种模式的业务差异 | 模型选择 |
| Agent 可做事项和不可做事项 | 提示词模板 |
| 摘要、缺口、待办、建议、提醒、升级的业务含义 | 工具调用实现 |
| 自动推进条件 | 阈值和评分算法 |
| 人工升级条件 | 自动裁决实现 |
| 权限继承关系 | 独立 Agent 账号体系 |

Agent 设计必须继承：

| 业务边界 | 说明 |
|---|---|
| Agent 不是单独用户 | Agent 是每个角色身边的业务助理。 |
| Agent 不伪造事实 | 事实来自业务角色。 |
| Agent 不越权 | 权限继承所属角色。 |
| Agent 不绕过规则 | 规则阻断必须尊重。 |
| Agent 不替资金方高风险决策 | 高风险进入人工或资金方。 |
| Agent 不裁决重大争议 | 重大争议进入人工或后续规则。 |

## 10. 权限设计边界

| 可以展开 | 不得展开 |
|---|---|
| 角色可见范围 | 鉴权中间件实现 |
| 角色可执行业务动作 | 权限表结构 |
| 越权场景和业务解释 | session 或 token 设计 |
| Agent 权限继承规则 | 钱包签名实现 |
| 平台运营边界 | 管理后台实现 |

权限设计必须继承：

| 边界 | 说明 |
|---|---|
| 单钱包单角色 | 注册后永久绑定单一业务身份。 |
| 案件相关可见 | 无关案件不可见。 |
| 事实来源不可篡改 | 谁提供事实，谁负责事实。 |
| 平台不替资金方审批 | 平台只运营规则、资金池、暂停和异常。 |
| Agent 不超过角色权限 | Agent 不能看所属角色不可见内容。 |

## 11. 原型实现设计边界

| 可以展开 | 不得展开 |
|---|---|
| 控制塔业务表达 | 具体页面路由 |
| 角色视角差异 | UI 组件结构 |
| 路径差异表达 | 前端状态管理 |
| Agent 输出表达 | Agent UI 组件实现 |
| DeFi 输出表达 | 合约调用实现 |
| 异常回流表达 | 交互代码 |
| 后置范围表达 | 旧页面迁移 |

原型实现设计必须继承：

| 边界 | 说明 |
|---|---|
| 不做登录前展示页 | 原型从注册后业务工作区开始。 |
| 大数据控制塔风格 | 能看源头到终端和中间节点。 |
| 角色局部可见 | 有的角色看局部，有的平台角色看全局。 |
| 不继承旧版代码口径 | 当前分支是全新瀑布重建。 |
| 不以 hash / proof 为主流程 | 当前主流程是供应链金融和 DeFi 执行。 |

## 12. 测试设计边界

| 可以展开 | 不得展开 |
|---|---|
| 业务验收场景 | 测试框架 |
| 正常路径检查 | 测试代码 |
| 异常路径检查 | API 自动化脚本 |
| 越权路径检查 | 鉴权测试实现 |
| DeFi 输出业务检查 | 链上测试脚本 |
| Agent 边界检查 | 模型测试实现 |
| 后置范围检查 | 发布流程 |

测试设计必须继承：

| 边界 | 说明 |
|---|---|
| 测业务闭环 | 不只测页面点击。 |
| 测路径差异 | 银行辅助、纯 DeFi、低人工都要区分。 |
| 测权限隔离 | 无关案件不可见，越权动作阻断。 |
| 测异常回流 | 每个异常要有原因、责任方、下一步。 |
| 测后置范围 | P1/P2 和技术细节不得混入 P0。 |

## 13. 技术设计回流边界

| 发现类型 | 必须回流 | 不得直接做 |
|---|---|---|
| 新角色 | 角色需求和权限矩阵。 | 直接新增角色实现。 |
| 新路径 | 场景路径矩阵和业务规则。 | 直接改流程实现。 |
| 新状态 | 业务事件和业务规则清单。 | 直接新增状态字段。 |
| 新异常 | 异常回流和业务规则。 | 直接新增错误码。 |
| 新合约输出 | 智能合约业务角色和 DeFi 规则。 | 直接新增函数或事件。 |
| Agent 权限扩大 | Agent Operating Model 和权限矩阵。 | 直接放开自动化。 |
| 数据记录不足 | 业务数据需求和审计要求。 | 直接新增表字段。 |
| P1/P2 提前 | MVP 范围和后置清单。 | 直接塞入 P0 实现。 |

## 14. 阶段进入和退出标准

### 14.1 可以进入技术设计

| 条件 | 标准 |
|---|---|
| 业务原型冻结 | 45 号收口检查允许冻结。 |
| 交接完成 | 46 号交接说明已明确接收方和范围。 |
| 技术设计准备完成 | 47 号准备说明已明确输入和不得开始内容。 |
| 阻断项关闭 | D 类变更已关闭。 |
| 需求缺口已补 | B 类需求缺口已回流补文档。 |
| 后置项清楚 | P1/P2 和技术实现后置项已记录。 |

### 14.2 技术设计阶段可退出

技术设计阶段未来可退出的业务标准应包括：

| 条件 | 标准 |
|---|---|
| 各设计域边界清楚 | 接口、数据、合约、Agent、权限、原型、测试各有边界。 |
| 业务输入映射完成 | 每个冻结业务输入都有对应设计域接收。 |
| 不得实现内容未越界 | 没有提前写 endpoint、表结构、Solidity、组件或代码。 |
| 回流项已处理 | 发现的业务缺口已回到 waterfall 文档。 |
| 下一阶段可接收 | 正式实现阶段知道哪些设计可接收、哪些仍后置。 |

## 15. 技术设计阻断条件

出现以下情况，技术设计不得继续推进。

| 阻断条件 | 原因 |
|---|---|
| 为了技术方便改变产品定位 | 破坏业务冻结。 |
| 把银行设成所有路径必经 | 破坏纯 DeFi 和低人工场景。 |
| 把合约设计成贸易真实性判断者 | 破坏事实来源边界。 |
| 把 Agent 设计成高风险审批人 | 破坏 Agent 边界。 |
| 放开角色查看无关案件 | 破坏权限隔离。 |
| 平台被设计成事实制造者 | 破坏可信边界。 |
| 异常只有技术错误，没有业务解释 | 破坏业务可处理性。 |
| 直接进入 endpoint、表结构、Solidity、组件或代码 | 阶段越界。 |
| 旧代码口径回流 | 破坏全新瀑布重建设计分支。 |

## 16. MVP 范围

### 16.1 P0 技术设计边界必须覆盖

| 范围 | 说明 |
|---|---|
| 接口设计边界 | 业务动作和业务结果可以展开，endpoint 和请求响应后置。 |
| 数据设计边界 | 信息域和记录责任可以展开，表结构和字段后置。 |
| 合约设计边界 | 业务输出和执行约束可以展开，Solidity 和 ABI 后置。 |
| Agent 设计边界 | 模式和边界可以展开，模型和工具实现后置。 |
| 权限设计边界 | 可见范围和越权场景可以展开，鉴权实现后置。 |
| 原型实现设计边界 | 业务表达可以展开，页面路由和组件后置。 |
| 测试设计边界 | 业务验收场景可以展开，测试代码后置。 |
| 回流边界 | 需求缺口必须回到 waterfall 文档。 |

### 16.2 P0 技术设计边界暂不覆盖

| 范围 | 原因 |
|---|---|
| 技术架构选型细节 | 后续正式技术设计处理。 |
| 前端组件实现 | 当前不写代码。 |
| API endpoint 和请求响应 | 后续接口规格阶段处理。 |
| 数据库表结构 | 后续数据模型阶段处理。 |
| Solidity / ABI / 部署 | 后续合约实现阶段处理。 |
| 钱包签名流程 | 后续钱包实现阶段处理。 |
| hash / proof 产品页 | 后续证明产品阶段处理。 |

## 17. 本步结论

P0 技术设计阶段边界的本质不是“开始写技术方案”。

它的本质是：

> 给 ChainTrace 后续技术设计划定可展开和不可展开的边界：可以整理业务输入、设计域职责、业务输出、失败解释、权限边界、回流规则和验收口径；但不能提前进入 endpoint、请求响应、数据库表结构、Solidity、ABI、钱包签名、页面组件或代码实现。

下一步应进入：

> P0 技术设计交付物清单，用于定义后续技术设计阶段最终应产出哪些业务对齐材料和边界记录，仍不进入实现代码。
