# 66 P0 Agent 实现准备

## 1. 定位

本文件定义 ChainTrace P0 规格准备阶段中的 Agent 实现准备边界。

它不是模型选型，不是提示词模板，不是工具调用协议，不是 Agent memory 结构，不是自动化工作流代码，不是向量库设计，不是队列任务实现，也不是聊天 UI 组件说明。

它的作用是：

> 从 `57-p0-agent-business-design.md` 已定义的 Agent 业务能力出发，建立“Agent 业务能力 → 实现规格候选”的准备框架，明确后续 Agent 实现应承接哪些业务能力、必须遵守哪些权限和安全边界、哪些内容仍然不能进入实现。

本文件只做 Agent 实现准备，不写 Agent 实现。

## 3. Agent 实现准备原则

| 原则 | 说明 |
|---|---|
| 从业务能力派生 | 不凭空设计 Agent，实现准备必须来自 57 的业务能力。 |
| Agent 是角色助理 | Agent 不是独立超级用户。 |
| 权限先于自动化 | 所属角色无权，Agent 一定无权。 |
| 授权先于推进 | 没有用户授权，不允许自动推进。 |
| 低风险才自动 | 高风险、dispute、失败、置信不足必须人工升级。 |
| 输出不是事实 | Agent 摘要、建议、缺口不等于原始事实或审批结论。 |
| 合约是执行边界 | Agent 不能绕过合约、规则、资金池或平台暂停。 |
| 行为必须可追踪 | 自动推进、请求补充、升级必须留痕。 |
| 不写实现细节 | 不写模型、提示词、工具、memory、代码、工作流引擎。 |
| 缺口必须回流 | 自动化边界不清时回流 57/60。 |

## 4. Agent 实现规格候选分组

| 分组 | 业务目的 | 当前阶段输出 |
|---|---|---|
| 角色上下文规格候选 | 确定 Agent 所属角色和可见范围。 | 角色绑定、权限继承、工作区边界。 |
| Agent 模式规格候选 | 表达全人工、半自动、全自动模式。 | 模式语义和授权范围。 |
| 材料摘要规格候选 | 整理贸易材料和案件信息。 | 摘要边界和来源说明。 |
| 缺口检查规格候选 | 找出材料、事实、确认的缺口。 | 缺口类型、责任方、下一步。 |
| 待办生成规格候选 | 将缺口和异常转成任务。 | 任务责任方和状态语义。 |
| 请求补充规格候选 | 向相关角色请求补材料或事实。 | 请求边界和权限条件。 |
| 状态解释规格候选 | 把系统、合约、异常状态转成业务语言。 | 控制塔可解释语义。 |
| 自动推进规格候选 | 在授权、低风险、事实完整、规则明确时推进。 | 自动条件和阻断条件。 |
| 人工升级规格候选 | 高风险、争议、失败、置信不足时交给人。 | 升级原因、责任方和解除条件。 |
| Agent 审计规格候选 | 记录 Agent 做过什么。 | 自动化留痕要求。 |

## 5. 通用 Agent 规格元信息

后续每个 Agent 实现规格候选至少描述以下业务元信息。

| 元信息 | 说明 |
|---|---|
| Agent 能力名称 | 该能力承接哪个业务能力。 |
| 所属角色 | 出口商、资金方、质检方、平台方或事实方。 |
| 可见范围 | Agent 可以读取哪些案件、材料、事实、状态。 |
| 可触发动作 | Agent 可辅助或触发哪些业务动作。 |
| 授权模式 | 全人工、半自动、全自动下是否可用。 |
| 业务输入来源 | 来自材料、事实、资金方决策、合约输出、异常、平台状态等。 |
| 业务输出类型 | 摘要、缺口、建议、待办、提醒、请求、升级、解释。 |
| 自动化边界 | 何时可自动，何时必须停止。 |
| 人工升级条件 | 高风险、争议、失败、权限不清、置信不足等。 |
| 审计要求 | 是否需要记录自动动作、请求和升级。 |
| 后置实现项 | 模型、提示词、工具、memory、代码、队列、UI 全部后置。 |

## 6. 角色上下文规格候选

Agent 的第一层规格不是模型，而是角色上下文。

| 所属角色 | Agent 可见范围 | Agent 不能越过 |
|---|---|---|
| 出口商 Agent | 出口商自己的案件、材料、资金结果、相关卡点。 | 资金方内部审查、其他出口商案件。 |
| 资金方 Agent | 分配/匹配给资金方的案件、证据完整性、风险建议、执行条件。 | 出口商原始材料修改权、其他资金方案件。 |
| 质检方 Agent | 自己的质检任务、清单、报告、补充请求。 | 放款决策、资金池状态、无关案件。 |
| 平台 Agent | 系统运行、异常、资金池、规则、合约、Agent 自动化状态。 | 替资金方审批、伪造事实。 |
| 物流/保险/买方 Agent | 对应事实方任务和确认状态。 | 伪造事实、越权处理其他角色任务。 |

通过标准：

```text
每个 Agent 能力必须先绑定所属角色，再判断可见范围和可触发动作。
```

## 7. Agent 模式规格候选

| 模式 | Agent 可做 | Agent 不可做 |
|---|---|---|
| 全人工确认 | 摘要、缺口、解释、建议、提醒。 | 自动提交、自动请求补充、自动推进。 |
| 半自动托管 | 自动摘要、缺口、待办、提醒、低风险请求补充。 | 高风险自动批准、自动裁决 dispute、绕过确认。 |
| 全自动 Agent | 在授权和低风险条件下推进常规步骤。 | 越权、伪造事实、绕过合约、处理重大争议。 |

模式边界：

- 模式只影响 Agent 自动化程度。
- 模式不改变所属业务角色。
- 模式不扩大数据可见范围。
- 全自动仍受权限、风险、事实、规则、合约和暂停约束。

## 8. 材料摘要规格候选

| 摘要对象 | 业务输入 | Agent 输出 | 边界 |
|---|---|---|---|
| 融资案件摘要 | 案件、融资目标、路径、状态。 | 案件是什么、当前阶段、资金结果或卡点。 | 摘要不改变案件状态。 |
| 贸易材料摘要 | 合同、订单、发票、装箱单、补充说明。 | 交易主体、金额、商品、数量、付款条件。 | 摘要不替代原始材料。 |
| 事实状态摘要 | 质检、物流、保险、买方、通关占位。 | 哪些事实已确认、待确认、dispute。 | 摘要不生成事实。 |
| 资金方审查摘要 | 材料、事实、风险、缺口、执行条件。 | 资金方可读的审查概览。 | 不等于资金方决策。 |
| 平台异常摘要 | 资金池、规则、合约、Agent、异常队列。 | 影响范围和处理优先级。 | 不等于平台自动处置。 |

实现准备要求：

- 每个摘要必须标明来源语义。
- 摘要必须能被用户追溯到原始材料或状态。
- 摘要不能覆盖原始记录。

## 9. 缺口检查规格候选

| 缺口类型 | 业务含义 | 责任方 | Agent 输出 |
|---|---|---|---|
| 缺贸易材料 | 合同、订单、发票、装箱单或补充说明不足。 | 出口商。 | 缺什么、为什么需要、如何补。 |
| 材料冲突 | 金额、数量、币种、主体、日期等不一致。 | 出口商 / 资金方。 | 冲突点和解释要求。 |
| 缺质检事实 | 质检任务、报告、照片或结论不足。 | 质检方。 | 质检补充待办。 |
| 缺物流事实 | 物流文件或节点不足。 | 物流方 / 出口商。 | P1 占位或补充请求。 |
| 缺买方确认 | 订单、收货、付款责任未确认。 | 买方。 | P1 占位或请求确认。 |
| 缺保险状态 | 承保或理赔状态不足。 | 保险方。 | P1 占位或补充请求。 |
| 缺执行条件 | 规则、资金池、暂停或授权不足。 | 平台 / 资金方。 | 执行条件卡点解释。 |

边界：

- 缺口不等于拒绝结论。
- 缺口必须有责任方和下一步。
- 责任方不清时不得实现，应回流。

## 10. 待办生成规格候选

| 待办类型 | 触发来源 | 责任方 | 完成含义 |
|---|---|---|---|
| 补合同/订单/发票 | 材料缺口。 | 出口商。 | 材料补齐或解释完成。 |
| 解释金额不一致 | 材料冲突。 | 出口商 / 资金方。 | 冲突解释被接受或进入人工。 |
| 补质检报告/照片 | 质检缺口。 | 质检方。 | 质检材料补齐。 |
| 处理质检 dispute | 事实争议。 | 质检方 / 出口商 / 资金方。 | dispute 解除或继续人工。 |
| 资金方人工复核 | 高风险或置信不足。 | 资金方。 | 允许、拒绝、补材料、降额。 |
| 平台处理资金池不足 | 资金池异常。 | 平台方。 | 恢复、等待、关闭或切换路径。 |
| 平台处理规则阻断 | 规则暂停/冲突。 | 平台方。 | 恢复、继续阻断或后置。 |
| 处理执行失败 | 合约/执行层失败。 | 平台方 / 执行责任方。 | 重试、关闭或人工处理。 |

待办边界：

- 待办必须有责任方。
- 待办必须有来源原因。
- 待办完成不一定代表案件完成。
- 待办不能分派给无权角色。

## 11. 请求补充规格候选

| 请求对象 | Agent 可否自动请求 | 条件 | 边界 |
|---|---|---|---|
| 出口商补材料 | 半自动/全自动可请求。 | 缺口明确且责任方为出口商。 | 不能代替出口商生成材料。 |
| 质检方补充说明 | 半自动/全自动可请求。 | 质检任务相关且缺口明确。 | 不能代替质检方出结论。 |
| 物流方补节点 | P1 完整后。 | P0 只占位。 | 不进入 P0 完整工作区。 |
| 保险方补状态 | P1 完整后。 | P0 只占位。 | 不自动承保。 |
| 买方确认 | P1 完整后。 | P0 只占位。 | 不替买方确认付款责任。 |
| 平台处理异常 | 可生成平台待办。 | 资金池、规则、执行失败等。 | 高影响操作需人工确认。 |

边界：

- 请求补充不是事实确认。
- 请求补充不能无视角色权限。
- 请求补充的结果仍要由责任方提交或确认。

## 12. 状态解释规格候选

Agent 必须把系统状态解释成业务语言。

| 状态来源 | 原始含义 | Agent 业务解释 |
|---|---|---|
| Need materials | 缺少材料或事实。 | 当前还缺少某项材料或确认，补齐后才能继续。 |
| Dispute | 存在争议。 | 当前事实不一致，需要责任方补充说明或人工复核。 |
| Rule blocked | 规则阻断。 | 当前规则不允许继续，需要查看规则原因或平台处理。 |
| Pool insufficient | 资金池不足。 | 资金方可能已批准，但资金池暂时不足或不可用。 |
| Platform paused | 平台暂停。 | 平台暂停了案件、资金池、规则或执行能力，Agent 不会继续推进。 |
| Execution failed | 执行失败。 | 放款执行失败，需要平台或执行责任方处理。 |
| Completed | 执行完成。 | 当前资金执行已完成，可以查看结果。 |

边界：

- Agent 不能改写合约输出含义。
- Agent 不能把执行失败解释成欺诈或成功。
- Agent 解释必须保留责任方和下一步。

## 13. 自动推进规格候选

Agent 自动推进必须同时满足以下条件。

| 条件 | 说明 | 不满足时 |
|---|---|---|
| 用户授权 | 所属角色选择半自动或全自动并授权该类动作。 | 停止并请求确认。 |
| 角色有权 | 所属角色有权执行该动作。 | 阻断并解释角色不匹配。 |
| 案件可见 | 案件属于该角色可见范围。 | 阻断。 |
| 事实完整 | 当前节点所需事实完整。 | 生成缺口或待办。 |
| 风险较低 | 未触发高金额、高风险、dispute 等条件。 | 人工升级。 |
| 规则明确 | 无规则冲突或不适用。 | 规则阻断或人工升级。 |
| 合约允许 | 执行类动作必须通过合约/规则边界。 | 停止推进。 |
| 平台未暂停 | 案件、资金池、规则、执行能力未暂停。 | 停止推进。 |
| 行为可追踪 | 自动动作可记录。 | 不允许自动推进。 |

自动推进适用：

| 可自动推进 | 不可自动推进 |
|---|---|
| 摘要生成。 | 高风险放款批准。 |
| 缺口检查。 | dispute 裁决。 |
| 待办生成。 | 事实伪造。 |
| 常规提醒。 | 绕过合约执行。 |
| 明确责任方的补充请求。 | 平台高影响暂停/恢复。 |
| 低风险状态推进。 | 权限不清动作。 |

## 14. 人工升级规格候选

| 升级条件 | 升级原因 | 责任方 |
|---|---|---|
| 高金额案件 | 风险高，不适合自动。 | 资金方。 |
| 高风险国家/品类/买方 | 需要人工判断。 | 资金方 / 平台。 |
| 质检 Fail | 事实不支持自动推进。 | 资金方 / 质检方。 |
| 质检 Dispute | 存在事实争议。 | 质检方 / 出口商 / 资金方。 |
| 金额不一致 | 材料冲突。 | 出口商 / 资金方。 |
| 买方争议 | 付款责任不清。 | 买方 / 资金方。 |
| 保险不足 | 风险缓释不足。 | 保险方 / 资金方。 |
| 资金池不足 | 执行条件不足。 | 平台 / 资金方。 |
| 规则阻断 | 当前规则不允许继续。 | 平台。 |
| 平台暂停 | 系统级边界。 | 平台。 |
| 执行失败 | 执行层失败。 | 平台 / 执行责任方。 |
| 置信不足 | Agent 判断不可靠。 | 所属角色。 |
| 权限不清 | 责任或可见范围不明确。 | 回流 waterfall。 |

升级输出必须包含：

| 输出项 | 说明 |
|---|---|
| 升级原因 | 为什么不能自动继续。 |
| 责任方 | 谁处理。 |
| 影响范围 | 影响提交、审查、执行、到账还是平台运营。 |
| 解除条件 | 满足什么才能继续。 |
| 下一步 | 人要做什么。 |

## 15. Agent 与接口候选关系

| 接口候选组 | Agent 参与方式 | 边界 |
|---|---|---|
| 身份与注册接口候选 | 可解释账户状态和 Agent 模式。 | 不改变永久角色。 |
| 融资案件接口候选 | 可创建草稿、补充信息、提醒提交。 | 不替用户伪造融资事实。 |
| 贸易材料接口候选 | 可摘要、缺口、补充请求。 | 不修改原始材料。 |
| 事实确认接口候选 | 可提醒事实方处理任务。 | 不生成事实结论。 |
| 资金方决策接口候选 | 可生成审查摘要和风险建议。 | 不替资金方审批高风险。 |
| 合约/规则执行接口候选 | 可请求检查和解释结果。 | 不绕过合约输出。 |
| 平台运营接口候选 | 可汇总异常和建议处理。 | 高影响操作需人工确认。 |
| 控制塔查询接口候选 | 可生成业务解释和待办。 | 不泄露无权数据。 |
| 异常处理接口候选 | 可生成待办和升级。 | 不裁决重大争议。 |

## 16. Agent 与概念数据对象关系

| 概念对象 | Agent 使用方式 | Agent 不得做什么 |
|---|---|---|
| Financing Case | 摘要、状态解释、待办。 | 无权创建或修改不相关案件。 |
| Trade Material Set | 摘要、缺口检查。 | 修改或伪造原始材料。 |
| Material Version Record | 引用提交历史。 | 覆盖历史版本。 |
| Fact Node | 读取状态和缺口。 | 生成事实。 |
| Inspection Fact | 解释 pass/fail/dispute。 | 替质检方确认结论。 |
| Funding Decision | 解释资金方决策。 | 替资金方决策。 |
| Contract Execution State | 解释 Allowed/Blocked/Failed。 | 改写或绕过结果。 |
| Exception Case | 生成待办和升级。 | 删除异常历史。 |
| Platform Operation Record | 汇总运营异常。 | 替平台执行高影响操作。 |
| Audit Record | 留下自动化记录。 | 无痕执行。 |

## 17. Agent 审计规格候选

Agent 行为必须可追踪。

| Agent 行为 | 留痕原因 |
|---|---|
| 生成摘要 | 说明摘要来源和时间。 |
| 生成缺口 | 说明缺口依据和责任方。 |
| 生成待办 | 说明待办来源和处理人。 |
| 请求补充 | 说明请求对象、原因和状态。 |
| 自动推进 | 证明授权、条件和动作。 |
| 人工升级 | 证明为什么不能自动继续。 |
| 解释合约输出 | 证明解释来源于合约/规则状态。 |
| 停止自动化 | 证明触发了暂停、规则阻断、dispute 或风险条件。 |

边界：

- 当前不定义日志表。
- 当前不定义 event 或消息队列。
- 当前只定义业务留痕要求。

## 21. Agent 实现准备通过标准

| 检查项 | 通过标准 |
|---|---|
| 来源清楚 | 每个能力来自 57 的业务设计。 |
| 所属角色清楚 | Agent 必须绑定业务角色。 |
| 权限继承清楚 | Agent 不扩大所属角色权限。 |
| 授权模式清楚 | 全人工、半自动、全自动边界清楚。 |
| 输出边界清楚 | 摘要、建议、缺口不等于事实或审批。 |
| 自动推进条件清楚 | 授权、低风险、事实完整、规则明确、未暂停。 |
| 人工升级条件清楚 | 高风险、dispute、失败、置信不足、权限不清。 |
| 合约边界清楚 | Agent 不绕过合约/规则/资金池/暂停。 |
| 审计要求清楚 | 自动动作和升级可追踪。 |
| 实现细节后置 | 未写模型、提示词、工具、memory、代码。 |

## 22. 一票否决项

| 一票否决项 | 原因 |
|---|---|
| Agent 实现准备把 Agent 设计成独立超级用户 | 破坏权限模型。 |
| Agent 可读取所属角色无权数据 | 数据越权。 |
| Agent 可伪造贸易、质检、物流、买方或保险事实 | 破坏事实来源。 |
| Agent 可自动批准高风险融资 | 破坏资金方责任。 |
| Agent 可自动裁决重大 dispute | 破坏争议处理边界。 |
| Agent 可绕过合约、规则、资金池或平台暂停 | 破坏执行边界。 |
| Agent 输出被当成原始事实或最终审批 | 破坏业务责任。 |
| 直接写模型、提示词、工具调用、memory 或代码 | 阶段越界。 |
| P1/P2 Agent 能力被塞入 P0 必做 | MVP 范围失控。 |
| 旧 proof/hash demo Agent 成为 P0 主线 | 破坏当前融资主线。 |

## 24. 本步结论

P0 Agent 实现准备的本质不是“开始写 AI Agent”。

它的本质是：

> 从 P0 Agent 业务设计出发，准备后续 Agent 实现必须承接的角色上下文、授权模式、摘要、缺口、待办、请求补充、状态解释、低风险自动推进、人工升级和行为留痕，同时继续确保 Agent 不越权、不伪造事实、不替代审批、不裁决重大争议、不绕过合约和平台暂停。
