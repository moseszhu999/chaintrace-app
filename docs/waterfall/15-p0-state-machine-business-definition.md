# 15 P0 状态机业务定义

## 1. 定位

本文件定义 ChainTrace P0 MVP 的业务状态机。

状态机不是技术实现，不是数据库字段设计，也不是智能合约代码。

它的核心作用是：

> 用统一业务语言说明融资案件从草稿到到账或终止的状态、进入条件、离开条件、责任角色和用户可理解解释。

本文件不写 UI、数据库、合约实现、钱包签名、hash 或 proof 细节。

## 2. 状态设计原则

| 原则 | 说明 |
|---|---|
| 业务语言优先 | 状态必须让出口商、资金方、质检方、平台方理解。 |
| 卡点可解释 | 每个阻断状态必须说明卡在哪里、找谁处理。 |
| Agent 可生成待办 | 状态变化应该能转成明确任务。 |
| 人工和自动边界清楚 | 正常推进可自动，高风险和授权必须人处理。 |
| 不混淆事实和决策 | 质检事实、资金方决策、平台执行条件要分开。 |
| 不跳到技术实现 | 不定义表结构、事件编码、合约函数。 |

## 3. P0 主状态列表

| 状态 | 中文含义 | 主要责任角色 |
|---|---|---|
| Draft | 案件草稿 | 出口商 |
| Preparing materials | 材料准备中 | 出口商 / Agent |
| Materials incomplete | 材料不完整 | 出口商 / Agent |
| Ready to submit | 可以提交 | 出口商 / Agent |
| Submitted | 已提交 | 出口商 |
| Inspection pending | 等待质检 | 质检方 |
| Inspection submitted | 质检已提交 | 质检方 / Agent |
| Under review | 资金方审查中 | 资金方 |
| Need materials | 需要补材料 | 对应责任角色 |
| Manual review | 人工复核中 | 资金方 / 平台方 |
| Approved | 资金方已允许放款 | 资金方 |
| Rejected | 融资被拒绝 | 资金方 |
| Execution checking | 执行条件检查中 | 平台 / 规则 / 合约业务层 |
| Execution waiting | 等待执行条件 | 平台方 / 资金方 |
| Blocked | 被规则、风险或平台状态阻断 | 平台方 / 资金方 |
| Disbursement pending | 放款执行中 | 平台 / 执行层 |
| Funds arrived | 资金已到账 | 出口商 / 资金方 |
| Closed | 案件关闭 | 相关角色 |

## 4. 状态定义

### 4.1 Draft

| 项目 | 说明 |
|---|---|
| 含义 | 出口商创建融资意图，但尚未形成可提交案件。 |
| 进入条件 | 出口商创建融资意图。 |
| 离开条件 | 开始上传贸易材料。 |
| Agent 动作 | 提醒需要哪些基础材料，生成准备清单。 |
| 用户解释 | 你的融资案件还在草稿阶段，需要先准备贸易材料。 |

### 4.2 Preparing materials

| 项目 | 说明 |
|---|---|
| 含义 | 出口商正在准备合同、订单、发票、装箱单等材料。 |
| 进入条件 | 出口商开始上传或补充材料。 |
| 离开条件 | Agent 完成基础完整性检查。 |
| Agent 动作 | 整理文件摘要，检查关键缺口。 |
| 用户解释 | Agent 正在整理你的贸易材料并检查是否可以提交。 |

### 4.3 Materials incomplete

| 项目 | 说明 |
|---|---|
| 含义 | 案件缺少提交融资所需的基础材料。 |
| 进入条件 | Agent 检查发现核心材料缺失或明显不一致。 |
| 离开条件 | 出口商或相关角色补齐材料。 |
| Agent 动作 | 生成缺口清单和补材料待办。 |
| 用户解释 | 当前材料还不够提交融资，需要补充指定材料。 |

### 4.4 Ready to submit

| 项目 | 说明 |
|---|---|
| 含义 | 案件达到最低提交条件。 |
| 进入条件 | Agent 判断基础材料完整，且没有明显阻断项。 |
| 离开条件 | 出口商确认提交，或 Agent 在授权范围内提交。 |
| Agent 动作 | 生成提交前摘要和提交建议。 |
| 用户解释 | 当前案件基本可以提交给资金方。 |

### 4.5 Submitted

| 项目 | 说明 |
|---|---|
| 含义 | 出口商已提交融资案件。 |
| 进入条件 | 出口商确认提交或授权 Agent 提交。 |
| 离开条件 | 进入质检、资金方审查或补材料状态。 |
| Agent 动作 | 锁定提交版本的业务摘要，分发后续任务。 |
| 用户解释 | 案件已提交，后续将进入质检和资金方审查。 |

### 4.6 Inspection pending

| 项目 | 说明 |
|---|---|
| 含义 | 案件正在等待质检方提供货物事实。 |
| 进入条件 | 案件需要质检且质检任务已生成。 |
| 离开条件 | 质检方提交 pass / fail / dispute。 |
| Agent 动作 | 提醒质检方处理，提醒出口商当前卡点。 |
| 用户解释 | 当前案件正在等待质检方提交质检结论。 |

### 4.7 Inspection submitted

| 项目 | 说明 |
|---|---|
| 含义 | 质检方已经提交结论和材料。 |
| 进入条件 | 质检方提交报告、照片、结论。 |
| 离开条件 | Agent 将质检事实纳入资金方摘要。 |
| Agent 动作 | 检查质检材料完整性，标记 pass / fail / dispute。 |
| 用户解释 | 质检结果已提交，系统正在整理给资金方审查。 |

### 4.8 Under review

| 项目 | 说明 |
|---|---|
| 含义 | 资金方正在审查融资案件。 |
| 进入条件 | 案件摘要、材料和质检事实已提交给资金方。 |
| 离开条件 | 资金方决定允许放款、拒绝、补材料或人工复核。 |
| Agent 动作 | 给资金方生成完整性、风险和建议摘要。 |
| 用户解释 | 资金方正在审查案件。 |

### 4.9 Need materials

| 项目 | 说明 |
|---|---|
| 含义 | 资金方、质检方、平台方或 Agent 要求补充材料。 |
| 进入条件 | 任一关键角色发现材料不足或异常说明不足。 |
| 离开条件 | 责任角色补齐材料并重新检查。 |
| Agent 动作 | 明确缺什么、找谁补、补完后回到哪个状态。 |
| 用户解释 | 当前案件需要补材料，补齐后才能继续。 |

### 4.10 Manual review

| 项目 | 说明 |
|---|---|
| 含义 | 案件需要人工判断，不能由 Agent 自动推进。 |
| 进入条件 | 高金额、质检 dispute、金额不一致、规则例外、疑似欺诈等。 |
| 离开条件 | 人工给出继续、拒绝、补材料或阻断结论。 |
| Agent 动作 | 汇总争议点、材料和建议，不自动裁决。 |
| 用户解释 | 当前案件存在需要人工判断的问题。 |

### 4.11 Approved

| 项目 | 说明 |
|---|---|
| 含义 | 资金方允许该案件进入放款执行流程。 |
| 进入条件 | 资金方明确选择允许放款。 |
| 离开条件 | 进入执行条件检查。 |
| Agent 动作 | 提醒平台/规则/合约业务层检查执行条件。 |
| 用户解释 | 资金方已允许融资，正在检查是否可以执行放款。 |

### 4.12 Rejected

| 项目 | 说明 |
|---|---|
| 含义 | 资金方拒绝融资。 |
| 进入条件 | 资金方明确选择拒绝。 |
| 离开条件 | 案件关闭，或出口商基于允许的业务规则重新发起。 |
| Agent 动作 | 汇总拒绝原因和可改进材料。 |
| 用户解释 | 资金方拒绝了本次融资申请，原因如下。 |

### 4.13 Execution checking

| 项目 | 说明 |
|---|---|
| 含义 | 平台、规则和合约业务层正在检查是否允许执行。 |
| 进入条件 | 资金方已允许放款。 |
| 离开条件 | 允许执行、等待条件、阻断或执行失败。 |
| Agent 动作 | 将检查结果转成用户可理解卡点。 |
| 用户解释 | 放款前正在检查资金池、规则和平台状态。 |

### 4.14 Execution waiting

| 项目 | 说明 |
|---|---|
| 含义 | 资金方已允许，但仍在等待某个执行条件。 |
| 进入条件 | 资金池不足、规则等待、平台维护、材料锁定未完成等。 |
| 离开条件 | 条件满足后继续执行，或转为阻断。 |
| Agent 动作 | 提醒责任方处理等待条件。 |
| 用户解释 | 资金方已允许，但当前还缺执行条件。 |

### 4.15 Blocked

| 项目 | 说明 |
|---|---|
| 含义 | 案件被规则、风险、平台状态或重大异常阻断。 |
| 进入条件 | 规则阻断、平台暂停、重大争议、质检 fail 等。 |
| 离开条件 | 平台或责任角色处理后恢复，或案件关闭。 |
| Agent 动作 | 说明阻断原因，不自动绕过。 |
| 用户解释 | 当前案件被阻断，原因如下，不能继续自动推进。 |

### 4.16 Disbursement pending

| 项目 | 说明 |
|---|---|
| 含义 | 放款已经进入执行中。 |
| 进入条件 | 执行条件检查通过。 |
| 离开条件 | 资金到账、执行失败或平台暂停。 |
| Agent 动作 | 提醒资金方和出口商执行状态。 |
| 用户解释 | 放款正在执行，请等待结果。 |

### 4.17 Funds arrived

| 项目 | 说明 |
|---|---|
| 含义 | 出口商已经收到资金。 |
| 进入条件 | 放款执行完成。 |
| 离开条件 | 进入贷后跟踪或案件关闭。 |
| Agent 动作 | 通知出口商和资金方，生成结果摘要。 |
| 用户解释 | 资金已到账。 |

### 4.18 Closed

| 项目 | 说明 |
|---|---|
| 含义 | 案件已经结束，不再继续推进。 |
| 进入条件 | 到账后关闭、拒绝后关闭、人工终止或长期无法处理。 |
| 离开条件 | 无。 |
| Agent 动作 | 生成结案摘要。 |
| 用户解释 | 案件已关闭。 |

## 5. 关键状态流转

### 5.1 正常通过路径

```text
Draft
→ Preparing materials
→ Ready to submit
→ Submitted
→ Inspection pending
→ Inspection submitted
→ Under review
→ Approved
→ Execution checking
→ Disbursement pending
→ Funds arrived
→ Closed
```

### 5.2 补材料路径

```text
Preparing materials
→ Materials incomplete
→ Need materials
→ Preparing materials
→ Ready to submit
```

资金方审查中也可以进入：

```text
Under review
→ Need materials
→ Under review
```

### 5.3 争议和人工复核路径

```text
Inspection submitted
→ Manual review
→ Need materials / Under review / Rejected / Blocked
```

### 5.4 执行阻断路径

```text
Approved
→ Execution checking
→ Execution waiting / Blocked / Disbursement pending
```

## 6. 状态解释要求

每个状态必须对用户输出三件事：

| 信息 | 说明 |
|---|---|
| 当前状态 | 用业务语言说明当前处于哪里。 |
| 当前原因 | 为什么处于这个状态。 |
| 下一步动作 | 谁需要做什么。 |

示例：

```text
当前状态：Need materials
当前原因：资金方要求补充质检照片，当前照片不足以确认货物标签。
下一步动作：质检方补充标签照片，Agent 将在补充后重新检查材料完整性。
```

## 7. MVP 范围

### P0 必须做

| 范围 | 说明 |
|---|---|
| 定义主状态 | 覆盖草稿、提交、质检、审查、执行、到账、关闭。 |
| 定义状态解释 | 每个状态有业务解释和下一步动作。 |
| 定义补材料回路 | 能从缺口回到继续审查。 |
| 定义人工复核回路 | 高风险和争议必须人处理。 |
| 定义执行检查回路 | 资金方批准后仍检查规则、资金池、平台状态。 |

### 暂不做

| 范围 | 原因 |
|---|---|
| 数据库状态字段设计 | 当前不写实现。 |
| 合约状态变量 | 当前不写合约代码。 |
| 页面状态展示组件 | 当前不做 UI。 |
| hash / proof 状态 | 当前不进入底层证明设计。 |

## 8. 本步结论

P0 状态机的本质不是技术状态枚举。

它的本质是：

> 让每个角色都知道案件现在在哪里、为什么卡住、下一步由谁处理；Agent 根据状态生成待办，规则和平台根据状态约束执行。
